PyTorch 共享库结构: 源码目录 → .so → torch._C

范围说明:本文按 Linux CUDA wheel 与 PyTorch v2.12.0 源码来写。macOS、Windows、ROCm、CPU-only wheel 和本地源码编译的文件名、可选库、链接方式可能不同;遇到差异时,以你安装包里的 torch/lib/ 和对应版本源码为准。

1. 起点:import torch 到底加载了哪些 .so

装一份 CUDA 版 PyTorch wheel,看 torch/lib/ 下面是这样:

$ ls torch/lib/*.so
torch/lib/libc10.so
torch/lib/libc10_cuda.so
torch/lib/libtorch.so
torch/lib/libtorch_cpu.so
torch/lib/libtorch_cuda.so
torch/lib/libtorch_python.so
torch/lib/libtorch_global_deps.so
torch/lib/libshm.so
torch/lib/libcaffe2_nvrtc.so
torch/lib/libnnapi_backend.so
# 可选(BUILD_LAZY_CUDA_LINALG=ON):
torch/lib/libtorch_cuda_linalg.so
# 可选(USE_NVSHMEM):
torch/lib/libtorch_nvshmem.so

这一篇要回答的问题就两个:

  1. 每个 .so 是从源码树哪个目录编出来的?
  2. 它们之间的链接依赖是怎样的?

顺手把一个反直觉的事实先放在这儿:libtorch.so 是一个空文件。它只是个 aggregator,把 libtorch_cpu.so 和 libtorch_cuda.so 打成一个对外引用,使得"链 libtorch 就同时有 CPU 和 CUDA"。 下面的依赖图直接给出全貌,§3 再展开每个库的细节。

PyTorch Shared Library Structure: Source Trees, .so Files, and torch._C

Scope: this article describes the Linux CUDA wheel and PyTorch v2.12.0 source. macOS, Windows, ROCm, CPU-only wheels, and local source builds can differ in file names, optional libraries, and link structure. If your install disagrees with this post, trust your own torch/lib/ directory and the matching source tag.

1. Start: which .so files import torch loads

Install a CUDA-enabled PyTorch wheel and torch/lib/ looks like this:

This article answers two questions:

  1. Which source-tree directory does each .so compile from?
  2. How do they link to each other?

One counterintuitive fact up front: libtorch.so is an empty file. It is an aggregator that bundles libtorch_cpu.so and libtorch_cuda.so behind one name, so that "link libtorch and you get both CPU and CUDA" stays a stable API. The graph below shows the full picture; §3 expands each library individually.

3. 每个库的源码目录详解

下面这一节是侧栏 detail 面板的展开版本 —— 同样的信息加上 prose 和表格上下文,适合通读。

3.1 libc10: Core Tensor 抽象

源码: c10/CMakeLists.txt 用 GLOB 收:

file(GLOB C10_SRCS
  *.cpp
  core/*.cpp
  core/impl/*.cpp
  mobile/*.cpp
  macros/*.cpp
  util/*.cpp)
# 显式排除 c10/cuda/、c10/hip/

包含:TensorImpl、Storage、Device、DispatchKey、Allocator、intrusive_ptr、TypeMeta、所有 c10::util 工具(Optional、SmallVector、ArrayRef、错误处理宏)。 是整个 PyTorch C++ API 最底层、没有任何 CUDA / Python / ATen 依赖的一层。

链接:fmt、nlohmann_json、moodycamel(concurrent queue),可选 gflags / glog / cpuinfo。

3.2 libc10_cuda: CUDA 基础设施

源码: c10/cuda/CMakeLists.txt 显式列出(不用 glob):

add_library(c10_cuda
  CUDAAllocatorConfig.cpp
  CUDACachingAllocator.cpp
  CUDAFunctions.cpp
  CUDAStream.cpp
  driver_api.cpp
  impl/CUDAGuardImpl.cpp
  ...)

包含:CUDA 流(CUDAStream)、CUDA caching allocator(这就是 torch.cuda.empty_cache() 操作的对象)、CUDAGuard、CUDA device assertion、driver API 包装。 这是 PyTorch CUDA 路径的"底座"。

链接:c10 PUBLIC + torch::cudart PUBLIC。

3.5 libtorch: 空的 aggregator

这是最魔幻的一个:

# caffe2/CMakeLists.txt:923
# Wrapper library for people who link against torch and expect
# both CPU and CUDA support. Contains 'torch_cpu' and 'torch_cuda'.
add_library(torch ${DUMMY_EMPTY_FILE})
target_link_libraries(torch PUBLIC torch_cpu_library)
if (USE_CUDA)
  target_link_libraries(torch PUBLIC torch_cuda_library)
endif()

它编出的 libtorch.so 几乎没有自己的代码 —— 只有一个空 .cpp 占位符。它存在的全部理由是:让用户写 target_link_libraries(my_app torch) 就同时能拿到 CPU 和 CUDA。 它是 PyTorch 工程对"链 libtorch 就够了"这个 API 承诺的实现机制。

在 BUILD_LIBTORCHLESS split-build 模式下,torch 会被 find_library() 找到一个已安装的外部 libtorch.so,但概念不变。

链接:${TORCH_LIB} PRIVATE,这个变量在标准 build 里就是空 aggregator torch —— 所以 libtorch_python 链接的是空壳 libtorch.so,通过它间接拿到 CPU 和 CUDA。 另外链 Python::Module、pybind11、shm、cuDNN / cuSPARSELt / cuFile / NVTX(可选)、UCC / NumPy。

3.7 libtorch_global_deps: RTLD_GLOBAL 预加载技巧

这是另一个反直觉的库:

add_library(torch_global_deps SHARED torch/csrc/empty.c)
target_link_libraries(torch_global_deps PRIVATE
  ${MPI_LIBRARIES}
  ${MKL_LIBRARIES}
  CUDA::cudart
  CUDA::nvToolsExt
  ...)

它的 C 代码本身是空的(只有一个 empty.c 占位符),但它链接了 MPI / MKL / cudart / nvToolsExt 这些"想要进全局符号空间"的库。 关键事实:没有任何其它 PyTorch 库链接它。 它的存在目的不是被链,而是被 Python 启动时手动 dlopen 进 RTLD_GLOBAL:

# torch/__init__.py (节选)
def _load_global_deps():
    here = os.path.abspath(__file__)
    lib_path = os.path.join(os.path.dirname(here), "lib", "libtorch_global_deps.so")
    ctypes.CDLL(lib_path, mode=ctypes.RTLD_GLOBAL)
# ... 然后:
from torch._C import *

为什么要这么搞:MKL、MPI、cudart 等依赖里有"必须被多个 .so 共享同一个 OpenMP / CUDA runtime 实例"的痛点。 如果 Python 用默认 RTLD_LOCAL 加载 libtorch_python.so,这些 runtime 符号会被局部隔离,可能跟用户 numpy / scipy 里加载的 MKL 撞符号或拿到不同实例。 把它们的符号通过 libtorch_global_deps 提前 RTLD_GLOBAL 进进程,后续所有 .so 都共享同一份。

3.8 其它的

  • libshm.so:torch/lib/libshm/,跨进程共享内存 Tensor 的 manager。 link torch_cpu PRIVATE。 配套还有 torch_shm_manager 可执行文件。
  • libcaffe2_nvrtc.so:aten/src/ATen/cuda/nvrtc_stub/ATenNVRTC.cpp,这是个延迟绑定 stub —— libtorch 不直接链 NVRTC,而是用 dlopen 这个 stub 间接调,这样没装 NVRTC 的环境也不会一加载就崩。
  • libtorch_cuda_linalg.so:可选(BUILD_LAZY_CUDA_LINALG=ON),把 MAGMA / cusolver 相关的 linalg 算子从 libtorch_cuda.so 里拆出来 lazy load,主库不必硬链 MAGMA。
  • libnnapi_backend.so:torch/csrc/jit/backends/nnapi/,Android NNAPI 的 TorchScript backend。 Linux 上居然也无条件 ship —— 历史遗留。
  • libtorch_nvshmem.so:可选(USE_NVSHMEM),用 NVSHMEM 做集体通信。

3.5 libtorch: the empty aggregator

The most surprising entry:

# caffe2/CMakeLists.txt:923
# Wrapper library for people who link against torch and expect
# both CPU and CUDA support. Contains 'torch_cpu' and 'torch_cuda'.
add_library(torch ${DUMMY_EMPTY_FILE})
target_link_libraries(torch PUBLIC torch_cpu_library)
if (USE_CUDA)
  target_link_libraries(torch PUBLIC torch_cuda_library)
endif()

The resulting libtorch.so contains almost no code of its own: just one empty .cpp. Its entire purpose is to give users a single name to link against: target_link_libraries(my_app torch) transitively pulls both CPU and CUDA. It is the implementation of PyTorch's "link libtorch and that is all you need" API contract.

In BUILD_LIBTORCHLESS split-build mode, torch is resolved by find_library() against an externally installed libtorch.so, but the concept is identical.

3.7 libtorch_global_deps: the RTLD_GLOBAL preload trick

Another counterintuitive library:

add_library(torch_global_deps SHARED torch/csrc/empty.c)
target_link_libraries(torch_global_deps PRIVATE
  ${MPI_LIBRARIES}
  ${MKL_LIBRARIES}
  CUDA::cudart
  CUDA::nvToolsExt
  ...)

The C code is empty (one placeholder empty.c), but the library links MPI, MKL, cudart, nvToolsExt, all the runtimes that want a single global instance. The key fact: no other PyTorch library links it. Its purpose is not to be linked but to be manually dlopened with RTLD_GLOBAL at Python startup:

# torch/__init__.py (abridged)
def _load_global_deps():
    here = os.path.abspath(__file__)
    lib_path = os.path.join(os.path.dirname(here), "lib", "libtorch_global_deps.so")
    ctypes.CDLL(lib_path, mode=ctypes.RTLD_GLOBAL)
# ... then:
from torch._C import *

Why: MKL, MPI, cudart, and similar runtimes require "all shared libraries see the same instance" semantics. If Python loads libtorch_python.so with the default RTLD_LOCAL, those runtime symbols become locally scoped and may collide with, or be a different instance from, the MKL already loaded by the user's numpy or scipy. Pre-dlopen-ing them with RTLD_GLOBAL through libtorch_global_deps ensures every later .so shares one instance.

3.8 The smaller libraries

  • libshm.so: under torch/lib/libshm/, the shared-memory manager for cross-process tensors; links torch_cpu PRIVATE. Ships alongside the torch_shm_manager executable.
  • libcaffe2_nvrtc.so: aten/src/ATen/cuda/nvrtc_stub/ATenNVRTC.cpp. A lazy-binding stub: libtorch never links NVRTC directly; it dlopens this stub at runtime, so environments without NVRTC do not crash on import.
  • libtorch_cuda_linalg.so: optional (BUILD_LAZY_CUDA_LINALG=ON); splits MAGMA / cusolver linalg kernels out of libtorch_cuda.so so the main library does not have to hard-link MAGMA.
  • libnnapi_backend.so: torch/csrc/jit/backends/nnapi/, the Android NNAPI TorchScript backend. Surprisingly ships unconditionally on Linux.
  • libtorch_nvshmem.so: optional (USE_NVSHMEM), NVSHMEM-based collective communication.

4. torch._C 到底是什么

把它拆开:

  1. torch._C 是个 Python 扩展模块;在本文这个 Linux CUDA wheel 布局里,它由 libtorch_python.so 提供。
  2. libtorch_python.so 是按 CPython 扩展模块的接口编出来的(链 Python::Module),在这个 wheel 里落在 torch/lib/libtorch_python.so。其它平台或构建方式可能换文件名或安装位置。
  3. 它的 Python 入口符号是 PyInit__C,定义在 torch/csrc/Module.cpp —— 这个名字告诉 CPython 它是 _C 这个名字的扩展模块,而 setuptools 把它注册到 torch._C 这个完整路径下。
  4. Python 端 from torch._C import * 触发动态链接器加载 libtorch_python.so → 它的 NEEDED 项把 libtorch.so 拖进来 → 再把 libtorch_cpu.so 和 libtorch_cuda.so 拖进来 → 再底下是 libc10.so 和 libc10_cuda.so。

所以在这个 Linux wheel 语境下,Python 角度看的 torch._C,从 C++ / 动态链接器角度看主要落到 libtorch_python.so 这个扩展模块;再底下一连串 .so 是动态链接器的工作。

torch/__init__.py 的实际启动流程:

# 1. 先用 RTLD_GLOBAL dlopen libtorch_global_deps,把 MKL/cudart 等全局化
_load_global_deps()

# 2. 加载 _C(即 libtorch_python),触发整条 .so 加载链
from torch._C import *

# 3. 显式 init ATen / c10 的全局状态
_C._initExtension(_manager_path())

5. 几个反直觉的点

  1. libtorch.so 是空的。 你 nm libtorch.so 看不到几个符号,它只是个链接 manifest。
  2. 定义 torch_cpu 和 torch_cuda 的 CMake 在 caffe2/CMakeLists.txt 里。 看见 caffe2 别犯怵,这不是给 Caffe2 框架用的,是真 libtorch 的家。
  3. libtorch_python 不直接链 libtorch_cpu 或 libtorch_cuda。 它链空壳 libtorch,通过 aggregator 间接拿到。 在 split-build 模式下这一抽象层让 build 还能切换到外部安装的 libtorch。
  4. libtorch_global_deps 没有任何库链它。 它的存在是为了被 ctypes.CDLL(..., RTLD_GLOBAL) 单独 dlopen,做 MKL / cudart 等的全局符号预加载。
  5. libcaffe2_nvrtc 也是个 stub。 它的意义在于 lazy bind NVRTC,这样运行时不带 NVRTC 的环境也能 import torch 而不是直接挂掉。 想真用 NVRTC 时 PyTorch 会再 dlopen 进来。
  6. libnnapi_backend.so 在 Linux 上也无条件 ship。 这是 Android NNAPI 的 TorchScript backend,Linux 上其实没人用,纯历史遗留。
  7. 源文件列表的"事实唯一来源"是 build_variables.bzl。 CMake 和 Bazel 都从这个 .bzl 读 libtorch_python_core_sources / libtorch_python_cuda_core_sources / libtorch_cuda_distributed_*_sources 等列表。 想加 / 移源文件,改这里就行。

4. What torch._C actually is

Breaking it down:

  1. torch._C is a Python extension module; in the Linux CUDA wheel layout discussed here, it is provided by libtorch_python.so.
  2. libtorch_python.so is built as a CPython extension (links Python::Module) and installed to torch/lib/libtorch_python.so in this wheel. Other platforms or build modes may use different file names or install locations.
  3. Its Python entry symbol is PyInit__C, defined in torch/csrc/Module.cpp. The name tells CPython this is the _C extension; setuptools registers it under the full path torch._C.
  4. On the Python side, from torch._C import * triggers the dynamic linker to load libtorch_python.so; its NEEDED entries drag in libtorch.so; the aggregator drags in libtorch_cpu.so and libtorch_cuda.so; and those drag in libc10.so and libc10_cuda.so.

So, in this Linux wheel context, torch._C from Python's perspective mostly lands on the libtorch_python.so extension module from the C++ / dynamic-linker perspective; the rest of the chain is the dynamic linker doing its job.

The startup sequence in torch/__init__.py:

# 1. dlopen libtorch_global_deps with RTLD_GLOBAL so MKL/cudart symbols are global
_load_global_deps()

# 2. load _C (i.e. libtorch_python), which triggers the full .so chain
from torch._C import *

# 3. explicitly init ATen / c10 global state
_C._initExtension(_manager_path())

5. A few counterintuitive points

  1. libtorch.so is empty. nm libtorch.so finds almost nothing; it is a linker manifest.
  2. The CMake that defines torch_cpu and torch_cuda lives in caffe2/CMakeLists.txt. Do not be confused by the directory name; it is the actual home of libtorch, not anything to do with the historical Caffe2 framework.
  3. libtorch_python does not link libtorch_cpu or libtorch_cuda directly. It links the empty libtorch aggregator and gets both halves transitively. This indirection is what lets split-build mode swap in an externally installed libtorch.
  4. Nothing links libtorch_global_deps. It exists solely to be loaded by ctypes.CDLL(..., RTLD_GLOBAL) for MKL / cudart symbol preloading.
  5. libcaffe2_nvrtc is a stub. Its purpose is to defer NVRTC binding, so environments without NVRTC can still import torch without crashing. PyTorch dlopens NVRTC through this stub only when the user actually needs it.
  6. libnnapi_backend.so ships unconditionally on Linux. It is the Android NNAPI TorchScript backend; no one uses it on Linux. Historical artefact.
  7. The single source of truth for source-file lists is build_variables.bzl. Both CMake and Bazel read libtorch_python_core_sources, libtorch_python_cuda_core_sources, libtorch_cuda_distributed_*_sources, and friends from this .bzl. Adding or removing source files happens here.
