PyTorch Dispatcher 全栈剖析:从 schema 到 CUDA kernel

本文 PyTorch 源码按写作时的 pytorch/pytorch main 分支核对;源码文件名、行号和部分内部 API 可能随版本漂移。读源码时请优先对照你实际使用的 PyTorch tag / commit。本文重点剖析 dispatcher 数据结构、注册宏的语义、以及一个 native op 从 Python 调到 CUDA 的完整调用栈;不涉及量化/分布式/inductor 等下游话题。

1. 调度键:运行时多维分发的基础

本章自下而上建立 dispatcher 的运行期模型。我们先从用户视角看"为什么 if/else 行不通"——这一节会自然地导出 dispatcher 的核心思想:把"该走哪条路"的决策从代码里抽出来,变成一张按 key 索引的查找表。然后我们看 PyTorch 用什么数据结构表达这张表(DispatchKey 枚举、DispatchKeySet 位图),怎么算出"激活了哪些 key",怎么决定"先走哪一层",以及一个 kernel 跑完后怎么不死循环地把控制权传给下一层。

读完本章,你应该能在脑子里精确模拟 z = x + y 在 dispatcher 内部走过的每一步。

1.1 为什么需要"调度"

我们从一个看起来再普通不过的 PyTorch 操作开始:

y = x + 1

如果 x 是一个普通的 CPU float32 张量、且不需要梯度,那这行代码只需要做一件事:跑一个简单的循环 y[i] = x[i] + 1。一个 if/else 足以判断"该走 CPU 加法还是 CUDA 加法":

// 玩具实现 —— PyTorch 没有这么写,后面解释为什么
Tensor add(const Tensor& x, Scalar s) {
  if (x.is_cuda()) return add_cuda(x, s);
  else            return add_cpu(x, s);
}

问题是,真实情况要复杂得多。同一个 x + 1,在不同的上下文中,内部要做截然不同的事:

  • 如果 x.requires_grad == True,加法之前必须先建一个 AddBackward0 节点,把 x 登记为它的"上游",方便日后 loss.backward() 时反向追溯。
  • 如果调用嵌在 with torch.autocast('cuda'): 里,加法之前要先把 fp32 输入转成 fp16,然后才算加法 —— 这样训练阶段才能拿到混合精度的吞吐收益。
  • 如果嵌在 with FakeTensorMode(): 里,根本不能真算 —— 只能"假装"算,推断输出 shape/dtype 并返回一个"形似而无内容"的张量(用于编译/分析阶段)。
  • 如果 x 是个稀疏张量(torch.sparse_coo_tensor(...)),加法走的是完全不同的算法 —— 不是 elementwise loop。
  • 如果 x 经过了 torch.func.vmap(...),加法要在额外一个 batch 维度上广播。

更要命的是,这些条件可以叠加。一个真实的训练 step 里完全可能同时打开 requires_grad、autocast、CUDA。每一种"上下文"都要按正确的顺序插入到这次调用里 —— autocast 必须在 autograd 之前(否则反向图记错了 dtype);autograd 必须在 CUDA 之前(否则记不到反向上游);FakeMode 必须在所有之上(否则就真算了)。

按 if/else 写,代码会长成下面这样,而且必须给每一个算子都写一遍:

Tensor add(const Tensor& x, Scalar s) {
  // FakeTensorMode 拦截:只算 meta,不算数值
  if (in_fake_mode()) return add_fake(x, s);

  // autocast:先降精度
  if (autocast_enabled() && x.is_cuda()) {
    x = x.to(at::kHalf);
  }

  // autograd:记反向
  if (x.requires_grad()) {
    register_backward<AddBackward0>(x);
  }

  // 真的算:按 layout × device 分支
  if (x.is_sparse()) return add_sparse(x, s);
  if (x.is_cuda())   return add_cuda(x, s);
  return add_cpu(x, s);
}

这种写法有两个致命缺陷,而且 PyTorch 团队没办法绕过:

  1. 重复。ATen(PyTorch 的算子库)有 ~2000 个算子。每一个都要把这段嵌套 if/else 复制一遍。任何一处忘记 if (autocast_enabled()),这个算子就会在 autocast 下静默走错精度;任何一处少写 if (in_fake_mode()),这个算子在编译 trace 阶段会真的去算 —— 失败模式难抓到极点。
  2. 封闭。如果今天 Habana / Apple Metal / 华为昇腾要给 PyTorch 加一个新后端,它们不能修改 PyTorch 源码 —— 必须能从外部"插入"自己的 is_my_backend() 分支。if/else 链做不到这一点。同理,第三方的 autocast 实现、JIT tracer、新型变换(vmap/grad/jvp)也都没法横切到 PyTorch 算子里。
小例 1.1 想象 PyTorch 想支持一个新后端 Foo。在 if/else 方案下,你要去把所有 ~2000 个算子的源码翻一遍、补上 if (x.is_foo()) return add_foo(...)。然后下次 PyTorch 升级到新版本,你的所有补丁要重做。每次合并冲突都是噩梦。这就是 if/else 方案在工程上立不住的核心原因 —— 不是"丑",而是"扩展性破产"。

1.2 思路:按 key 索引的函数表

PyTorch 的解法是把"该走哪条路"的决策从代码里抽出来,放进一张查找表。

核心机制。每个算子(例如 aten::add)在 PyTorch 启动后都拥有一张函数指针表。表的下标叫做调度键(DispatchKey),代表"一种行为层":CUDA 后端、CPU 后端、Autograd 包装、Autocast 包装、FakeMode 拦截 …… 每一项都是一种可能的"该做什么"。

例如,aten::add 的查找表大致长这样(简化,真实条目更多):

DispatchKey          →  函数指针 (kernel)
─────────────────────────────────────────────────
CPU                  →  add_cpu
CUDA                 →  add_cuda
SparseCPU            →  add_sparse_cpu
SparseCUDA           →  add_sparse_cuda
MPS                  →  add_mps                  // Apple Metal
AutogradCPU          →  VariableType::add_cpu_wrapper
AutogradCUDA         →  VariableType::add_cuda_wrapper
AutocastCPU          →  autocast::add_cpu_wrapper
AutocastCUDA         →  autocast::add_cuda_wrapper
Fake                 →  fake::add_meta_only
...等几十项

调用 torch.add(x, y) 时,dispatcher 干三件事:

  1. 算出激活的 keys。从输入张量的属性(在 CUDA 上、requires_grad)+ 当前线程的"模式栈"(autocast 是否开启、是否在 FakeTensorMode 里)算出一个键的集合,叫 DispatchKeySet。例如 {AutocastCUDA, AutogradCUDA, CUDA}。
  2. 取优先级最高的一个。例如这个集合里 AutocastCUDA > AutogradCUDA > CUDA,选 AutocastCUDA。
  3. 跳过去。查表得到 AutocastCUDA 对应的函数指针 autocast::add_cuda_wrapper,调用它。

关键巧思:这个 autocast 包装函数做完自己的事(把输入降到 fp16)后,要继续走 autograd → CUDA。它不会再调一次 torch.add(...) 那样回到顶层 —— 那会死循环。它做的是:把"AutocastCUDA"这个 key 从位图里 mask 掉,然后告诉 dispatcher 再算一次最高优先级 key。这次最高优先级变成 AutogradCUDA,跳到 autograd 包装;autograd 包装记完反向图后,再把 AutogradCUDA mask 掉,这次落到 CUDA,跑真的 elementwise kernel。

整条调用链像剥洋葱一样,一层做一层,然后把自己"擦掉"让下一层接力。下面这张图把 with torch.autocast('cuda'): z = x + y(x, y 在 CUDA 上、requires_grad=True)的完整路径画出来:

每个 kernel 只管自己那层。CUDA kernel 不需要知道 autocast 存在;autocast 不需要知道 autograd 存在。 • 可扩展。新加一个后端只是往每个算子的查找表里塞新行,完全不动调用点。新模式装饰器(类似 vmap)也是新加一个 key。 • 可组合。"位图里有哪些 key"决定了"经过哪些层";顺序由优先级决定,跟代码完全解耦。 • 顺序由设计保证。"autocast 必须在 autograd 之前"这个约束被编码进 key 的优先级,不再依赖每个算子作者记得。 这种"逐层下传、每层 mask 自己"的模式,正式名字叫 redispatch。1.7 节会拆开它的实现。
边栏 1.2:为什么不用虚函数 / std::function 你可能会想:这不就是 C++ 的虚函数表(vtable)吗?为什么不直接用语言原生机制?

差别在两点:
① vtable 的"动态类型"只来自一个对象(this 指针)。但 dispatcher 的动态类型来自所有输入张量的并集 + 一组线程局部状态。torch.add(x, y) 的实际行为既看 x,也看 y,还看是否在 autocast 里 —— 这是多分派(multiple dispatch),vtable 表达不了。
② std::function 能表达多分派,但每次调用都要堆分配一个类型擦除的闭包。对每秒几百万次算子调用的训练场景,这种开销不可接受。

手撸位图 + 数组查表,每次 dispatch 大约 5-10 ns,比虚函数还便宜。

1.3 DispatchKey 枚举的真身

知道了大思路,现在看具体定义。所有可能的 key 列在 c10/core/DispatchKey.h。这里先看一个简化版(实际定义在 1.4 节会展开,有更精巧的二维拆分,但思路完全一致):

// 简化示意 —— 真实定义见 1.4
enum class DispatchKey : uint16_t {
  Undefined = 0,

  // === 后端层(真的算数值)===
  CPU,
  CUDA,
  MPS,            // Apple Metal Performance Shaders
  XLA,            // Google TPU / XLA backend
  Meta,           // 只算 shape 不算数值
  ...
  SparseCPU,
  SparseCUDA,
  ...

  // === Autograd 层(在后端之前,负责建反向图)===
  AutogradCPU,
  AutogradCUDA,
  AutogradXLA,
  ...

  // === 模式层(在 autograd 之前,影响每一次算子调用)===
  ADInplaceOrView,   // in-place / view 的 autograd 元信息
  Tracer,            // 旧 JIT trace 记录
  AutocastCPU,
  AutocastCUDA,
  Fake,              // FakeTensorMode
  Functionalize,     // 消除 alias/mutation
  Python,            // TorchDispatchMode (子类化拦截)
  ...                // 总共约 60 个

  NumDispatchKeys
};

三个观察:

  • 类型是 uint16_t(16 位无符号整数)。一会儿要把 key 打包进 64 位位图,16 位的下标足够编码所有可能 key 加上头部位标记。
  • 枚举是有序的。后面会看到,"优先级最高的 key"就是位图里最高位 1 对应的枚举值 —— 排在枚举后面的 key 优先级高。所以你看到 Autocast* 排在 Autograd* 后面、Autograd* 排在 CPU/CUDA 后面 —— 这正是 1.1 里"autocast 必须在 autograd 之前发生"的代码层翻译。设计师把"语义上的层级顺序"直接编码进了枚举的字面顺序。
  • 约 60 个 keys(NumDispatchKeys 占位常量)。看起来很多,但其中大多数是"模式层"(autocast / fake / functionalize / python 等),平常调用根本不激活;只有打开对应 with-block 才会被 push 进位图。
小例 1.3 一个普通 CPU float32 张量、不需要梯度,跑 x + 1 时,位图只有 {CPU} 这一个 key,最高优先级就是 CPU,直接跳到 add_cpu。如果 x.requires_grad_(True),位图变成 {AutogradCPU, CPU},最高优先级是 AutogradCPU,先进 autograd 包装,再 redispatch 到 CPU。整个 dispatcher 走了两轮,但用户代码完全没变。

1.4 二维拆分:functionality × backend

上面那个枚举有个尺寸问题:如果按"每种行为 × 每种 backend"做笛卡儿积,枚举会爆炸。例如 PyTorch 想支持 Habana(HPU),它需要新增这一堆条目:

HPU              // dense × Habana
SparseHPU        // sparse × Habana
SparseCsrHPU
QuantizedHPU
NestedTensorHPU
AutogradHPU      // autograd × Habana

一个新 backend 就要新增 ~6 个枚举值,每加一个就要改 enum 定义、改所有相关 codegen。每加一种新功能(假设以后要支持 Adagrad×Backend 这种)又是 N 倍。这不可持续。

PyTorch 团队 2022 年做了一次重要重构:把枚举拆成两个正交的小枚举,合并到一张 64 位位图里(DispatchKey.h):

  1. BackendComponent —— 仅仅"是哪个后端"。当前的位:CPU / CUDA / HIP / XLA / MPS / IPU / XPU / HPU / VE / Lazy / MTIA / MAIA / PrivateUse1‑3 / Meta。一个 static_assert 限定后端数 ≤ 16(DispatchKey.h 里有显式 16 位预算)。注意它单独不构成分派级别 —— 必须和某个 functionality 组合才有意义。
  2. DispatchKey(功能 functionality)。绝大多数 functionality 只占一格(全局唯一 handler);六个是 "per-backend",需要和某个 BackendComponent 组合才落成一个运行时 key。这六个是:Dense、Quantized、Sparse、SparseCsr、NestedTensor、AutogradFunctionality。

组合规则(toRuntimePerBackendFunctionalityKey()):

(Dense,                CUDABit)  →  CUDA           // 你熟悉的 "CUDA" key 真身
(Dense,                CPUBit)   →  CPU
(AutogradFunctionality, CUDABit) →  AutogradCUDA
(AutogradFunctionality, CPUBit)  →  AutogradCPU
(Sparse,               CUDABit)  →  SparseCUDA
(Quantized,            CPUBit)   →  QuantizedCPU
...

也就是说,你写 TORCH_LIBRARY_IMPL(aten, CUDA, m) 其实是在 (Dense, CUDA) 这个二维坐标上注册。代码层面你看到的还是 CUDA 这一个 key —— 但底层位图把它编码成 "Dense 这个 functionality 位 + CUDA 这个 backend 位 同时为 1"。新增一个 backend(HPU)?只要给 BackendComponent 加一个位 —— 那 6 个 per-backend functionality 就自动为它定义了 6 个新组合 key,根本不用改 enum。

小例 1.4 要给 PyTorch 加 Habana(HPU)后端的"sparse"实现:在 BackendComponent 里加一个 HPU 位 → 立刻就有了 (Sparse, HPU) 这个组合 key,可以用 TORCH_LIBRARY_IMPL(aten, SparseHPU, m) 注册稀疏 kernel。enum DispatchKey 里既不需要新加 SparseHPU,也不需要新加 AutogradHPU —— 它们都是组合出来的运行时键。
边栏 1.4:drift 警告 ezyang 2019 年的 PyTorch internals 没有 BackendComponent 这一层,2020 年的 dispatcher 文章也没有。本节描述的二维拆分是 2022+ 后的最新机制,看老文章时注意这点已经过期 —— 它们把 SparseCUDA 当作枚举里的一个具体值,在今天的代码里它实际是 (Sparse, CUDABit) 的组合。

1.5 DispatchKeySet:64 位位图

现在我们有了 DispatchKey(约 60 个 functionality)和 BackendComponent(16 个后端)。把这些塞进一个 64 位的整数(uint64_t),就得到了 DispatchKeySet。它是 dispatcher 的核心数据结构 —— 每次算子调用,dispatcher 拿到的就是这么一个 64 位整数。

这个 64 位整数从哪里来?三个来源汇总:

  1. 所有输入张量的 TensorImpl::key_set_ 的并集。每个张量的 TensorImpl 里都缓存了一个 DispatchKeySet,记录"我是 CUDA 上的"+"我 requires_grad"等属性。dispatcher 把所有参数张量的 key_set or 起来 —— 这就实现了"多分派"(任何输入是 CUDA 都会激活 CUDA key)。
  2. TLS local include(线程局部"添加"):由模式 guard 往里加。例如进入 with torch.autocast('cuda'):,这个 with 块的实现会把 AutocastCUDA 这一位 push 进 TLS;每次算子调用,dispatcher 把 TLS include or 进位图。
  3. TLS local exclude(线程局部"剔除"):用于 redispatch。autograd kernel 内部调 at::AutoDispatchBelowAutograd guard;,这个 RAII 对象把所有 autograd 位塞进 TLS exclude。dispatcher 把 exclude 从位图里减掉,让最高位"自然"落到下一层。

第四个隐藏因素:fallthrough mask。每个算子在注册期就预先算好"哪些 key 在我这里是 fallthrough(没注册具体 kernel,等价于'我这层跳过')"。这些 key 在位图进 dispatcher 时被一次性 mask 掉,实现 O(1) 跳过。这是 dispatcher 能"白板支持几十个 functionality 而不付出运行时代价"的关键 —— 你的算子如果没注册 Conjugate 上的 kernel,Conjugate 位就被 fallthrough mask 直接 clear。

具体到代码,核心一行在 DispatchKeyExtractor.h:23-44:

// aten/src/ATen/core/dispatch/DispatchKeyExtractor.h
inline DispatchKeySet computeDispatchKeySet(
    DispatchKeySet ks,              // 输入参数张量 or 起来的 key_set
    DispatchKeySet key_mask) {      // 算子的 fallthrough mask
  c10::impl::LocalDispatchKeySet local = c10::impl::tls_local_dispatch_key_set();
  return (((ks | local.included_) - local.excluded_) & key_mask);
  //        ↑ 加 include    ↑ 减 exclude     ↑ 应用 fallthrough mask
}

这就是 1.5 节的全部"魔法":三段位运算,得到一个最终位图。然后 lookup() 在这个位图里找最高位 —— 1.6 节展开。

小例 1.5 设 x 是 CUDA fp32 张量,requires_grad=True,代码在 with torch.autocast('cuda'): 里调 z = x + 1。

1. x 的 key_set_ = {CUDABit, Dense, AutogradFunctionality} = 位图里 CUDABit、Dense、AutogradFunctionality 三位为 1。
2. autocast 上下文 push 了 AutocastCUDA 到 TLS include。
3. 没有 exclude,没有 fallthrough mask 截掉这几位。
4. 最终位图组合 functionality + backend 后,等价于运行时 key 集合 {AutocastCUDA, AutogradCUDA, CUDA}。

对照 1.2 节的"剥洋葱"图 —— 这就是那张图开始时的初始位图。

1.6 优先级阶梯

给定一个 64 位位图,dispatcher 用 highestPriorityTypeId() 取"最高位 1 对应的 functionality"。"最高"指的是位图里最高编号的 set bit —— 功能位是按优先级排好的,bit 编号 = 优先级。

下图把常见 functionality 按优先级画出来,从上(最高)到下(最低):

找 key:取位图最高位 1 → 走那一层 kernel;kernel 体里若调 at::redispatch(...) 则带着剔除掉当前 key 的位图,自然落到下一层。 • autograd "在 backend 之上"是设计灵魂:它能在算数值之前抓住输入、挂上 next_edges、建反向图,然后 redispatch 跑 CUDA。 • autocast 又在 autograd 之上:输入降精度先于 autograd 抓输入,反向图才会用降精度后的 dtype。这是 1.1 节"顺序约束"的代码翻译。 • fallthrough kernel = "我这层不做事,把我的位 mask 掉再分发" —— O(1) skip,大量模式 key 默认就是 fallthrough。

"catchall"这个词在 dispatcher 语境里指alias key 的兜底。Alias key 不是真正的 functionality,而是"注册时展开成多个具体 key"的便利写法:

  • Autograd(别名)= 注册到所有 Autograd<Backend> 上 —— 你写 TORCH_LIBRARY_IMPL(ns, Autograd, m) 时,这条注册被展开成对 AutogradCPU、AutogradCUDA、…… 都注册同一份 kernel。
  • CompositeImplicitAutograd(别名)= 所有 backend + 所有 autograd keys 上都生效 —— 适合用 at::* 组合写出的算子(组合内部的 op 会再走 dispatcher,从而获得 autograd 和后端无关性)。
  • CompositeExplicitAutograd(别名)= 所有 backend(不含 autograd)—— 后端无关实现,但作者需自己处理 autograd。

这三个 alias 让 PyTorch 对"一份代码覆盖所有后端"有了简洁的注册语法,本质上是批量 m.impl 的语法糖。运行时 dispatcher 里只会看到具体的 AutogradCUDA、CUDA 等 key,不会真的看到 Autograd 这个 alias。

1.7 redispatch:层间不死循环递归

1.2 节我们建立了"剥洋葱"的心智模型 —— 一层做完,把自己 mask 掉,让下一层接力。本节看具体怎么实现。

核心问题:autograd kernel 跑完自己那点事(建 grad_fn、记录 next_edges)之后,要怎么"调到 backend kernel"?直接调 at::add(x, y) 不行 —— 那会回到顶层 Dispatcher::call,最高位算出来又是 AutogradCUDA,再进 autograd kernel,死循环。

答案是 Dispatcher::redispatch(Dispatcher.h:833-848):

// 关键注释(从源码):"redispatch treats the inputted DispatchKeySet as is,
//  and does NOT mask out the highest priority bit"
//  —— 调用方负责把"当前 key"从位图剔掉,再传进来
template <class Return, class... Args>
inline Return Dispatcher::redispatch(
    const TypedOperatorHandle<Return(Args...)>& op,
    DispatchKeySet currentDispatchKeySet,    // ← 调用方已经把当前 key mask 掉
    Args... args) const {
  const KernelFunction& kernel = op.operatorDef_->op.lookup(currentDispatchKeySet);
  return kernel.template call<Return, Args...>(op, currentDispatchKeySet, ...);
}

对比 Dispatcher::call(顶层入口): call 会重新算位图(or 输入张量的 key_set + TLS 处理);而 redispatch 直接用调用方传来的位图。这是故意的——它给了 kernel "我已经决定下一步该跳哪儿了"的灵活性。

那"怎么剔掉当前 key"?Autograd kernel 内部用 RAII guard 来翻动 TLS exclude:

  • at::AutoDispatchBelowAutograd(老名字 AutoNonVariableTypeMode,在老博文里常看到):构造时把所有 autograd functionality 位都塞进 TLS excluded_;析构时还原。生命周期内,所有算子调用的位图都会被自动减掉 autograd 位。
  • at::AutoDispatchBelowADInplaceOrView:进一步把 ADInplaceOrView 也塞进 exclude。这是 codegen 出来的 VariableType wrapper 标准用法 —— autograd kernel 调它,然后 redispatch 直接落到 backend。

把这两个机制串起来,一个真实 autograd kernel 的骨架长这样(简化自 VariableType_*.cpp 的 codegen 输出):

// AutogradCUDA kernel for aten::add.Tensor (简化)
Tensor VariableType::add_Tensor(
    c10::DispatchKeySet ks,
    const Tensor& self, const Tensor& other, const Scalar& alpha) {

  // ① 建反向节点(只在需要 grad 时)
  std::shared_ptr<AddBackward0> grad_fn;
  if (compute_requires_grad(self, other)) {
    grad_fn = std::make_shared<AddBackward0>();
    grad_fn->set_next_edges(collect_next_edges(self, other));
  }

  // ② 进入 "below autograd" 范围 → autograd 位被剔出位图
  auto result = [&]() {
    at::AutoDispatchBelowADInplaceOrView guard;
    return at::redispatch::add(
        ks & c10::after_autograd_keyset,    // 显式 mask 当前 key
        self, other, alpha);
  }();

  // ③ 把 grad_fn 挂到输出
  if (grad_fn) set_history(result, grad_fn);
  return result;
}
小例 1.7:完整 redispatch 流程 跟着这段代码走一遍 z = x + y(x, y 在 CUDA、requires_grad=True):

(a) 顶层 Dispatcher::call 算出位图 = {AutogradCUDA, CUDA},最高位 = AutogradCUDA,跳到 VariableType::add_Tensor。
(b) 函数体进入 RAII guard AutoDispatchBelowADInplaceOrView:TLS exclude 里加上所有 autograd 位 + ADInplaceOrView 位。
(c) 调 at::redispatch::add(ks & after_autograd_keyset, ...)。after_autograd_keyset 是预算好的"清掉所有 autograd 及以上位"的掩码,显式确保 AutogradCUDA 不会再被选中。
(d) Dispatcher::redispatch 直接用这个位图 lookup,最高位现在是 CUDA,跳到 wrapper_CUDA_add_Tensor。
(e) CUDA wrapper 跑完返回。
(f) 回到 autograd kernel,set_history(result, grad_fn) 把反向节点挂上去,返回。

guard 在 (e) → (f) 之间析构,TLS exclude 还原。这样下一次顶层调用的位图计算又干净了。
记住这条心智模型:dispatcher 不是"一次跳转",而是"每次跳转都有可能触发下一次跳转 —— 由 kernel 决定要不要 redispatch"。模式装饰器、autograd、backend 形成一条由 redispatch 显式驱动的链。链的顺序由 functionality 在 64 位位图里的 bit 编号决定;每一层用 RAII guard 把自己擦掉,让下一层自然成为新的最高位。

2. Schema:算子的类型签名

第 1 章给出了 dispatcher 的运行期机制 —— 但 dispatcher 怎么知道"aten::add.Tensor"这个算子接受什么参数?返回什么?哪些参数会被改写?这些问题的答案叫做 schema。

Schema 是算子的接口契约。它不光是给文档看的字符串 —— 它被 dispatcher、autograd、JIT、functionalization、Python 绑定层同时读取,每个组件都从中提取自己关心的信息:dispatcher 从中知道有几个张量参数(决定 keyset);autograd 从中知道哪些参数会被改写(决定 view 处理);Python 绑定层从中生成参数解析代码。所以 schema 写错一个字 —— 比如把 Tensor(a!) 写成 Tensor(a) —— 不是文档错误,而是正确性错误:functionalization 会以为这个参数没被改写,生成错的图。

本章拆开 schema 的字符串文法、变体命名规则、以及它在 yaml 中的实际落地形式。

2.1 一个 schema 的逐字解读

看一个真实的 schema 字符串:

aten::add.Tensor(Tensor self, Tensor other, *, Scalar alpha=1) -> Tensor

看起来像个 Python 函数签名,但每一处标点都有独立的语义。我们从左到右一字一句拆:

位置片段语义
1aten命名空间。PyTorch 内置算子都在 aten 下;自定义算子(扩展)在自己命名空间下,比如 myops::foo。命名空间隔离了"哪些 op 能被找到",也决定了用哪个 TORCH_LIBRARY* 宏注册(见 3 章)。
2::命名空间分隔符,跟 C++ 一样。
3addop 名。同一个 op 名下可以有多个 overload(像 C++ 的函数重载)。
4.Tensoroverload 名。这是关键 —— add 有 add.Tensor(两个张量相加)和 add.Scalar(张量加标量)两个 overload。可省略,等于 overload 名为 ""(空串)。dispatcher 把"aten::add.Tensor"作为完整 key 来索引函数表。
5(...)参数列表。
6Tensor self第一个 positional 参数,类型 Tensor,名字 self。约定:类似"Python 方法"中的 self —— autograd 会把它当成"主"输入。
7Tensor other第二个 positional 参数,类型 Tensor。
8*关键词分隔符。从这里开始,后面的参数必须用关键字传(add(x, y, alpha=2)),不能位置传。和 Python 函数定义里的 * 完全一样。
9Scalar alpha=1关键词参数,类型 Scalar,默认值 1。Python 里写 torch.add(x, y, alpha=3) 时,这个值落到这。
10-> Tensor返回值类型。多个返回值用 tuple 写,比如 -> (Tensor, Tensor)。

类型语法覆盖一系列原语和复合类型:

类型含义
Tensor张量(强制非空)
Tensor?可空张量(对应 C++ std::optional<Tensor>)
Tensor[]张量列表(std::vector<Tensor>)
Tensor(a)alias 标记:此参数和某个名为 a 的"内存位"共享存储,但不写它(只读 view)
Tensor(a!)可变 alias 标记:共享存储且会改写(in-place / out)
Scalar标量(任意数值类型)
int / int[]整数 / 整数列表(C++ int64_t / IntArrayRef)
SymInt"符号整数",支持动态 shape 编译(torch.compile 用)
bool, str布尔、字符串
Device, ScalarType, Layout, MemoryFormat张量属性,通常出现在 factory function 里
Generator?随机数发生器,通常用在 dropout / randn / multinomial 上
记住这一句:alias 注解((a) 和 (a!))是 schema 里最重要的标记。它们告诉多个下游组件不同的事:
• functionalization 用它决定"这个 op 是否产生副作用,要不要展开成函数式形式"
• autograd 用它决定"这个 op 是否产生 view,要不要建 DifferentiableViewMeta"
• JIT 用它做 alias analysis(两个变量会不会指向同一块内存,影响死代码消除等优化)
• inductor / functorch 都依赖这套注解

错写一个 (a) 为 (a!),后果是 functionalization 把一个本来 in-place 的 op 当 functional 处理,反向图错;反过来,本来 functional 的写成 in-place,可能误报 alias 冲突。
小例 2.1:alias 注解的作用 看 aten::transpose 的 schema: 
aten::transpose.int(Tensor(a) self, int dim0, int dim1) -> Tensor(a)
Tensor(a) 出现在 self 和返回值上 —— 它们共享同一个名为 a 的"内存位"。这告诉 dispatcher 下游:转置后的张量和原张量共享存储,改其中一个的元素另一个也变。

对比 aten::clone: 
aten::clone(Tensor self, *, MemoryFormat? memory_format=None) -> Tensor
返回值没有 alias 标记 → 它是新分配的内存,改它不影响 self。

注册时,这个字符串会被 torch::jit::parseSchema 解析进一个 c10::FunctionSchema 结构体,记录每个参数的名字、类型、alias 信息、默认值;然后才登记到 dispatcher 的内部 op 表里。所以"schema 字符串"是源头,FunctionSchema 是它的 in-memory 表示。

2.2 三种变体:functional / in-place / out

很多算子有"副作用"能力(in-place 改写、写入预分配的 out 张量)。PyTorch 给这类算子同时提供三种 schema 变体,功能等价但接口不同。看 add 的三个变体:

变体schema
functionaladd.Tensor(Tensor self, Tensor other, *, Scalar alpha=1) -> Tensor
in-placeadd_.Tensor(Tensor(a!) self, Tensor other, *, Scalar alpha=1) -> Tensor(a!)
outadd.out(Tensor self, Tensor other, *, Scalar alpha=1, Tensor(a!) out) -> Tensor(a!)

命名约定:

  • functional:返回新张量。Python 里写 z = x.add(y)。最常用,对应 schema 的"基础形态"。
  • in-place:op 名末尾加 _(下划线),self 加 (a!),返回也是 (a!) 表示"返回的就是改后的 self"。Python 里写 x.add_(y),等价 x = x + y 但不分配新内存。
  • out:overload 加 .out,多一个 kw-only 的 Tensor(a!) out。Python 里写 torch.add(x, y, out=z),把结果写进预分配的 z。常用于复用内存的优化场景。

为什么要写三套?有三个原因:

  1. 用户写法差异。Python 用户三种写法都要支持。如果只写 functional,x.add_(y) 就不存在。
  2. 性能。functional 总是分配新内存,in-place 和 out 可复用,对训练阶段的内存峰值差距很大。
  3. autograd 行为。三个变体的 autograd 处理完全不同:functional 直接建 grad_fn,in-place 要先升 version counter、可能拒绝(RuntimeError: a leaf Variable that requires grad...),out 介于两者之间。
小例 2.2:三个变体怎么共用实现 虽然 schema 有三套,实际数学只写一次。底层用 structured kernel 模式: 
add.out      :  写入预分配的 out tensor (核心实现)
add.Tensor   :  分配一个新 out → 调 add.out → 返回 out
add_.Tensor  :  把 self 当成 out → 调 add.out → 返回 self
"分配 out + 调 out 版本"这一步是 codegen 自动生成的(看 yaml 里的 structured_delegate: add.out),作者只需要写 out 版本的实现。
边栏 2.2:autogen 关键字 某些算子作者只写 functional 一份,告诉 codegen "请按规则补齐 in-place 和 out 变体": 
- func: my_op(Tensor self) -> Tensor
  autogen: my_op.out, my_op_
codegen 看到 autogen: 后会自动生成 my_op.out 和 my_op_ 的 schema 注册以及"out 包装 functional"的实现,作者一行实现都不用写。这对纯函数式 op(没有 in-place 复杂性)非常省力。

2.3 native_functions.yaml:批量声明

对比一个非 structured 的简单 op —— relu:

# native_functions.yaml — relu (非 structured,直接 dispatch)
- func: relu(Tensor self) -> Tensor
  device_check: NoCheck
  variants: function, method
  dispatch:
    CPU, CUDA: relu                 # 同一个函数名,绑到两个后端

这里 dispatch: CPU, CUDA: relu 把同一个名字 at::native::relu 同时绑到 (Dense, CPU) 和 (Dense, CUDA)。具体实现是两个文件里的两个 overload(aten/src/ATen/native/Activation.cpp 给 CPU,aten/src/ATen/native/cuda/Activation.cu 给 CUDA),靠 link 时按 device 决定。这种非 structured 写法适合简单算子;但失去了 ufunc 自动生成的便利。

边栏 2.3:为什么有 structured 这一层 PyTorch 早年(2018 之前)每个算子的 functional / inplace / out 三套实现都得作者手写,大量代码长这样:add 的 functional 分配输出 → 调 add_kernel;add_ 把 self 当输出 → 调 add_kernel;add_out 检查 out shape → 调 add_kernel。这种重复代码每个算子要写 3 遍,每次 add 一个新算子都要写 ~100 行胶水。

2020 后引入的 structured kernel 模式标准化了这套胶水:作者只在 .out 变体上写实际数学(impl()),meta 阶段(算输出 shape/dtype)写在 meta()。codegen 自动生成 functional 和 inplace 变体作为 out 的包装。新算子只需要 ~10 行 yaml + 一份 impl,代码量降到原来的 1/5。

2.4 derivatives.yaml:反向公式

到目前为止 schema 只描述了"前向行为"。反向(autograd)信息在另一个文件里:tools/autograd/derivatives.yaml。每个 differentiable op 在这里给一段反向公式:

- name: add.Tensor(Tensor self, Tensor other, *, Scalar alpha=1) -> Tensor
  self:  handle_r_to_c(self.scalar_type(),  grad)
  other: handle_r_to_c(other.scalar_type(), maybe_multiply(grad, alpha.conj()))
  result: self_t + maybe_multiply(other_t, alpha)            # forward-mode tangent

- name: relu(Tensor self) -> Tensor
  self: threshold_backward(grad, result, 0)
  result: auto_element_wise

逐字解读:

  • name: 必须和 native_functions.yaml 里的 schema 完全一致 —— 这是匹配两边的"主键"。
  • self:、other: 是输入参数名,值是一段表达式,描述"如果这个输入要算梯度,梯度 = 什么"。例如 add 的 other 输入梯度 = grad * alpha.conj()(用 maybe_multiply 包了一层,alpha == 1 时跳过乘法优化)。grad 是个魔法变量,代表"这次 backward 传入的上游梯度"。
  • result: 是前向模式(jvp / tangent)的公式,用于 torch.func.jvp。self_t、other_t 也是魔法变量,代表"输入 self 的切向量"。如果不需要支持 jvp,这一行可以省。
  • auto_element_wise 是个特殊标记,告诉 codegen "这是个 elementwise op,jvp 公式按规则自动推导"。

这些公式不是 Python,而是 codegen 用的受限 DSL —— 它支持调用其他 at::* op(如 threshold_backward)、访问输入的属性(如 self.scalar_type())、用 grad/result/self_t 这样的预定义变量。

codegen 怎么用它?对每个 differentiable op,生成两个东西:

  1. 一个 autograd Node 子类(例如 AddBackward0、ReluBackward0),它的 apply() 方法体就是公式翻译成的 C++ 代码。
  2. 一个 autograd kernel wrapper(注册到 AutogradCPU/AutogradCUDA 等 key),在前向时建这个 Node、挂 next_edges、redispatch 到 backend(就是 1.7 节那段骨架代码)。
小例 2.4:derivatives.yaml 与生成代码的对应 derivatives.yaml 里 add 的反向公式: 
other: handle_r_to_c(other.scalar_type(), maybe_multiply(grad, alpha.conj()))
被 codegen 翻译成 AddBackward0::apply() 里的: 
if (should_compute_output(1)) {
  grad_inputs[1] = handle_r_to_c(other_scalar_type,
                                 maybe_multiply(grad, alpha.conj()));
}
几乎是一比一翻译(只多了"要不要算这个梯度"的判断,因为某些下游可能不需要 other 的梯度,跳过能省计算)。

5.8 节会展开看 AddBackward0::apply() 的完整生成代码,以及 autograd engine 在 loss.backward() 时怎么调用它。

本章带走的三个心智模型:
① Schema 是多方共读的契约 —— dispatcher、autograd、JIT、functionalization、Python 绑定都从同一字符串里抽不同信息。
② (a!) alias 注解是 schema 里"最危险"的标记 —— 写错直接影响 functionalization、autograd、alias analysis 三个正确性。
③ native_functions.yaml + derivatives.yaml 是一切 codegen 的输入源,后面所有生成的 C++ 文件(包括 dispatcher 注册、autograd wrapper、Python 绑定)都从这两个文件长出来。

3. C++ 注册接口:TORCH_LIBRARY 家族

第 2 章我们看到 schema 是契约。本章回答"谁来执行这份契约" —— PyTorch 用一组宏(全部定义在 torch/library.h)把"这个 schema 在 X 后端的实现是这一段 C++ 代码"喂给 dispatcher。

这组宏看起来花样很多 —— TORCH_LIBRARY、TORCH_LIBRARY_FRAGMENT、TORCH_LIBRARY_IMPL、m.def、m.impl、m.fallback、TORCH_FN、makeFromBoxedFunction ...... 但它们底层全是一个模式:

"用一个 file-static 对象的构造函数,在 main() 之前把信息塞进全局 dispatcher。"

所有差异都在"塞什么信息":开一个新命名空间、续写已有命名空间、还是给已有 op 挂某个 key 上的 kernel。本章把这三种"塞法"逐个拆开,然后讲三个相关概念(boxed vs unboxed、fallback vs fallthrough、def vs impl),最后用一个完整的自定义 op 把所有元素拼起来。

3.1 Library 的三种身份:DEF / FRAGMENT / IMPL

注册的核心数据结构是 torch::Library 对象。每次你写一个 TORCH_LIBRARY* 宏,展开后都构造一个 Library 实例 —— 但它有三种身份(torch/library.h:551-553):

// torch/library.h
class Library {
 public:
  enum Kind { DEF, IMPL, FRAGMENT };
  ...
};

三种 Kind 对应三种语义:

Kind语义触发宏典型用户
DEF给某个 namespace开一个"主"库。整个进程里一个 namespace 名只允许出现一个 DEF。重复链接立即 abort。TORCH_LIBRARY(ns, m)自定义扩展(给自己开一个 myops 命名空间)
FRAGMENT给同 namespace追加一片"片段"。同名 fragment 可以来自任意多个翻译单元(TU),拼到一起算一个库。TORCH_LIBRARY_FRAGMENT(ns, m)PyTorch 内部:aten 命名空间下的 ~2000 个 op 由几十个 codegen 文件拼出
IMPL给"已存在的某个 op"挂一个 DispatchKey 上的 kernel。不能在这里定义新 schema,只能 m.impl(...) 给现有 op 注册实现。TORCH_LIBRARY_IMPL(ns, key, m)给某后端注册具体实现(aten 的 CUDA wrapper、自定义 op 的 CUDA kernel 等)
小例 3.1:三种 Kind 的实际用例 你写一个 PyTorch 扩展 my_extension,它注册一个 myops::custom_relu 算子,在 CPU、CUDA 上各有实现。三个 .cpp 文件分工:

register_schema.cpp 用 TORCH_LIBRARY(myops, m) → DEF,声明 schema。这一行存在,myops 命名空间才"开"了。
kernel_cpu.cpp 用 TORCH_LIBRARY_IMPL(myops, CPU, m) → IMPL,给 CPU 注册实现。
kernel_cuda.cu 用 TORCH_LIBRARY_IMPL(myops, CUDA, m) → IMPL,给 CUDA 注册实现。

如果你把 schema 和 CPU kernel 都写在一个文件里(TORCH_LIBRARY(myops, m) { m.def(...); m.impl(...); }),也可以 —— 但 m.impl 在 DEF 块里不能指定 DispatchKey,只能用作"catch-all kernel"(所有 key 都走它)。所以多后端场景下,你必须用 IMPL 把不同 backend 拆开注册。
边栏 3.1:为什么不是单一宏 你可能会问:既然底层都是构造 Library 对象,为什么不能合并成一个 TORCH_LIBRARY(ns, kind, key, m)?

有两个原因:
① 静态校验。DEF 不允许指定 DispatchKey(它声明 schema,不挂 kernel);IMPL 必须指定 DispatchKey。把它们做成三个独立宏,编译期就能阻止参数错乱(IDE 也能更好地补全)。
② 链接期"唯一性"检查。DEF 走 file-static 对象的"无 uniquifier"路径,这样多次链接会触发重复符号错误,从而强制"一个 ns 只能有一个 DEF"。FRAGMENT 和 IMPL 都用 C10_UID 唯一化符号名,允许同名宏在多个文件里共存。这个区别是用宏的展开方式编码的,合并成一个宏就破坏了。

3.2 TORCH_LIBRARY:开新命名空间

看 TORCH_LIBRARY 宏的真身(torch/library.h:982-991):

// torch/library.h
#define TORCH_LIBRARY(ns, m)                                                   \
  static void TORCH_LIBRARY_init_##ns(torch::Library&);                        \
  static const torch::detail::TorchLibraryInit TORCH_LIBRARY_static_init_##ns( \
      torch::Library::DEF,                                                     \
      &TORCH_LIBRARY_init_##ns,                                                \
      C10_STRINGIZE(ns), std::nullopt,                                         \
      __FILE__, __LINE__);                                                     \
  void TORCH_LIBRARY_init_##ns(torch::Library& m)

看起来吓人,但拆开看就是三个声明:

  1. 第 1 行:static void TORCH_LIBRARY_init_##ns(torch::Library&); —— 前向声明一个函数(##ns 是 token 拼接,如果 ns=myops,这就是 TORCH_LIBRARY_init_myops)。
  2. 第 2-5 行:static const torch::detail::TorchLibraryInit TORCH_LIBRARY_static_init_##ns(DEF, ...); —— 构造一个文件级 static 对象。它的构造函数收到这些参数:
    • torch::Library::DEF —— 身份标记,这是 DEF 不是 IMPL。
    • &TORCH_LIBRARY_init_##ns —— 用户提供的"注册函数"(第 1 行声明的那个),构造函数会调用它。
    • C10_STRINGIZE(ns) —— namespace 字符串字面量("myops")。
    • std::nullopt —— DispatchKey,DEF 时为空。
    • __FILE__, __LINE__ —— 报错时的位置信息。
  3. 第 6 行:void TORCH_LIBRARY_init_##ns(torch::Library& m) —— 这是函数的定义头,用户的 { ... } 紧接在宏之后提供函数体。

所以用户写的 TORCH_LIBRARY(myops, m) { m.def("..."); } 展开成:

static void TORCH_LIBRARY_init_myops(torch::Library&);
static const torch::detail::TorchLibraryInit TORCH_LIBRARY_static_init_myops(
    torch::Library::DEF,
    &TORCH_LIBRARY_init_myops,
    "myops", std::nullopt,
    "/path/to/myops.cpp", 42);

// 用户的 { ... } 提供这个函数的 body:
void TORCH_LIBRARY_init_myops(torch::Library& m)
{
  m.def("...");
}

关键魔法在文件级 static 对象的初始化时机:C++ 标准保证 main() 调用之前所有 file-static 对象都已构造完。所以当 PyTorch 的 main / 用户的 Python 解释器启动时,TorchLibraryInit 的构造函数已经跑过、已经把这个 schema 塞进全局 dispatcher 表了。用户的注册代码"自动"在程序启动期生效,完全不需要手动调用。

小例 3.2:用法 
// myops_register.cpp
#include <torch/library.h>

TORCH_LIBRARY(myops, m) {
  m.def("custom_add(Tensor self, Tensor other) -> Tensor");
  m.def("custom_relu(Tensor self) -> Tensor");
}
这一段编译并链接进 Python 进程后,Python 端就能用 torch.ops.myops.custom_add(...)、torch.ops.myops.custom_relu(...) —— 即使还没注册任何后端 kernel。当然调起来会报 "no kernel registered" 错;但 schema 本身已经在 dispatcher 里了。
关键限制:同一个 namespace 在整个进程里只能出现一个 TORCH_LIBRARY(ns, ...)。再写第二个会触发运行时 std::abort(具体在 Library 构造时检查重复)。如果你需要把同 namespace 的注册拆到多个 .cpp 文件,请用下一节的 TORCH_LIBRARY_FRAGMENT。

3.3 TORCH_LIBRARY_FRAGMENT:同命名空间拼接

FRAGMENT 解决"同 namespace 跨文件"的需求。看宏(torch/library.h:1001-1023,简化):

// torch/library.h (简化)
#define TORCH_LIBRARY_FRAGMENT(ns, m) _TORCH_LIBRARY_FRAGMENT(ns, m, C10_UID)
#define _TORCH_LIBRARY_FRAGMENT(ns, m, uid)                                \
  static void C10_CONCATENATE(TORCH_LIBRARY_FRAGMENT_init_##ns##_, uid)(   \
      torch::Library&);                                                    \
  static const torch::detail::TorchLibraryInit                             \
      C10_CONCATENATE(TORCH_LIBRARY_FRAGMENT_static_init_##ns##_, uid)(    \
          torch::Library::FRAGMENT,                                        \
          ...,                                                             \
          C10_STRINGIZE(ns), std::nullopt, ...);                           \
  void C10_CONCATENATE(...)(torch::Library& m)

和 TORCH_LIBRARY 比,只有两点不同:

  1. Kind 从 DEF 改成 FRAGMENT —— 这告诉 Library 构造函数"不要因为重复 namespace 而 abort"。
  2. 符号名用 C10_UID 唯一化。C10_UID 是 PyTorch 内部的"行号 + counter"宏,每次展开都生成不同的 token。这样同一个 TU 里写 5 个 TORCH_LIBRARY_FRAGMENT(aten, m) { ... },它们的 init 函数名互不冲突。
边栏 3.3:为什么 aten 强制用 FRAGMENT 你看 PyTorch codegen 出来的 RegisterCPU.cpp / RegisterCUDA.cpp / RegisterSchema.cpp 等几十个文件,开头全是 TORCH_LIBRARY_FRAGMENT(aten, m),没有一个 TORCH_LIBRARY(aten, m)。这不是巧合 ——torchgen 在 torchgen/gen.py 里专门写了 special case:
# torchgen/gen.py 1761-1776 附近(简化)
if namespace == "aten":
    aten_schema_registrations = schema_registrations_body
else:
    torch_library_macro = (
        "TORCH_LIBRARY_FRAGMENT"
        if namespace in FRAGMENT_NAMESPACES
        else "TORCH_LIBRARY"
    )
原因很现实:aten 命名空间下有 ~2000 个 op,每个 codegen 文件贡献几十到几百个 schema 注册。一个文件几千行,塞进单个 TORCH_LIBRARY 块编译会爆炸(MSVC 单函数指令数有上限)。拆成几十个 fragment,每个独立编译,完美。
小例 3.3:同 namespace 跨文件的 fragment 你想给 myops 命名空间下分模块注册算子(算子族 A 在 a.cpp,算子族 B 在 b.cpp): 
// myops_a.cpp
TORCH_LIBRARY_FRAGMENT(myops, m) {
  m.def("op_a1(Tensor x) -> Tensor");
  m.def("op_a2(Tensor x) -> Tensor");
}

// myops_b.cpp
TORCH_LIBRARY_FRAGMENT(myops, m) {
  m.def("op_b1(Tensor x) -> Tensor");
}
两个文件并存,各自的 TorchLibraryInit 都在 main 之前跑,把对应 schema 添加到 myops 命名空间。

但注意:整个进程仍然只能有一个 TORCH_LIBRARY(myops, ...)(DEF)。FRAGMENT 不能"自己开新命名空间" —— 它假定 namespace 已经被某个 DEF 创立。如果只有 FRAGMENT 没有 DEF,会报 "namespace not found" 错。

对于 aten namespace,它的 DEF 其实是 PyTorch 内部初始化逻辑创建的(不是来自某个特定 .cpp);所以 codegen 的 fragment 都能直接续写。

3.4 TORCH_LIBRARY_IMPL:挂载 kernel

schema 是"声明",kernel 才是"实现"。TORCH_LIBRARY_IMPL 是给已存在的 op 挂某个 DispatchKey 上的 kernel 的宏(torch/library.h:1064-1084,简化):

// torch/library.h (简化)
#define TORCH_LIBRARY_IMPL(ns, k, m) _TORCH_LIBRARY_IMPL(ns, k, m, C10_UID)
#define _TORCH_LIBRARY_IMPL(ns, k, m, uid)                                  \
  static void C10_CONCATENATE(TORCH_LIBRARY_IMPL_init_##ns##_##k##_, uid)(  \
      torch::Library&);                                                    \
  static const torch::detail::TorchLibraryInit                              \
      C10_CONCATENATE(TORCH_LIBRARY_IMPL_static_init_##ns##_##k##_, uid)(   \
          torch::Library::IMPL,                                             \
          ...,                                                              \
          C10_STRINGIZE(ns),                                                \
          c10::make_optional(c10::DispatchKey::k),                          \
          ...);                                                             \
  void C10_CONCATENATE(...)(torch::Library& m)

和 FRAGMENT 比的差别:

  • Kind = IMPL。
  • 多了一个参数 c10::DispatchKey::k(k 是用户传的第二个宏参数,如 CPU、CUDA、AutogradCUDA)。块内的每一次 m.impl("op_name", kernel) 都会被附加到 op 在这个 DispatchKey 下的 dispatch 表项里。
小例 3.4:典型用法 
// myops_cpu.cpp — 给 myops::* 在 CPU key 上注册 kernel
TORCH_LIBRARY_IMPL(myops, CPU, m) {
  m.impl("custom_add",  &my_add_cpu);
  m.impl("custom_relu", &my_relu_cpu);
}

// myops_cuda.cu — 给 myops::* 在 CUDA key 上注册 kernel
TORCH_LIBRARY_IMPL(myops, CUDA, m) {
  m.impl("custom_add",  &my_add_cuda);
  m.impl("custom_relu", &my_relu_cuda);
}

// myops_autograd.cpp — 单独一个 autograd 包装,用来挂 grad_fn
TORCH_LIBRARY_IMPL(myops, Autograd, m) {       // Autograd 是 alias key,自动展开到 AutogradCPU/CUDA/...
  m.impl("custom_add", &my_add_autograd);
}
三个文件链接进来之后,dispatcher 表里的 myops::custom_add 行就有了三个 kernel:CPU、CUDA、Autograd*(自动展开到所有具体 autograd key)。一次调用根据输入张量在哪个设备、是否 require grad,自动找最高位 key 的 kernel 跳过去。
特殊形式 TORCH_LIBRARY_IMPL(_, KEY, m):第一个参数下划线表示"所有 namespace 都套用",通常配合 m.fallback(...) 给某个 DispatchKey 注册全局兜底 kernel。一个例子:
// aten/src/ATen/core/VariableFallbackKernel.cpp
TORCH_LIBRARY_IMPL(_, AutogradCUDA, m) {
  m.fallback(autogradNotImplementedFallback());
}
含义:对于任何命名空间下的算子,如果在 AutogradCUDA 这个 key 上没显式注册 kernel,就走这个 fallback。这个 fallback 干的事是"当作 CompositeImplicitAutograd 处理 —— 让里面的子 op 自己走 dispatcher" —— 这是 PyTorch "没明确写反向公式的算子默认也能自动求导"的实现机制。

3.5 m.def 与 m.impl 的语义差异

m.def 和 m.impl 是 Library 对象上最常用的两个方法。它们职责完全不同:

m.defm.impl
在哪用仅 TORCH_LIBRARY / TORCH_LIBRARY_FRAGMENT 块内仅 TORCH_LIBRARY_IMPL 块内
作用声明 schema:给一个新的 op-name 注册类型签名,把它"注册"到 dispatcher 表里给已声明的 op 挂 kernel:在当前块的 DispatchKey 下登记一个具体函数指针
关键参数schema 字符串("custom_add(Tensor self, Tensor other) -> Tensor")op 名("custom_add")+ 函数指针(&my_kernel)
schema 校验第一次 def 锁定签名;后续不能改kernel 签名必须和 schema 解析后的类型兼容(否则注册时 abort)
能否同时挂 kernel可以(m.def("schema", &kernel)),但这个 kernel 没有 DispatchKey,被认为是 "catch-all" —— 任何 key 都走它本身就是给特定 key 挂 kernel

三种常见 m.impl 写法:

// (a) raw function pointer —— 框架推断函数签名
m.impl("op_name", &cpu_kernel);

// (b) TORCH_FN 包装 —— 把签名固化成 compile-time 类型,启用更快的 unboxed 调用路径
m.impl("op_name", TORCH_FN(cpu_kernel));

// (c) 强制走 boxed(用于 fallback、interpreter 等不预知签名的场景)
m.impl("op_name", torch::CppFunction::makeFromBoxedFunction<&boxed_fn>());
小例 3.5:schema 类型不兼容 你声明 schema: 
m.def("custom_add(Tensor self, Tensor other) -> Tensor");
后来注册 kernel: 
at::Tensor wrong_kernel(const at::Tensor& self, int other) { ... }
m.impl("custom_add", TORCH_FN(wrong_kernel));   // 类型不匹配!
程序启动时(static initializer 执行时)会立即 abort,错误信息大概是: 
Tried to register a kernel for operator myops::custom_add with type
Tensor(const Tensor&, int) — but the schema declares type Tensor(Tensor, Tensor).
这是 PyTorch 安全设计的关键:schema 是单一可信源,所有 kernel 都要对齐它。

3.6 Boxed 与 unboxed 调用路径

1.5 节我们说过 dispatcher 查表得到一个 KernelFunction,然后调用它。KernelFunction 内部其实同时存了两份函数指针:一份 unboxed,一份 boxed。dispatcher 根据"调用方知道类型签名 vs 调用方不知道"自动选择走哪条路径。

UnboxedBoxed
函数签名原生 C++:Tensor (*)(const Tensor&, const Tensor&, Scalar)统一:void (*)(const OperatorHandle&, DispatchKeySet, Stack*)
参数传递模板直接展开,零拷贝、无类型擦除所有参数被打包成 IValue 压在 Stack(std::vector<IValue>)上,kernel 自己从 stack pop
性能每次调用约 5-10 ns多 ~50-100 ns(参数装箱/拆箱开销)
什么时候用用户代码、codegen 出来的 wrapper —— 调用现场知道签名fallback kernel、functorch interpreter、JIT runtime、profiler hook —— op-agnostic 的场景

TORCH_FN(fn) 这个宏的作用是用模板提取 fn 的字面类型签名(在编译期),从而让 KernelFunction 可以同时存下 unboxed 路径(直接调用)和 boxed 路径(自动生成的 IValue 包装版本)。如果你写 m.impl("name", &fn)(没 TORCH_FN),框架也能用 SFINAE 推断签名,但 TORCH_FN 写法更明确,签名错误能在编译期被发现。

边栏 3.6:m.impl_UNBOXED 的历史 PyTorch 早期(2019 前)有一个 m.impl_UNBOXED("name", &fn) 接口,语义是"这个 kernel 只能 boxed 调用 —— 不要尝试生成 unboxed 路径"。当时是因为有些 kernel 的签名无法被模板推断(比如带 c10::optional<Tensor> 早期某些版本)。

现在这个 API 已被删除。等价写法是 m.impl("name", torch::CppFunction::makeFromBoxedFunction<&fn>()) —— 显式说"用这个 boxed 函数包装,不生成 unboxed 路径"。你在 2020 年前的博客里看到 impl_UNBOXED,知道它对应现在的什么就行。

3.7 Fallback 与 fallthrough

除了 m.def 和 m.impl,Library 还有一个 m.fallback。它的语义是:给整个 DispatchKey 注册一个 op-agnostic 的 kernel,在该 key 下"没有单独 impl 的 op"都走它。

常见两种用法:

m.fallback(torch::CppFunction::makeFallthrough());
    // "我这层不做事,把我的 bit 从位图清掉再走下一层"
    // O(1) 实现:dispatcher 看到这是 fallthrough,直接在 fallthrough_mask 里把这个 key 永久清零

m.fallback(torch::CppFunction::makeFromBoxedFunction<&my_boxed_fallback>());
    // 真的写个 boxed kernel 兜底,比如 functorch 的 vmap interpreter

fallthrough 是 dispatcher 高效的关键。它不是"运行时跳过" —— 而是"注册期就告诉 dispatcher 这个 key 在我这里没意义,以后位图里这个位直接 mask 掉"。代价 = 0。

典型 fallthrough 用户:

  • BackendSelect:绝大多数 op 不需要它(只有 factory function 如 torch.randn 才用)。对所有其他 op,BackendSelect fallthrough。
  • Named:没开 named tensor 时全部 fallthrough。
  • Conjugate / Negative / ZeroTensor:没用对应惰性标记时全部 fallthrough。
  • Tracer:没在 JIT trace 模式时 fallthrough。

这就是为什么 dispatcher 表面上"支持几十个 functionality 层" —— 实际上 99% 的算子调用都只激活两三层(backend + 可能的 autograd),其他层全是 fallthrough,在位图算出来后位都不会被点亮。

设计哲学:dispatcher 的扩展性来自"你不付不要的代价"。注册时声明 fallthrough → 你这层永远不会拖慢调用。你真的需要某层时(开 autocast、开 vmap),只有该层会被激活,其他仍然 fallthrough。这跟"if/else 链每次调用都要走完"是质的不同。

3.8 完整自定义 op 走一遍

现在把 3.1-3.7 学到的零件拼起来,看一个完整的自定义算子。这是一个"自定义 relu",在 CPU 和 CUDA 上都有实现,带 autograd 支持。涉及四个 .cpp 文件:

// ============ myops_schema.cpp ============
// (1) 用 TORCH_LIBRARY 给 myops 命名空间开 DEF,声明 schema 一次
#include <torch/library.h>

TORCH_LIBRARY(myops, m) {
  m.def("custom_relu(Tensor self) -> Tensor");
}

这一文件提供两件事:① 命名空间 myops 被开通(DEF);② 算子 custom_relu 的 schema 进入 dispatcher 表(但还没有任何 kernel)。Python 端此时调 torch.ops.myops.custom_relu(x) 会报 "kernel not found"。

// ============ myops_cpu.cpp ============
// (2) CPU 实现 + 注册到 (myops, CPU)
#include <ATen/ATen.h>
#include <torch/library.h>

at::Tensor custom_relu_cpu(const at::Tensor& self) {
  return at::clamp_min(self, 0);   // 简单起见:借用现成 at::clamp_min 当实现
}

TORCH_LIBRARY_IMPL(myops, CPU, m) {
  m.impl("custom_relu", TORCH_FN(custom_relu_cpu));
}

这一文件做的事:给 myops::custom_relu 在 CPU key(= 二维 (Dense, CPUBit))下挂一个具体 kernel。TORCH_FN 让 dispatcher 知道 unboxed 签名,运行时快。注意实现是用 at::clamp_min —— 它本身是个 dispatched op,内部会再走一遍 dispatcher 调用 CPU 的 clamp_min。

// ============ myops_cuda.cu ============
// (3) CUDA 实现 + 注册到 (myops, CUDA)
#include <ATen/ATen.h>
#include <torch/library.h>

at::Tensor custom_relu_cuda(const at::Tensor& self) {
  return at::clamp_min(self, 0);   // 同上,clamp_min 自己有 CUDA kernel
}

TORCH_LIBRARY_IMPL(myops, CUDA, m) {
  m.impl("custom_relu", TORCH_FN(custom_relu_cuda));
}

对称地给 CUDA key 挂 kernel。现在 dispatcher 表里 myops::custom_relu 的两个 backend 项都有 kernel。

// ============ myops_autograd.cpp ============
// (4) Autograd 实现:接住 forward,建 grad_fn,前向数学透传到 backend
#include <torch/autograd.h>
#include <torch/library.h>

class CustomReluFn : public torch::autograd::Function<CustomReluFn> {
 public:
  static at::Tensor forward(torch::autograd::AutogradContext* ctx,
                            const at::Tensor& self) {
    // 进入 forward 时手动排除 "AutogradXxx + ADInplaceOrView" key —— 否则下面
    // dispatcher 调用会再次落回到本 Function,死循环。
    at::AutoDispatchBelowADInplaceOrView guard;
    // guard 生效期内, AutogradCUDA 已从 TLS 位图剔除,
    // 调 dispatcher 会落到 CUDA / CPU 之一(由 self.device() 决定)。
    static auto op = c10::Dispatcher::singleton()
        .findSchemaOrThrow("myops::custom_relu", "")
        .typed<at::Tensor(const at::Tensor&)>();
    auto result = op.call(self);
    ctx->save_for_backward({result});
    return result;
  }

  static torch::autograd::variable_list backward(
      torch::autograd::AutogradContext* ctx,
      torch::autograd::variable_list grad_outputs) {
    auto saved = ctx->get_saved_variables();
    auto& result = saved[0];
    // d/dx relu(x) = 1 if x > 0 else 0;但我们存的是 result,等价于 (result > 0)
    return { grad_outputs[0] * (result > 0).to(grad_outputs[0].dtype()) };
  }
};

at::Tensor custom_relu_autograd(const at::Tensor& self) {
  return CustomReluFn::apply(self);
}

TORCH_LIBRARY_IMPL(myops, Autograd, m) {   // Autograd 是 alias key,展开到所有 AutogradXxx
  m.impl("custom_relu", &custom_relu_autograd);
}

这是关键一文件,几个细节:

  • torch::autograd::Function:PyTorch 提供的 CRTP 模板基类,子类定义 forward 和 backward。apply() 是基类生成的入口,它会自动处理 grad_fn 建立、next_edges 收集、上下文管理。
  • at::AutoDispatchBelowADInplaceOrView guard;:RAII 对象,构造时把所有 autograd functionality 位 + ADInplaceOrView 位 push 进 TLS exclude。整个 forward 函数体内,dispatcher 不会再走 AutogradCUDA 这一层。
  • 用 c10::Dispatcher::singleton().findSchemaOrThrow 主动查 schema,然后 typed<...>().call(...) 跑。这是 PyTorch 推荐的"在 C++ 里调自定义 op"写法 —— 等价于 Python 端的 torch.ops.myops.custom_relu(self)。
  • 注册到 Autograd alias:不写具体 AutogradCPU/AutogradCUDA,而是用 alias Autograd —— 它会被自动展开到所有具体 autograd keys,一行覆盖全部 backend 的 autograd 情况。

四个文件链接到一起,Python 端使用:

>>> import torch, my_extension
>>> x = torch.randn(3, requires_grad=True, device='cuda')
>>> y = torch.ops.myops.custom_relu(x)
    # dispatcher 算位图 → 最高位 = AutogradCUDA → 跳 custom_relu_autograd
    # → CustomReluFn::apply → forward()
    # → AutoDispatchBelowADInplaceOrView guard 生效
    # → op.call(self) (myops::custom_relu) → 最高位现在 = CUDA → 跳 custom_relu_cuda
    # → at::clamp_min(self, 0) → 走 clamp_min 的 CUDA kernel → 返回
>>> y.sum().backward()
    # autograd engine 调 CustomReluFn::backward(...)
    # → 计算梯度 → 传给 next_edges
更省事的写法:CompositeImplicitAutograd:既然 custom_relu 的实现完全是 at::* op 的组合(at::clamp_min),且它的反向能完全靠 clamp_min 自己的反向来推 —— 你可以直接注册到 alias CompositeImplicitAutograd,一次注册同时覆盖所有 backend、所有 autograd: 
at::Tensor custom_relu_composite(const at::Tensor& self) {
  return at::clamp_min(self, 0);
}

TORCH_LIBRARY_IMPL(myops, CompositeImplicitAutograd, m) {
  m.impl("custom_relu", &custom_relu_composite);
}
就一个 .cpp 文件,前面三个 IMPL 块全部不需要 —— 因为里头的 at::clamp_min 自己会走一遍 dispatcher,获得 backend 路由 + autograd 双重便利。对于"用现成 op 拼出来"的算子,这是首选写法。只有当你的实现是真正"原生"的(直接写 CUDA kernel、不是 at::* 组合),才需要 3.8 节那种分四个文件的全套写法。

4. Python 注册接口:torch.library 与 register_fake

第 3 章讲的是 C++ 端 TORCH_LIBRARY 那套宏。 完全等价的能力在 Python 端通过 torch.library 暴露出来 —— 走的是同一个 c10::Dispatcher 注册表,没有"Python op vs C++ op"之分,只是入口语言不同。 这一章梳理它对 C++ 端宏的一一对应,以及 Python 端独有的 register_fake 接口(meta 推断,Dynamo / export / Inductor / AOTI 的必备件)。

4.1 两套 Python API:Library 与 custom_op

4.4 Meta 这个 dispatcher:register_fake 和 torch.export

Meta 是 dispatcher 注册表里一个特殊的 backend key:跟 CPU / CUDA 平级,但表示"tensor 只有 metadata(shape / stride / dtype / device),没有 storage 也不实际算"。 这一节先把 Meta key 在 dispatcher 表里的位置说清楚,然后讲 register_fake 怎么挂上去,最后重点讲它在 torch.export 里的角色 —— export 的整条路径几乎都靠它撑着。

4.4.1 Meta key 在 dispatcher 表里的位置

从 §1.3 那张 DispatchKey 表往下看,Meta 跟 CPU / CUDA / XPU 是同一行的 backend key,值大约是 DispatchKey::Meta。 一个 op 在 dispatcher 表里同时有 CPU / CUDA / Meta 三种实现是正常的 —— meta tensor 路过 dispatcher 时,DispatchKeySet 算出来的 backend bit 就是 Meta,routing 到这条 entry。

aten::add.Tensor 在 OperatorEntry 里的 kernel 表(节选):

  DispatchKey::CPU             -> at::native::add_kernel (CPU vectorized)
  DispatchKey::CUDA            -> at::native::add_kernel_cuda
  DispatchKey::Meta            -> at::native::add_meta       # shape inference only
  DispatchKey::AutogradCPU     -> AutogradAdd::apply
  DispatchKey::AutogradCUDA    -> AutogradAdd::apply
  ...

对原生算子(aten::*),Meta kernel 大部分是 codegen 自动生成的(torchgen,见 §5),逻辑就是"调一次 TensorIterator 算出 broadcasting 后的 shape,返回一个 Meta tensor"。 自定义算子没有这条 codegen 路径,所以需要用户手写 —— 这就是 register_fake 的入口。

4.4.2 FakeTensor:Meta key 之上的薄层

实际上 export / Inductor / AOTI 用的不是裸 Meta tensor,而是 FakeTensor(torch._subclasses.fake_tensor.FakeTensor),它是 Meta tensor 的 __torch_dispatch__ 子类。 二者的关系:

裸 meta tensorFakeTensor
设备device='meta'保留 原始 device(cuda:0 / cpu / etc.),storage 仍然空
storage空空
dispatcher 行为走 Meta key外层走 Python key(__torch_dispatch__ 拦截),内部 fall through 到 Meta key
能不能保留 device 上下文不能能(这是 FakeTensor 存在的主要原因)
谁创建torch.empty(..., device='meta')FakeTensorMode 上下文里创建

关键点:FakeTensor 走的 dispatcher 路径里,真正算 shape 的还是 Meta kernel。 FakeTensor 的 __torch_dispatch__ 把 input tensors "降"到 meta(保留 device 信息),调用 op,得到 meta 结果,再"升"回 FakeTensor(把 device 信息塞回去)。 所以一个 op 只要有 Meta kernel,就自动能被 FakeTensor 处理 —— 反之,register_fake 注册的就是 Meta kernel。

4.4.3 register_fake 到底注册到哪

register_fake(fn) 在 dispatcher 表里挂的是一个 Meta 条目:

@scale_add.register_fake
def _(x, scale, bias):
    return torch.empty_like(x)

# 内部等价于:
torch.library.impl("myops::scale_add", "Meta")(_fn)
# 进一步等价于 C++ 端:
// TORCH_LIBRARY_IMPL(myops, Meta, m) { m.impl("scale_add", &scale_add_meta); }

名字叫 register_fake 不叫 register_meta 的原因是历史:更早的 API 就叫 impl_abstract / register_meta,2.4 改成现在的 custom_op 系列时统一改名 register_fake —— 强调它注册的是"FakeTensor 用的 shape inference 函数",更贴近用户视角。 dispatcher 表里那一格仍然是 Meta key,改的只是 Python API 名字。

4.4.4 在 torch.export 里的角色 —— 整条路径都靠 Meta

torch.export(model, example_inputs) 内部不跑真实数值,跑的是一遍 FakeTensor。 整条路径上 Meta key 的位置:

torch.export(model, example_inputs)
   │
   ├─ 1. 用 example_inputs 创建对应的 FakeTensor(保留 shape / dtype / device)
   │
   ├─ 2. 进入 FakeTensorMode 上下文
   │
   ├─ 3. Dynamo 用 symbolic_shapes 给可变维度装 SymInt
   │     (e.g. batch 维 -> s0,而不是具体 8)
   │
   ├─ 4. AOT autograd trace:把 model 每个 op 用 FakeTensor 跑一遍
   │     ├─ 每个 op 进 dispatcher
   │     ├─ FakeTensor 的 __torch_dispatch__ 把 inputs 降到 meta(带 device)
   │     ├─ dispatcher 算 DispatchKeySet:meta tensor -> Meta key
   │     ├─ 调对应的 Meta kernel —— aten::* 是 codegen,custom op 是 register_fake
   │     ├─ Meta kernel 返回 meta tensor(只有 shape/dtype)
   │     └─ FakeTensor wrap 回去
   │
   ├─ 5. 收集所有 op 的 Meta kernel 输出,构成 ExportedProgram.graph
   │     (FX graph,每个 node 都带 val:FakeTensor 形式的 shape annotation)
   │
   └─ 6. graph 序列化进 .pt2 / 进 Inductor / 进 AOTI

从这条路径能反推几个事实:

  • export 从头到尾没有调过真实 kernel。 模型权重的数据从来没进过 GPU(它们也是 FakeTensor)。 这就是 export 比 trace 快几个数量级的原因。
  • shape 推断的正确性 100% 由 Meta kernel 决定。 一个 op 的 Meta kernel 算错 shape,export 出来的 graph 后面整条都是错的 —— Inductor 按错 shape 分 tile、AOTI 按错 shape 分 cubin、运行时一堆 mismatched dim。
  • SymInt 是 Meta kernel 必须能处理的输入类型。 export 把 batch 维标成 s0(SymInt),Meta kernel 收到的就是 SymInt;如果实现里 if x.size(0) == 8: 这种 concrete branch,在 SymInt 上抛 GuardOnDataDependentSymNode。
  • custom op 没注册 Meta kernel,export 直接挂。 dispatcher 走到 Meta key 找不到 entry,根据 fallback 规则要么报 "NotImplementedError: ... operator does not have a meta kernel"(2.4+),要么实际去调真实 kernel(老版本,这会让 export 把数值算一遍 —— 慢、且没法处理动态 shape)。

4.4.5 register_fake 的硬性约束(写错就 export 失败)

  • 不能调 aten::* 真实算子(x.mul(2) / torch.add(x, y)):这些走 dispatcher,在 Meta key 上会回头找同一个 Meta kernel,要么递归,要么走到错误分支。 用 Python 算子(x * 2)或直接造 tensor(torch.empty_like(x) / torch.empty(shape, dtype=, device=))。
  • shape 推断必须能在 SymInt 上跑:
    • 不能用 if x.size(0) == 8: 之类 concrete check
    • shape 运算用 torch.sym_int / x.shape[0] + 1 这种 SymInt 友好的形式
    • 如果某条逻辑 unavoidable 依赖具体数值,用 torch._check(...) 给 Dynamo 加 guard
  • 不能有副作用:写文件、改全局状态、print 都不行 —— export 会反复 trace,副作用会重复触发。
  • device 必须跟输入一致:输出 tensor 的 device 要从 input 推(device=x.device),硬编码成 cuda:0 会让 export 后 graph 没法迁移设备。
  • dtype 要算清楚:type promotion 跟真实 kernel 一致(torch.promote_types),不能默认返回 x.dtype 完事 —— 否则 export 出的 graph 跑起来 dtype mismatch。

4.4.6 几个常见错误案例

# ❌ 案例 1:在 fake 里调真实 op
@my_op.register_fake
def _(x):
    return x.mul(2)              # dispatch 回 Meta key,要么递归要么走错分支

# ✓ 改:
def _(x):
    return torch.empty_like(x)

# ❌ 案例 2:concrete check 撞 SymInt
@my_op.register_fake
def _(x):
    if x.size(0) % 8 == 0:    # SymInt % 8 -> SymBool,if 强制 guard,export warning/error
        ...

# ✓ 改:
def _(x):
    torch._check(x.size(0) % 8 == 0)
    return torch.empty_like(x)

# ❌ 案例 3:device 硬编码
@my_op.register_fake
def _(x):
    return torch.empty(x.shape, device="cuda")  # 永远在 cuda,导出的 graph 没法 cpu 跑

# ✓ 改:
def _(x):
    return torch.empty(x.shape, dtype=x.dtype, device=x.device)

# ❌ 案例 4:输出多个 tensor 时 shape 算错
@split.register_fake
def _(x, n):
    return [torch.empty_like(x) for _ in range(n)]   # 每片应该是 x.shape[0] // n

# ✓ 改:
def _(x, n):
    chunk = x.size(0) // n
    return [torch.empty((chunk,) + x.shape[1:], dtype=x.dtype, device=x.device)
            for _ in range(n)]

4.4.7 export 链路上 Meta 之外的入口

除了 register_fake,跟 Meta 推断相关的还有几个 hook 点,容易混:

入口挂的位置用途
register_fake / impl("Meta") dispatcher Meta key shape inference,export 必备
op.register_torch_dispatch(...) dispatcher Python key 定制 FakeTensor 行为(很少用)
FakeTensor._dispatch_impls 表 FakeTensor Python 层 少数 op 在 FakeTensor 层有特化(aten::nonzero 之类 data-dependent op)
Inductor lowering(torch._inductor.lowering) Inductor 把 fx node 翻译成 Inductor IR;register_fake 出错的 op 这里也跟着坏

正常 custom op 只需要写 register_fake 这一个 —— 其余几个是 PyTorch 内部 / 极端 case 才用。

4.5 Python → Dispatcher:实际穿透到 C++ 的路径

Python 端这些 API 不是另一套 dispatcher,而是 C++ 端 c10::Dispatcher 注册表的 Python wrapper。 调用栈一层层捏穿:

torch.library.Library("myops", "DEF")
   ↓ # torch/library.py
torch._C._dispatch_library("DEF", "myops", ...)
   ↓ # torch/csrc/utils/python_dispatch.cpp,在 libtorch_python.so 里
torch::Library lib(...)              // C++ Library 对象
   ↓
c10::Dispatcher::singleton()
       .registerName(c10::OperatorName{"myops", ""})    // 注册 namespace

lib.define("scale_add(Tensor x, float scale, Tensor bias) -> Tensor")
   ↓
torch._C._dispatch_def(lib_handle, schema_str)
   ↓
torch::jit::parseSchema(schema_str)             // 解析成 FunctionSchema
   ↓
c10::Dispatcher::singleton().registerDef(schema, ...)

op.register_kernel("cuda")(fn_py)
   ↓
torch._C._dispatch_impl(lib_handle, "scale_add", "CUDA", boxed_python_fn)
   ↓
c10::Dispatcher::singleton().registerImpl(
   OperatorName{"myops", "scale_add"},
   c10::DispatchKey::CUDA,
   KernelFunction::makeFromBoxedFunction(...));  // fn_py 通过 PythonFallbackKernel 间接调到

关键点:

  • Python 注册的 fn 并非直接挂到 dispatch table。 它先包成 PyObject 持有者,然后由 PythonFallbackKernel 在调用时反向跳进 Python 解释器执行。 这就是 [[pytorch-torch-dispatch-mode]] 那篇里讲的 C++ ↔ Python 双向调度机制。
  • 所以 Python 注册的 op 调用比纯 C++ 注册的多两次 dispatcher 转换 + 一次 GIL 获取,有性能损失。 inference 路径建议性能敏感时把 op 用 C++ 重写。

4.6 完整 Python 例子:对偶 §3.8

§3.8 那个 myops::custom_relu 用 C++ 写了四个文件;Python 端等价的一份 .py 文件:

import torch

## 1) Schema + 默认实现(custom_op 一次性给齐)
@torch.library.custom_op("myops::custom_relu", mutates_args=())
def custom_relu(x: torch.Tensor) -> torch.Tensor:
    # 默认实现:跑 CPU + CUDA 通用代码
    return x.clamp(min=0)

## 2) 为 CUDA 注册更优的实现(替换默认)
@custom_relu.register_kernel("cuda")
def _(x: torch.Tensor) -> torch.Tensor:
    # 这里可以调 Triton kernel 或者 cuBLAS / cuDNN 包装
    return x.relu()                          # 示意

## 3) register_fake:让 export / AOTI 推 shape
@custom_relu.register_fake
def _(x):
    return torch.empty_like(x)

## 4) register_autograd:给反向(可选,如训练需要)
def _setup_ctx(ctx, inputs, output):
    (x,) = inputs
    ctx.save_for_backward(x)

def _backward(ctx, grad):
    (x,) = ctx.saved_tensors
    return grad * (x > 0).to(grad.dtype),    # tuple,跟 inputs 一一对应

custom_relu.register_autograd(_backward, setup_context=_setup_ctx)

## 用法,跟 C++ 注册版完全一样:
y = torch.ops.myops.custom_relu(torch.randn(3, device="cuda", requires_grad=True))
y.sum().backward()

对比 §3.8 那四个 C++ 文件,Python 版的好处是:

  • 开发周期短:不用 nvcc / setup.py / 编译;改完就跑
  • 跟 export / FakeTensor 对齐:custom_op 提供稳定 schema,register_fake 负责抽象执行时的 shape / dtype / device 推断

代价:

  • 调用开销大(Python → dispatcher → PythonFallbackKernel → Python fn)
  • 对 torch.compile / torch.export 来说默认是 opaque op:compiler 不会自动偷看 Python 函数体,也不会因为 fake impl 自动生成 Inductor lowering
  • 不能脱离 Python runtime(AOTI 部署时要再加一层 C++ shim,见 [[aoti-cuda-deployment]] §9.4)

4.7 torch.library.opcheck:自动验证一致性

注册完一个 custom op 之后,跑一遍 opcheck 是非常推荐的:

torch.library.opcheck(
    custom_relu,
    args=(torch.randn(3, 4, device="cuda"),),
    test_utils=("test_schema", "test_faketensor", "test_aot_dispatch_static", "test_aot_dispatch_dynamic"),
)

这一个调用会跑一套不变量测试:

  • test_schema:schema 字符串跟实现函数的签名匹配(参数名、类型、可选性)
  • test_faketensor:fake impl 产出的 FakeTensor 跟真实 impl 产出的 Tensor 在 shape / dtype / device 上一致
  • test_aot_dispatch_static:走 AOT autograd 静态 trace 一遍,确认没问题
  • test_aot_dispatch_dynamic:同上但 dynamic shape
  • (可选)test_autograd_registration / gradcheck:验证 backward 的数值正确性

过不了 opcheck 的 custom op 在 export / Inductor / AOTI 阶段会出各种诡异错误,提前用它筛一遍能省大量调试时间。

5. 构建期自动化:torchgen

第 3 章我们看到自定义算子要写四个 .cpp:一个 schema、一个 CPU kernel、一个 CUDA kernel、一个 autograd wrapper。对 ATen 命名空间的 ~2000 个 in-tree 算子,如果都这么手写,会有几万行重复的胶水代码。

PyTorch 的解决方案是 构建期 codegen:你只需要在两个 yaml 文件里写"声明"(schema + dispatch 表 + 反向公式),构建系统跑一个 Python 程序 torchgen/gen.py,它会自动生成几十种 C++ / 头文件产物,所有 TORCH_LIBRARY_FRAGMENT 和 TORCH_LIBRARY_IMPL 注册都被它写好。本章看 torchgen 吃什么、吐什么。

5.1 输入:三份 yaml

torchgen 的主程序 torchgen/gen.py 在 PyTorch 编译期作为 python setup.py build 的子步骤跑,吃三份 yaml:

6. 端到端追踪:aten::add on CUDA

前四章我们建立了 dispatcher 的概念模型(§1)、schema 的接口契约(§2)、注册宏(§3)、构建期 codegen(§4)。本章把这些拼起来,选一个具体场景把整条调用链走完:

设定:两个 CUDA 张量 x, y,都 requires_grad=True。用户写: 
z = torch.add(x, y)
z.sum().backward()
我们要回答:这两行 Python 代码,内部到底跑过哪些 C++ / CUDA 函数?

调用链从 Python 进入,沿 dispatcher 优先级阶梯一层层下落,落到 CUDA elementwise kernel;反向时再回到 autograd engine 调 AddBackward0::apply。下面九小节按顺序拆开,每节锁定一层,讲清"这一层在做什么、对应哪个生成产物、哪段代码、注意事项是什么"。

6.1 起点:yaml 中的两行声明 + 反向公式

整条调用链的源头就是两份 yaml 里的几行声明。它们决定了后续生成产物长什么样。我们把这几行抄一遍,后面每一节都能回头对照。

native_functions.yaml 里的 add.Tensor(functional 变体)和 add.out(structured 锚点)条目(简化):

# native_functions.yaml
- func: add.Tensor(Tensor self, Tensor other, *, Scalar alpha=1) -> Tensor
  device_check: NoCheck
  structured_delegate: add.out      # 自己不实现,委托给 add.out (5.6 节会用到)
  variants: function, method        # 生成 at::add 和 Tensor::add 两种入口
  ...

- func: add.out(Tensor self, Tensor other, *, Scalar alpha=1, Tensor(a!) out) -> Tensor(a!)
  structured: True                  # 这是 structured kernel 的"出口"(5.7 节会用到)
  structured_inherits: TensorIteratorBase
  ufunc_inner_loop:                 # 触发 ufunc codegen(5.7 节会用到)
    Generic: add (AllAndComplex, BFloat16, Half, ComplexHalf)
    ScalarOnly: add (Bool)

derivatives.yaml 里 add.Tensor 的反向公式(5.5 和 5.8 节会用到):

# tools/autograd/derivatives.yaml
- name: add.Tensor(Tensor self, Tensor other, *, Scalar alpha=1) -> Tensor
  self:  handle_r_to_c(self.scalar_type(),  grad)
  other: handle_r_to_c(other.scalar_type(), maybe_multiply(grad, alpha.conj()))

就这么几行 yaml,加上 aten/src/ATen/native/ufunc/add.h 里一行真正的数学定义 —— 就构成了 aten::add 的全部"原始资料"。后面所有 C++ 函数都是 torchgen 根据这些行生成的。

6.2 Python → C 绑定层

用户写 torch.add(x, y)。CPython 解释器查到 torch.add 是一个 C 函数指针,跳到 THPVariable_add(注意 "THP" 前缀:Torch HPython,PyTorch 给所有 Python C 绑定的命名约定)。它的源码生成在:

build/.../torch/csrc/autograd/generated/python_torch_functions_2.cpp

_2 是分片号(整个文件太大,torchgen 把它拆成几片)。THPVariable_add 体大概干这四件事:

  1. 解析 Python 参数。用 PythonArgParser 按 schema(2.1 节那个字符串)解析,识别出第一个张量是 self、第二个是 other、可选 kw alpha 默认 1。
  2. 拆 Python 包装。用户的 x 实际是 Python 端的 THPVariable 对象(torch.Tensor),内部持有一个 at::Tensor(C++ 张量)。THPVariable_unpack 把 C++ 张量取出来。
  3. 调 C++ API。有了纯 C++ 张量,调 at::add(self, other, alpha)。控制权离开 Python,进入纯 C++ 世界。
  4. 把返回值包回 Python。C++ 返回的 at::Tensor 被 THPVariable_Wrap 包成新的 torch.Tensor 返还给 Python。
边栏 5.2:为什么需要这一层"包装/拆包" 你可能会问:既然 torch.Tensor 和 at::Tensor 本质是同一个对象,为什么不直接传?

两个原因:
① Python 引用计数 vs C++ 智能指针。Python 的 torch.Tensor 用 CPython 引用计数(Py_INCREF/Py_DECREF),C++ 的 at::Tensor 用 intrusive_ptr。两套生命周期管理必须在边界精确转换,否则会 use-after-free。
② 解析 Python 多态参数。Python 用户可以传 (Tensor, Tensor) 也可以传 (Tensor, Scalar) 给同一个 torch.add —— 选择哪个 overload(add.Tensor vs add.Scalar)是这一层决定的。C++ 函数本身只接受静态类型签名,需要 Python 端先做 overload resolution。

6.3 C++ API 与 Dispatcher::call

进入 C++ 世界,第一个落点是 at::add(生成在 build/aten/src/ATen/Functions.h):

// build/aten/src/ATen/Functions.h
inline at::Tensor add(const at::Tensor& self, const at::Tensor& other, const at::Scalar& alpha) {
  return at::_ops::add_Tensor::call(self, other, alpha);
}

这只是个 inline 转发,把调用甩给 at::_ops::add_Tensor::call(在 Operators.cpp ~第 612 行):

// build/aten/src/ATen/Operators.cpp
at::Tensor _ops::add_Tensor::call(const at::Tensor& self, const at::Tensor& other, const at::Scalar& alpha) {
  static auto op = create_add_Tensor_typed_handle();   // 进程内只算一次的 TypedOperatorHandle
  return op.call(self, other, alpha);                  // 转给 Dispatcher::call
}

这一层做的是"找到 op 的 TypedOperatorHandle"。TypedOperatorHandle 是 dispatcher 内部的"句柄" —— 它指向 aten::add.Tensor 这个 op 在 dispatcher 表里的那一行。static 保证这个查表只在第一次调用时跑一次,之后都是缓存。

op.call 最终调到 c10::Dispatcher::call(Dispatcher.h:774-828)—— 这就是 1.4 节里讲的 dispatcher 入口。从这一刻起,我们正式踏入 dispatcher 的位图魔法。

小例 5.3:这一层为什么需要单独存在 你可能会问:既然 at::add 只是个转发,为什么不直接让 THPVariable_add 调 Dispatcher::call?

两个原因:
① 它给了用户友好的 C++ API。第三方 C++ 代码不想知道 dispatcher 内部细节,只想写 at::add(x, y)。
② 它隐藏了 schema lookup 的缓存。at::_ops::add_Tensor::call 在第一次调用时找 schema(create_add_Tensor_typed_handle()),之后缓存。如果每次都 Dispatcher::call 都要先查 schema,性能会差很多。

6.4 DispatchKeySet 的现场计算

进入 Dispatcher::call(Dispatcher.h:774-828),核心两行:

// Dispatcher::call 核心
auto dispatchKeySet =
    op.operatorDef_->op.dispatchKeyExtractor()
        .template getDispatchKeySetUnboxed<Args...>(args...);
const KernelFunction& kernel = op.operatorDef_->op.lookup(dispatchKeySet);
return kernel.template call<Return, Args...>(op, dispatchKeySet, ...);

第一行算位图(1.5 节那个 DispatchKeySet),第二行查表取最高位的 kernel,第三行跳过去。对我们这次调用,具体发生:

  1. 位图初始值:每个输入张量 self、other 的 TensorImpl::key_set_。它们都是 CUDA 张量、都 require grad,所以 key_set_ = {AutogradFunctionality, Dense, CUDABit}(用 1.4 的二维拆分写法)。两个张量 or 起来,位图还是这三个位为 1。运行时翻译就是 {AutogradCUDA, CUDA}。
  2. TLS local include:没开 autocast、没在 functorch 里,所以 include 为空。
  3. TLS local exclude:第一次进 dispatcher,exclude 也是空。
  4. Fallthrough mask:每个 op 的 dispatch table 在注册期预算了 fallthrough_mask。对 add,BackendSelect、Named、Conjugate 等位都被 mask 清掉。位图维持 {AutogradCUDA, CUDA}。
  5. highestPriorityTypeId():在剩下的位里找最高 functionality 位 —— 是 AutogradFunctionality(对应运行时 AutogradCUDA),因为它在 enum 里排在 Dense 后面(更高位)。

lookup 返回 aten::add.Tensor 在 AutogradCUDA 槽位下的 kernel 函数指针。这个 kernel 是谁注册的?codegen 自动注册的:

// build/.../VariableType_2.cpp(分片号会变)
TORCH_LIBRARY_IMPL(aten, AutogradCUDA, m) {
  ...
  m.impl("add.Tensor", TORCH_FN(VariableType::add_Tensor));
  ...
}

跳过去,进入 VariableType::add_Tensor。

6.5 AutogradCUDA kernel:建图与 redispatch

VariableType::add_Tensor 的函数体是 codegen 出来的(根据 derivatives.yaml 里 add.Tensor 那条反向公式 + 标准模板)。简化骨架:

at::Tensor VariableType::add_Tensor(
    c10::DispatchKeySet ks,
    const at::Tensor& self, const at::Tensor& other, const at::Scalar& alpha) {

  // === 步骤 1)若任何输入 requires_grad,准备一个新的 grad_fn 反向节点 ===
  std::shared_ptr<AddBackward0> grad_fn;
  if (compute_requires_grad(self, other)) {
    grad_fn = std::shared_ptr<AddBackward0>(new AddBackward0(), deleteNode);
    grad_fn->set_next_edges(collect_next_edges(self, other));   // 反向边
    grad_fn->alpha = alpha;                                      // 存反向公式要用的
    grad_fn->other_scalar_type = other.scalar_type();
    grad_fn->self_scalar_type  = self.scalar_type();
  }

  // === 步骤 2)把"AutogradCUDA + ADInplaceOrView"从位图剔掉,redispatch ===
  auto _tmp = ([&]() {
    at::AutoDispatchBelowADInplaceOrView guard;     // RAII:进 guard = TLS exclude 加 autograd + ADInplaceOrView
    return at::redispatch::add(
        ks & c10::after_autograd_keyset,             // 显式 mask 当前层及以上
        self, other, alpha);
  })();
  auto result = std::move(_tmp);

  // === 步骤 3)把新算出的 result 挂到 grad_fn,加进 autograd 历史 ===
  if (grad_fn) {
    set_history(flatten_tensor_args(result), grad_fn);   // result.grad_fn = grad_fn
  }
  return result;
}

逐行解读:

  1. compute_requires_grad(self, other):看任何一个输入张量是不是 require grad、且不在 torch.no_grad() 上下文里。我们的场景两个输入都 require grad → true。
  2. 建 AddBackward0 节点:这是 codegen 生成的 Node 子类(5.8 节看它的 apply())。set_next_edges 把 self.grad_fn 和 other.grad_fn 设为 AddBackward0 的下游边 —— 反向时梯度按这两条边传出去。alpha、self_scalar_type、other_scalar_type 是反向公式里要用的"上下文",保存到节点里。
  3. at::AutoDispatchBelowADInplaceOrView guard;:RAII 对象,构造时修改 TLS —— 把所有 autograd functionality 位 + ADInplaceOrView 位 push 进 exclude 集合。从这一行到 guard 析构,所有 dispatcher 调用都不会再走 autograd 层。
  4. at::redispatch::add(ks & c10::after_autograd_keyset, ...):at::redispatch::add 是另一份 codegen 生成的入口(在 RedispatchFunctions.h),它和 at::add 区别只在于多带一个 DispatchKeySet 参数透传给 Dispatcher::redispatch。after_autograd_keyset 是预先编译好的常量掩码,作用是"清掉所有 autograd 及以上的位"。这等于显式告诉 dispatcher:接下来跳 backend。
  5. set_history(result, grad_fn):把刚算出的 result 张量的 .grad_fn 字段设为我们建的 AddBackward0 节点。这样 result.backward() 时,autograd engine 找得到这个节点。
这一节最关键的概念:autograd kernel 不做任何数值计算。它只做三件事:① 准备反向节点;② 把控制权交给 backend kernel(借助 guard + redispatch);③ 把结果挂到反向图。所有数值算术由下游的 CUDA kernel 完成。这种"包装层"模式让 autograd 完全和具体 backend 解耦 —— 同一个 autograd wrapper 适用于 CPU、CUDA、MPS 等所有 backend。

6.6 CUDA 后端 wrapper

at::redispatch::add 把控制权交给 Dispatcher::redispatch。这次位图里 AutogradCUDA 已经清掉(被 guard 和 after_autograd_keyset 双重保险),highestPriorityTypeId() 返回 CUDA。

CUDA 槽位的 kernel 注册在 build/.../RegisterCUDA.cpp:

TORCH_LIBRARY_IMPL(aten, CUDA, m) {
  ...
  m.impl("add.Tensor", TORCH_FN(wrapper_CUDA_add_Tensor));
  m.impl("add.out",    TORCH_FN(wrapper_CUDA_add_out_out));
  ...
}

跳到 wrapper_CUDA_add_Tensor(注意:还不是真正算数值的 CUDA kernel,只是 functional → out 的转发,因为 yaml 里 structured_delegate: add.out):

at::Tensor wrapper_CUDA_add_Tensor(
    const at::Tensor& self, const at::Tensor& other, const at::Scalar& alpha) {
  // 1) 分配一个新的输出张量(在 CUDA 上,common dtype 自动 promote)
  auto common_dtype = at::native::result_type(self, other);
  auto out = at::detail::empty_cuda({0}, common_dtype, self.device());
  // 2) 转给 out 变体真正算
  return wrapper_CUDA_add_out_out(self, other, alpha, out);
}

所以 add.Tensor(functional)的实现 = "分配 out + 调 add.out"。这是 structured_delegate: add.out 在 codegen 端的具体落地 —— 三种变体共享同一份数学,只有内存分配策略不同。

小例 5.6:对比 in-place 变体的实现 对 add_.Tensor(in-place),codegen 会生成类似的 wrapper,但不分配新 out,而是把 self 当成 out 传给 add.out: 
at::Tensor& wrapper_CUDA_add__Tensor(
    at::Tensor& self, const at::Tensor& other, const at::Scalar& alpha) {
  return wrapper_CUDA_add_out_out(self, other, alpha, self);   // self 当 out
}
三种变体三套 wrapper,但真正算数值的只有 add.out 这一份 impl。这是 structured kernel 的核心收益。

6.7 Structured kernel 与 ufunc

wrapper_CUDA_add_out_out 是真正"开始算"的地方。它的体也是 codegen 生成的:

at::Tensor& wrapper_CUDA_add_out_out(
    const at::Tensor& self, const at::Tensor& other, const at::Scalar& alpha,
    at::Tensor& out) {
  structured_add_out_out op(out);            // 实例化 structured kernel 类
  op.meta(self, other, alpha);               // 阶段 1: meta —— 算输出 shape/dtype,设 TensorIterator
  op.impl(self, other, alpha, op.maybe_get_output(0));   // 阶段 2: impl —— 真的算
  return out;
}

structured kernel 把"算法"拆成两个标准化阶段:

  1. meta():广播 shape、确定 dtype、初始化 TensorIterator。调用方提供输出 tensor(可能是空的或已分配的),meta 根据输入决定输出该是什么形状,必要时 resize。
  2. impl():数值计算。读 TensorIterator 拿到逐元素迭代器,跑实际的算法。

structured_add_out_out::impl 的体由 UfuncCUDA_add.cu 生成(这就是 4.2 节"structured + ufunc"产物):

// build/.../UfuncCUDA_add.cu (简化)
void structured_add_out_out::impl(
    const Tensor& self, const Tensor& other, const Scalar& alpha, const Tensor& out) {
  AT_DISPATCH_ALL_TYPES_AND_COMPLEX_AND4(
      kBFloat16, kHalf, kComplexHalf, kBool,                      // 支持的 dtype 集合
      iter.common_dtype(), "add_cuda",                            // 当前 dtype + 标签
      [&] {                                                      // per-dtype 实例化的 lambda
        scalar_t alpha_val = alpha.to<scalar_t>();
        gpu_kernel(iter, [alpha_val] GPU_LAMBDA (scalar_t a, scalar_t b) -> scalar_t {
          return at::native::ufunc::add(a, b, alpha_val);         // 真的数学
        });
      });
}

这里有两层 dispatch:

  1. AT_DISPATCH_ALL_TYPES_AND_COMPLEX_AND4:这是编译期 dispatch(aten/src/ATen/Dispatch.h),按 iter.common_dtype() 展开成 switch (scalar_type),对每个 dtype(float、double、int8、kHalf 等)实例化 lambda 一次。这一层不进 dispatcher 表 —— dtype 不是 DispatchKey,它只是 C++ 模板参数。
  2. gpu_kernel(iter, lambda):这是启动 CUDA kernel(aten/src/ATen/native/cuda/CUDALoops.cuh)。它接 TensorIterator(描述逐元素遍历计划)+ 用户 lambda,展开成 vectorized_elementwise_kernel<...><<<blocks, threads>>>(...)。每个 CUDA 线程跑用户 lambda 一次。

那"用户 lambda"做什么?调 at::native::ufunc::add。这是真正的数学函数,定义在 ATen/native/ufunc/add.h,只有四行:

// aten/src/ATen/native/ufunc/add.h
template <typename T>
C10_HOST_DEVICE C10_ALWAYS_INLINE T add(T self, T other, T alpha) {
  return self + alpha * other;
}
沿调用栈所有这么多代码层,真正的"算 add"就是这一行 self + alpha * other。其他所有层都是为了"让这一行能在正确的 dtype、正确的 backend、记好反向、可叠加 autocast/vmap 等模式下被正确调用"。这就是 PyTorch 工程量的本质。
小例 5.7:为什么 CPU 和 CUDA 共享同一个 ufunc::add ufunc/add.h 用 C10_HOST_DEVICE 注解,这是 PyTorch 包装 CUDA __host__ __device__ 的宏,告诉 nvcc:"这个函数同时编译成 CPU 版和 GPU 版"。

CPU 端的 UfuncCPU_add.cpp 也 include 这同一个 ufunc/add.h,然后用 SIMD 矢量化的 vectorized_loop2d 而不是 gpu_kernel 来调它。数学定义只写一次,CPU/CUDA 用同一份。 加一个新后端(比如 MPS)只需要写新的"循环驱动"(怎么跑 lambda),数学函数不动。

到这里,前向计算完成。控制权层层返回:CUDA kernel 返回 → impl 返回 → wrapper_CUDA_add_out_out 返回 out → wrapper_CUDA_add_Tensor 返回 → Dispatcher::redispatch 返回 → at::redispatch::add 返回到 VariableType::add_Tensor。然后:

  1. guard 析构,TLS exclude 还原(后续顶层调用又干净了)。
  2. set_history(result, grad_fn) 把反向节点挂到结果。
  3. 返回到顶层 Dispatcher::call,再返回到 at::_ops::add_Tensor::call,再返回到 at::add,再返回到 THPVariable_add,再包装成 Python torch.Tensor 返回给用户。

前向结束。z 拿到手,它的 .grad_fn 是 AddBackward0。

6.8 反向:AddBackward0::apply

用户接着调 z.sum().backward()。backward() 触发 autograd engine(torch/csrc/autograd/engine.cpp):

  1. 从 loss(z.sum() 那个标量)的 .grad_fn 开始,拓扑排序整张反向图。
  2. 逆拓扑序遍历,对每个 Node 调它的 apply(grads) 方法。
  3. apply 返回的梯度通过 next_edges 传给上游节点继续累加。

对我们的 AddBackward0::apply,codegen 生成的代码(在 build/.../generated/Functions.cpp)长这样:

variable_list AddBackward0::apply(variable_list&& grads) {
  auto& grad = grads[0];                                          // 上游传下来的 dz
  variable_list grad_inputs(2);                                    // 两个输入,两个梯度

  if (should_compute_output(0)) {                                  // self 需要梯度?
    grad_inputs[0] = handle_r_to_c(self_scalar_type, grad);
    // 翻译自 derivatives.yaml:  self: handle_r_to_c(self.scalar_type(), grad)
  }

  if (should_compute_output(1)) {                                  // other 需要梯度?
    grad_inputs[1] = handle_r_to_c(other_scalar_type,
                                   maybe_multiply(grad, alpha.conj()));
    // 翻译自 derivatives.yaml:  other: handle_r_to_c(other.scalar_type(),
    //                                     maybe_multiply(grad, alpha.conj()))
  }

  return grad_inputs;
}

对照 5.1 节抄的反向公式:

self:  handle_r_to_c(self.scalar_type(),  grad)
other: handle_r_to_c(other.scalar_type(), maybe_multiply(grad, alpha.conj()))

这是一字不差的翻译。codegen 把 yaml 里的表达式直接转成 C++ 代码 —— 这就是为什么 PyTorch 团队让"反向公式 DSL"是受限的:它必须能机械地翻成 C++。

注意几个细节:

  • should_compute_output(i):autograd engine 知道哪些叶子张量真的 require grad。如果 other 是个常量(不 require grad),那 grad_inputs[1] 根本不算 —— 性能优化。
  • maybe_multiply(grad, alpha.conj()):maybe_multiply 是个小优化:alpha == 1 时直接返回 grad,跳过浮点乘法。这是 codegen 自动加的"啄木鸟"。
  • handle_r_to_c:处理"实数 tensor → 复数 grad"的边界情况(数值实部投影),实数运算下基本是 no-op。
  • 没有 at::redispatch:反向公式里的子运算(像 maybe_multiply 内部的 at::mul)直接走顶层 at::* API —— 它们自己会走完整 dispatcher,获得"在 CUDA 上跑、记到反向图"的能力。这意味着反向自动支持高阶导:对 AddBackward0::apply 求二阶导,内部的 at::mul 又会建一个 MulBackward0。

返回 grad_inputs 后,engine 把它们沿 next_edges 传给上游(x.grad_fn 和 y.grad_fn,如果有的话)。如果 x、y 是 leaf tensor(没有 grad_fn),梯度直接累加到 x.grad、y.grad。反向结束。

PyTorch Dispatcher Deep Dive: From Schema to CUDA Kernel

PyTorch source references in this article were checked against the pytorch/pytorch main branch at writing time; file names, line numbers, and some internal APIs can drift across releases. When reading the source, prefer the PyTorch tag / commit you actually use. The focus here is on the dispatcher data structures, registration macro semantics, and the complete call stack of a native op from Python down to CUDA; quantization, distributed, inductor and other downstream topics are out of scope.

1. Dispatch keys: foundation of runtime multi-dimensional dispatch

This chapter builds up the dispatcher's runtime model from the bottom up. We start from the user's perspective ("why doesn't if/else work?"), which naturally leads to the dispatcher's core idea: pull the "which path do we take" decision out of the code and turn it into a lookup table indexed by key. We then look at the data structures PyTorch uses to express this table (the DispatchKey enum, the DispatchKeySet bitmap), how it computes "which keys are active," how it decides "which layer to run first," and how a kernel hands off control to the next layer without falling into an infinite loop.

After this chapter, you should be able to mentally simulate every step that z = x + y goes through inside the dispatcher.

1.1 Why we need "dispatch"

We start from an operation that looks utterly ordinary in PyTorch:

y = x + 1

If x is a plain CPU float32 tensor that doesn't need a gradient, this line just needs to do one thing: run a simple loop y[i] = x[i] + 1. A single if/else suffices to decide "CPU add or CUDA add":

// Toy implementation — PyTorch does not do this; we'll explain why
Tensor add(const Tensor& x, Scalar s) {
  if (x.is_cuda()) return add_cuda(x, s);
  else            return add_cpu(x, s);
}

The problem is that reality is much messier. The same x + 1, in different contexts, must internally do completely different things:

  • If x.requires_grad == True, before computing the addition we must first build an AddBackward0 node and register x as its "upstream," so that loss.backward() can trace back later.
  • If the call is nested inside with torch.autocast('cuda'):, we must first convert the fp32 inputs to fp16 before computing. That's how training gets the mixed-precision throughput benefit.
  • If nested inside with FakeTensorMode():, we must not actually compute. We can only "pretend" to compute, infer output shape/dtype, and return a tensor that "looks right but has no contents" (used during compilation/analysis).
  • If x is a sparse tensor (torch.sparse_coo_tensor(...)), the addition goes through a completely different algorithm, not an elementwise loop.
  • If x has been through torch.func.vmap(...), the addition has to broadcast over an extra batch dimension.

What's worse, these conditions stack. A real training step may simultaneously have requires_grad, autocast and CUDA all active. Each "context" must be inserted into the call in the right order: autocast must come before autograd (otherwise the backward graph would record the wrong dtype); autograd must come before CUDA (otherwise the backward upstream wouldn't be recorded); FakeMode must be above everything (otherwise we'd actually compute).

Written with if/else, the code grows into something like the following, and you'd have to write it for every single operator:

Tensor add(const Tensor& x, Scalar s) {
  // FakeTensorMode intercept: only compute meta, no actual values
  if (in_fake_mode()) return add_fake(x, s);

  // autocast: lower precision first
  if (autocast_enabled() && x.is_cuda()) {
    x = x.to(at::kHalf);
  }

  // autograd: record backward
  if (x.requires_grad()) {
    register_backward<AddBackward0>(x);
  }

  // Actually compute: branch on layout × device
  if (x.is_sparse()) return add_sparse(x, s);
  if (x.is_cuda())   return add_cuda(x, s);
  return add_cpu(x, s);
}

This style has two fatal defects that the PyTorch team cannot work around:

  1. Repetition. ATen (PyTorch's operator library) has ~2000 operators. Every single one would need this nested if/else duplicated. Forget if (autocast_enabled()) in any one of them, and that operator silently runs at the wrong precision under autocast; miss if (in_fake_mode()) in any one, and that operator actually computes during tracing. Both failure modes are nearly impossible to catch.
  2. Closedness. If Habana / Apple Metal / Huawei Ascend wants to add a new backend to PyTorch today, they cannot modify PyTorch's source: they need to "insert" their own is_my_backend() branch from the outside. An if/else chain can't do that. Likewise, third-party autocast implementations, JIT tracers and new transforms (vmap/grad/jvp) can't cut across PyTorch ops.
Example 1.1 Imagine PyTorch wants to support a new backend Foo. Under the if/else scheme, you'd have to scroll through the source of every one of ~2000 operators and add if (x.is_foo()) return add_foo(...). Then the next time PyTorch upgrades, you redo all your patches. Every merge conflict is a nightmare. That's the core reason the if/else scheme doesn't hold up in practice: not "ugly," but "extensibility bankrupt."

1.2 The idea: a key-indexed function table

PyTorch's answer is to pull the "which path do we take" decision out of the code and put it in a lookup table.

Core mechanism. After PyTorch starts up, every operator (e.g. aten::add) owns a function-pointer table. The index of this table is called a dispatch key (DispatchKey), representing "a behavioral layer": CUDA backend, CPU backend, Autograd wrapper, Autocast wrapper, FakeMode intercept... each entry is one possible "what to do."

For instance, the lookup table for aten::add looks roughly like this (simplified, the real one has more entries):

DispatchKey          →  function pointer (kernel)
─────────────────────────────────────────────────
CPU                  →  add_cpu
CUDA                 →  add_cuda
SparseCPU            →  add_sparse_cpu
SparseCUDA           →  add_sparse_cuda
MPS                  →  add_mps                  // Apple Metal
AutogradCPU          →  VariableType::add_cpu_wrapper
AutogradCUDA         →  VariableType::add_cuda_wrapper
AutocastCPU          →  autocast::add_cpu_wrapper
AutocastCUDA         →  autocast::add_cuda_wrapper
Fake                 →  fake::add_meta_only
... dozens more

When you call torch.add(x, y), the dispatcher does three things:

  1. Compute the active keys. From the input tensors' properties (on CUDA, requires_grad) plus the current thread's "mode stack" (whether autocast is on, whether inside FakeTensorMode), it computes a set of keys, called the DispatchKeySet. For instance {AutocastCUDA, AutogradCUDA, CUDA}.
  2. Pick the highest-priority one. For instance in this set AutocastCUDA > AutogradCUDA > CUDA, so pick AutocastCUDA.
  3. Jump. Look up the function pointer for AutocastCUDA, autocast::add_cuda_wrapper, and call it.

The key trick: after this autocast wrapper finishes its work (lowering inputs to fp16), it has to continue down to autograd → CUDA. It does not call torch.add(...) again to re-enter the top level (that would loop forever). Instead it does this: mask the AutocastCUDA bit out of the bitmap, then tell the dispatcher to recompute the highest-priority key. This time the highest priority becomes AutogradCUDA, jumping to the autograd wrapper; after the autograd wrapper records the backward graph, it masks AutogradCUDA out and falls to CUDA, which runs the real elementwise kernel.

The whole call chain is like peeling an onion: each layer does its bit, then "erases itself" so the next layer can take over. The figure below traces the full path for with torch.autocast('cuda'): z = x + y (with x, y on CUDA and requires_grad=True):

Each kernel only manages its own layer. The CUDA kernel doesn't need to know autocast exists; autocast doesn't need to know autograd exists. • Extensible. A new backend just adds a row to every op's lookup table; call sites untouched. New mode decorators (like vmap) are just new keys. • Composable. The set of bits in the bitmap determines which layers are traversed; their order is determined by priority, fully decoupled from code. • Order guaranteed by design. "autocast must come before autograd" is encoded as the priority of the keys, no longer relying on every op author remembering it. This "layer-by-layer descent, each layer masking itself" pattern has a proper name: redispatch. §1.7 will unpack its implementation.
Aside 1.2: Why not virtual functions / std::function You might wonder: isn't this just C++'s vtable? Why not use the language's native mechanism?

The differences are twofold:
① The vtable's "dynamic type" comes from a single object (the this pointer). But the dispatcher's dynamic type comes from the union of all input tensors plus a set of thread-local states. The actual behavior of torch.add(x, y) depends on x, on y, and on whether we are inside autocast. This is multiple dispatch, which a vtable cannot express.
② std::function can express multiple dispatch, but every call requires heap-allocating a type-erased closure. In a training scenario with millions of op calls per second, this overhead is unacceptable.

Hand-rolled bitmap plus array lookup, each dispatch costs about 5 to 10 ns, cheaper than a virtual call.

1.3 The actual DispatchKey enum

Now that we have the big picture, let's look at the concrete definitions. All possible keys are listed in c10/core/DispatchKey.h. Here is a simplified version (the real definition will be unpacked in §1.4, which has a more elegant two-dimensional split, but the idea is identical):

// Simplified — real definition in §1.4
enum class DispatchKey : uint16_t {
  Undefined = 0,

  // === Backend layer (actually compute values) ===
  CPU,
  CUDA,
  MPS,            // Apple Metal Performance Shaders
  XLA,            // Google TPU / XLA backend
  Meta,           // Only compute shapes, no values
  ...
  SparseCPU,
  SparseCUDA,
  ...

  // === Autograd layer (above backend, builds backward graph) ===
  AutogradCPU,
  AutogradCUDA,
  AutogradXLA,
  ...

  // === Mode layer (above autograd, intercepts every op call) ===
  ADInplaceOrView,   // in-place / view autograd metadata
  Tracer,            // Old JIT trace recording
  AutocastCPU,
  AutocastCUDA,
  Fake,              // FakeTensorMode
  Functionalize,     // Eliminate alias/mutation
  Python,            // TorchDispatchMode (subclass interception)
  ...                // ~60 in total

  NumDispatchKeys
};

Three observations:

  • The type is uint16_t (16-bit unsigned). Soon we'll pack keys into a 64-bit bitmap, and a 16-bit index is enough to encode every possible key plus header bits.
  • The enum is ordered. As we'll see, the "highest-priority key" is the enum value corresponding to the highest set bit in the bitmap: keys later in the enum have higher priority. That's why Autocast* comes after Autograd*, and Autograd* comes after CPU/CUDA. This is precisely the code-level translation of "autocast must happen before autograd" from §1.1. The designers encoded the "semantic layer ordering" directly into the enum's literal order.
  • About 60 keys (NumDispatchKeys sentinel). It sounds like a lot, but most of them are "mode-layer" keys (autocast / fake / functionalize / python, etc.) that aren't activated during normal calls; they only get pushed into the bitmap when the corresponding with-block is opened.
Example 1.3 For a plain CPU float32 tensor that doesn't require grad, when running x + 1, the bitmap holds only {CPU}; the highest priority is CPU, so we jump directly to add_cpu. If x.requires_grad_(True), the bitmap becomes {AutogradCPU, CPU}, the highest priority is AutogradCPU, so we go into the autograd wrapper first, then redispatch to CPU. The dispatcher runs two rounds, but the user code is unchanged.

1.4 Two-dimensional split: functionality × backend

The enum above has a size problem: if you take the Cartesian product of "each behavior × each backend," the enum explodes. For example, if PyTorch wants to support Habana (HPU), it would need to add this pile of entries:

HPU              // dense × Habana
SparseHPU        // sparse × Habana
SparseCsrHPU
QuantizedHPU
NestedTensorHPU
AutogradHPU      // autograd × Habana

One new backend requires ~6 new enum values, and each addition means editing the enum definition and all related codegen. Each new functionality (say if we wanted Adagrad×Backend in the future) multiplies it by N. Unsustainable.

The PyTorch team did a major refactor in 2022: split the enum into two orthogonal smaller enums and merge them into a single 64-bit bitmap (DispatchKey.h):

  1. BackendComponent: just "which backend." The current bits: CPU / CUDA / HIP / XLA / MPS / IPU / XPU / HPU / VE / Lazy / MTIA / MAIA / PrivateUse1‑3 / Meta. A static_assert caps the backend count at ≤ 16 (DispatchKey.h has an explicit 16-bit budget). Note that this alone doesn't form a dispatch level; it must combine with some functionality to be meaningful.
  2. DispatchKey (functionality). The vast majority of functionalities occupy a single slot (a globally unique handler); six are "per-backend" and need to combine with a BackendComponent to form a runtime key. The six are: Dense, Quantized, Sparse, SparseCsr, NestedTensor, AutogradFunctionality.

The combination rule (toRuntimePerBackendFunctionalityKey()):

(Dense,                CUDABit)  →  CUDA           // the real identity of the "CUDA" key you know
(Dense,                CPUBit)   →  CPU
(AutogradFunctionality, CUDABit) →  AutogradCUDA
(AutogradFunctionality, CPUBit)  →  AutogradCPU
(Sparse,               CUDABit)  →  SparseCUDA
(Quantized,            CPUBit)   →  QuantizedCPU
...

In other words, when you write TORCH_LIBRARY_IMPL(aten, CUDA, m) you're actually registering at the two-dimensional coordinate (Dense, CUDA). At the source level you still see a single CUDA key, but the bitmap underneath encodes it as "the Dense functionality bit + the CUDA backend bit both set to 1." Adding a new backend (HPU)? Just add a bit to BackendComponent, and the six per-backend functionalities automatically define six new combined keys for it, with no enum changes needed.

Example 1.4 To add a Habana (HPU) backend "sparse" implementation to PyTorch: add an HPU bit to BackendComponent → instantly the combined key (Sparse, HPU) exists, and you can register the sparse kernel with TORCH_LIBRARY_IMPL(aten, SparseHPU, m). Neither SparseHPU nor AutogradHPU needs to be added to enum DispatchKey; both are combined runtime keys.
Aside 1.4: drift warning ezyang's 2019 PyTorch internals does not have the BackendComponent layer, and neither does the 2020 dispatcher article. The two-dimensional split described in this section is the post-2022 mechanism; when reading older articles, note that this has shifted: they treat SparseCUDA as a concrete enum value, whereas in today's code it is the combination (Sparse, CUDABit).

1.5 DispatchKeySet: a 64-bit bitmap

Now we have DispatchKey (~60 functionalities) and BackendComponent (16 backends). Pack them all into a 64-bit integer (uint64_t), and you get DispatchKeySet. It is the core data structure of the dispatcher: what the dispatcher receives at every op call is exactly this 64-bit integer.

The bitmap layout (per the class comments in DispatchKeySet.h):

Where does this 64-bit integer come from? The union of three sources:

  1. The union of all input tensors' TensorImpl::key_set_. Each tensor's TensorImpl caches a DispatchKeySet that records "I'm on CUDA" + "I requires_grad" and so on. The dispatcher or's together the key_sets of all argument tensors. This is what implements "multiple dispatch" (any input being on CUDA activates the CUDA key).
  2. TLS local include (thread-local "add"): pushed by mode guards. For example, entering with torch.autocast('cuda'):, the with block's implementation pushes the AutocastCUDA bit into TLS; on every op call the dispatcher or's the TLS include into the bitmap.
  3. TLS local exclude (thread-local "remove"): used for redispatch. Inside an autograd kernel, at::AutoDispatchBelowAutograd guard; stuffs every autograd bit into TLS exclude via an RAII object. The dispatcher subtracts exclude from the bitmap, so the highest bit "naturally" falls to the next layer.

A fourth hidden factor: fallthrough mask. For each op, at registration time the dispatcher precomputes "which keys are fallthrough at this op (no concrete kernel registered, equivalent to 'this layer skips')." These keys are masked out of the bitmap as soon as it enters the dispatcher, achieving O(1) skip. This is the key reason the dispatcher can "support dozens of functionalities by default without paying runtime cost": if your op doesn't register a kernel for Conjugate, the Conjugate bit is cleared directly by the fallthrough mask.

Concretely, the core line lives in DispatchKeyExtractor.h:23-44:

// aten/src/ATen/core/dispatch/DispatchKeyExtractor.h
inline DispatchKeySet computeDispatchKeySet(
    DispatchKeySet ks,              // or'd key_set of input tensors
    DispatchKeySet key_mask) {      // the op's fallthrough mask
  c10::impl::LocalDispatchKeySet local = c10::impl::tls_local_dispatch_key_set();
  return (((ks | local.included_) - local.excluded_) & key_mask);
  //        ↑ add include  ↑ subtract exclude  ↑ apply fallthrough mask
}

That's all the "magic" of §1.5: three bit operations producing a final bitmap. Then lookup() finds the highest bit in this bitmap, unpacked in §1.6.

Example 1.5 Let x be a CUDA fp32 tensor with requires_grad=True, and run z = x + 1 inside with torch.autocast('cuda'):.

1. x's key_set_ = {CUDABit, Dense, AutogradFunctionality} = three bits set in the bitmap (CUDABit, Dense, AutogradFunctionality).
2. The autocast context pushed AutocastCUDA into TLS include.
3. No exclude, and no fallthrough mask removes these bits.
4. The final bitmap, after combining functionality + backend, is equivalent to the runtime key set {AutocastCUDA, AutogradCUDA, CUDA}.

Compare with the "peeling the onion" diagram in §1.2: this is exactly that diagram's initial bitmap.

1.6 The priority ladder

Given a 64-bit bitmap, the dispatcher uses highestPriorityTypeId() to take "the functionality corresponding to the highest 1 bit." "Highest" means the highest-numbered set bit: functionality bits are arranged by priority, with bit number = priority.

The figure below draws common functionalities by priority, from top (highest) to bottom (lowest):

Find a key: take the highest 1 bit → run that layer's kernel; if the kernel body calls at::redispatch(...), it carries a bitmap with the current key cleared, falling naturally to the next layer. • autograd "above backend" is the soul of the design: it can intercept inputs before any number-crunching, attach next_edges, build the backward graph, then redispatch to run CUDA. • autocast is above autograd: lowering input precision happens before autograd captures inputs, so the backward graph records the lower-precision dtype. This is the code-level translation of §1.1's "order constraint." • fallthrough kernel = "this layer does nothing, mask my bit before dispatching" (O(1) skip); most mode keys default to fallthrough.

The word "catchall" in dispatcher context refers to the fallback for alias keys. Alias keys aren't real functionalities; they are convenience syntax for "expand at registration time into multiple concrete keys":

  • Autograd (alias) = register to all Autograd<Backend>. When you write TORCH_LIBRARY_IMPL(ns, Autograd, m), the registration is expanded into registering the same kernel for AutogradCPU, AutogradCUDA, etc.
  • CompositeImplicitAutograd (alias) = active on all backends + all autograd keys. Suitable for ops written as a composition of at::* calls (the internal ops re-enter the dispatcher and inherit autograd and backend-independence).
  • CompositeExplicitAutograd (alias) = all backends (without autograd). Backend-agnostic implementations where the author handles autograd themselves.

These three aliases give PyTorch a concise registration syntax for "one implementation covers all backends." Mechanically they are syntactic sugar for batched m.impl. At runtime the dispatcher only sees concrete keys like AutogradCUDA or CUDA, never the Autograd alias itself.

1.7 redispatch: cycle-free inter-layer recursion

In §1.2 we built the mental model of "peeling the onion": each layer does its work, masks itself, and lets the next layer take over. This section looks at how that is actually implemented.

The core question: after the autograd kernel finishes its bit (building grad_fn, recording next_edges), how does it "drop down to the backend kernel"? Calling at::add(x, y) directly won't work. That re-enters top-level Dispatcher::call, the highest bit is computed again as AutogradCUDA, we re-enter the autograd kernel, and we loop forever.

The answer is Dispatcher::redispatch (Dispatcher.h:833-848):

// Key comment (from source): "redispatch treats the inputted DispatchKeySet as is,
//  and does NOT mask out the highest priority bit"
//  — the caller is responsible for masking out the "current key" from the bitmap before passing it in
template <class Return, class... Args>
inline Return Dispatcher::redispatch(
    const TypedOperatorHandle<Return(Args...)>& op,
    DispatchKeySet currentDispatchKeySet,    // ← caller has already masked the current key
    Args... args) const {
  const KernelFunction& kernel = op.operatorDef_->op.lookup(currentDispatchKeySet);
  return kernel.template call<Return, Args...>(op, currentDispatchKeySet, ...);
}

Compared to Dispatcher::call (the top-level entry): call recomputes the bitmap (or-ing input tensors' key_set + TLS processing); redispatch uses the bitmap passed in directly. This is deliberate: it gives the kernel the flexibility to say "I've decided where to jump next."

So how does one "mask out the current key"? Inside the autograd kernel, RAII guards flip TLS exclude:

  • at::AutoDispatchBelowAutograd (older name AutoNonVariableTypeMode, often seen in old blog posts): on construction it pushes every autograd functionality bit into TLS excluded_; on destruction it restores them. During its lifetime, every op call has its bitmap automatically stripped of the autograd bits.
  • at::AutoDispatchBelowADInplaceOrView: additionally pushes ADInplaceOrView into exclude. This is the standard usage of the codegen-emitted VariableType wrapper: the autograd kernel uses it, then redispatch drops directly to the backend.

Stitching these mechanisms together, the skeleton of a real autograd kernel looks like this (simplified from the codegen output in VariableType_*.cpp):

// AutogradCUDA kernel for aten::add.Tensor (simplified)
Tensor VariableType::add_Tensor(
    c10::DispatchKeySet ks,
    const Tensor& self, const Tensor& other, const Scalar& alpha) {

  // ① Build backward node (only if grad is needed)
  std::shared_ptr<AddBackward0> grad_fn;
  if (compute_requires_grad(self, other)) {
    grad_fn = std::make_shared<AddBackward0>();
    grad_fn->set_next_edges(collect_next_edges(self, other));
  }

  // ② Enter "below autograd" scope → autograd bits removed from the bitmap
  auto result = [&]() {
    at::AutoDispatchBelowADInplaceOrView guard;
    return at::redispatch::add(
        ks & c10::after_autograd_keyset,    // explicitly mask the current key
        self, other, alpha);
  }();

  // ③ Attach grad_fn to the output
  if (grad_fn) set_history(result, grad_fn);
  return result;
}
Example 1.7: full redispatch flow Walk through this code for z = x + y (with x, y on CUDA, requires_grad=True):

(a) Top-level Dispatcher::call computes the bitmap = {AutogradCUDA, CUDA}, highest bit = AutogradCUDA, jumps to VariableType::add_Tensor.
(b) The function body enters the RAII guard AutoDispatchBelowADInplaceOrView: TLS exclude now contains all autograd bits + the ADInplaceOrView bit.
(c) Call at::redispatch::add(ks & after_autograd_keyset, ...). after_autograd_keyset is a precomputed mask that "clears all autograd bits and above," explicitly ensuring AutogradCUDA won't be picked again.
(d) Dispatcher::redispatch looks up directly with this bitmap; the highest bit is now CUDA, jumping to wrapper_CUDA_add_Tensor.
(e) The CUDA wrapper finishes and returns.
(f) Back in the autograd kernel, set_history(result, grad_fn) attaches the backward node and returns.

The guard destructs between (e) and (f), and TLS exclude is restored. The bitmap computation for the next top-level call starts clean again.
Remember this mental model: the dispatcher is not "one jump," but "every jump may trigger the next jump"; the kernel decides whether to redispatch. Mode decorators, autograd, and backends form a chain explicitly driven by redispatch. The chain's order is determined by the bit number of each functionality in the 64-bit bitmap; each layer uses an RAII guard to erase itself so that the next layer naturally becomes the new highest bit.

2. Schema: the operator type signature

Chapter 1 laid out the runtime mechanics of the dispatcher. But how does the dispatcher know what arguments "aten::add.Tensor" accepts? What it returns? Which arguments are mutated? The answer to these questions is the schema.

The schema is the operator's interface contract. It is not just a string for documentation: it is read by the dispatcher, autograd, JIT, functionalization, and the Python binding layer at the same time, and each component extracts the information it cares about. The dispatcher learns how many tensor arguments there are (which determines the keyset); autograd learns which arguments are mutated (which determines view handling); the Python binding layer generates argument-parsing code from it. So getting one character wrong in the schema (for example writing Tensor(a!) as Tensor(a)) is not a documentation error but a correctness error: functionalization will believe this argument is not mutated and emit the wrong graph.

This chapter unpacks the schema's string grammar, its variant naming rules, and its concrete realization in yaml.

2.1 Reading a schema, character by character

Take a real schema string:

aten::add.Tensor(Tensor self, Tensor other, *, Scalar alpha=1) -> Tensor

It looks like a Python function signature, but every piece of punctuation has independent semantics. Let's break it down left to right, one piece at a time:

#PieceSemantics
1atenNamespace. PyTorch's built-in ops live under aten; custom ops (extensions) live in their own namespace, e.g. myops::foo. The namespace isolates "which ops are findable" and determines which TORCH_LIBRARY* macro to register with (see chapter 3).
2::Namespace separator, just like C++.
3addThe op name. Multiple overloads can share the same name (like C++ function overloading).
4.TensorThe overload name. This is critical: add has both add.Tensor (adding two tensors) and add.Scalar (adding a tensor and a scalar). Omittable, equivalent to overload name "" (empty string). The dispatcher uses the full string "aten::add.Tensor" as the key into the function table.
5(...)Argument list.
6Tensor selfFirst positional argument, of type Tensor, named self. Convention: like Python method's self, autograd treats it as the "main" input.
7Tensor otherSecond positional argument, type Tensor.
8*Keyword separator. From here, subsequent arguments must be passed by keyword (add(x, y, alpha=2)), not positionally. Exactly the same as Python's * in function definitions.
9Scalar alpha=1Keyword argument, type Scalar, default value 1. When Python writes torch.add(x, y, alpha=3), this value lands here.
10-> TensorReturn type. Multiple return values use tuple syntax, e.g. -> (Tensor, Tensor).

The type syntax covers a range of primitives and composites:

TensorTensor (required, non-null)
Tensor?Nullable tensor (corresponds to C++ std::optional<Tensor>)
Tensor[]Tensor list (std::vector<Tensor>)
Tensor(a)Alias annotation: this argument shares storage with a "memory slot" named a, but does not write it (read-only view)
Tensor(a!)Mutable alias annotation: shares storage and writes it (in-place / out)

Remember this: the alias annotations ((a) and (a!)) are the most important markers in the schema. They tell multiple downstream components different things:
• functionalization uses them to decide "does this op produce side effects, do we need to expand into functional form"
• autograd uses them to decide "does this op produce a view, do we need to build a DifferentiableViewMeta"
• JIT uses them for alias analysis (whether two variables may point to the same memory, affecting dead-code elimination and other optimizations)
• inductor and functorch both rely on this annotation system

Writing (a) as (a!) by mistake makes functionalization treat an in-place op as functional, getting the backward graph wrong; the other way around, marking a functional op as in-place may produce false alias-conflict reports.
Example 2.1: what alias annotations do Look at aten::transpose's schema: 
aten::transpose.int(Tensor(a) self, int dim0, int dim1) -> Tensor(a)
Tensor(a) appears on both self and the return: they share the same memory slot named a. This tells the dispatcher's downstream: the transposed tensor and the original share storage; mutating one mutates the other.

Contrast with aten::clone: 
aten::clone(Tensor self, *, MemoryFormat? memory_format=None) -> Tensor
The return has no alias annotation → it's a freshly allocated tensor; modifying it doesn't affect self.

At registration time, this string is parsed by torch::jit::parseSchema into a c10::FunctionSchema struct that records each argument's name, type, alias info, and default value; only then is it registered into the dispatcher's internal op table. So the "schema string" is the source of truth, and FunctionSchema is its in-memory representation.

2.2 Three variants: functional / in-place / out

Many operators have "side-effect" capabilities (in-place mutation, writing into a preallocated out tensor). PyTorch gives such operators three schema variants simultaneously, functionally equivalent but with different interfaces. Look at the three variants of add:

VariantSchema
functionaladd.Tensor(Tensor self, Tensor other, *, Scalar alpha=1) -> Tensor
in-placeadd_.Tensor(Tensor(a!) self, Tensor other, *, Scalar alpha=1) -> Tensor(a!)
outadd.out(Tensor self, Tensor other, *, Scalar alpha=1, Tensor(a!) out) -> Tensor(a!)

Naming conventions:

  • functional: returns a new tensor. Python writes z = x.add(y). Most common, corresponds to the schema's "base form."
  • in-place: op name ends with _ (underscore), self annotated with (a!), return also (a!) indicating "the returned tensor is the mutated self." Python writes x.add_(y), equivalent to x = x + y but with no new allocation.
  • out: overload adds .out, with an extra kw-only Tensor(a!) out. Python writes torch.add(x, y, out=z), writing the result into the preallocated z. Commonly used in memory-reuse optimization scenarios.

Why write three? Three reasons:

  1. User-side variation. Python users want all three styles supported. Without functional, x.add_(y) wouldn't exist.
  2. Performance. Functional always allocates new memory; in-place and out can reuse, and the difference in peak training memory is significant.
  3. Autograd behavior. The three variants are handled completely differently by autograd: functional directly builds grad_fn; in-place must bump the version counter and may reject (RuntimeError: a leaf Variable that requires grad...); out is somewhere in between.
Example 2.2: how the three variants share an implementation Although the schema is in triplicate, the actual math is written once. Under the hood this uses the structured kernel pattern: 
add.out      :  writes into the preallocated out tensor (core implementation)
add.Tensor   :  allocates a new out → calls add.out → returns out
add_.Tensor  :  treats self as out → calls add.out → returns self
The "allocate out + call the out version" step is auto-generated by codegen (see structured_delegate: add.out in yaml); the author only writes the out version's implementation.
Aside 2.2: the autogen keyword Some op authors only write the functional variant and tell codegen to "fill in the in-place and out variants by rule": 
- func: my_op(Tensor self) -> Tensor
  autogen: my_op.out, my_op_
On seeing autogen:, codegen automatically generates schema registrations for my_op.out and my_op_, plus the "out wraps functional" implementations, without the author writing a single line. This is very labor-saving for purely functional ops (no in-place complexity).

Aside 2.3: why structured exists In early PyTorch (pre-2018), each operator's three variants (functional / inplace / out) had to be written by hand; vast amounts of code looked like this: add's functional allocates output → calls add_kernel; add_ treats self as output → calls add_kernel; add_out checks out shape → calls add_kernel. This boilerplate had to be written three times per op, and every new op meant ~100 lines of glue.

Introduced post-2020, the structured kernel pattern standardizes this glue: the author writes the actual math (impl()) only on the .out variant, and the meta phase (computing output shape/dtype) in meta(). Codegen auto-generates the functional and inplace variants as wrappers of out. A new op then needs only ~10 lines of yaml + a single impl, cutting code volume to one-fifth of before.

2.4 derivatives.yaml: backward formulas

So far the schema only describes "forward behavior." Backward (autograd) information lives in another file: tools/autograd/derivatives.yaml. Each differentiable op gets a backward formula here:

- name: add.Tensor(Tensor self, Tensor other, *, Scalar alpha=1) -> Tensor
  self:  handle_r_to_c(self.scalar_type(),  grad)
  other: handle_r_to_c(other.scalar_type(), maybe_multiply(grad, alpha.conj()))
  result: self_t + maybe_multiply(other_t, alpha)            # forward-mode tangent

- name: relu(Tensor self) -> Tensor
  self: threshold_backward(grad, result, 0)
  result: auto_element_wise

Reading it field by field:

  • name: must exactly match the schema in native_functions.yaml: this is the "primary key" matching the two sides.
  • self:, other: are the input argument names; the value is an expression describing "if this input needs a gradient, the gradient = what." For example, add's gradient for other = grad * alpha.conj() (wrapped with maybe_multiply as an optimization that skips the multiplication when alpha == 1). grad is a magic variable meaning "the upstream gradient passed in by this backward."
  • result: is the forward-mode (jvp / tangent) formula, used by torch.func.jvp. self_t, other_t are also magic variables meaning "the tangent vector of input self." If jvp support isn't needed, this line can be omitted.
  • auto_element_wise is a special tag that tells codegen "this is an elementwise op; auto-derive the jvp formula by rule."

These formulas are not Python; they are a restricted DSL consumed by codegen. It supports calling other at::* ops (like threshold_backward), accessing input attributes (like self.scalar_type()), and using predefined variables like grad/result/self_t.

How does codegen use it? For each differentiable op, it generates two things:

  1. An autograd Node subclass (e.g. AddBackward0, ReluBackward0) whose apply() method body is the formula translated into C++.
  2. An autograd kernel wrapper (registered to AutogradCPU/AutogradCUDA etc.) that during forward builds the Node, attaches next_edges, and redispatches to the backend (the skeleton from §1.7).
Example 2.4: the correspondence between derivatives.yaml and generated code The backward formula for add in derivatives.yaml: 
other: handle_r_to_c(other.scalar_type(), maybe_multiply(grad, alpha.conj()))
gets translated by codegen into this code inside AddBackward0::apply(): 
if (should_compute_output(1)) {
  grad_inputs[1] = handle_r_to_c(other_scalar_type,
                                 maybe_multiply(grad, alpha.conj()));
}
Almost one-to-one (the only addition is a "do we need this gradient" guard, since some downstream consumers may not need other's gradient, allowing a compute-saving skip).

§5.8 will unfold the full generated code of AddBackward0::apply() and how the autograd engine calls it during loss.backward().

Three mental models to take from this chapter:
① The schema is a contract read by many parties: dispatcher, autograd, JIT, functionalization, Python bindings all extract different information from the same string.
② The (a!) alias annotation is the "most dangerous" marker in the schema: getting it wrong directly affects the correctness of functionalization, autograd, and alias analysis.
③ native_functions.yaml + derivatives.yaml are the input sources for all codegen; every generated C++ file that follows (dispatcher registration, autograd wrapper, Python bindings) sprouts from these two files.

3. The C++ registration interface: the TORCH_LIBRARY family

Chapter 2 showed that the schema is the contract. This chapter answers "who executes that contract": PyTorch uses a family of macros (all defined in torch/library.h) to feed "this schema's implementation on backend X is this snippet of C++ code" to the dispatcher.

The family looks elaborate (TORCH_LIBRARY, TORCH_LIBRARY_FRAGMENT, TORCH_LIBRARY_IMPL, m.def, m.impl, m.fallback, TORCH_FN, makeFromBoxedFunction ......) but they all share the same underlying pattern:

"Use a file-static object's constructor to stuff information into the global dispatcher before main()."

All differences are in "what to stuff in": open a new namespace, extend an existing namespace, or attach a kernel for a given key to an existing op. This chapter unpacks each of these three "stuffings" in turn, then covers three related concepts (boxed vs unboxed, fallback vs fallthrough, def vs impl), and finally walks through a complete custom op that ties everything together.

3.1 The three identities of Library: DEF / FRAGMENT / IMPL

The core data structure for registration is the torch::Library object. Every time you write a TORCH_LIBRARY* macro, the expansion constructs a Library instance, but it has three identities (torch/library.h:551-553):

// torch/library.h
class Library {
 public:
  enum Kind { DEF, IMPL, FRAGMENT };
  ...
};

The three Kinds correspond to three semantics:

KindSemanticsTriggering macroTypical user
DEFOpens a "main" library for a namespace. Each namespace name may appear in at most one DEF in the entire process; duplicate linking aborts immediately.TORCH_LIBRARY(ns, m)Custom extensions (opening your own myops namespace)
FRAGMENTAppends a "fragment" to the same namespace. Same-name fragments may come from any number of translation units (TUs) and are stitched together into one library.TORCH_LIBRARY_FRAGMENT(ns, m)PyTorch internals: the ~2000 ops under the aten namespace are assembled from dozens of codegen files
IMPLAttaches a kernel under a DispatchKey to an already existing op. Cannot define a new schema here, only m.impl(...) implementations for existing ops.TORCH_LIBRARY_IMPL(ns, key, m)Registering concrete implementations for a backend (CUDA wrappers in aten, CUDA kernels for custom ops, etc.)
Example 3.1: practical uses for the three Kinds You write a PyTorch extension my_extension that registers an operator myops::custom_relu with implementations on both CPU and CUDA. The three .cpp files split as follows:

register_schema.cpp uses TORCH_LIBRARY(myops, m) → DEF, declares the schema. With this line in place, the myops namespace is "opened."
kernel_cpu.cpp uses TORCH_LIBRARY_IMPL(myops, CPU, m) → IMPL, registers the CPU implementation.
kernel_cuda.cu uses TORCH_LIBRARY_IMPL(myops, CUDA, m) → IMPL, registers the CUDA implementation.

If you write schema and CPU kernel in the same file (TORCH_LIBRARY(myops, m) { m.def(...); m.impl(...); }), that's also valid, but m.impl inside a DEF block can't specify a DispatchKey, and is treated as a "catch-all kernel" (all keys go through it). So for multi-backend scenarios, you must use IMPL to register different backends separately.
Aside 3.1: why not a single unified macro You might ask: since they all construct Library objects underneath, why not merge into a single TORCH_LIBRARY(ns, kind, key, m)?

Two reasons:
① Static validation. DEF is not allowed to specify a DispatchKey (it declares the schema, doesn't attach a kernel); IMPL must specify a DispatchKey. Splitting into three independent macros lets compile-time prevent argument confusion (and lets IDEs autocomplete more accurately).
② Link-time "uniqueness" checking. DEF uses a "no uniquifier" path for the file-static object, so multiple links trigger a duplicate-symbol error, enforcing "one DEF per namespace." FRAGMENT and IMPL both use C10_UID to uniquify symbol names, allowing same-name macros to coexist across files. This difference is encoded in how the macros expand; merging into one macro would break it.

3.2 TORCH_LIBRARY: opening a new namespace

Look at the actual TORCH_LIBRARY macro (torch/library.h:982-991):

// torch/library.h
#define TORCH_LIBRARY(ns, m)                                                   \
  static void TORCH_LIBRARY_init_##ns(torch::Library&);                        \
  static const torch::detail::TorchLibraryInit TORCH_LIBRARY_static_init_##ns( \
      torch::Library::DEF,                                                     \
      &TORCH_LIBRARY_init_##ns,                                                \
      C10_STRINGIZE(ns), std::nullopt,                                         \
      __FILE__, __LINE__);                                                     \
  void TORCH_LIBRARY_init_##ns(torch::Library& m)

It looks scary, but breaks down into three declarations:

  1. Line 1: static void TORCH_LIBRARY_init_##ns(torch::Library&); forward-declares a function (##ns is token pasting; if ns=myops, this becomes TORCH_LIBRARY_init_myops).
  2. Lines 2 to 5: static const torch::detail::TorchLibraryInit TORCH_LIBRARY_static_init_##ns(DEF, ...); constructs a file-static object. Its constructor receives:
    • torch::Library::DEF: identity marker; this is a DEF, not an IMPL.
    • &TORCH_LIBRARY_init_##ns: the user-provided "registration function" (declared on line 1); the constructor calls it.
    • C10_STRINGIZE(ns): namespace string literal ("myops").
    • std::nullopt: DispatchKey, empty for DEF.
    • __FILE__, __LINE__: location info for errors.
  3. Line 6: void TORCH_LIBRARY_init_##ns(torch::Library& m) is the function's definition header; the user's { ... } follows immediately after the macro as the function body.

So the user's TORCH_LIBRARY(myops, m) { m.def("..."); } expands to:

static void TORCH_LIBRARY_init_myops(torch::Library&);
static const torch::detail::TorchLibraryInit TORCH_LIBRARY_static_init_myops(
    torch::Library::DEF,
    &TORCH_LIBRARY_init_myops,
    "myops", std::nullopt,
    "/path/to/myops.cpp", 42);

// The user's { ... } supplies the body of this function:
void TORCH_LIBRARY_init_myops(torch::Library& m)
{
  m.def("...");
}

The key magic is the initialization timing of file-static objects: the C++ standard guarantees all file-static objects are constructed before main() runs. So by the time PyTorch's main / the user's Python interpreter starts, the constructor of TorchLibraryInit has already run and pushed this schema into the global dispatcher table. The user's registration code "automatically" takes effect during program startup, no manual call needed.

Example 3.2: usage 
// myops_register.cpp
#include <torch/library.h>

TORCH_LIBRARY(myops, m) {
  m.def("custom_add(Tensor self, Tensor other) -> Tensor");
  m.def("custom_relu(Tensor self) -> Tensor");
}
Once this is compiled and linked into the Python process, Python code can call torch.ops.myops.custom_add(...) and torch.ops.myops.custom_relu(...), even without any backend kernel registered. Of course actually calling will produce "no kernel registered" errors; but the schema itself is in the dispatcher.
Key restriction: only one TORCH_LIBRARY(ns, ...) per namespace per process. Writing a second triggers a runtime std::abort (checked during Library construction). If you need to split a namespace's registrations across multiple .cpp files, use TORCH_LIBRARY_FRAGMENT from the next section.

3.3 TORCH_LIBRARY_FRAGMENT: extending the same namespace

FRAGMENT addresses the "same namespace across multiple files" need. Look at the macro (torch/library.h:1001-1023, simplified):

// torch/library.h (simplified)
#define TORCH_LIBRARY_FRAGMENT(ns, m) _TORCH_LIBRARY_FRAGMENT(ns, m, C10_UID)
#define _TORCH_LIBRARY_FRAGMENT(ns, m, uid)                                \
  static void C10_CONCATENATE(TORCH_LIBRARY_FRAGMENT_init_##ns##_, uid)(   \
      torch::Library&);                                                    \
  static const torch::detail::TorchLibraryInit                             \
      C10_CONCATENATE(TORCH_LIBRARY_FRAGMENT_static_init_##ns##_, uid)(    \
          torch::Library::FRAGMENT,                                        \
          ...,                                                             \
          C10_STRINGIZE(ns), std::nullopt, ...);                           \
  void C10_CONCATENATE(...)(torch::Library& m)

Compared with TORCH_LIBRARY, only two things differ:

  1. Kind changes from DEF to FRAGMENT: telling the Library constructor "don't abort on a duplicate namespace."
  2. Symbol names are uniquified with C10_UID. C10_UID is PyTorch's internal "line-number + counter" macro, generating a different token at each expansion. So writing five TORCH_LIBRARY_FRAGMENT(aten, m) { ... } blocks in the same TU yields five non-conflicting init function names.
Aside 3.3: why aten is forced to use FRAGMENT Look at the dozens of PyTorch codegen-generated files RegisterCPU.cpp / RegisterCUDA.cpp / RegisterSchema.cpp: they all start with TORCH_LIBRARY_FRAGMENT(aten, m), never TORCH_LIBRARY(aten, m). This is not by accident; torchgen has a special case in torchgen/gen.py:
# torchgen/gen.py around 1761-1776 (simplified)
if namespace == "aten":
    aten_schema_registrations = schema_registrations_body
else:
    torch_library_macro = (
        "TORCH_LIBRARY_FRAGMENT"
        if namespace in FRAGMENT_NAMESPACES
        else "TORCH_LIBRARY"
    )
The reason is practical: the aten namespace has ~2000 ops, and each codegen file contributes tens to hundreds of schema registrations. A single file with thousands of lines stuffed into a single TORCH_LIBRARY block would explode the compiler (MSVC has an instruction-count limit per function). Splitting into dozens of fragments, each compiled independently, works perfectly.
Example 3.3: fragments across files in the same namespace You want to register ops under the myops namespace by module (op family A in a.cpp, op family B in b.cpp): 
// myops_a.cpp
TORCH_LIBRARY_FRAGMENT(myops, m) {
  m.def("op_a1(Tensor x) -> Tensor");
  m.def("op_a2(Tensor x) -> Tensor");
}

// myops_b.cpp
TORCH_LIBRARY_FRAGMENT(myops, m) {
  m.def("op_b1(Tensor x) -> Tensor");
}
Both files coexist; each TorchLibraryInit runs before main and adds the corresponding schemas into the myops namespace.

But note: the entire process still allows only one TORCH_LIBRARY(myops, ...) (DEF). FRAGMENT cannot "open a new namespace itself"; it assumes the namespace has been created by some DEF. With only FRAGMENT and no DEF, you get a "namespace not found" error.

For the aten namespace, its DEF is actually created by PyTorch's internal init logic (not from any specific .cpp); so all codegen fragments can simply extend it.

3.4 TORCH_LIBRARY_IMPL: attaching a kernel

The schema is the "declaration"; the kernel is the "implementation." TORCH_LIBRARY_IMPL is the macro for attaching a kernel for a given DispatchKey to an existing op (torch/library.h:1064-1084, simplified):

// torch/library.h (simplified)
#define TORCH_LIBRARY_IMPL(ns, k, m) _TORCH_LIBRARY_IMPL(ns, k, m, C10_UID)
#define _TORCH_LIBRARY_IMPL(ns, k, m, uid)                                  \
  static void C10_CONCATENATE(TORCH_LIBRARY_IMPL_init_##ns##_##k##_, uid)(  \
      torch::Library&);                                                    \
  static const torch::detail::TorchLibraryInit                              \
      C10_CONCATENATE(TORCH_LIBRARY_IMPL_static_init_##ns##_##k##_, uid)(   \
          torch::Library::IMPL,                                             \
          ...,                                                              \
          C10_STRINGIZE(ns),                                                \
          c10::make_optional(c10::DispatchKey::k),                          \
          ...);                                                             \
  void C10_CONCATENATE(...)(torch::Library& m)

Differences from FRAGMENT:

  • Kind = IMPL.
  • An extra parameter c10::DispatchKey::k (where k is the user's second macro argument, e.g. CPU, CUDA, AutogradCUDA). Each m.impl("op_name", kernel) inside the block is attached to the op's dispatch entry under this DispatchKey.
Example 3.4: typical usage 
// myops_cpu.cpp — register kernels for myops::* under the CPU key
TORCH_LIBRARY_IMPL(myops, CPU, m) {
  m.impl("custom_add",  &my_add_cpu);
  m.impl("custom_relu", &my_relu_cpu);
}

// myops_cuda.cu — register kernels for myops::* under the CUDA key
TORCH_LIBRARY_IMPL(myops, CUDA, m) {
  m.impl("custom_add",  &my_add_cuda);
  m.impl("custom_relu", &my_relu_cuda);
}

// myops_autograd.cpp — separate autograd wrapper, for attaching grad_fn
TORCH_LIBRARY_IMPL(myops, Autograd, m) {       // Autograd is an alias key, auto-expanded to AutogradCPU/CUDA/...
  m.impl("custom_add", &my_add_autograd);
}
Once the three files are linked together, the dispatcher table for myops::custom_add has three kernels: CPU, CUDA, Autograd* (auto-expanded to every concrete autograd key). On each call, the dispatcher finds the highest-bit key based on the input tensors' device and whether they require grad, and jumps to the matching kernel.
The special form TORCH_LIBRARY_IMPL(_, KEY, m): the underscore as the first argument means "applies to all namespaces," typically paired with m.fallback(...) to register a global fallback kernel for a DispatchKey. An example:
// aten/src/ATen/core/VariableFallbackKernel.cpp
TORCH_LIBRARY_IMPL(_, AutogradCUDA, m) {
  m.fallback(autogradNotImplementedFallback());
}
Meaning: for any operator in any namespace, if no explicit kernel is registered for the AutogradCUDA key, this fallback runs. The fallback's behavior is "treat as CompositeImplicitAutograd, letting the inner sub-ops walk through the dispatcher themselves." This is the mechanism implementing PyTorch's "operators without explicit backward formulas can still auto-differentiate."

3.5 m.def vs m.impl semantics

m.def and m.impl are the two most-used methods on the Library object. Their responsibilities are completely different:

m.defm.impl
Where to useOnly inside TORCH_LIBRARY / TORCH_LIBRARY_FRAGMENT blocksOnly inside TORCH_LIBRARY_IMPL blocks
PurposeDeclare the schema: register a type signature for a new op name, putting it into the dispatcher tableAttach a kernel to a declared op: register a concrete function pointer at the current block's DispatchKey
Key argumentThe schema string ("custom_add(Tensor self, Tensor other) -> Tensor")Op name ("custom_add") + function pointer (&my_kernel)
Schema validationThe first def locks the signature; later ones cannot change itThe kernel's signature must be compatible with the parsed schema (otherwise registration aborts)
Can it also attach a kernelYes (m.def("schema", &kernel)), but this kernel has no DispatchKey and is treated as "catch-all" (all keys go through it)By design, attaches a kernel for the specific key

Three common m.impl forms:

// (a) raw function pointer — framework infers the function signature
m.impl("op_name", &cpu_kernel);

// (b) TORCH_FN wrapper — bakes the signature into a compile-time type, enabling the faster unboxed call path
m.impl("op_name", TORCH_FN(cpu_kernel));

// (c) Force boxed (for fallback, interpreter, and other scenarios where the signature isn't known in advance)
m.impl("op_name", torch::CppFunction::makeFromBoxedFunction<&boxed_fn>());
Example 3.5: schema type incompatibility You declare: 
m.def("custom_add(Tensor self, Tensor other) -> Tensor");
Then register a kernel: 
at::Tensor wrong_kernel(const at::Tensor& self, int other) { ... }
m.impl("custom_add", TORCH_FN(wrong_kernel));   // type mismatch!
The program aborts immediately at startup (when the static initializer runs), with an error roughly like: 
Tried to register a kernel for operator myops::custom_add with type
Tensor(const Tensor&, int) — but the schema declares type Tensor(Tensor, Tensor).
This is the cornerstone of PyTorch's safety design: the schema is the single source of truth; every kernel must align with it.

3.6 Boxed vs unboxed call paths

In §1.5 we said the dispatcher's table lookup returns a KernelFunction, which is then called. The KernelFunction internally stores two function pointers at once: one unboxed, one boxed. The dispatcher automatically picks which path based on whether the caller knows the signature.

UnboxedBoxed
Function signatureNative C++: Tensor (*)(const Tensor&, const Tensor&, Scalar)Uniform: void (*)(const OperatorHandle&, DispatchKeySet, Stack*)
Argument passingTemplates expand directly, zero-copy, no type erasureAll arguments packed as IValues pushed onto the Stack (std::vector<IValue>); the kernel pops them itself
Performance~5 to 10 ns per call~50 to 100 ns extra (boxing/unboxing overhead)
When usedUser code, codegen-emitted wrappers (call site knows the signature)Fallback kernels, functorch interpreter, JIT runtime, profiler hooks (op-agnostic scenarios)

TORCH_FN(fn) uses templates to extract fn's literal type signature (at compile time), enabling KernelFunction to simultaneously store the unboxed path (direct call) and the boxed path (auto-generated IValue-wrapped version). If you write m.impl("name", &fn) (no TORCH_FN), the framework can still infer the signature via SFINAE, but the TORCH_FN form is more explicit, and signature errors can be detected at compile time.

Aside 3.6: the history of m.impl_UNBOXED Early PyTorch (pre-2019) had an m.impl_UNBOXED("name", &fn) API, meaning "this kernel can only be boxed-called; don't try to generate the unboxed path." This existed because some kernel signatures couldn't be deduced by templates (e.g. early versions of c10::optional<Tensor>).

This API has since been removed. The equivalent is m.impl("name", torch::CppFunction::makeFromBoxedFunction<&fn>()), explicitly saying "wrap with this boxed function, don't generate the unboxed path." When you see impl_UNBOXED in pre-2020 blog posts, knowing what it maps to today is enough.

3.7 Fallback and fallthrough

Beyond m.def and m.impl, Library also has m.fallback. Its meaning is: register an op-agnostic kernel for the entire DispatchKey; any op without its own impl under that key falls through to this one.

Two common forms:

m.fallback(torch::CppFunction::makeFallthrough());
    // "this layer does nothing; clear my bit from the bitmap and proceed to the next layer"
    // O(1) implementation: the dispatcher sees fallthrough and permanently clears this key from fallthrough_mask

m.fallback(torch::CppFunction::makeFromBoxedFunction<&my_boxed_fallback>());
    // Actually write a boxed kernel as a fallback, e.g. functorch's vmap interpreter

Fallthrough is the key to the dispatcher's efficiency. It is not "skip at runtime"; it is "tell the dispatcher at registration time that this key is meaningless here; from now on, mask this bit out of the bitmap." Cost = 0.

Typical fallthrough users:

  • BackendSelect: most ops don't need it (only factory functions like torch.randn use it). For all other ops, BackendSelect is fallthrough.
  • Named: fully fallthrough when named tensors aren't enabled.
  • Conjugate / Negative / ZeroTensor: fully fallthrough when their lazy markers aren't used.
  • Tracer: fallthrough when not in JIT trace mode.

This is why the dispatcher "supports dozens of functionality layers" on paper while in practice 99% of op calls activate only two or three (backend + maybe autograd), with every other layer fallthrough; after the bitmap is computed, those bits aren't even set.

Design philosophy: the dispatcher's extensibility comes from "you don't pay for what you don't need." Declaring fallthrough at registration → that layer never slows down your call. When you do need a layer (turn on autocast, enable vmap), only that layer activates, others remain fallthrough. This is qualitatively different from "an if/else chain that walks through every check on every call."

3.8 A complete custom op walkthrough

Now let's stitch §3.1 through §3.7 together with a complete custom operator. This is a "custom relu" with implementations on both CPU and CUDA, plus autograd support. It involves four .cpp files:

// ============ myops_schema.cpp ============
// (1) Use TORCH_LIBRARY to open DEF for the myops namespace and declare the schema once
#include <torch/library.h>

TORCH_LIBRARY(myops, m) {
  m.def("custom_relu(Tensor self) -> Tensor");
}

This file provides two things: ① the namespace myops is opened (DEF); ② the operator custom_relu's schema is entered into the dispatcher table (but no kernel yet). Calling torch.ops.myops.custom_relu(x) from Python at this point reports "kernel not found."

// ============ myops_cpu.cpp ============
// (2) CPU implementation + registration to (myops, CPU)
#include <ATen/ATen.h>
#include <torch/library.h>

at::Tensor custom_relu_cpu(const at::Tensor& self) {
  return at::clamp_min(self, 0);   // For simplicity, borrow at::clamp_min as the implementation
}

TORCH_LIBRARY_IMPL(myops, CPU, m) {
  m.impl("custom_relu", TORCH_FN(custom_relu_cpu));
}

This file: attaches a concrete kernel for myops::custom_relu under the CPU key (= the two-dimensional (Dense, CPUBit)). TORCH_FN exposes the unboxed signature to the dispatcher for fast runtime. Note the implementation uses at::clamp_min, itself a dispatched op that internally re-enters the dispatcher to call CPU's clamp_min.

// ============ myops_cuda.cu ============
// (3) CUDA implementation + registration to (myops, CUDA)
#include <ATen/ATen.h>
#include <torch/library.h>

at::Tensor custom_relu_cuda(const at::Tensor& self) {
  return at::clamp_min(self, 0);   // Same as above — clamp_min has its own CUDA kernel
}

TORCH_LIBRARY_IMPL(myops, CUDA, m) {
  m.impl("custom_relu", TORCH_FN(custom_relu_cuda));
}

Symmetrically attaches a kernel to the CUDA key. Now both backend entries for myops::custom_relu in the dispatcher table have kernels.

// ============ myops_autograd.cpp ============
// (4) Autograd implementation: capture forward, build grad_fn, pass forward math through to backend
#include <torch/autograd.h>
#include <torch/library.h>

class CustomReluFn : public torch::autograd::Function<CustomReluFn> {
 public:
  static at::Tensor forward(torch::autograd::AutogradContext* ctx,
                            const at::Tensor& self) {
    // On entering forward, manually exclude "AutogradXxx + ADInplaceOrView" keys — otherwise
    // the dispatcher call below would land back in this Function, looping forever.
    at::AutoDispatchBelowADInplaceOrView guard;
    // Inside the guard, AutogradCUDA has been removed from the TLS bitmap,
    // so the dispatcher call falls to CUDA or CPU (depending on self.device()).
    static auto op = c10::Dispatcher::singleton()
        .findSchemaOrThrow("myops::custom_relu", "")
        .typed<at::Tensor(const at::Tensor&)>();
    auto result = op.call(self);
    ctx->save_for_backward({result});
    return result;
  }

  static torch::autograd::variable_list backward(
      torch::autograd::AutogradContext* ctx,
      torch::autograd::variable_list grad_outputs) {
    auto saved = ctx->get_saved_variables();
    auto& result = saved[0];
    // d/dx relu(x) = 1 if x > 0 else 0; but we saved result, equivalent to (result > 0)
    return { grad_outputs[0] * (result > 0).to(grad_outputs[0].dtype()) };
  }
};

at::Tensor custom_relu_autograd(const at::Tensor& self) {
  return CustomReluFn::apply(self);
}

TORCH_LIBRARY_IMPL(myops, Autograd, m) {   // Autograd is an alias key, expanded to all AutogradXxx
  m.impl("custom_relu", &custom_relu_autograd);
}

This file is the key one; a few details:

  • torch::autograd::Function: a PyTorch-provided CRTP template base class; subclasses define forward and backward. apply() is the base-class-generated entry that automatically handles grad_fn construction, next_edges collection, and context management.
  • at::AutoDispatchBelowADInplaceOrView guard;: an RAII object that pushes all autograd functionality bits + the ADInplaceOrView bit into TLS exclude. Throughout the forward body, the dispatcher won't revisit the AutogradCUDA layer.
  • Use c10::Dispatcher::singleton().findSchemaOrThrow to look up the schema, then typed<...>().call(...) to run it. This is PyTorch's recommended form for "calling custom ops in C++", equivalent to Python's torch.ops.myops.custom_relu(self).
  • Register to the Autograd alias: rather than specific AutogradCPU/AutogradCUDA, use the alias Autograd. It auto-expands to every concrete autograd key, covering autograd cases for all backends in one line.

With the four files linked together, the Python side:

>>> import torch, my_extension
>>> x = torch.randn(3, requires_grad=True, device='cuda')
>>> y = torch.ops.myops.custom_relu(x)
    # dispatcher computes bitmap → highest bit = AutogradCUDA → jump to custom_relu_autograd
    # → CustomReluFn::apply → forward()
    # → AutoDispatchBelowADInplaceOrView guard takes effect
    # → op.call(self) (myops::custom_relu) → highest bit now = CUDA → jump to custom_relu_cuda
    # → at::clamp_min(self, 0) → runs clamp_min's CUDA kernel → returns
>>> y.sum().backward()
    # autograd engine calls CustomReluFn::backward(...)
    # → computes gradient → passes through next_edges
A more economical form: CompositeImplicitAutograd. Since custom_relu's implementation is entirely a composition of at::* ops (at::clamp_min), and its backward can be derived entirely from clamp_min's own backward, you can register directly to the alias CompositeImplicitAutograd and cover all backends and all autograd at once: 
at::Tensor custom_relu_composite(const at::Tensor& self) {
  return at::clamp_min(self, 0);
}

TORCH_LIBRARY_IMPL(myops, CompositeImplicitAutograd, m) {
  m.impl("custom_relu", &custom_relu_composite);
}
Just one .cpp file (none of the three previous IMPL blocks are needed), because the inner at::clamp_min goes through the dispatcher itself, gaining both backend routing and autograd benefits. For operators "built from existing ops," this is the preferred form. Only when your implementation is truly "native" (directly writing a CUDA kernel, not an at::* composition) do you need the full four-file pattern of §3.8.

4. Python registration: torch.library and register_fake

Chapter 3 covered the C++-side TORCH_LIBRARY macros. The same capabilities are exposed in Python through torch.library, all of it backed by the same c10::Dispatcher registry. There is no "Python op vs C++ op" distinction; the only difference is the language at the entry point. This chapter maps each Python API to its C++ macro counterpart and then dives into register_fake, the Python-only entry point that the meta key needs for Dynamo, export, Inductor, and AOTI.

4.1 Two Python APIs: Library and custom_op

PyTorch ships two Python registration APIs; the newer one is syntactic sugar over the older one:

4.4 The Meta dispatcher: register_fake and torch.export

Meta is a special backend key in the dispatcher table, on the same level as CPU / CUDA, representing "a tensor that carries metadata (shape, stride, dtype, device) but no storage, and never actually computes." This section first locates Meta in the dispatcher table, then explains how register_fake hooks into it, and finally focuses on its role in torch.export. Almost the entire export path rides on it.

4.4.1 Where Meta sits in the dispatcher table

Following the DispatchKey table from §1.3, Meta is in the same row as CPU / CUDA / XPU, a backend key with value roughly DispatchKey::Meta. It is normal for one op to carry CPU, CUDA, and Meta entries side by side: when a meta tensor passes through the dispatcher, the computed DispatchKeySet's backend bit is Meta, and routing lands on this entry.

aten::add.Tensor kernel table inside OperatorEntry (excerpt):

  DispatchKey::CPU             -> at::native::add_kernel (CPU vectorized)
  DispatchKey::CUDA            -> at::native::add_kernel_cuda
  DispatchKey::Meta            -> at::native::add_meta       # shape inference only
  DispatchKey::AutogradCPU     -> AutogradAdd::apply
  DispatchKey::AutogradCUDA    -> AutogradAdd::apply
  ...

For native operators (aten::*), most Meta kernels are auto-generated by codegen (torchgen, see §5). The logic is straightforward: run TensorIterator once to figure out the broadcast shape and return a meta tensor. Custom operators do not get this codegen path, so the user must write it by hand. That is the entry point of register_fake.

4.4.2 FakeTensor: a thin wrapper above the Meta key

What export / Inductor / AOTI actually use is not a bare meta tensor but FakeTensor (torch._subclasses.fake_tensor.FakeTensor), a __torch_dispatch__ subclass of meta tensor. The relationship:

Bare meta tensorFakeTensor
Devicedevice='meta'Preserves the original device (cuda:0 / cpu / etc.), storage stays empty

Key insight: the dispatcher path inside FakeTensor still computes shape via the Meta kernel. FakeTensor's __torch_dispatch__ "lowers" input tensors to meta (preserving device info), calls the op, gets back a meta result, then "lifts" it back to FakeTensor (re-injecting device info). So any op with a Meta kernel is automatically FakeTensor-compatible, and conversely, register_fake registers the Meta kernel.

4.4.3 What register_fake actually registers

register_fake(fn) attaches a Meta entry in the dispatcher table:

@scale_add.register_fake
def _(x, scale, bias):
    return torch.empty_like(x)

# Internally equivalent to:
torch.library.impl("myops::scale_add", "Meta")(_fn)
# Which is the C++ equivalent of:
// TORCH_LIBRARY_IMPL(myops, Meta, m) { m.impl("scale_add", &scale_add_meta); }

The reason the API is named register_fake rather than register_meta is historical: earlier APIs went through impl_abstract / register_meta, and the 2.4 custom_op family renamed everything to register_fake to emphasize "the shape inference function FakeTensor uses," closer to the user's perspective. The slot in the dispatcher table is still the Meta key; only the Python name changed.

4.4.4 The role of Meta in torch.export: the entire path rides on it

torch.export(model, example_inputs) runs no real numerics internally. It runs one pass with FakeTensors. Where Meta sits along that path:

torch.export(model, example_inputs)
   │
   ├─ 1. Create matching FakeTensors from example_inputs (preserve shape / dtype / device)
   │
   ├─ 2. Enter FakeTensorMode context
   │
   ├─ 3. Dynamo + symbolic_shapes wrap variable dimensions as SymInt
   │     (e.g. batch dim -> s0, not the concrete 8)
   │
   ├─ 4. AOT autograd trace: every op of the model is run once with FakeTensor
   │     ├─ Op enters dispatcher
   │     ├─ FakeTensor's __torch_dispatch__ lowers inputs to meta (keeping device)
   │     ├─ Dispatcher computes DispatchKeySet: meta tensor -> Meta key
   │     ├─ Calls the matching Meta kernel: aten::* is codegen, custom op is register_fake
   │     ├─ Meta kernel returns a meta tensor (shape/dtype only)
   │     └─ FakeTensor wraps it back
   │
   ├─ 5. Collect every op's Meta kernel output to build ExportedProgram.graph
   │     (an FX graph where every node carries val: a FakeTensor-shaped annotation)
   │
   └─ 6. Serialize the graph into .pt2 / feed to Inductor / feed to AOTI

A few facts fall out of this path:

  • Export never invokes a real kernel. Model weights never reach the GPU during export (they are FakeTensors too). This is why export is orders of magnitude faster than a trace.
  • Shape inference correctness is 100% owned by Meta kernels. A wrong Meta kernel for one op makes the entire downstream graph wrong: Inductor tiles by the wrong shape, AOTI compiles cubins for the wrong shape, runtime hits a wall of mismatched dims.
  • A Meta kernel must accept SymInt inputs. Export marks the batch dim as s0 (SymInt); the Meta kernel receives a SymInt. A concrete branch like if x.size(0) == 8: raises GuardOnDataDependentSymNode on a SymInt.
  • A custom op without a Meta kernel breaks export. The dispatcher reaches Meta and finds no entry; depending on fallback rules the result is either NotImplementedError: ... operator does not have a meta kernel (2.4+) or, on older versions, a fallthrough to the real kernel, which forces export to compute numerics (slow, and incompatible with dynamic shapes).

4.4.5 register_fake's hard constraints (violations break export)

  • Do not call real aten::* ops (x.mul(2) / torch.add(x, y)): those re-enter the dispatcher, land on Meta again, and either recurse or hit the wrong branch. Use Python operators (x * 2) or construct tensors directly (torch.empty_like(x) / torch.empty(shape, dtype=, device=)).
  • Shape inference must work over SymInts:
    • No concrete checks like if x.size(0) == 8:
    • Use torch.sym_int or SymInt-friendly forms like x.shape[0] + 1 for shape arithmetic
    • If a branch unavoidably depends on a concrete value, use torch._check(...) to register the guard with Dynamo
  • No side effects: no file writes, no global state mutation, no print. Export retraces, and side effects fire repeatedly.
  • Output device must match input device: derive output device from the inputs (device=x.device); hard-coding cuda:0 makes the exported graph non-portable across devices.
  • dtype must follow real promotion rules: use torch.promote_types to match the real kernel; defaulting to x.dtype leads to dtype mismatch at runtime.

4.4.6 Common failure modes

# ❌ Case 1: calling a real op inside a fake
@my_op.register_fake
def _(x):
    return x.mul(2)              # dispatches back to Meta — recurses or hits wrong branch

# ✓ Fix:
def _(x):
    return torch.empty_like(x)

# ❌ Case 2: concrete check colliding with SymInt
@my_op.register_fake
def _(x):
    if x.size(0) % 8 == 0:    # SymInt % 8 -> SymBool, if forces a guard, export warns/errors
        ...

# ✓ Fix:
def _(x):
    torch._check(x.size(0) % 8 == 0)
    return torch.empty_like(x)

# ❌ Case 3: hard-coded device
@my_op.register_fake
def _(x):
    return torch.empty(x.shape, device="cuda")  # always cuda; exported graph cannot run on cpu

# ✓ Fix:
def _(x):
    return torch.empty(x.shape, dtype=x.dtype, device=x.device)

# ❌ Case 4: wrong shape on a multi-output op
@split.register_fake
def _(x, n):
    return [torch.empty_like(x) for _ in range(n)]   # each shard should be x.shape[0] // n

# ✓ Fix:
def _(x, n):
    chunk = x.size(0) // n
    return [torch.empty((chunk,) + x.shape[1:], dtype=x.dtype, device=x.device)
            for _ in range(n)]

4.5 Python → dispatcher: how the call actually reaches C++

The Python APIs are wrappers over the same c10::Dispatcher registry the C++ macros use. The full call stack:

torch.library.Library("myops", "DEF")
   ↓ # torch/library.py
torch._C._dispatch_library("DEF", "myops", ...)
   ↓ # torch/csrc/utils/python_dispatch.cpp, inside libtorch_python.so
torch::Library lib(...)              // C++ Library object
   ↓
c10::Dispatcher::singleton()
       .registerName(c10::OperatorName{"myops", ""})    // register namespace

lib.define("scale_add(Tensor x, float scale, Tensor bias) -> Tensor")
   ↓
torch._C._dispatch_def(lib_handle, schema_str)
   ↓
torch::jit::parseSchema(schema_str)             // parse into FunctionSchema
   ↓
c10::Dispatcher::singleton().registerDef(schema, ...)

op.register_kernel("cuda")(fn_py)
   ↓
torch._C._dispatch_impl(lib_handle, "scale_add", "CUDA", boxed_python_fn)
   ↓
c10::Dispatcher::singleton().registerImpl(
   OperatorName{"myops", "scale_add"},
   c10::DispatchKey::CUDA,
   KernelFunction::makeFromBoxedFunction(...));  // fn_py reached indirectly via PythonFallbackKernel

Two consequences:

  • The Python fn is not hung directly off the dispatch table. It is wrapped in a PyObject holder; PythonFallbackKernel bounces calls back into the Python interpreter. This is exactly the C++↔Python bidirectional dispatch described in [[pytorch-torch-dispatch-mode]].
  • A Python-registered op pays two extra dispatcher hops plus a GIL acquisition compared to a pure C++ registration. For inference-critical paths, rewrite the op in C++.

4.6 Full Python example: dual of §3.8

The C++ version of myops::custom_relu in §3.8 spans four files. The Python equivalent is one .py file:

import torch

## 1) Schema + default implementation (custom_op rolls them together)
@torch.library.custom_op("myops::custom_relu", mutates_args=())
def custom_relu(x: torch.Tensor) -> torch.Tensor:
    # Default: shared CPU + CUDA implementation
    return x.clamp(min=0)

## 2) Register a CUDA-specialised implementation (overrides default)
@custom_relu.register_kernel("cuda")
def _(x: torch.Tensor) -> torch.Tensor:
    # Could dispatch into a Triton kernel or a cuBLAS / cuDNN wrapper here
    return x.relu()                          # illustrative

## 3) register_fake: lets export / AOTI infer shape
@custom_relu.register_fake
def _(x):
    return torch.empty_like(x)

## 4) register_autograd: provide a backward (optional, needed for training)
def _setup_ctx(ctx, inputs, output):
    (x,) = inputs
    ctx.save_for_backward(x)

def _backward(ctx, grad):
    (x,) = ctx.saved_tensors
    return grad * (x > 0).to(grad.dtype),    # tuple matching the inputs

custom_relu.register_autograd(_backward, setup_context=_setup_ctx)

## Usage, identical to the C++-registered version:
y = torch.ops.myops.custom_relu(torch.randn(3, device="cuda", requires_grad=True))
y.sum().backward()

Trade-offs versus the four-file C++ version in §3.8:

  • Development cycle is shorter: no nvcc, no setup.py, no compilation; edit and run.
  • Aligned with export / FakeTensor: custom_op gives the op a stable schema, while register_fake supplies shape / dtype / device inference for abstract execution.

Costs:

  • Per-call overhead is high (Python → dispatcher → PythonFallbackKernel → Python fn).
  • For torch.compile / torch.export, the op is opaque by default: the compiler does not inspect the Python body and a fake impl does not automatically create an Inductor lowering.
  • Cannot run outside a Python runtime; AOTI deployment needs an extra C++ shim (see [[aoti-cuda-deployment]] §9.4).

4.7 torch.library.opcheck: automatic invariant verification

After registering a custom op, run opcheck:

torch.library.opcheck(
    custom_relu,
    args=(torch.randn(3, 4, device="cuda"),),
    test_utils=("test_schema", "test_faketensor", "test_aot_dispatch_static", "test_aot_dispatch_dynamic"),
)

The call exercises a suite of invariants:

  • test_schema: the schema string matches the implementation's signature (parameter names, types, optionality).
  • test_faketensor: the fake impl's output FakeTensor agrees with the real impl in shape / dtype / device.
  • test_aot_dispatch_static: a full pass through AOT autograd in static shape mode.
  • test_aot_dispatch_dynamic: same with dynamic shapes.
  • (optional) test_autograd_registration / gradcheck: numerically validates the backward.

Custom ops that fail opcheck tend to misbehave during export, Inductor, or AOTI in subtle ways; running it upfront removes a lot of downstream debugging.

6. End-to-end trace: aten::add on CUDA

The previous four chapters built up the dispatcher's conceptual model (§1), the schema's interface contract (§2), the registration macros (§3), and build-time codegen (§4). This chapter stitches them together by picking one concrete scenario and walking through the entire call chain:

Setting: two CUDA tensors x, y, both requires_grad=True. The user writes: 
z = torch.add(x, y)
z.sum().backward()
We want to answer: what C++ / CUDA functions actually run for these two lines of Python?

The call chain enters from Python, descends layer by layer along the dispatcher's priority ladder, and lands at the CUDA elementwise kernel; backward then returns to the autograd engine to call AddBackward0::apply. The nine subsections below proceed in order; each pins one layer, clarifies what it does, which generated artifact it corresponds to, which source line is involved, and what to watch for.

6.1 Starting point: two lines of yaml + the backward formula

The source of the entire call chain is a handful of yaml lines. They determine what the downstream generated artifacts look like. Let's copy them once for easy reference in every later subsection.

The add.Tensor (functional variant) and add.out (structured anchor) entries in native_functions.yaml (simplified):

# native_functions.yaml
- func: add.Tensor(Tensor self, Tensor other, *, Scalar alpha=1) -> Tensor
  device_check: NoCheck
  structured_delegate: add.out      # Doesn't implement itself; delegates to add.out (used in §5.6)
  variants: function, method        # Generates two entries: at::add and Tensor::add
  ...

- func: add.out(Tensor self, Tensor other, *, Scalar alpha=1, Tensor(a!) out) -> Tensor(a!)
  structured: True                  # The "exit" of the structured kernel (used in §5.7)
  structured_inherits: TensorIteratorBase
  ufunc_inner_loop:                 # Triggers ufunc codegen (used in §5.7)
    Generic: add (AllAndComplex, BFloat16, Half, ComplexHalf)

Just these few yaml lines, plus a single math definition in aten/src/ATen/native/ufunc/add.h, constitute the full "source material" for aten::add. All subsequent C++ functions are generated by torchgen from these lines.

6.2 Python → C binding layer

The user writes torch.add(x, y). The CPython interpreter finds torch.add as a C function pointer and jumps to THPVariable_add (note the "THP" prefix: Torch HPython, PyTorch's naming convention for all Python C bindings). Its source is generated at:

build/.../torch/csrc/autograd/generated/python_torch_functions_2.cpp

_2 is the shard number (the entire file is too large, so torchgen splits it into several shards). The body of THPVariable_add roughly does these four things:

  1. Parse Python arguments. Use PythonArgParser to parse per the schema (the string from §2.1), identifying the first tensor as self, the second as other, with optional kw alpha defaulting to 1.
  2. Unwrap the Python wrapper. The user's x is actually a Python-side THPVariable object (torch.Tensor) holding an at::Tensor (the C++ tensor). THPVariable_unpack extracts the C++ tensor.
  3. Call the C++ API. With pure C++ tensors in hand, call at::add(self, other, alpha). Control leaves Python and enters the pure C++ world.
  4. Wrap the return back to Python. The C++-returned at::Tensor is wrapped by THPVariable_Wrap into a new torch.Tensor and returned to Python.
Aside 5.2: why we need this wrap/unwrap layer You may ask: since torch.Tensor and at::Tensor are essentially the same object, why not pass it directly?

Two reasons:
① Python reference counting vs C++ smart pointers. Python's torch.Tensor uses CPython refcounting (Py_INCREF/Py_DECREF); C++'s at::Tensor uses intrusive_ptr. The two lifetime-management systems must be converted precisely at the boundary, or you get use-after-free.
② Parsing polymorphic Python arguments. A Python user can pass either (Tensor, Tensor) or (Tensor, Scalar) to the same torch.add; which overload (add.Tensor vs add.Scalar) to pick is decided by this layer. The C++ function itself accepts only static signatures, so Python-side overload resolution must happen first.

6.3 C++ API and Dispatcher::call

Entering the C++ world, the first landing point is at::add (generated in build/aten/src/ATen/Functions.h):

// build/aten/src/ATen/Functions.h
inline at::Tensor add(const at::Tensor& self, const at::Tensor& other, const at::Scalar& alpha) {
  return at::_ops::add_Tensor::call(self, other, alpha);
}

This is just an inline forwarder, delegating to at::_ops::add_Tensor::call (around line 612 of Operators.cpp):

// build/aten/src/ATen/Operators.cpp
at::Tensor _ops::add_Tensor::call(const at::Tensor& self, const at::Tensor& other, const at::Scalar& alpha) {
  static auto op = create_add_Tensor_typed_handle();   // Process-wide TypedOperatorHandle computed once
  return op.call(self, other, alpha);                  // Hand off to Dispatcher::call
}

This layer's job is "find the op's TypedOperatorHandle". TypedOperatorHandle is the dispatcher's internal "handle": it points to the row for aten::add.Tensor in the dispatcher's table. The static ensures this lookup runs only on the first call; subsequent calls hit the cache.

op.call eventually invokes c10::Dispatcher::call (Dispatcher.h:774-828), the dispatcher entry from §1.4. From this point on, we officially step into the dispatcher's bitmap magic.

Example 5.3: why this layer exists separately You may ask: since at::add is just a forwarder, why not have THPVariable_add call Dispatcher::call directly?

Two reasons:
① It provides a user-friendly C++ API. Third-party C++ code doesn't want to know dispatcher internals: it just wants to write at::add(x, y).
② It hides the schema-lookup cache. at::_ops::add_Tensor::call looks up the schema on the first call (create_add_Tensor_typed_handle()) and caches it. If every call went straight to Dispatcher::call and had to look up the schema each time, performance would degrade significantly.

6.4 On-the-fly DispatchKeySet computation

Entering Dispatcher::call (Dispatcher.h:774-828), the core is two lines:

// Core of Dispatcher::call
auto dispatchKeySet =
    op.operatorDef_->op.dispatchKeyExtractor()
        .template getDispatchKeySetUnboxed<Args...>(args...);
const KernelFunction& kernel = op.operatorDef_->op.lookup(dispatchKeySet);
return kernel.template call<Return, Args...>(op, dispatchKeySet, ...);

The first line computes the bitmap (the DispatchKeySet from §1.5), the second looks up the kernel for the highest bit, the third jumps. For our call, concretely:

  1. Initial bitmap value: each input tensor's TensorImpl::key_set_ (self, other). Both are CUDA tensors that require grad, so key_set_ = {AutogradFunctionality, Dense, CUDABit} (using §1.4's two-dim split). Or'ing the two tensors keeps the same three bits set. The runtime translation is {AutogradCUDA, CUDA}.
  2. TLS local include: autocast off, not in functorch, so include is empty.
  3. TLS local exclude: first time entering the dispatcher, so exclude is empty too.
  4. Fallthrough mask: each op's dispatch table precomputed its fallthrough_mask at registration. For add, bits like BackendSelect, Named, Conjugate are masked out. The bitmap stays at {AutogradCUDA, CUDA}.
  5. highestPriorityTypeId(): among the remaining bits, the highest functionality bit is AutogradFunctionality (corresponding to runtime AutogradCUDA), since it sits after Dense in the enum (a higher bit).

lookup returns the kernel function pointer in the AutogradCUDA slot of aten::add.Tensor. Who registered this kernel? Codegen registered it automatically:

  ...

  ...
}

Jump in, and we enter VariableType::add_Tensor.

6.5 AutogradCUDA kernel: build graph and redispatch

VariableType::add_Tensor's body is codegen-generated (based on add.Tensor's backward formula in derivatives.yaml + a standard template). Simplified skeleton:

at::Tensor VariableType::add_Tensor(
    c10::DispatchKeySet ks,
    const at::Tensor& self, const at::Tensor& other, const at::Scalar& alpha) {

  // === Step 1) If any input requires_grad, prepare a fresh grad_fn backward node ===
  std::shared_ptr<AddBackward0> grad_fn;
  if (compute_requires_grad(self, other)) {
    grad_fn = std::shared_ptr<AddBackward0>(new AddBackward0(), deleteNode);
    grad_fn->set_next_edges(collect_next_edges(self, other));   // backward edges
    grad_fn->alpha = alpha;                                      // stored for the backward formula
    grad_fn->other_scalar_type = other.scalar_type();
    grad_fn->self_scalar_type  = self.scalar_type();
  }

  // === Step 2) Remove "AutogradCUDA + ADInplaceOrView" from the bitmap, then redispatch ===
  auto _tmp = ([&]() {
    at::AutoDispatchBelowADInplaceOrView guard;     // RAII: enter guard = add autograd + ADInplaceOrView to TLS exclude
    return at::redispatch::add(
        ks & c10::after_autograd_keyset,             // explicitly mask the current layer and above
        self, other, alpha);
  })();
  auto result = std::move(_tmp);

  // === Step 3) Attach the freshly computed result to grad_fn, push into autograd history ===
  if (grad_fn) {
    set_history(flatten_tensor_args(result), grad_fn);   // result.grad_fn = grad_fn
  }
  return result;
}

Line by line:

  1. compute_requires_grad(self, other): checks whether any input tensor requires_grad and we're not under torch.no_grad(). In our scenario both inputs require grad → true.
  2. Build the AddBackward0 node: this is a codegen-generated Node subclass (its apply() shown in §5.8). set_next_edges sets self.grad_fn and other.grad_fn as the downstream edges of AddBackward0; backward propagates gradients out along these two edges. alpha, self_scalar_type, other_scalar_type are the "context" needed by the backward formula, stored on the node.
  3. at::AutoDispatchBelowADInplaceOrView guard;: an RAII object that on construction modifies TLS, pushing all autograd functionality bits + the ADInplaceOrView bit into the exclude set. From this line until guard destruction, no dispatcher call goes through the autograd layer.
  4. at::redispatch::add(ks & c10::after_autograd_keyset, ...): at::redispatch::add is another codegen-generated entry (in RedispatchFunctions.h), differing from at::add only in taking an extra DispatchKeySet argument that it forwards to Dispatcher::redispatch. after_autograd_keyset is a precompiled constant mask that "clears all autograd-and-above bits." Equivalently, it explicitly tells the dispatcher: jump to backend next.
  5. set_history(result, grad_fn): sets the freshly computed result tensor's .grad_fn field to the AddBackward0 node we built. This way result.backward() will find this node via the autograd engine.
The key concept of this section: the autograd kernel does no arithmetic at all. It only does three things: ① prepare the backward node; ② hand control off to the backend kernel (via guard + redispatch); ③ attach the result to the backward graph. All numerical computation is done by the downstream CUDA kernel. This "wrapper layer" pattern fully decouples autograd from any specific backend; the same autograd wrapper works for CPU, CUDA, MPS, etc.

6.6 The CUDA backend wrapper

at::redispatch::add hands control to Dispatcher::redispatch. This time AutogradCUDA has been cleared from the bitmap (doubly insured by guard and after_autograd_keyset), and highestPriorityTypeId() returns CUDA.

The kernel registered to the CUDA slot lives in build/.../RegisterCUDA.cpp:

TORCH_LIBRARY_IMPL(aten, CUDA, m) {
  ...
  m.impl("add.Tensor", TORCH_FN(wrapper_CUDA_add_Tensor));
  m.impl("add.out",    TORCH_FN(wrapper_CUDA_add_out_out));
  ...
}

Jump to wrapper_CUDA_add_Tensor (note: this is not the real number-crunching CUDA kernel; just functional → out forwarding, because yaml declares structured_delegate: add.out):

at::Tensor wrapper_CUDA_add_Tensor(
    const at::Tensor& self, const at::Tensor& other, const at::Scalar& alpha) {
  // 1) Allocate a new output tensor (on CUDA, common dtype auto-promoted)
  auto common_dtype = at::native::result_type(self, other);
  auto out = at::detail::empty_cuda({0}, common_dtype, self.device());
  // 2) Forward to the out variant for the real computation
  return wrapper_CUDA_add_out_out(self, other, alpha, out);
}

So add.Tensor (functional)'s implementation = "allocate out + call add.out." This is exactly how structured_delegate: add.out materializes in codegen: the three variants share the same math, differing only in memory-allocation strategy.

Example 5.6: contrast with the in-place variant For add_.Tensor (in-place), codegen produces a similar wrapper, but without allocating a new out; instead, self is passed as out to add.out: 
at::Tensor& wrapper_CUDA_add__Tensor(
    at::Tensor& self, const at::Tensor& other, const at::Scalar& alpha) {
  return wrapper_CUDA_add_out_out(self, other, alpha, self);   // self as out
}
Three variants with three wrappers, but only one number-crunching impl in add.out. That's the core benefit of structured kernels.

6.7 Structured kernel and ufunc

wrapper_CUDA_add_out_out is where computation truly begins. Its body is also codegen-generated:

at::Tensor& wrapper_CUDA_add_out_out(
    const at::Tensor& self, const at::Tensor& other, const at::Scalar& alpha,
    at::Tensor& out) {
  structured_add_out_out op(out);            // instantiate the structured kernel class
  op.meta(self, other, alpha);               // Phase 1: meta — compute output shape/dtype, set up TensorIterator
  op.impl(self, other, alpha, op.maybe_get_output(0));   // Phase 2: impl — the real computation
  return out;
}

Structured kernels split "the algorithm" into two standardized phases:

  1. meta(): broadcasts shape, determines dtype, initializes the TensorIterator. The caller provides an output tensor (possibly empty or already allocated); meta decides the expected output shape from the inputs and resizes if needed.
  2. impl(): numerical computation. Reads the TensorIterator for element-wise iterators and runs the actual algorithm.

structured_add_out_out::impl's body is generated by UfuncCUDA_add.cu (the "structured + ufunc" artifact from §4.2):

// build/.../UfuncCUDA_add.cu (simplified)
void structured_add_out_out::impl(
    const Tensor& self, const Tensor& other, const Scalar& alpha, const Tensor& out) {
  AT_DISPATCH_ALL_TYPES_AND_COMPLEX_AND4(
      kBFloat16, kHalf, kComplexHalf, kBool,                      // supported dtype set
      iter.common_dtype(), "add_cuda",                            // current dtype + label
      [&] {                                                      // per-dtype instantiated lambda
        scalar_t alpha_val = alpha.to<scalar_t>();
        gpu_kernel(iter, [alpha_val] GPU_LAMBDA (scalar_t a, scalar_t b) -> scalar_t {
          return at::native::ufunc::add(a, b, alpha_val);         // the real math
        });
      });
}

There are two layers of dispatch here:

  1. AT_DISPATCH_ALL_TYPES_AND_COMPLEX_AND4: this is compile-time dispatch (aten/src/ATen/Dispatch.h), expanding into a switch (scalar_type) on iter.common_dtype() and instantiating the lambda once per dtype (float, double, int8, kHalf, ...). This layer does not enter the dispatcher table: dtype is not a DispatchKey, just a C++ template parameter.
  2. gpu_kernel(iter, lambda): this is launching a CUDA kernel (aten/src/ATen/native/cuda/CUDALoops.cuh). It takes a TensorIterator (describing the elementwise traversal plan) + a user lambda, and expands into vectorized_elementwise_kernel<...><<<blocks, threads>>>(...). Each CUDA thread runs the user lambda once.

And what does the "user lambda" do? It calls at::native::ufunc::add. This is the real math function, defined in ATen/native/ufunc/add.h, in four lines:

// aten/src/ATen/native/ufunc/add.h
template <typename T>
C10_HOST_DEVICE C10_ALWAYS_INLINE T add(T self, T other, T alpha) {
  return self + alpha * other;
}
For all the layers of code along the call stack, the real "computation of add" is this one line: self + alpha * other. Every other layer exists so this one line can be invoked at the right dtype, on the right backend, with backward recorded, composable with autocast/vmap and other modes. That is the essence of PyTorch's engineering effort.
Example 5.7: why CPU and CUDA share the same ufunc::add ufunc/add.h uses the C10_HOST_DEVICE annotation (PyTorch's wrapper for CUDA's __host__ __device__), telling nvcc: "compile this function for both CPU and GPU."

The CPU-side UfuncCPU_add.cpp includes the same ufunc/add.h, then drives it with SIMD-vectorized vectorized_loop2d rather than gpu_kernel. The math is written once; CPU/CUDA share the same definition. Adding a new backend (say MPS) needs only a new "loop driver" (how to run the lambda); the math function is unchanged.

Forward computation is now complete. Control returns layer by layer: the CUDA kernel returns → impl returns → wrapper_CUDA_add_out_out returns out → wrapper_CUDA_add_Tensor returns → Dispatcher::redispatch returns → at::redispatch::add returns to VariableType::add_Tensor. Then:

  1. The guard destructs; TLS exclude is restored (subsequent top-level calls compute a clean bitmap again).
  2. set_history(result, grad_fn) attaches the backward node to the result.
  3. Returns to top-level Dispatcher::call, then to at::_ops::add_Tensor::call, then to at::add, then to THPVariable_add, then is wrapped as a Python torch.Tensor and returned to the user.

Forward complete. z is in hand, with its .grad_fn set to AddBackward0.

6.8 Backward: AddBackward0::apply

The user now calls z.sum().backward(). backward() triggers the autograd engine (torch/csrc/autograd/engine.cpp):

  1. Starting from the loss's (the z.sum() scalar's) .grad_fn, topologically sort the whole backward graph.
  2. Traverse in reverse topological order, calling each Node's apply(grads) method.
  3. The gradients apply returns are passed via next_edges to upstream nodes for accumulation.

For our AddBackward0::apply, codegen produces (in build/.../generated/Functions.cpp):

variable_list AddBackward0::apply(variable_list&& grads) {
  auto& grad = grads[0];                                          // upstream-incoming dz
  variable_list grad_inputs(2);                                    // two inputs, two gradients

  if (should_compute_output(0)) {                                  // does self need a grad?
    grad_inputs[0] = handle_r_to_c(self_scalar_type, grad);
    // translated from derivatives.yaml:  self: handle_r_to_c(self.scalar_type(), grad)
  }

  if (should_compute_output(1)) {                                  // does other need a grad?
    grad_inputs[1] = handle_r_to_c(other_scalar_type,
                                   maybe_multiply(grad, alpha.conj()));
    // translated from derivatives.yaml:  other: handle_r_to_c(other.scalar_type(),
    //                                     maybe_multiply(grad, alpha.conj()))
  }

  return grad_inputs;
}

Compare with the backward formula copied in §5.1:

self:  handle_r_to_c(self.scalar_type(),  grad)
other: handle_r_to_c(other.scalar_type(), maybe_multiply(grad, alpha.conj()))

This is a character-by-character translation. Codegen turns the yaml expressions directly into C++ code. That is why the PyTorch team makes the "backward formula DSL" restricted: it must mechanically translate to C++.

Note a few details:

  • should_compute_output(i): the autograd engine knows which leaf tensors actually require grad. If other is a constant (doesn't require grad), grad_inputs[1] isn't computed at all (a performance optimization).
  • maybe_multiply(grad, alpha.conj()): maybe_multiply is a small optimization. When alpha == 1, it returns grad directly, skipping the float multiply. Codegen automatically inserts this "woodpecker."
  • handle_r_to_c: handles the "real tensor → complex grad" edge case (projects to the real part); for real arithmetic this is essentially a no-op.
  • No at::redispatch: sub-operations in the backward formula (e.g. at::mul inside maybe_multiply) directly use the top-level at::* API. They themselves traverse the full dispatcher, gaining "run on CUDA, record in backward graph" capabilities. This means the backward automatically supports higher-order derivatives: differentiating AddBackward0::apply a second time has the inner at::mul build a new MulBackward0.

After returning grad_inputs, the engine propagates them along next_edges to upstream (x.grad_fn and y.grad_fn, if any). If x, y are leaf tensors (no grad_fn), the gradients accumulate directly into x.grad, y.grad. Backward complete.
