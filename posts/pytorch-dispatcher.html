PyTorch Dispatcher 全栈剖析:从 schema 到 CUDA kernel

本文 PyTorch 源码以 pytorch/pytorch main 分支为准。所有 file:line 引用都可点开直接看到当前源。本文重点剖析 dispatcher 数据结构、注册宏的语义、以及一个 native op 从 Python 调到 CUDA 的完整调用栈;不涉及量化/分布式/inductor 等下游话题。

1. 调度键:运行时多维分发的基础

本章自下而上建立 dispatcher 的运行期模型。我们先从用户视角看"为什么 if/else 行不通"——这一节会自然地导出 dispatcher 的核心思想:把"该走哪条路"的决策从代码里抽出来,变成一张按 key 索引的查找表。然后我们看 PyTorch 用什么数据结构表达这张表(DispatchKey 枚举、DispatchKeySet 位图),怎么算出"激活了哪些 key",怎么决定"先走哪一层",以及一个 kernel 跑完后怎么不死循环地把控制权传给下一层。

读完本章,你应该能在脑子里精确模拟 z = x + y 在 dispatcher 内部走过的每一步。

1.1 为什么需要"调度"

我们从一个看起来再普通不过的 PyTorch 操作开始:

y = x + 1

如果 x 是一个普通的 CPU float32 张量、且不需要梯度,那这行代码只需要做一件事:跑一个简单的循环 y[i] = x[i] + 1。一个 if/else 足以判断"该走 CPU 加法还是 CUDA 加法":

// 玩具实现 —— PyTorch 没有这么写,后面解释为什么
Tensor add(const Tensor& x, Scalar s) {
  if (x.is_cuda()) return add_cuda(x, s);
  else            return add_cpu(x, s);
}

问题是,真实情况要复杂得多。同一个 x + 1,在不同的上下文中,内部要做截然不同的事:

  • 如果 x.requires_grad == True,加法之前必须先建一个 AddBackward0 节点,把 x 登记为它的"上游",方便日后 loss.backward() 时反向追溯。
  • 如果调用嵌在 with torch.autocast('cuda'): 里,加法之前要先把 fp32 输入转成 fp16,然后才算加法 —— 这样训练阶段才能拿到混合精度的吞吐收益。
  • 如果嵌在 with FakeTensorMode(): 里,根本不能真算 —— 只能"假装"算,推断输出 shape/dtype 并返回一个"形似而无内容"的张量(用于编译/分析阶段)。
  • 如果 x 是个稀疏张量(torch.sparse_coo_tensor(...)),加法走的是完全不同的算法 —— 不是 elementwise loop。
  • 如果 x 经过了 torch.func.vmap(...),加法要在额外一个 batch 维度上广播。

更要命的是,这些条件可以叠加。一个真实的训练 step 里完全可能同时打开 requires_grad、autocast、CUDA。每一种"上下文"都要按正确的顺序插入到这次调用里 —— autocast 必须在 autograd 之前(否则反向图记错了 dtype);autograd 必须在 CUDA 之前(否则记不到反向上游);FakeMode 必须在所有之上(否则就真算了)。

按 if/else 写,代码会长成下面这样,而且必须给每一个算子都写一遍:

Tensor add(const Tensor& x, Scalar s) {
  // FakeTensorMode 拦截:只算 meta,不算数值
  if (in_fake_mode()) return add_fake(x, s);

  // autocast:先降精度
  if (autocast_enabled() && x.is_cuda()) {
    x = x.to(at::kHalf);
  }

  // autograd:记反向
  if (x.requires_grad()) {
    register_backward<AddBackward0>(x);
  }

  // 真的算:按 layout × device 分支
  if (x.is_sparse()) return add_sparse(x, s);
  if (x.is_cuda())   return add_cuda(x, s);
  return add_cpu(x, s);
}

这种写法有两个致命缺陷,而且 PyTorch 团队没办法绕过:

  1. 重复。ATen(PyTorch 的算子库)有 ~2000 个算子。每一个都要把这段嵌套 if/else 复制一遍。任何一处忘记 if (autocast_enabled()),这个算子就会在 autocast 下静默走错精度;任何一处少写 if (in_fake_mode()),这个算子在编译 trace 阶段会真的去算 —— 失败模式难抓到极点。
  2. 封闭。如果今天 Habana / Apple Metal / 华为昇腾要给 PyTorch 加一个新后端,它们不能修改 PyTorch 源码 —— 必须能从外部"插入"自己的 is_my_backend() 分支。if/else 链做不到这一点。同理,第三方的 autocast 实现、JIT tracer、新型变换(vmap/grad/jvp)也都没法横切到 PyTorch 算子里。
小例 1.1 想象 PyTorch 想支持一个新后端 Foo。在 if/else 方案下,你要去把所有 ~2000 个算子的源码翻一遍、补上 if (x.is_foo()) return add_foo(...)。然后下次 PyTorch 升级到新版本,你的所有补丁要重做。每次合并冲突都是噩梦。这就是 if/else 方案在工程上立不住的核心原因 —— 不是"丑",而是"扩展性破产"。

1.2 思路:按 key 索引的函数表

PyTorch 的解法是把"该走哪条路"的决策从代码里抽出来,放进一张查找表。

核心机制。每个算子(例如 aten::add)在 PyTorch 启动后都拥有一张函数指针表。表的下标叫做调度键(DispatchKey),代表"一种行为层":CUDA 后端、CPU 后端、Autograd 包装、Autocast 包装、FakeMode 拦截 …… 每一项都是一种可能的"该做什么"。

例如,aten::add 的查找表大致长这样(简化,真实条目更多):

DispatchKey          →  函数指针 (kernel)
─────────────────────────────────────────────────
CPU                  →  add_cpu
CUDA                 →  add_cuda
SparseCPU            →  add_sparse_cpu
SparseCUDA           →  add_sparse_cuda
MPS                  →  add_mps                  // Apple Metal
AutogradCPU          →  VariableType::add_cpu_wrapper
AutogradCUDA         →  VariableType::add_cuda_wrapper
AutocastCPU          →  autocast::add_cpu_wrapper
AutocastCUDA         →  autocast::add_cuda_wrapper
Fake                 →  fake::add_meta_only
...等几十项

调用 torch.add(x, y) 时,dispatcher 干三件事:

  1. 算出激活的 keys。从输入张量的属性(在 CUDA 上、requires_grad)+ 当前线程的"模式栈"(autocast 是否开启、是否在 FakeTensorMode 里)算出一个键的集合,叫 DispatchKeySet。例如 {AutocastCUDA, AutogradCUDA, CUDA}。
  2. 取优先级最高的一个。例如这个集合里 AutocastCUDA > AutogradCUDA > CUDA,选 AutocastCUDA。
  3. 跳过去。查表得到 AutocastCUDA 对应的函数指针 autocast::add_cuda_wrapper,调用它。

关键巧思:这个 autocast 包装函数做完自己的事(把输入降到 fp16)后,要继续走 autograd → CUDA。它不会再调一次 torch.add(...) 那样回到顶层 —— 那会死循环。它做的是:把"AutocastCUDA"这个 key 从位图里 mask 掉,然后告诉 dispatcher 再算一次最高优先级 key。这次最高优先级变成 AutogradCUDA,跳到 autograd 包装;autograd 包装记完反向图后,再把 AutogradCUDA mask 掉,这次落到 CUDA,跑真的 elementwise kernel。

整条调用链像剥洋葱一样,一层做一层,然后把自己"擦掉"让下一层接力。下面这张图把 with torch.autocast('cuda'): z = x + y(x, y 在 CUDA 上、requires_grad=True)的完整路径画出来:

每个 kernel 只管自己那层。CUDA kernel 不需要知道 autocast 存在;autocast 不需要知道 autograd 存在。 • 可扩展。新加一个后端只是往每个算子的查找表里塞新行,完全不动调用点。新模式装饰器(类似 vmap)也是新加一个 key。 • 可组合。"位图里有哪些 key"决定了"经过哪些层";顺序由优先级决定,跟代码完全解耦。 • 顺序由设计保证。"autocast 必须在 autograd 之前"这个约束被编码进 key 的优先级,不再依赖每个算子作者记得。 这种"逐层下传、每层 mask 自己"的模式,正式名字叫 redispatch。1.7 节会拆开它的实现。
边栏 1.2 — 为什么不用虚函数 / std::function 你可能会想:这不就是 C++ 的虚函数表(vtable)吗?为什么不直接用语言原生机制?

差别在两点:
① vtable 的"动态类型"只来自一个对象(this 指针)。但 dispatcher 的动态类型来自所有输入张量的并集 + 一组线程局部状态。torch.add(x, y) 的实际行为既看 x,也看 y,还看是否在 autocast 里 —— 这是多分派(multiple dispatch),vtable 表达不了。
② std::function 能表达多分派,但每次调用都要堆分配一个类型擦除的闭包。对每秒几百万次算子调用的训练场景,这种开销不可接受。

手撸位图 + 数组查表,每次 dispatch 大约 5-10 ns,比虚函数还便宜。

1.3 DispatchKey 枚举的真身

知道了大思路,现在看具体定义。所有可能的 key 列在 c10/core/DispatchKey.h。这里先看一个简化版(实际定义在 1.4 节会展开,有更精巧的二维拆分,但思路完全一致):

// 简化示意 —— 真实定义见 1.4
enum class DispatchKey : uint16_t {
  Undefined = 0,

  // === 后端层(真的算数值)===
  CPU,
  CUDA,
  MPS,            // Apple Metal Performance Shaders
  XLA,            // Google TPU / XLA backend
  Meta,           // 只算 shape 不算数值
  ...
  SparseCPU,
  SparseCUDA,
  ...

  // === Autograd 层(在后端之前,负责建反向图)===
  AutogradCPU,
  AutogradCUDA,
  AutogradXLA,
  ...

  // === 模式层(在 autograd 之前,影响每一次算子调用)===
  ADInplaceOrView,   // in-place / view 的 autograd 元信息
  Tracer,            // 旧 JIT trace 记录
  AutocastCPU,
  AutocastCUDA,
  Fake,              // FakeTensorMode
  Functionalize,     // 消除 alias/mutation
  Python,            // TorchDispatchMode (子类化拦截)
  ...                // 总共约 60 个

  NumDispatchKeys
};

三个观察:

  • 类型是 uint16_t(16 位无符号整数)。一会儿要把 key 打包进 64 位位图,16 位的下标足够编码所有可能 key 加上头部位标记。
  • 枚举是有序的。后面会看到,"优先级最高的 key"就是位图里最高位 1 对应的枚举值 —— 排在枚举后面的 key 优先级高。所以你看到 Autocast* 排在 Autograd* 后面、Autograd* 排在 CPU/CUDA 后面 —— 这正是 1.1 里"autocast 必须在 autograd 之前发生"的代码层翻译。设计师把"语义上的层级顺序"直接编码进了枚举的字面顺序。
  • 约 60 个 keys(NumDispatchKeys 占位常量)。看起来很多,但其中大多数是"模式层"(autocast / fake / functionalize / python 等),平常调用根本不激活;只有打开对应 with-block 才会被 push 进位图。
小例 1.3 一个普通 CPU float32 张量、不需要梯度,跑 x + 1 时,位图只有 {CPU} 这一个 key,最高优先级就是 CPU,直接跳到 add_cpu。如果 x.requires_grad_(True),位图变成 {AutogradCPU, CPU},最高优先级是 AutogradCPU,先进 autograd 包装,再 redispatch 到 CPU。整个 dispatcher 走了两轮,但用户代码完全没变。

1.4 二维拆分:functionality × backend

上面那个枚举有个尺寸问题:如果按"每种行为 × 每种 backend"做笛卡儿积,枚举会爆炸。例如 PyTorch 想支持 Habana(HPU),它需要新增这一堆条目:

HPU              // dense × Habana
SparseHPU        // sparse × Habana
SparseCsrHPU
QuantizedHPU
NestedTensorHPU
AutogradHPU      // autograd × Habana

一个新 backend 就要新增 ~6 个枚举值,每加一个就要改 enum 定义、改所有相关 codegen。每加一种新功能(假设以后要支持 Adagrad×Backend 这种)又是 N 倍。这不可持续。

PyTorch 团队 2022 年做了一次重要重构:把枚举拆成两个正交的小枚举,合并到一张 64 位位图里(DispatchKey.h):

  1. BackendComponent —— 仅仅"是哪个后端"。当前的位:CPU / CUDA / HIP / XLA / MPS / IPU / XPU / HPU / VE / Lazy / MTIA / MAIA / PrivateUse1‑3 / Meta。一个 static_assert 限定后端数 ≤ 16(DispatchKey.h 里有显式 16 位预算)。注意它单独不构成分派级别 —— 必须和某个 functionality 组合才有意义。
  2. DispatchKey(功能 functionality)。绝大多数 functionality 只占一格(全局唯一 handler);六个是 "per-backend",需要和某个 BackendComponent 组合才落成一个运行时 key。这六个是:Dense、Quantized、Sparse、SparseCsr、NestedTensor、AutogradFunctionality。

组合规则(toRuntimePerBackendFunctionalityKey()):

(Dense,                CUDABit)  →  CUDA           // 你熟悉的 "CUDA" key 真身
(Dense,                CPUBit)   →  CPU
(AutogradFunctionality, CUDABit) →  AutogradCUDA
(AutogradFunctionality, CPUBit)  →  AutogradCPU
(Sparse,               CUDABit)  →  SparseCUDA
(Quantized,            CPUBit)   →  QuantizedCPU
...

也就是说,你写 TORCH_LIBRARY_IMPL(aten, CUDA, m) 其实是在 (Dense, CUDA) 这个二维坐标上注册。代码层面你看到的还是 CUDA 这一个 key —— 但底层位图把它编码成 "Dense 这个 functionality 位 + CUDA 这个 backend 位 同时为 1"。新增一个 backend(HPU)?只要给 BackendComponent 加一个位 —— 那 6 个 per-backend functionality 就自动为它定义了 6 个新组合 key,根本不用改 enum。

小例 1.4 要给 PyTorch 加 Habana(HPU)后端的"sparse"实现:在 BackendComponent 里加一个 HPU 位 → 立刻就有了 (Sparse, HPU) 这个组合 key,可以用 TORCH_LIBRARY_IMPL(aten, SparseHPU, m) 注册稀疏 kernel。enum DispatchKey 里既不需要新加 SparseHPU,也不需要新加 AutogradHPU —— 它们都是组合出来的运行时键。
边栏 1.4 — drift 警告 ezyang 2019 年的 PyTorch internals 没有 BackendComponent 这一层,2020 年的 dispatcher 文章也没有。本节描述的二维拆分是 2022+ 后的最新机制,看老文章时注意这点已经过期 —— 它们把 SparseCUDA 当作枚举里的一个具体值,在今天的代码里它实际是 (Sparse, CUDABit) 的组合。

1.5 DispatchKeySet:64 位位图

现在我们有了 DispatchKey(约 60 个 functionality)和 BackendComponent(16 个后端)。把这些塞进一个 64 位的整数(uint64_t),就得到了 DispatchKeySet。它是 dispatcher 的核心数据结构 —— 每次算子调用,dispatcher 拿到的就是这么一个 64 位整数。

这个 64 位整数从哪里来?三个来源汇总:

  1. 所有输入张量的 TensorImpl::key_set_ 的并集。每个张量的 TensorImpl 里都缓存了一个 DispatchKeySet,记录"我是 CUDA 上的"+"我 requires_grad"等属性。dispatcher 把所有参数张量的 key_set or 起来 —— 这就实现了"多分派"(任何输入是 CUDA 都会激活 CUDA key)。
  2. TLS local include(线程局部"添加"):由模式 guard 往里加。例如进入 with torch.autocast('cuda'):,这个 with 块的实现会把 AutocastCUDA 这一位 push 进 TLS;每次算子调用,dispatcher 把 TLS include or 进位图。
  3. TLS local exclude(线程局部"剔除"):用于 redispatch。autograd kernel 内部调 at::AutoDispatchBelowAutograd guard;,这个 RAII 对象把所有 autograd 位塞进 TLS exclude。dispatcher 把 exclude 从位图里减掉,让最高位"自然"落到下一层。

第四个隐藏因素:fallthrough mask。每个算子在注册期就预先算好"哪些 key 在我这里是 fallthrough(没注册具体 kernel,等价于'我这层跳过')"。这些 key 在位图进 dispatcher 时被一次性 mask 掉,实现 O(1) 跳过。这是 dispatcher 能"白板支持几十个 functionality 而不付出运行时代价"的关键 —— 你的算子如果没注册 Conjugate 上的 kernel,Conjugate 位就被 fallthrough mask 直接 clear。

具体到代码,核心一行在 DispatchKeyExtractor.h:23-44:

// aten/src/ATen/core/dispatch/DispatchKeyExtractor.h
inline DispatchKeySet computeDispatchKeySet(
    DispatchKeySet ks,              // 输入参数张量 or 起来的 key_set
    DispatchKeySet key_mask) {      // 算子的 fallthrough mask
  c10::impl::LocalDispatchKeySet local = c10::impl::tls_local_dispatch_key_set();
  return (((ks | local.included_) - local.excluded_) & key_mask);
  //        ↑ 加 include    ↑ 减 exclude     ↑ 应用 fallthrough mask
}

这就是 1.5 节的全部"魔法":三段位运算,得到一个最终位图。然后 lookup() 在这个位图里找最高位 —— 1.6 节展开。

小例 1.5 设 x 是 CUDA fp32 张量,requires_grad=True,代码在 with torch.autocast('cuda'): 里调 z = x + 1。

1. x 的 key_set_ = {CUDABit, Dense, AutogradFunctionality} = 位图里 CUDABit、Dense、AutogradFunctionality 三位为 1。
2. autocast 上下文 push 了 AutocastCUDA 到 TLS include。
3. 没有 exclude,没有 fallthrough mask 截掉这几位。
4. 最终位图组合 functionality + backend 后,等价于运行时 key 集合 {AutocastCUDA, AutogradCUDA, CUDA}。

对照 1.2 节的"剥洋葱"图 —— 这就是那张图开始时的初始位图。

1.6 优先级阶梯

给定一个 64 位位图,dispatcher 用 highestPriorityTypeId() 取"最高位 1 对应的 functionality"。"最高"指的是位图里最高编号的 set bit —— 功能位是按优先级排好的,bit 编号 = 优先级。

下图把常见 functionality 按优先级画出来,从上(最高)到下(最低):

找 key:取位图最高位 1 → 走那一层 kernel;kernel 体里若调 at::redispatch(...) 则带着剔除掉当前 key 的位图,自然落到下一层。 • autograd "在 backend 之上"是设计灵魂:它能在算数值之前抓住输入、挂上 next_edges、建反向图,然后 redispatch 跑 CUDA。 • autocast 又在 autograd 之上:输入降精度先于 autograd 抓输入,反向图才会用降精度后的 dtype。这是 1.1 节"顺序约束"的代码翻译。 • fallthrough kernel = "我这层不做事,把我的位 mask 掉再分发" —— O(1) skip,大量模式 key 默认就是 fallthrough。

"catchall"这个词在 dispatcher 语境里指alias key 的兜底。Alias key 不是真正的 functionality,而是"注册时展开成多个具体 key"的便利写法:

  • Autograd(别名)= 注册到所有 Autograd<Backend> 上 —— 你写 TORCH_LIBRARY_IMPL(ns, Autograd, m) 时,这条注册被展开成对 AutogradCPU、AutogradCUDA、…… 都注册同一份 kernel。
  • CompositeImplicitAutograd(别名)= 所有 backend + 所有 autograd keys 上都生效 —— 适合用 at::* 组合写出的算子(组合内部的 op 会再走 dispatcher,从而获得 autograd 和后端无关性)。
  • CompositeExplicitAutograd(别名)= 所有 backend(不含 autograd)—— 后端无关实现,但作者需自己处理 autograd。

这三个 alias 让 PyTorch 对"一份代码覆盖所有后端"有了简洁的注册语法,本质上是批量 m.impl 的语法糖。运行时 dispatcher 里只会看到具体的 AutogradCUDA、CUDA 等 key,不会真的看到 Autograd 这个 alias。

1.7 redispatch:层间不死循环递归

1.2 节我们建立了"剥洋葱"的心智模型 —— 一层做完,把自己 mask 掉,让下一层接力。本节看具体怎么实现。

核心问题:autograd kernel 跑完自己那点事(建 grad_fn、记录 next_edges)之后,要怎么"调到 backend kernel"?直接调 at::add(x, y) 不行 —— 那会回到顶层 Dispatcher::call,最高位算出来又是 AutogradCUDA,再进 autograd kernel,死循环。

答案是 Dispatcher::redispatch(Dispatcher.h:833-848):

// 关键注释(从源码):"redispatch treats the inputted DispatchKeySet as is,
//  and does NOT mask out the highest priority bit"
//  —— 调用方负责把"当前 key"从位图剔掉,再传进来
template <class Return, class... Args>
inline Return Dispatcher::redispatch(
    const TypedOperatorHandle<Return(Args...)>& op,
    DispatchKeySet currentDispatchKeySet,    // ← 调用方已经把当前 key mask 掉
    Args... args) const {
  const KernelFunction& kernel = op.operatorDef_->op.lookup(currentDispatchKeySet);
  return kernel.template call<Return, Args...>(op, currentDispatchKeySet, ...);
}

对比 Dispatcher::call(顶层入口): call 会重新算位图(or 输入张量的 key_set + TLS 处理);而 redispatch 直接用调用方传来的位图。这是故意的——它给了 kernel "我已经决定下一步该跳哪儿了"的灵活性。

那"怎么剔掉当前 key"?Autograd kernel 内部用 RAII guard 来翻动 TLS exclude:

  • at::AutoDispatchBelowAutograd(老名字 AutoNonVariableTypeMode,在老博文里常看到):构造时把所有 autograd functionality 位都塞进 TLS excluded_;析构时还原。生命周期内,所有算子调用的位图都会被自动减掉 autograd 位。
  • at::AutoDispatchBelowADInplaceOrView:进一步把 ADInplaceOrView 也塞进 exclude。这是 codegen 出来的 VariableType wrapper 标准用法 —— autograd kernel 调它,然后 redispatch 直接落到 backend。

把这两个机制串起来,一个真实 autograd kernel 的骨架长这样(简化自 VariableType_*.cpp 的 codegen 输出):

// AutogradCUDA kernel for aten::add.Tensor (简化)
Tensor VariableType::add_Tensor(
    c10::DispatchKeySet ks,
    const Tensor& self, const Tensor& other, const Scalar& alpha) {

  // ① 建反向节点(只在需要 grad 时)
  std::shared_ptr<AddBackward0> grad_fn;
  if (compute_requires_grad(self, other)) {
    grad_fn = std::make_shared<AddBackward0>();
    grad_fn->set_next_edges(collect_next_edges(self, other));
  }

  // ② 进入 "below autograd" 范围 → autograd 位被剔出位图
  auto result = [&]() {
    at::AutoDispatchBelowADInplaceOrView guard;
    return at::redispatch::add(
        ks & c10::after_autograd_keyset,    // 显式 mask 当前 key
        self, other, alpha);
  }();

  // ③ 把 grad_fn 挂到输出
  if (grad_fn) set_history(result, grad_fn);
  return result;
}
小例 1.7 — 完整 redispatch 流程 跟着这段代码走一遍 z = x + y(x, y 在 CUDA、requires_grad=True):

(a) 顶层 Dispatcher::call 算出位图 = {AutogradCUDA, CUDA},最高位 = AutogradCUDA,跳到 VariableType::add_Tensor。
(b) 函数体进入 RAII guard AutoDispatchBelowADInplaceOrView:TLS exclude 里加上所有 autograd 位 + ADInplaceOrView 位。
(c) 调 at::redispatch::add(ks & after_autograd_keyset, ...)。after_autograd_keyset 是预算好的"清掉所有 autograd 及以上位"的掩码,显式确保 AutogradCUDA 不会再被选中。
(d) Dispatcher::redispatch 直接用这个位图 lookup,最高位现在是 CUDA,跳到 wrapper_CUDA_add_Tensor。
(e) CUDA wrapper 跑完返回。
(f) 回到 autograd kernel,set_history(result, grad_fn) 把反向节点挂上去,返回。

guard 在 (e) → (f) 之间析构,TLS exclude 还原。这样下一次顶层调用的位图计算又干净了。
记住这条心智模型:dispatcher 不是"一次跳转",而是"每次跳转都有可能触发下一次跳转 —— 由 kernel 决定要不要 redispatch"。模式装饰器、autograd、backend 形成一条由 redispatch 显式驱动的链。链的顺序由 functionality 在 64 位位图里的 bit 编号决定;每一层用 RAII guard 把自己擦掉,让下一层自然成为新的最高位。

2. Schema:算子的类型签名

第 1 章给出了 dispatcher 的运行期机制 —— 但 dispatcher 怎么知道"aten::add.Tensor"这个算子接受什么参数?返回什么?哪些参数会被改写?这些问题的答案叫做 schema。

Schema 是算子的接口契约。它不光是给文档看的字符串 —— 它被 dispatcher、autograd、JIT、functionalization、Python 绑定层同时读取,每个组件都从中提取自己关心的信息:dispatcher 从中知道有几个张量参数(决定 keyset);autograd 从中知道哪些参数会被改写(决定 view 处理);Python 绑定层从中生成参数解析代码。所以 schema 写错一个字 —— 比如把 Tensor(a!) 写成 Tensor(a) —— 不是文档错误,而是正确性错误:functionalization 会以为这个参数没被改写,生成错的图。

本章拆开 schema 的字符串文法、变体命名规则、以及它在 yaml 中的实际落地形式。

2.1 一个 schema 的逐字解读

看一个真实的 schema 字符串:

aten::add.Tensor(Tensor self, Tensor other, *, Scalar alpha=1) -> Tensor

看起来像个 Python 函数签名,但每一处标点都有独立的语义。我们从左到右一字一句拆:

位置片段语义
1aten命名空间。PyTorch 内置算子都在 aten 下;自定义算子(扩展)在自己命名空间下,比如 myops::foo。命名空间隔离了"哪些 op 能被找到",也决定了用哪个 TORCH_LIBRARY* 宏注册(见 3 章)。
2::命名空间分隔符,跟 C++ 一样。
3addop 名。同一个 op 名下可以有多个 overload(像 C++ 的函数重载)。
4.Tensoroverload 名。这是关键 —— add 有 add.Tensor(两个张量相加)和 add.Scalar(张量加标量)两个 overload。可省略,等于 overload 名为 ""(空串)。dispatcher 把"aten::add.Tensor"作为完整 key 来索引函数表。
5(...)参数列表。
6Tensor self第一个 positional 参数,类型 Tensor,名字 self。约定:类似"Python 方法"中的 self —— autograd 会把它当成"主"输入。
7Tensor other第二个 positional 参数,类型 Tensor。
8*关键词分隔符。从这里开始,后面的参数必须用关键字传(add(x, y, alpha=2)),不能位置传。和 Python 函数定义里的 * 完全一样。
9Scalar alpha=1关键词参数,类型 Scalar,默认值 1。Python 里写 torch.add(x, y, alpha=3) 时,这个值落到这。
10-> Tensor返回值类型。多个返回值用 tuple 写,比如 -> (Tensor, Tensor)。

类型语法覆盖一系列原语和复合类型:

类型含义
Tensor张量(强制非空)
Tensor?可空张量(对应 C++ std::optional<Tensor>)
Tensor[]张量列表(std::vector<Tensor>)
Tensor(a)alias 标记:此参数和某个名为 a 的"内存位"共享存储,但不写它(只读 view)
Tensor(a!)可变 alias 标记:共享存储且会改写(in-place / out)
Scalar标量(任意数值类型)
int / int[]整数 / 整数列表(C++ int64_t / IntArrayRef)

bool, str布尔、字符串
Device, ScalarType, Layout, MemoryFormat张量属性,通常出现在 factory function 里
Generator?随机数发生器,通常用在 dropout / randn / multinomial 上
记住这一句:alias 注解((a) 和 (a!))是 schema 里最重要的标记。它们告诉多个下游组件不同的事:
• functionalization 用它决定"这个 op 是否产生副作用,要不要展开成函数式形式"
• autograd 用它决定"这个 op 是否产生 view,要不要建 DifferentiableViewMeta"
• JIT 用它做 alias analysis(两个变量会不会指向同一块内存,影响死代码消除等优化)
• inductor / functorch 都依赖这套注解

错写一个 (a) 为 (a!),后果是 functionalization 把一个本来 in-place 的 op 当 functional 处理,反向图错;反过来,本来 functional 的写成 in-place,可能误报 alias 冲突。
小例 2.1 — alias 注解的作用 看 aten::transpose 的 schema: 
aten::transpose.int(Tensor(a) self, int dim0, int dim1) -> Tensor(a)
Tensor(a) 出现在 self 和返回值上 —— 它们共享同一个名为 a 的"内存位"。这告诉 dispatcher 下游:转置后的张量和原张量共享存储,改其中一个的元素另一个也变。

对比 aten::clone: 
aten::clone(Tensor self, *, MemoryFormat? memory_format=None) -> Tensor
返回值没有 alias 标记 → 它是新分配的内存,改它不影响 self。

注册时,这个字符串会被 torch::jit::parseSchema 解析进一个 c10::FunctionSchema 结构体,记录每个参数的名字、类型、alias 信息、默认值;然后才登记到 dispatcher 的内部 op 表里。所以"schema 字符串"是源头,FunctionSchema 是它的 in-memory 表示。

2.2 三种变体:functional / in-place / out

很多算子有"副作用"能力(in-place 改写、写入预分配的 out 张量)。PyTorch 给这类算子同时提供三种 schema 变体,功能等价但接口不同。看 add 的三个变体:

变体schema
functionaladd.Tensor(Tensor self, Tensor other, *, Scalar alpha=1) -> Tensor
in-placeadd_.Tensor(Tensor(a!) self, Tensor other, *, Scalar alpha=1) -> Tensor(a!)
outadd.out(Tensor self, Tensor other, *, Scalar alpha=1, Tensor(a!) out) -> Tensor(a!)

命名约定:

  • functional:返回新张量。Python 里写 z = x.add(y)。最常用,对应 schema 的"基础形态"。
  • in-place:op 名末尾加 _(下划线),self 加 (a!),返回也是 (a!) 表示"返回的就是改后的 self"。Python 里写 x.add_(y),等价 x = x + y 但不分配新内存。
  • out:overload 加 .out,多一个 kw-only 的 Tensor(a!) out。Python 里写 torch.add(x, y, out=z),把结果写进预分配的 z。常用于复用内存的优化场景。

为什么要写三套?有三个原因:

  1. 用户写法差异。Python 用户三种写法都要支持。如果只写 functional,x.add_(y) 就不存在。
  2. 性能。functional 总是分配新内存,in-place 和 out 可复用,对训练阶段的内存峰值差距很大。
  3. autograd 行为。三个变体的 autograd 处理完全不同:functional 直接建 grad_fn,in-place 要先升 version counter、可能拒绝(RuntimeError: a leaf Variable that requires grad...),out 介于两者之间。
小例 2.2 — 三个变体怎么共用实现 虽然 schema 有三套,实际数学只写一次。底层用 structured kernel 模式: 
add.out      :  写入预分配的 out tensor (核心实现)
add.Tensor   :  分配一个新 out → 调 add.out → 返回 out
add_.Tensor  :  把 self 当成 out → 调 add.out → 返回 self
"分配 out + 调 out 版本"这一步是 codegen 自动生成的(看 yaml 里的 structured_delegate: add.out),作者只需要写 out 版本的实现。
边栏 2.2 — autogen 关键字 某些算子作者只写 functional 一份,告诉 codegen "请按规则补齐 in-place 和 out 变体": 
- func: my_op(Tensor self) -> Tensor
  autogen: my_op.out, my_op_
codegen 看到 autogen: 后会自动生成 my_op.out 和 my_op_ 的 schema 注册以及"out 包装 functional"的实现,作者一行实现都不用写。这对纯函数式 op(没有 in-place 复杂性)非常省力。

2.3 native_functions.yaml:批量声明

PyTorch 内置的 ~2000 个算子,它们的 schema 全部声明在一个文件里:aten/src/ATen/native/native_functions.yaml。这是 in-tree op 的"出生证"。每个 yaml 条目除了 schema,还带了一堆元数据,告诉构建期 codegen 该生成什么:

# native_functions.yaml — add 的完整声明(简化)
- func: add.Tensor(Tensor self, Tensor other, *, Scalar alpha=1) -> Tensor
  device_check: NoCheck             # 不强制要求所有输入同设备(TensorIterator 自处理)
  structured_delegate: add.out      # 转给 out 变体实现
  variants: function, method        # 生成 at::add(...) 和 self.add(...)
  dispatch:                         # 对于"特殊后端",指定专门的 kernel
    SparseCPU, SparseCUDA: add_sparse
    SparseCsrCPU, SparseCsrCUDA: add_sparse_csr
    MkldnnCPU: mkldnn_add
    ZeroTensor: add_zerotensor
    NestedTensorCPU, NestedTensorCUDA: NestedTensor_add_Tensor
  tags: [core, pointwise]           # 给编译期 pass 用的标签

- func: add.out(Tensor self, Tensor other, *, Scalar alpha=1, Tensor(a!) out) -> Tensor(a!)
  device_check: NoCheck
  structured: True                  # ★ 这是 structured kernel 的锚点
  structured_inherits: TensorIteratorBase
  ufunc_inner_loop:                 # ★ 用 elementwise 通用流水线
    Generic: add (AllAndComplex, BFloat16, Half, ComplexHalf)
    ScalarOnly: add (Bool)
  dispatch:
    SparseCPU, SparseMeta: add_out_sparse_cpu
    SparseCUDA: add_out_sparse_cuda
    MPS: add_out_mps
    MTIA: add_out_mtia
    # 注意:这里没有 CPU / CUDA 条目 ——
    #       Dense × {CPU, CUDA} 的实现完全由 ufunc_inner_loop 自动 codegen 出来,
    #       数学函数本身只需要在 ATen/native/ufunc/add.h 里写一份。

关键字段逐个解释(后面章节会用):

对比一个非 structured 的简单 op —— relu:

# native_functions.yaml — relu (非 structured,直接 dispatch)
- func: relu(Tensor self) -> Tensor
  device_check: NoCheck
  variants: function, method
  dispatch:
    CPU, CUDA: relu                 # 同一个函数名,绑到两个后端
    MPS: relu_mps
    MkldnnCPU: mkldnn_relu
    QuantizedCPU: relu_quantized_cpu
  tags: [core, pointwise]

这里 dispatch: CPU, CUDA: relu 把同一个名字 at::native::relu 同时绑到 (Dense, CPU) 和 (Dense, CUDA)。具体实现是两个文件里的两个 overload(aten/src/ATen/native/Activation.cpp 给 CPU,aten/src/ATen/native/cuda/Activation.cu 给 CUDA),靠 link 时按 device 决定。这种非 structured 写法适合简单算子;但失去了 ufunc 自动生成的便利。

边栏 2.3 — 为什么有 structured 这一层 PyTorch 早年(2018 之前)每个算子的 functional / inplace / out 三套实现都得作者手写,大量代码长这样:add 的 functional 分配输出 → 调 add_kernel;add_ 把 self 当输出 → 调 add_kernel;add_out 检查 out shape → 调 add_kernel。这种重复代码每个算子要写 3 遍,每次 add 一个新算子都要写 ~100 行胶水。

2020 后引入的 structured kernel 模式标准化了这套胶水:作者只在 .out 变体上写实际数学(impl()),meta 阶段(算输出 shape/dtype)写在 meta()。codegen 自动生成 functional 和 inplace 变体作为 out 的包装。新算子只需要 ~10 行 yaml + 一份 impl,代码量降到原来的 1/5。

2.4 derivatives.yaml:反向公式

到目前为止 schema 只描述了"前向行为"。反向(autograd)信息在另一个文件里:tools/autograd/derivatives.yaml。每个 differentiable op 在这里给一段反向公式:

- name: add.Tensor(Tensor self, Tensor other, *, Scalar alpha=1) -> Tensor
  self:  handle_r_to_c(self.scalar_type(),  grad)
  other: handle_r_to_c(other.scalar_type(), maybe_multiply(grad, alpha.conj()))
  result: self_t + maybe_multiply(other_t, alpha)            # forward-mode tangent

- name: relu(Tensor self) -> Tensor
  self: threshold_backward(grad, result, 0)
  result: auto_element_wise

逐字解读:

  • name: 必须和 native_functions.yaml 里的 schema 完全一致 —— 这是匹配两边的"主键"。
  • self:、other: 是输入参数名,值是一段表达式,描述"如果这个输入要算梯度,梯度 = 什么"。例如 add 的 other 输入梯度 = grad * alpha.conj()(用 maybe_multiply 包了一层,alpha == 1 时跳过乘法优化)。grad 是个魔法变量,代表"这次 backward 传入的上游梯度"。
  • result: 是前向模式(jvp / tangent)的公式,用于 torch.func.jvp。self_t、other_t 也是魔法变量,代表"输入 self 的切向量"。如果不需要支持 jvp,这一行可以省。
  • auto_element_wise 是个特殊标记,告诉 codegen "这是个 elementwise op,jvp 公式按规则自动推导"。

这些公式不是 Python,而是 codegen 用的受限 DSL —— 它支持调用其他 at::* op(如 threshold_backward)、访问输入的属性(如 self.scalar_type())、用 grad/result/self_t 这样的预定义变量。

codegen 怎么用它?对每个 differentiable op,生成两个东西:

  1. 一个 autograd Node 子类(例如 AddBackward0、ReluBackward0),它的 apply() 方法体就是公式翻译成的 C++ 代码。
  2. 一个 autograd kernel wrapper(注册到 AutogradCPU/AutogradCUDA 等 key),在前向时建这个 Node、挂 next_edges、redispatch 到 backend(就是 1.7 节那段骨架代码)。
小例 2.4 — derivatives.yaml 与生成代码的对应 derivatives.yaml 里 add 的反向公式: 
other: handle_r_to_c(other.scalar_type(), maybe_multiply(grad, alpha.conj()))
被 codegen 翻译成 AddBackward0::apply() 里的: 
if (should_compute_output(1)) {
  grad_inputs[1] = handle_r_to_c(other_scalar_type,
                                 maybe_multiply(grad, alpha.conj()));
}
几乎是一比一翻译(只多了"要不要算这个梯度"的判断,因为某些下游可能不需要 other 的梯度,跳过能省计算)。

5.8 节会展开看 AddBackward0::apply() 的完整生成代码,以及 autograd engine 在 loss.backward() 时怎么调用它。

本章带走的三个心智模型:
① Schema 是多方共读的契约 —— dispatcher、autograd、JIT、functionalization、Python 绑定都从同一字符串里抽不同信息。
② (a!) alias 注解是 schema 里"最危险"的标记 —— 写错直接影响 functionalization、autograd、alias analysis 三个正确性。
③ native_functions.yaml + derivatives.yaml 是一切 codegen 的输入源,后面所有生成的 C++ 文件(包括 dispatcher 注册、autograd wrapper、Python 绑定)都从这两个文件长出来。

3. C++ 注册接口:TORCH_LIBRARY 家族

第 2 章我们看到 schema 是契约。本章回答"谁来执行这份契约" —— PyTorch 用一组宏(全部定义在 torch/library.h)把"这个 schema 在 X 后端的实现是这一段 C++ 代码"喂给 dispatcher。

这组宏看起来花样很多 —— TORCH_LIBRARY、TORCH_LIBRARY_FRAGMENT、TORCH_LIBRARY_IMPL、m.def、m.impl、m.fallback、TORCH_FN、makeFromBoxedFunction ...... 但它们底层全是一个模式:

"用一个 file-static 对象的构造函数,在 main() 之前把信息塞进全局 dispatcher。"

所有差异都在"塞什么信息":开一个新命名空间、续写已有命名空间、还是给已有 op 挂某个 key 上的 kernel。本章把这三种"塞法"逐个拆开,然后讲三个相关概念(boxed vs unboxed、fallback vs fallthrough、def vs impl),最后用一个完整的自定义 op 把所有元素拼起来。

3.1 Library 的三种身份:DEF / FRAGMENT / IMPL

注册的核心数据结构是 torch::Library 对象。每次你写一个 TORCH_LIBRARY* 宏,展开后都构造一个 Library 实例 —— 但它有三种身份(torch/library.h:551-553):

// torch/library.h
class Library {
 public:
  enum Kind { DEF, IMPL, FRAGMENT };
  ...
};

三种 Kind 对应三种语义:

Kind语义触发宏典型用户
DEF给某个 namespace开一个"主"库。整个进程里一个 namespace 名只允许出现一个 DEF。重复链接立即 abort。TORCH_LIBRARY(ns, m)自定义扩展(给自己开一个 myops 命名空间)
FRAGMENT给同 namespace追加一片"片段"。同名 fragment 可以来自任意多个翻译单元(TU),拼到一起算一个库。TORCH_LIBRARY_FRAGMENT(ns, m)PyTorch 内部:aten 命名空间下的 ~2000 个 op 由几十个 codegen 文件拼出
IMPL给"已存在的某个 op"挂一个 DispatchKey 上的 kernel。不能在这里定义新 schema,只能 m.impl(...) 给现有 op 注册实现。TORCH_LIBRARY_IMPL(ns, key, m)给某后端注册具体实现(aten 的 CUDA wrapper、自定义 op 的 CUDA kernel 等)
小例 3.1 — 三种 Kind 的实际用例 你写一个 PyTorch 扩展 my_extension,它注册一个 myops::custom_relu 算子,在 CPU、CUDA 上各有实现。三个 .cpp 文件分工:

register_schema.cpp 用 TORCH_LIBRARY(myops, m) → DEF,声明 schema。这一行存在,myops 命名空间才"开"了。
kernel_cpu.cpp 用 TORCH_LIBRARY_IMPL(myops, CPU, m) → IMPL,给 CPU 注册实现。
kernel_cuda.cu 用 TORCH_LIBRARY_IMPL(myops, CUDA, m) → IMPL,给 CUDA 注册实现。

如果你把 schema 和 CPU kernel 都写在一个文件里(TORCH_LIBRARY(myops, m) { m.def(...); m.impl(...); }),也可以 —— 但 m.impl 在 DEF 块里不能指定 DispatchKey,只能用作"catch-all kernel"(所有 key 都走它)。所以多后端场景下,你必须用 IMPL 把不同 backend 拆开注册。
边栏 3.1 — 为什么不是单一宏 你可能会问:既然底层都是构造 Library 对象,为什么不能合并成一个 TORCH_LIBRARY(ns, kind, key, m)?

有两个原因:
① 静态校验。DEF 不允许指定 DispatchKey(它声明 schema,不挂 kernel);IMPL 必须指定 DispatchKey。把它们做成三个独立宏,编译期就能阻止参数错乱(IDE 也能更好地补全)。
② 链接期"唯一性"检查。DEF 走 file-static 对象的"无 uniquifier"路径,这样多次链接会触发重复符号错误,从而强制"一个 ns 只能有一个 DEF"。FRAGMENT 和 IMPL 都用 C10_UID 唯一化符号名,允许同名宏在多个文件里共存。这个区别是用宏的展开方式编码的,合并成一个宏就破坏了。

3.2 TORCH_LIBRARY:开新命名空间

看 TORCH_LIBRARY 宏的真身(torch/library.h:982-991):

// torch/library.h
#define TORCH_LIBRARY(ns, m)                                                   \
  static void TORCH_LIBRARY_init_##ns(torch::Library&);                        \
  static const torch::detail::TorchLibraryInit TORCH_LIBRARY_static_init_##ns( \
      torch::Library::DEF,                                                     \
      &TORCH_LIBRARY_init_##ns,                                                \
      C10_STRINGIZE(ns), std::nullopt,                                         \
      __FILE__, __LINE__);                                                     \
  void TORCH_LIBRARY_init_##ns(torch::Library& m)

看起来吓人,但拆开看就是三个声明:

  1. 第 1 行:static void TORCH_LIBRARY_init_##ns(torch::Library&); —— 前向声明一个函数(##ns 是 token 拼接,如果 ns=myops,这就是 TORCH_LIBRARY_init_myops)。
  2. 第 2-5 行:static const torch::detail::TorchLibraryInit TORCH_LIBRARY_static_init_##ns(DEF, ...); —— 构造一个文件级 static 对象。它的构造函数收到这些参数:
    • torch::Library::DEF —— 身份标记,这是 DEF 不是 IMPL。
    • &TORCH_LIBRARY_init_##ns —— 用户提供的"注册函数"(第 1 行声明的那个),构造函数会调用它。
    • C10_STRINGIZE(ns) —— namespace 字符串字面量("myops")。
    • std::nullopt —— DispatchKey,DEF 时为空。
    • __FILE__, __LINE__ —— 报错时的位置信息。
  3. 第 6 行:void TORCH_LIBRARY_init_##ns(torch::Library& m) —— 这是函数的定义头,用户的 { ... } 紧接在宏之后提供函数体。

所以用户写的 TORCH_LIBRARY(myops, m) { m.def("..."); } 展开成:

static void TORCH_LIBRARY_init_myops(torch::Library&);
static const torch::detail::TorchLibraryInit TORCH_LIBRARY_static_init_myops(
    torch::Library::DEF,
    &TORCH_LIBRARY_init_myops,
    "myops", std::nullopt,
    "/path/to/myops.cpp", 42);

// 用户的 { ... } 提供这个函数的 body:
void TORCH_LIBRARY_init_myops(torch::Library& m)
{
  m.def("...");
}

关键魔法在文件级 static 对象的初始化时机:C++ 标准保证 main() 调用之前所有 file-static 对象都已构造完。所以当 PyTorch 的 main / 用户的 Python 解释器启动时,TorchLibraryInit 的构造函数已经跑过、已经把这个 schema 塞进全局 dispatcher 表了。用户的注册代码"自动"在程序启动期生效,完全不需要手动调用。

小例 3.2 — 用法 
// myops_register.cpp
#include <torch/library.h>

TORCH_LIBRARY(myops, m) {
  m.def("custom_add(Tensor self, Tensor other) -> Tensor");
  m.def("custom_relu(Tensor self) -> Tensor");
}
这一段编译并链接进 Python 进程后,Python 端就能用 torch.ops.myops.custom_add(...)、torch.ops.myops.custom_relu(...) —— 即使还没注册任何后端 kernel。当然调起来会报 "no kernel registered" 错;但 schema 本身已经在 dispatcher 里了。
关键限制:同一个 namespace 在整个进程里只能出现一个 TORCH_LIBRARY(ns, ...)。再写第二个会触发运行时 std::abort(具体在 Library 构造时检查重复)。如果你需要把同 namespace 的注册拆到多个 .cpp 文件,请用下一节的 TORCH_LIBRARY_FRAGMENT。

3.3 TORCH_LIBRARY_FRAGMENT:同命名空间拼接

FRAGMENT 解决"同 namespace 跨文件"的需求。看宏(torch/library.h:1001-1023,简化):

// torch/library.h (简化)
#define TORCH_LIBRARY_FRAGMENT(ns, m) _TORCH_LIBRARY_FRAGMENT(ns, m, C10_UID)
#define _TORCH_LIBRARY_FRAGMENT(ns, m, uid)                                \
  static void C10_CONCATENATE(TORCH_LIBRARY_FRAGMENT_init_##ns##_, uid)(   \
      torch::Library&);                                                    \
  static const torch::detail::TorchLibraryInit                             \
      C10_CONCATENATE(TORCH_LIBRARY_FRAGMENT_static_init_##ns##_, uid)(    \
          torch::Library::FRAGMENT,                                        \
          ...,                                                             \
          C10_STRINGIZE(ns), std::nullopt, ...);                           \
  void C10_CONCATENATE(...)(torch::Library& m)

和 TORCH_LIBRARY 比,只有两点不同:

  1. Kind 从 DEF 改成 FRAGMENT —— 这告诉 Library 构造函数"不要因为重复 namespace 而 abort"。
  2. 符号名用 C10_UID 唯一化。C10_UID 是 PyTorch 内部的"行号 + counter"宏,每次展开都生成不同的 token。这样同一个 TU 里写 5 个 TORCH_LIBRARY_FRAGMENT(aten, m) { ... },它们的 init 函数名互不冲突。
边栏 3.3 — 为什么 aten 强制用 FRAGMENT 你看 PyTorch codegen 出来的 RegisterCPU.cpp / RegisterCUDA.cpp / RegisterSchema.cpp 等几十个文件,开头全是 TORCH_LIBRARY_FRAGMENT(aten, m),没有一个 TORCH_LIBRARY(aten, m)。这不是巧合 ——torchgen 在 torchgen/gen.py 里专门写了 special case:
# torchgen/gen.py 1761-1776 附近(简化)
if namespace == "aten":
    aten_schema_registrations = schema_registrations_body
else:
    torch_library_macro = (
        "TORCH_LIBRARY_FRAGMENT"
        if namespace in FRAGMENT_NAMESPACES
        else "TORCH_LIBRARY"
    )
原因很现实:aten 命名空间下有 ~2000 个 op,每个 codegen 文件贡献几十到几百个 schema 注册。一个文件几千行,塞进单个 TORCH_LIBRARY 块编译会爆炸(MSVC 单函数指令数有上限)。拆成几十个 fragment,每个独立编译,完美。
小例 3.3 — 同 namespace 跨文件的 fragment 你想给 myops 命名空间下分模块注册算子(算子族 A 在 a.cpp,算子族 B 在 b.cpp): 
// myops_a.cpp
TORCH_LIBRARY_FRAGMENT(myops, m) {
  m.def("op_a1(Tensor x) -> Tensor");
  m.def("op_a2(Tensor x) -> Tensor");
}

// myops_b.cpp
TORCH_LIBRARY_FRAGMENT(myops, m) {
  m.def("op_b1(Tensor x) -> Tensor");
}
两个文件并存,各自的 TorchLibraryInit 都在 main 之前跑,把对应 schema 添加到 myops 命名空间。

但注意:整个进程仍然只能有一个 TORCH_LIBRARY(myops, ...)(DEF)。FRAGMENT 不能"自己开新命名空间" —— 它假定 namespace 已经被某个 DEF 创立。如果只有 FRAGMENT 没有 DEF,会报 "namespace not found" 错。

对于 aten namespace,它的 DEF 其实是 PyTorch 内部初始化逻辑创建的(不是来自某个特定 .cpp);所以 codegen 的 fragment 都能直接续写。

3.4 TORCH_LIBRARY_IMPL:挂载 kernel

schema 是"声明",kernel 才是"实现"。TORCH_LIBRARY_IMPL 是给已存在的 op 挂某个 DispatchKey 上的 kernel 的宏(torch/library.h:1064-1084,简化):

// torch/library.h (简化)
#define TORCH_LIBRARY_IMPL(ns, k, m) _TORCH_LIBRARY_IMPL(ns, k, m, C10_UID)
#define _TORCH_LIBRARY_IMPL(ns, k, m, uid)                                  \
  static void C10_CONCATENATE(TORCH_LIBRARY_IMPL_init_##ns##_##k##_, uid)(  \
      torch::Library&);                                                    \
  static const torch::detail::TorchLibraryInit                              \
      C10_CONCATENATE(TORCH_LIBRARY_IMPL_static_init_##ns##_##k##_, uid)(   \
          torch::Library::IMPL,                                             \
          ...,                                                              \
          C10_STRINGIZE(ns),                                                \
          c10::make_optional(c10::DispatchKey::k),                          \
          ...);                                                             \
  void C10_CONCATENATE(...)(torch::Library& m)

和 FRAGMENT 比的差别:

  • Kind = IMPL。
  • 多了一个参数 c10::DispatchKey::k(k 是用户传的第二个宏参数,如 CPU、CUDA、AutogradCUDA)。块内的每一次 m.impl("op_name", kernel) 都会被附加到 op 在这个 DispatchKey 下的 dispatch 表项里。
小例 3.4 — 典型用法 
// myops_cpu.cpp — 给 myops::* 在 CPU key 上注册 kernel
TORCH_LIBRARY_IMPL(myops, CPU, m) {
  m.impl("custom_add",  &my_add_cpu);
  m.impl("custom_relu", &my_relu_cpu);
}

// myops_cuda.cu — 给 myops::* 在 CUDA key 上注册 kernel
TORCH_LIBRARY_IMPL(myops, CUDA, m) {
  m.impl("custom_add",  &my_add_cuda);
  m.impl("custom_relu", &my_relu_cuda);
}

// myops_autograd.cpp — 单独一个 autograd 包装,用来挂 grad_fn
TORCH_LIBRARY_IMPL(myops, Autograd, m) {       // Autograd 是 alias key,自动展开到 AutogradCPU/CUDA/...
  m.impl("custom_add", &my_add_autograd);
}
三个文件链接进来之后,dispatcher 表里的 myops::custom_add 行就有了三个 kernel:CPU、CUDA、Autograd*(自动展开到所有具体 autograd key)。一次调用根据输入张量在哪个设备、是否 require grad,自动找最高位 key 的 kernel 跳过去。
特殊形式 TORCH_LIBRARY_IMPL(_, KEY, m):第一个参数下划线表示"所有 namespace 都套用",通常配合 m.fallback(...) 给某个 DispatchKey 注册全局兜底 kernel。一个例子:
// aten/src/ATen/core/VariableFallbackKernel.cpp
TORCH_LIBRARY_IMPL(_, AutogradCUDA, m) {
  m.fallback(autogradNotImplementedFallback());
}
含义:对于任何命名空间下的算子,如果在 AutogradCUDA 这个 key 上没显式注册 kernel,就走这个 fallback。这个 fallback 干的事是"当作 CompositeImplicitAutograd 处理 —— 让里面的子 op 自己走 dispatcher" —— 这是 PyTorch "没明确写反向公式的算子默认也能自动求导"的实现机制。

3.5 m.def 与 m.impl 的语义差异

m.def 和 m.impl 是 Library 对象上最常用的两个方法。它们职责完全不同:

m.defm.impl
在哪用仅 TORCH_LIBRARY / TORCH_LIBRARY_FRAGMENT 块内仅 TORCH_LIBRARY_IMPL 块内
作用声明 schema:给一个新的 op-name 注册类型签名,把它"注册"到 dispatcher 表里给已声明的 op 挂 kernel:在当前块的 DispatchKey 下登记一个具体函数指针
关键参数schema 字符串("custom_add(Tensor self, Tensor other) -> Tensor")op 名("custom_add")+ 函数指针(&my_kernel)
schema 校验第一次 def 锁定签名;后续不能改kernel 签名必须和 schema 解析后的类型兼容(否则注册时 abort)
能否同时挂 kernel可以(m.def("schema", &kernel)),但这个 kernel 没有 DispatchKey,被认为是 "catch-all" —— 任何 key 都走它本身就是给特定 key 挂 kernel

三种常见 m.impl 写法:

// (a) raw function pointer —— 框架推断函数签名
m.impl("op_name", &cpu_kernel);

// (b) TORCH_FN 包装 —— 把签名固化成 compile-time 类型,启用更快的 unboxed 调用路径
m.impl("op_name", TORCH_FN(cpu_kernel));

// (c) 强制走 boxed(用于 fallback、interpreter 等不预知签名的场景)
m.impl("op_name", torch::CppFunction::makeFromBoxedFunction<&boxed_fn>());
小例 3.5 — schema 类型不兼容 你声明 schema: 
m.def("custom_add(Tensor self, Tensor other) -> Tensor");
后来注册 kernel: 
at::Tensor wrong_kernel(const at::Tensor& self, int other) { ... }
m.impl("custom_add", TORCH_FN(wrong_kernel));   // 类型不匹配!
程序启动时(static initializer 执行时)会立即 abort,错误信息大概是: 
Tried to register a kernel for operator myops::custom_add with type
Tensor(const Tensor&, int) — but the schema declares type Tensor(Tensor, Tensor).
这是 PyTorch 安全设计的关键:schema 是单一可信源,所有 kernel 都要对齐它。

3.6 Boxed 与 unboxed 调用路径

1.5 节我们说过 dispatcher 查表得到一个 KernelFunction,然后调用它。KernelFunction 内部其实同时存了两份函数指针:一份 unboxed,一份 boxed。dispatcher 根据"调用方知道类型签名 vs 调用方不知道"自动选择走哪条路径。

UnboxedBoxed
函数签名原生 C++:Tensor (*)(const Tensor&, const Tensor&, Scalar)统一:void (*)(const OperatorHandle&, DispatchKeySet, Stack*)
参数传递模板直接展开,零拷贝、无类型擦除所有参数被打包成 IValue 压在 Stack(std::vector<IValue>)上,kernel 自己从 stack pop
性能每次调用约 5-10 ns多 ~50-100 ns(参数装箱/拆箱开销)
什么时候用用户代码、codegen 出来的 wrapper —— 调用现场知道签名fallback kernel、functorch interpreter、JIT runtime、profiler hook —— op-agnostic 的场景

TORCH_FN(fn) 这个宏的作用是用模板提取 fn 的字面类型签名(在编译期),从而让 KernelFunction 可以同时存下 unboxed 路径(直接调用)和 boxed 路径(自动生成的 IValue 包装版本)。如果你写 m.impl("name", &fn)(没 TORCH_FN),框架也能用 SFINAE 推断签名,但 TORCH_FN 写法更明确,签名错误能在编译期被发现。

边栏 3.6 — m.impl_UNBOXED 的历史 PyTorch 早期(2019 前)有一个 m.impl_UNBOXED("name", &fn) 接口,语义是"这个 kernel 只能 boxed 调用 —— 不要尝试生成 unboxed 路径"。当时是因为有些 kernel 的签名无法被模板推断(比如带 c10::optional<Tensor> 早期某些版本)。

现在这个 API 已被删除。等价写法是 m.impl("name", torch::CppFunction::makeFromBoxedFunction<&fn>()) —— 显式说"用这个 boxed 函数包装,不生成 unboxed 路径"。你在 2020 年前的博客里看到 impl_UNBOXED,知道它对应现在的什么就行。

3.7 Fallback 与 fallthrough

除了 m.def 和 m.impl,Library 还有一个 m.fallback。它的语义是:给整个 DispatchKey 注册一个 op-agnostic 的 kernel,在该 key 下"没有单独 impl 的 op"都走它。

常见两种用法:

m.fallback(torch::CppFunction::makeFallthrough());
    // "我这层不做事,把我的 bit 从位图清掉再走下一层"
    // O(1) 实现:dispatcher 看到这是 fallthrough,直接在 fallthrough_mask 里把这个 key 永久清零

m.fallback(torch::CppFunction::makeFromBoxedFunction<&my_boxed_fallback>());
    // 真的写个 boxed kernel 兜底,比如 functorch 的 vmap interpreter

fallthrough 是 dispatcher 高效的关键。它不是"运行时跳过" —— 而是"注册期就告诉 dispatcher 这个 key 在我这里没意义,以后位图里这个位直接 mask 掉"。代价 = 0。

典型 fallthrough 用户:

  • BackendSelect:绝大多数 op 不需要它(只有 factory function 如 torch.randn 才用)。对所有其他 op,BackendSelect fallthrough。
  • Named:没开 named tensor 时全部 fallthrough。
  • Conjugate / Negative / ZeroTensor:没用对应惰性标记时全部 fallthrough。
  • Tracer:没在 JIT trace 模式时 fallthrough。

这就是为什么 dispatcher 表面上"支持几十个 functionality 层" —— 实际上 99% 的算子调用都只激活两三层(backend + 可能的 autograd),其他层全是 fallthrough,在位图算出来后位都不会被点亮。

设计哲学:dispatcher 的扩展性来自"你不付不要的代价"。注册时声明 fallthrough → 你这层永远不会拖慢调用。你真的需要某层时(开 autocast、开 vmap),只有该层会被激活,其他仍然 fallthrough。这跟"if/else 链每次调用都要走完"是质的不同。

3.8 完整自定义 op 走一遍

现在把 3.1-3.7 学到的零件拼起来,看一个完整的自定义算子。这是一个"自定义 relu",在 CPU 和 CUDA 上都有实现,带 autograd 支持。涉及四个 .cpp 文件:

// ============ myops_schema.cpp ============
// (1) 用 TORCH_LIBRARY 给 myops 命名空间开 DEF,声明 schema 一次
#include <torch/library.h>

TORCH_LIBRARY(myops, m) {
  m.def("custom_relu(Tensor self) -> Tensor");
}

这一文件提供两件事:① 命名空间 myops 被开通(DEF);② 算子 custom_relu 的 schema 进入 dispatcher 表(但还没有任何 kernel)。Python 端此时调 torch.ops.myops.custom_relu(x) 会报 "kernel not found"。

// ============ myops_cpu.cpp ============
// (2) CPU 实现 + 注册到 (myops, CPU)
#include <ATen/ATen.h>
#include <torch/library.h>

at::Tensor custom_relu_cpu(const at::Tensor& self) {
  return at::clamp_min(self, 0);   // 简单起见:借用现成 at::clamp_min 当实现
}

TORCH_LIBRARY_IMPL(myops, CPU, m) {
  m.impl("custom_relu", TORCH_FN(custom_relu_cpu));
}

这一文件做的事:给 myops::custom_relu 在 CPU key(= 二维 (Dense, CPUBit))下挂一个具体 kernel。TORCH_FN 让 dispatcher 知道 unboxed 签名,运行时快。注意实现是用 at::clamp_min —— 它本身是个 dispatched op,内部会再走一遍 dispatcher 调用 CPU 的 clamp_min。

// ============ myops_cuda.cu ============
// (3) CUDA 实现 + 注册到 (myops, CUDA)
#include <ATen/ATen.h>
#include <torch/library.h>

at::Tensor custom_relu_cuda(const at::Tensor& self) {
  return at::clamp_min(self, 0);   // 同上,clamp_min 自己有 CUDA kernel
}

TORCH_LIBRARY_IMPL(myops, CUDA, m) {
  m.impl("custom_relu", TORCH_FN(custom_relu_cuda));
}

对称地给 CUDA key 挂 kernel。现在 dispatcher 表里 myops::custom_relu 的两个 backend 项都有 kernel。

// ============ myops_autograd.cpp ============
// (4) Autograd 实现:接住 forward,建 grad_fn,前向数学透传到 backend
#include <torch/autograd.h>
#include <torch/library.h>

class CustomReluFn : public torch::autograd::Function<CustomReluFn> {
 public:
  static at::Tensor forward(torch::autograd::AutogradContext* ctx,
                            const at::Tensor& self) {
    // 进入 forward 时手动排除 "AutogradXxx + ADInplaceOrView" key —— 否则下面
    // dispatcher 调用会再次落回到本 Function,死循环。
    at::AutoDispatchBelowADInplaceOrView guard;
    // guard 生效期内, AutogradCUDA 已从 TLS 位图剔除,
    // 调 dispatcher 会落到 CUDA / CPU 之一(由 self.device() 决定)。
    static auto op = c10::Dispatcher::singleton()
        .findSchemaOrThrow("myops::custom_relu", "")
        .typed<at::Tensor(const at::Tensor&)>();
    auto result = op.call(self);
    ctx->save_for_backward({result});
    return result;
  }

  static torch::autograd::variable_list backward(
      torch::autograd::AutogradContext* ctx,
      torch::autograd::variable_list grad_outputs) {
    auto saved = ctx->get_saved_variables();
    auto& result = saved[0];
    // d/dx relu(x) = 1 if x > 0 else 0;但我们存的是 result,等价于 (result > 0)
    return { grad_outputs[0] * (result > 0).to(grad_outputs[0].dtype()) };
  }
};

at::Tensor custom_relu_autograd(const at::Tensor& self) {
  return CustomReluFn::apply(self);
}

TORCH_LIBRARY_IMPL(myops, Autograd, m) {   // Autograd 是 alias key,展开到所有 AutogradXxx
  m.impl("custom_relu", &custom_relu_autograd);
}

这是关键一文件,几个细节:

  • torch::autograd::Function:PyTorch 提供的 CRTP 模板基类,子类定义 forward 和 backward。apply() 是基类生成的入口,它会自动处理 grad_fn 建立、next_edges 收集、上下文管理。
  • at::AutoDispatchBelowADInplaceOrView guard;:RAII 对象,构造时把所有 autograd functionality 位 + ADInplaceOrView 位 push 进 TLS exclude。整个 forward 函数体内,dispatcher 不会再走 AutogradCUDA 这一层。
  • 用 c10::Dispatcher::singleton().findSchemaOrThrow 主动查 schema,然后 typed<...>().call(...) 跑。这是 PyTorch 推荐的"在 C++ 里调自定义 op"写法 —— 等价于 Python 端的 torch.ops.myops.custom_relu(self)。
  • 注册到 Autograd alias:不写具体 AutogradCPU/AutogradCUDA,而是用 alias Autograd —— 它会被自动展开到所有具体 autograd keys,一行覆盖全部 backend 的 autograd 情况。

四个文件链接到一起,Python 端使用:

>>> import torch, my_extension
>>> x = torch.randn(3, requires_grad=True, device='cuda')
>>> y = torch.ops.myops.custom_relu(x)
    # dispatcher 算位图 → 最高位 = AutogradCUDA → 跳 custom_relu_autograd
    # → CustomReluFn::apply → forward()
    # → AutoDispatchBelowADInplaceOrView guard 生效
    # → op.call(self) (myops::custom_relu) → 最高位现在 = CUDA → 跳 custom_relu_cuda
    # → at::clamp_min(self, 0) → 走 clamp_min 的 CUDA kernel → 返回
>>> y.sum().backward()

    # → 计算梯度 → 传给 next_edges
更省事的写法 — CompositeImplicitAutograd:既然 custom_relu 的实现完全是 at::* op 的组合(at::clamp_min),且它的反向能完全靠 clamp_min 自己的反向来推 —— 你可以直接注册到 alias CompositeImplicitAutograd,一次注册同时覆盖所有 backend、所有 autograd: 
at::Tensor custom_relu_composite(const at::Tensor& self) {
  return at::clamp_min(self, 0);
}

TORCH_LIBRARY_IMPL(myops, CompositeImplicitAutograd, m) {
  m.impl("custom_relu", &custom_relu_composite);

3.5 m.def vs m.impl

m.defm.impl

m.impl("op_name", &cpu_kernel);                    // raw fn ptr,框架推断签名
m.impl("op_name", TORCH_FN(cpu_kernel));            // 把函数指针包成 compile-time 类型,
                                                    // 启用更快的 unboxed 调用路径
m.impl("op_name", torch::CppFunction::makeFromBoxedFunction<&boxed_fn>()); // 强制走 boxed

3.6 boxed vs unboxed kernel

每个 kernel 在 dispatcher 内部都用 KernelFunction 表示。它可能存两份函数指针:

  • Unboxed:原生 C++ 签名 Tensor (*)(const Tensor&, const Tensor&, Scalar)。Dispatcher 用模板直接把参数推进去,零拷贝、无类型擦除。所有热路径 op 都走这条。
  • Boxed:统一签名 void (*)(const OperatorHandle&, DispatchKeySet, Stack*)。参数被打包成 IValue 压在 Stack 上。慢一点,但不依赖具体类型 —— functorch interpreter、JIT runtime、profiling hook、fallback kernel 都必须走 boxed。

TORCH_FN(fn) 的作用是用模板提取 fn 的字面类型签名,从而让 dispatcher同时保留 unboxed 调用路径 —— 默认 m.impl("name", &fn) 也能走 unboxed,但 TORCH_FN 写法更明确、能让编译器在签名错的时候立刻报错。

m.impl_UNBOXED 是已删除的老 API,只允许 boxed 调用 —— 现在用 m.impl(..., torch::CppFunction::makeFromBoxedFunction<...>()) 表达同样语义。

3.7 fallback / fallthrough

m.fallback(...) 给当前 IMPL 块的整个 DispatchKey 注册一个 op-agnostic 的 kernel,在该 key 下"没单独 impl 的 op"都走它。常见两种:

m.fallback(torch::CppFunction::makeFallthrough());
    // "我这层不做事,把我的 bit 从位图清掉再走下一层"
    // O(1) 实现:dispatcher 看到这是 fallthrough,直接在 fallthrough_mask 里把这个 key 清零

m.fallback(torch::CppFunction::makeFromBoxedFunction<&my_boxed_fallback>());
    // 真的写个 boxed kernel 兜底,比如 functorch 的 vmap interpreter

典型 fallthrough 用户:BackendSelect(只在 factory function 上有 kernel,别的 op 自动跳过)、Named(没开 named tensor 时全跳过)、Conjugate(没用复数共轭标记时全跳过)。这是 dispatcher 能"零开销支持几十个 functionality"的关键。

#include <torch/autograd.h>
#include <torch/library.h>

class CustomReluFn : public torch::autograd::Function<CustomReluFn> {
 public:
  static at::Tensor forward(torch::autograd::AutogradContext* ctx,
                            const at::Tensor& self) {
    // 进 forward 时手动排除"AutogradXxx + ADInplaceOrView" —— 否则下面 dispatcher 调用
    // 会再次落回到本 Function,死循环。
    at::AutoDispatchBelowADInplaceOrView guard;
    // 直接调注册到 myops::custom_relu 的 dispatcher 入口 —— guard 在生效期内
    // AutogradCUDA 已被剔除,会落到 CUDA / CPU 那两个 IMPL 之一(由 self.device() 决定)。
    static auto op = c10::Dispatcher::singleton()
        .findSchemaOrThrow("myops::custom_relu", "")
        .typed<at::Tensor(const at::Tensor&)>();
    auto result = op.call(self);
    ctx->save_for_backward({result});
    return result;
  }
  static torch::autograd::variable_list backward(
      torch::autograd::AutogradContext* ctx,
      torch::autograd::variable_list grad_outputs) {
    auto saved = ctx->get_saved_variables();
    auto& result = saved[0];
    // d/dx relu(x) = (x > 0)
    return { grad_outputs[0] * (result > 0).to(grad_outputs[0].dtype()) };
  }
};

at::Tensor custom_relu_autograd(const at::Tensor& self) {
  return CustomReluFn::apply(self);
}

TORCH_LIBRARY_IMPL(myops, Autograd, m) {
  m.impl("custom_relu", &custom_relu_autograd);
}

5.5 AutogradCUDA kernel:建 grad_fn,redispatch

kernel 注册在 torch/csrc/autograd/generated/VariableType_2.cpp(具体分片号会随版本变):

TORCH_LIBRARY_IMPL(aten, AutogradCUDA, m) {
  ...
  m.impl("add.Tensor", TORCH_FN(VariableType::add_Tensor));
  ...
}

kernel 体是 codegen 出来的,典型骨架:

at::Tensor VariableType::add_Tensor(
    c10::DispatchKeySet ks,
    const at::Tensor& self, const at::Tensor& other, const at::Scalar& alpha) {
  // === 1) 若任何输入 requires_grad,准备一个新的 grad_fn 节点 ===
  std::shared_ptr<AddBackward0> grad_fn;
  if (compute_requires_grad(self, other)) {
    grad_fn = std::shared_ptr<AddBackward0>(new AddBackward0(), deleteNode);
    grad_fn->set_next_edges(collect_next_edges(self, other));
    grad_fn->alpha = alpha;
    grad_fn->other_scalar_type = other.scalar_type();
    grad_fn->self_scalar_type  = self.scalar_type();
  }

  // === 2) 把"AutogradCUDA + ADInplaceOrView"从位图剔掉,redispatch ===
  auto _tmp = ([&]() {
    at::AutoDispatchBelowADInplaceOrView guard;
    return at::redispatch::add(
        ks & c10::after_autograd_keyset,    // 留下 CUDA 等更低位
        self, other, alpha);
  })();
  auto result = std::move(_tmp);

  // === 3) 把新算出的 result 挂到 grad_fn,加进 history ===
  if (grad_fn) {
    set_history(flatten_tensor_args(result), grad_fn);
  }
  return result;
}

三步:建节点 → 排除自己 + redispatch 到下一层 → 把输出绑到节点。AddBackward0 是 codegen 出来的 Node 子类,定义在 torch/csrc/autograd/generated/Functions.h,5.8 节会看 apply() 体。

5.6 redispatch 到 CUDA backend

at::redispatch::add 在 RedispatchFunctions.h(codegen),做的事和 at::add 几乎一样,只是多带一个 DispatchKeySet 参数透传给 Dispatcher::redispatch。这次位图里 AutogradCUDA 已经清掉,highestPriorityTypeId 返回 CUDA。

CUDA 槽位上注册的 kernel 在 build/.../RegisterCUDA.cpp:

TORCH_LIBRARY_IMPL(aten, CUDA, m) {
  ...
  m.impl("add.Tensor", TORCH_FN(wrapper_CUDA_add_Tensor));
  m.impl("add.out",    TORCH_FN(wrapper_CUDA_add_out_out));
  ...
}

wrapper_CUDA_add_Tensor 是 functional → out 的转发(因为 yaml 里 structured_delegate: add.out):

at::Tensor wrapper_CUDA_add_Tensor(
    const at::Tensor& self, const at::Tensor& other, const at::Scalar& alpha) {
  // 1) 分配输出
  auto common_dtype = at::native::result_type(self, other);
  auto out = at::detail::empty_cuda({0}, common_dtype, self.device());
  // 2) 转给 out 变体
  return wrapper_CUDA_add_out_out(self, other, alpha, out);
}

5.7 structured kernel + ufunc

wrapper_CUDA_add_out_out 实例化 structured kernel base(继承 TensorIteratorBase):

at::Tensor& wrapper_CUDA_add_out_out(
    const at::Tensor& self, const at::Tensor& other, const at::Scalar& alpha,
    at::Tensor& out) {
  structured_add_out_out op(out);
  op.meta(self, other, alpha);   // 算输出 shape/dtype, 设 TensorIterator
  op.impl(self, other, alpha, op.maybe_get_output(0));   // 真的算
  return out;
}

impl 体由 UfuncCUDA_add.cu 生成 —— ufunc 模板的标准形态:

// build/.../UfuncCUDA_add.cu (示意)
void structured_add_out_out::impl(
    const Tensor& self, const Tensor& other, const Scalar& alpha, const Tensor& out) {
  AT_DISPATCH_ALL_TYPES_AND_COMPLEX_AND4(
      kBFloat16, kHalf, kComplexHalf, kBool,
      iter.common_dtype(), "add_cuda",
      [&] {
        scalar_t alpha_val = alpha.to<scalar_t>();
        gpu_kernel(iter, [alpha_val] GPU_LAMBDA (scalar_t a, scalar_t b) -> scalar_t {
          return at::native::ufunc::add(a, b, alpha_val);
        });
      });
}

AT_DISPATCH_ALL_TYPES_AND_COMPLEX_AND4(aten/src/ATen/Dispatch.h) 展开成一个 switch (scalar_type),对每个 dtype 实例化 lambda;gpu_kernel 接 TensorIterator + lambda,展开成 vectorized_elementwise_kernel<...><<<blocks, threads>>>(...)(aten/src/ATen/native/cuda/CUDALoops.cuh)。

真正的标量数学只有一行,在 ATen/native/ufunc/add.h:

template <typename T>
C10_HOST_DEVICE C10_ALWAYS_INLINE T add(T self, T other, T alpha) {
  return self + alpha * other;
}

所以 CPU 和 CUDA 共享同一个 ufunc::add 标量函数,只是各自的循环 wrapper 不同(CPU 是 SIMD 矢量化的 vectorized_loop2d,CUDA 是 launch 的 kernel)。

5.8 backward:AddBackward0

反向阶段:用户调 z.sum().backward(),autograd engine(torch/csrc/autograd/engine.cpp)从 loss 拓扑排序整张图,对每个 Node 调 apply(grads)。AddBackward0::apply(codegen 出在 generated/Functions.cpp)体大致是:

variable_list AddBackward0::apply(variable_list&& grads) {
  auto& grad = grads[0];
  variable_list grad_inputs(2);
  if (should_compute_output(0)) {
    grad_inputs[0] = handle_r_to_c(self_scalar_type, grad);
  }
  if (should_compute_output(1)) {
    grad_inputs[1] = handle_r_to_c(other_scalar_type, maybe_multiply(grad, alpha.conj()));
  }
  return grad_inputs;
}

对照 derivatives.yaml:

self:  handle_r_to_c(self.scalar_type(),  grad)
other: handle_r_to_c(other.scalar_type(), maybe_multiply(grad, alpha.conj()))

一字不差地翻译。返回的 grad_inputs 通过 next_edges 继续传递给上游节点。maybe_multiply 是个小优化:alpha == 1 时直接返回 grad,跳过乘法。

7. 与老博文的 drift

读旧资料时容易踩的几个坑:

  • 2019 年的 PyTorch internals 博文把 dispatch 描述为"先 device/layout 再 dtype 两轮"。今天的 dispatcher 第一轮已经是 30+ 个 functionality 的多层结构 —— device/layout 只是其中一层,dtype 完全不在 dispatcher 里(由 AT_DISPATCH_* 在 kernel 内部 switch)。
  • 2020 年的 dispatcher 文章还在用 AutoNonVariableTypeMode 这个名字,现在的代码用 AutoDispatchBelowAutograd 和 AutoDispatchBelowADInplaceOrView。老名字以 alias 保留。
  • 官方 tutorial(dispatcher 那篇)的 class MyAddFunction : public torch::autograd::Function<MyAddFunction> 例子里也还在用旧 AutoNonVariableTypeMode guard。
  • BackendComponent 拆分是 2022 年才做的。任何更早的资料把 DispatchKey 当一维大枚举看,理解机制没问题,但"为什么 AutogradCUDA 不是单独枚举值,是 AutogradFunctionality + CUDABit 组出来的"这种细节不能直接套。
  • 一些 key 名变过:VariableTensorId → Autograd / AutogradCPU 等;DefaultBackend → CompositeExplicitAutograd(alias 还在);Autocast → AutocastCUDA(alias 还在)。
  • m.impl_UNBOXED 已删,改用 m.impl(..., torch::CppFunction::makeFromBoxedFunction<...>())。
