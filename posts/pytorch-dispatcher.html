PyTorch Dispatcher 全栈剖析:从 schema 到 CUDA kernel

本文 PyTorch 源码以 pytorch/pytorch main 分支为准。所有 file:line 引用都可点开直接看到当前源。本文重点剖析 dispatcher 数据结构、注册宏的语义、以及一个 native op 从 Python 调到 CUDA 的完整调用栈;不涉及量化/分布式/inductor 等下游话题。

引言:为什么不能 if/else

一个 PyTorch tensor 上能跑的 op 同时跨好几个正交维度:

  • device × layout × dtype: CPU / CUDA / MPS / XPU / Meta × dense / sparse / sparse_csr / mkldnn × fp32 / fp16 / bf16 / int8 / ...
  • autograd: 真正的数值算之前要先记 grad_fn,反向才能回溯
  • 变换 / 模式: Tracer (JIT) / Functionalize (alias+mutation 消除) / Autocast (输入侧降精度) / ADInplaceOrView (autograd 给 view/in-place 打补丁) / Python (TorchDispatchMode) / FuncTorch (vmap / grad transforms) / Conjugate / Negative / ZeroTensor / BackendSelect ...

这是个 ~6 维的笛卡儿积,而且每一维都要可扩展(让 OOT backend、第三方扩展、模式装饰器都能挂上去)。任何 if (device == CUDA) { ... } else if ... 的写法都会瞬间崩溃 —— 调用方代码要被改 N 次,且第三方扩展根本插不进来。

Ezyang 的 Let's talk about the PyTorch Dispatcher 把它总结得很到位:dispatcher 的工作就是,根据输入 tensor(以及一些别的东西)算出一个 dispatch key,然后做一次间接跳转到那个 key 对应的函数指针。每个被 dispatch 的 op 都拥有一张"按 DispatchKey 索引"的函数指针表;扩展只需往表里塞新条目,完全不动调用点。

这套机制比 vtable 更强的地方在于,它的"动态类型"不只来自第一个对象 —— 它要看全部输入 tensor 的并集、加上一组线程局部的"模式 stack"。下面逐层拆开。

1. DispatchKey 与 DispatchKeySet

1.1 二维拆分:functionality × backend

2022 之后的 PyTorch 把"以前的一个大 DispatchKey 枚举"拆成了两个独立枚举,然后在一张 64-bit 位图里合并表示(c10/core/DispatchKey.h、DispatchKeySet.h)。

  • BackendComponent —— 仅仅"是哪个后端",不单独构成分派级别。当前支持的位:CPU / CUDA / HIP / XLA / MPS / IPU / XPU / HPU / VE / Lazy / MTIA / MAIA / PrivateUse1‑3 / Meta(C10_FORALL_BACKEND_COMPONENTS 宏)。一个 static_assert 限定后端数 ≤ 16(DispatchKey.h 内有显式 16-bit 预算)。
  • DispatchKey —— "功能"(functionality)。绝大多数只占一格(全局唯一 handler);六个是 "per-backend",需要和某个 BackendComponent 组合才落成一个运行时 key:Dense、Quantized、Sparse、SparseCsr、NestedTensor、AutogradFunctionality(C10_FORALL_FUNCTIONALITY_KEYS)。

组合规则:(Dense, CUDABit) → CUDA、(AutogradFunctionality, CUDABit) → AutogradCUDA、(Sparse, CUDABit) → SparseCUDA,通过 toRuntimePerBackendFunctionalityKey() 互译。也就是说,你写 TORCH_LIBRARY_IMPL(aten, CUDA, m) 其实是在 (Dense, CUDA) 这个二维坐标上注册。

drift 警告:ezyang 2019 年的 PyTorch internals 写的是"先 device/layout 再 dtype 两轮 dispatch",2020 年的 dispatcher 文章也还没有 BackendComponent 这一层 —— 本节描述的二维拆分是 2022+ 后的最新机制,看老文章时注意这点已经过期。

1.4 DispatchKeySet 的计算

真正决定"这次调用走哪条路"的是 DispatchKeySet —— 一个 64-bit 位图,综合了:

  1. 所有输入 tensor 的 TensorImpl::key_set_ 的并集。一个 CUDA 上 requires_grad=True 的 tensor 自带 {AutogradCUDA, CUDA}。
  2. TLS local include:由模式 guard(autocast 打开、functorch 层 push 等)往里加。
  3. TLS local exclude:由 AutoDispatchBelowAutograd / AutoDispatchBelowADInplaceOrView 这类 RAII guard 在 autograd kernel 内部往外剔。
  4. fallthrough mask:每个 op 的 dispatch table 在注册期就预算好"哪些 key 是 fallthrough" —— 这些 key 在 dispatcher 一进门就被位图清零,实现 O(1) skip。

核心代码在 DispatchKeyExtractor.h:23-44:

// aten/src/ATen/core/dispatch/DispatchKeyExtractor.h
inline DispatchKeySet computeDispatchKeySet(
    DispatchKeySet ks,
    DispatchKeySet key_mask) {
  c10::impl::LocalDispatchKeySet local = c10::impl::tls_local_dispatch_key_set();
  return (((ks | local.included_) - local.excluded_) & key_mask);
}

调用现场:

// Dispatcher.h:774-828 — Dispatcher::call
auto dispatchKeySet =
    op.operatorDef_->op.dispatchKeyExtractor()
        .template getDispatchKeySetUnboxed<Args...>(args...);
const KernelFunction& kernel = op.operatorDef_->op.lookup(dispatchKeySet);
return kernel.template call<Return, Args...>(op, dispatchKeySet, std::forward<Args>(args)...);

getDispatchKeySetUnboxed 会跑 MultiDispatchKeySet,把每个 tensor 参数的 key_set_ or 起来;lookup() 进去找位图最高位对应的 KernelFunction,再 call。整条路径都是 inline,热路径上的代价就是一次 or、一次掩码、一次 ctz、一次表查询。

1.5 redispatch 机制

一个 Autograd kernel 跑完它自己那点事(建 grad_fn、记录 next_edges)之后,要怎么"把控制权传给 backend kernel"?直接调 Dispatcher::call 不行 —— 那会再次进 dispatcher,找到的最高优先级 key 仍然是它自己,死循环。

答案是 redispatch(Dispatcher.h:833-848):

// 关键:"redispatch treats the inputted DispatchKeySet as is, and does NOT
//  mask out the highest priority bit"  — 调用方负责把当前 key 从位图剔掉
template <class Return, class... Args>
inline Return Dispatcher::redispatch(
    const TypedOperatorHandle<Return(Args...)>& op,
    DispatchKeySet currentDispatchKeySet,
    Args... args) const {
  const KernelFunction& kernel = op.operatorDef_->op.lookup(currentDispatchKeySet);
  return kernel.template call<Return, Args...>(op, currentDispatchKeySet, ...);
}

那"怎么剔掉当前 key"?Autograd kernel 内部用 RAII guard 来翻动 TLS exclude:

  • at::AutoDispatchBelowAutograd / 老名字 AutoNonVariableTypeMode:加入所有 autograd keys 到 local.excluded_。
  • at::AutoDispatchBelowADInplaceOrView:进一步把 ADInplaceOrView 也剔掉,用于 codegen 出的 VariableType wrapper。

然后 kernel 调 at::redispatch::<op>(currentDispatchKeySet & after_autograd_keyset, ...) —— 这条路径正好就是 codegen 出来的 VariableType*.cpp 里 每个 op wrapper 的典型骨架,5.5 节会展开。

记住这条心智模型:每个 op 的 dispatch 是一次"从优先级最高位往下找第一个非 fallthrough 的 kernel,kernel 自己负责(通过 guard + redispatch)往下一层传"——所以模式装饰器、autograd、backend 形成一条由数据依赖驱动的链。

2. Schema:op 的"类型签名"

2.1 字符串语法

一个 op 的"类型签名"以一行 schema 字符串描述,比如:

aten::add.Tensor(Tensor self, Tensor other, *, Scalar alpha=1) -> Tensor

各部分拆解:

片段含义
aten命名空间(自定义 op 在自己的 namespace 下,如 myops)
addop 名
.Tensoroverload 名(区分 add.Tensor vs add.Scalar),可省 = ""
Tensor self, Tensor otherpositional 参数
*分隔符,之后的都是 kw-only
Scalar alpha=1带默认值的 kw 参数
-> Tensor返回值

类型语法覆盖:Tensor、Tensor?(可空)、Tensor[](list)、Tensor(a)(alias 标记)、Tensor(a!)(可变 alias)、Scalar、int、int[]、SymInt、bool、str、Device、ScalarType、Layout、MemoryFormat、Generator? 等。

alias 注解最关键:

  • Tensor(a) = 与某个输入共享存储但不写它(view)
  • Tensor(a!) = 与某个输入共享存储且会改写(in-place / out)

这两条注解被 functionalization、autograd-view 追踪、JIT alias analysis 全部读取,所以错一个字直接影响正确性。

注册时这个字符串被 torch::jit::parseSchema 解析进 c10::FunctionSchema 结构,然后才登记到 dispatcher 表里。

2.2 functional / inplace / out 三件套

有副作用能力的 op 在 native_functions.yaml 里通常会出现三个变体:

变体schema 例命名规则
functionaladd.Tensor(Tensor self, Tensor other, *, Scalar alpha=1) -> Tensor无 trailing underscore;返回新 tensor
in-placeadd_.Tensor(Tensor(a!) self, Tensor other, *, Scalar alpha=1) -> Tensor(a!)op 名末尾 _;self 标 (a!);返回 (a!)
outadd.out(Tensor self, Tensor other, *, Scalar alpha=1, Tensor(a!) out) -> Tensor(a!)overload 名 .out;kw-only Tensor(a!) out

这三个一旦写齐,torchgen 还能用 autogen: 给你自动补齐缺的变体。它们三个底层通常共用同一个 structured kernel —— functional 和 inplace 分别分配/复用一个 out tensor,然后 delegate 到 add.out。

2.3 native_functions.yaml 入口

所有 in-tree op 的"出生证"在 aten/src/ATen/native/native_functions.yaml。add 的两个核心条目(第 555 / 578 行附近,版本会漂):

# native_functions.yaml — add.Tensor (functional)
- func: add.Tensor(Tensor self, Tensor other, *, Scalar alpha=1) -> Tensor
  device_check: NoCheck   # TensorIterator
  structured_delegate: add.out
  variants: function, method
  dispatch:
    SparseCPU, SparseCUDA, SparseMPS, SparseMeta: add_sparse
    SparseCsrCPU, SparseCsrCUDA, SparseCsrMeta: add_sparse_csr
    MkldnnCPU: mkldnn_add
    ZeroTensor: add_zerotensor
    NestedTensorCPU, NestedTensorHPU, NestedTensorCUDA: NestedTensor_add_Tensor
  tags: [core, pointwise]

# native_functions.yaml — add.out (structured 锚点)
- func: add.out(Tensor self, Tensor other, *, Scalar alpha=1, Tensor(a!) out) -> Tensor(a!)
  device_check: NoCheck
  structured: True
  structured_inherits: TensorIteratorBase
  ufunc_inner_loop:
    Generic: add (AllAndComplex, BFloat16, Half, ComplexHalf)
    ScalarOnly: add (Bool)
  dispatch:
    SparseCPU, SparseMeta: add_out_sparse_cpu
    SparseCUDA: add_out_sparse_cuda
    MPS: add_out_mps
    MTIA: add_out_mtia
    # 注意:这里没有 CPU / CUDA 条目 ——
    #       Dense × {CPU, CUDA} 的实现完全由 ufunc_inner_loop 自动 codegen 出来

关键字段:

  • variants: function, method:同时生成 at::add(self, other) 和 self.add(other) 两套绑定。
  • structured: True:声明这是一个 structured kernel —— 它有标准 meta() 阶段(算输出 shape/dtype)和 impl() 阶段(算数值),框架自动处理输出分配和广播。
  • structured_delegate: add.out:functional 变体本身不写实现,直接转给 out 变体(由 codegen 生成分配输出 + 调用 out)。
  • ufunc_inner_loop:声明可以走通用 elementwise 流水线(TensorIterator + 模板化的 GPU/CPU 循环),只需在 ATen/native/ufunc/add.h 里给一个 scalar_t-模板化的标量函数。
  • dispatch:把非 Dense × {CPU, CUDA} 之外的特殊后端实现单独列出。
  • tags:供编译期 pass 识别(pointwise、core、nondeterministic_seeded ...)。

非 structured 的简单 op 例子(relu,第 5228 行附近):

- func: relu(Tensor self) -> Tensor
  device_check: NoCheck
  variants: function, method
  dispatch:
    CPU, CUDA: relu       # 直接指向 at::native::relu 的两个 overload
    MPS: relu_mps
    MkldnnCPU: mkldnn_relu
    QuantizedCPU: relu_quantized_cpu
  tags: [core, pointwise]

这里 dispatch: CPU, CUDA: relu 把同一个名字 at::native::relu 同时绑到 (Dense, CPU) 和 (Dense, CUDA)。具体实现可能是同一个文件里的两个 overload(aten/src/ATen/native/Activation.cpp 和 aten/src/ATen/native/cuda/Activation.cu),靠 link 时按 device 决定。

2.4 derivatives.yaml 反向公式

autograd 不在 native_functions.yaml 里写。每个 differentiable op 在 tools/autograd/derivatives.yaml 里给反向公式:

- name: add.Tensor(Tensor self, Tensor other, *, Scalar alpha=1) -> Tensor
  self:  handle_r_to_c(self.scalar_type(),  grad)
  other: handle_r_to_c(other.scalar_type(), maybe_multiply(grad, alpha.conj()))
  result: self_t + maybe_multiply(other_t, alpha)            # forward-mode tangent

- name: relu(Tensor self) -> Tensor
  self: threshold_backward(grad, result, 0)
  result: auto_element_wise

这些行被 codegen 读进 AddBackward0 / ReluBackward0 这种 Node 子类的 apply() 方法体里。下游 5.8 节会看实际生成的代码长啥样。

3. C++ 注册宏全家桶

所有宏定义在 torch/library.h。它们都是 "通过一个 file-static 初始化对象,触发一段 lambda 注册到全局 dispatcher"这一个模式的不同变体。区别只在于:1) 是给 namespace 开新口子还是接进已有的;2) 默认 DispatchKey 是哪个。

3.1 Library::Kind 三种身份

// torch/library.h:551-553
enum Kind { DEF, IMPL, FRAGMENT };
Kind语义触发宏
DEF给某个 namespace 开一个"主"库 —— 全局 namespace 名只允许一个 DEF,重复链接立即 abort。这是大多数自定义 op 用的口子。TORCH_LIBRARY
FRAGMENT给同一个 namespace 追加 一片"片段" —— 同名 fragment 可以来自任意多个 TU,拼到一起算一个库。aten 全靠 fragment 拼,因为代码量太大要拆成几十个 codegen 文件。TORCH_LIBRARY_FRAGMENT
IMPL给"已存在的某个 op"挂某个 DispatchKey 上的 kernel —— 不定义 schema,只能 m.impl(...)。多个 backend 各自的 kernel 通常来自不同 TU 的多个 IMPL 块。TORCH_LIBRARY_IMPL

3.2 TORCH_LIBRARY:开新命名空间

// torch/library.h:982-991
#define TORCH_LIBRARY(ns, m)                                                   \
  static void TORCH_LIBRARY_init_##ns(torch::Library&);                        \
  static const torch::detail::TorchLibraryInit TORCH_LIBRARY_static_init_##ns( \
      torch::Library::DEF,                                                     \
      &TORCH_LIBRARY_init_##ns,                                                \
      C10_STRINGIZE(ns), std::nullopt,                                         \
      __FILE__, __LINE__);                                                     \
  void TORCH_LIBRARY_init_##ns(torch::Library& m)

展开后是一个文件级 static 对象,它的构造函数构造一个 Library(DEF, "ns", nullopt, __FILE__, __LINE__),然后调用你提供的 lambda 体(里面写 m.def(...))。Static 初始化顺序保证它在 main() 之前就把 schema 塞进 dispatcher。

用法 —— 给自定义 namespace 开口子:

// myops_register.cpp — 给一个新 namespace 注册两个 op 的 schema
TORCH_LIBRARY(myops, m) {
  m.def("custom_add(Tensor self, Tensor other) -> Tensor");
  m.def("custom_relu(Tensor self) -> Tensor");
}

注意:同一个 namespace 在整个进程里只能出现一个 TORCH_LIBRARY(ns, ...)。再写一个会触发运行时 abort。如果你需要把同 namespace 的注册拆到多个 .cpp,请用 TORCH_LIBRARY_FRAGMENT。

3.3 TORCH_LIBRARY_FRAGMENT:拼接片段

// torch/library.h:1001-1023 (简化)
#define TORCH_LIBRARY_FRAGMENT(ns, m) _TORCH_LIBRARY_FRAGMENT(ns, m, C10_UID)
#define _TORCH_LIBRARY_FRAGMENT(ns, m, uid)                                \
  static void C10_CONCATENATE(TORCH_LIBRARY_FRAGMENT_init_##ns##_, uid)(  \
      torch::Library&);                                                   \
  static const torch::detail::TorchLibraryInit                            \
      C10_CONCATENATE(TORCH_LIBRARY_FRAGMENT_static_init_##ns##_, uid)(    \
          torch::Library::FRAGMENT, ...,                                   \
          C10_STRINGIZE(ns), std::nullopt, ...);                           \
  void C10_CONCATENATE(...)(torch::Library& m)

和 TORCH_LIBRARY 只有两点不同:

  • Kind 从 DEF 改成 FRAGMENT —— 不抢"唯一所有权"。
  • 符号名用 C10_UID 唯一化,允许同 namespace 在同一个 TU 里出现多次。

codegen 对 aten namespace 强制走 FRAGMENT(torchgen/gen.py ~第 1761-1776 行):

if namespace == "aten":
    aten_schema_registrations = schema_registrations_body
else:
    torch_library_macro = (
        "TORCH_LIBRARY_FRAGMENT"
        if namespace in FRAGMENT_NAMESPACES
        else "TORCH_LIBRARY"
    )

原因很现实:aten 命名空间下有 ~2000+ 个 op,每个 codegen 出来的 RegisterSchema*.cpp 文件都要贡献几十个 schema —— 不可能塞进单个 TORCH_LIBRARY 块。

3.4 TORCH_LIBRARY_IMPL:给某 DispatchKey 注册实现

// torch/library.h:1064-1084 (简化)
#define TORCH_LIBRARY_IMPL(ns, k, m) _TORCH_LIBRARY_IMPL(ns, k, m, C10_UID)
#define _TORCH_LIBRARY_IMPL(ns, k, m, uid)                          \
  static void C10_CONCATENATE(TORCH_LIBRARY_IMPL_init_##ns##_##k##_, uid)(  \
      torch::Library&);                                                    \
  static const torch::detail::TorchLibraryInit                             \
      C10_CONCATENATE(TORCH_LIBRARY_IMPL_static_init_##ns##_##k##_, uid)(   \
          torch::Library::IMPL,                                              \
          ...,                                                               \
          C10_STRINGIZE(ns),                                                 \
          c10::make_optional(c10::DispatchKey::k),                          \
          ...);                                                              \
  void C10_CONCATENATE(...)(torch::Library& m)

构造函数现在带一个 DispatchKey::k 参数。块内的每一次 m.impl("op_name", kernel) 都会被附加到 op 在 该 key 下的 dispatch 表项里。

典型用法:

// myops_cpu.cpp
TORCH_LIBRARY_IMPL(myops, CPU, m) {
  m.impl("custom_add",  &my_add_cpu);
  m.impl("custom_relu", &my_relu_cpu);
}

// myops_cuda.cu
TORCH_LIBRARY_IMPL(myops, CUDA, m) {
  m.impl("custom_add",  &my_add_cuda);
  m.impl("custom_relu", &my_relu_cuda);
}

// myops_autograd.cpp — 单独一个 autograd kernel,用来挂 grad_fn
TORCH_LIBRARY_IMPL(myops, Autograd, m) {
  m.impl("custom_add", &my_add_autograd);
}

特殊形式 TORCH_LIBRARY_IMPL(_, KEY, m):第一个参数下划线表示"所有 namespace 都套用",通常配合 m.fallback(...) 给某个 DispatchKey 注册全局兜底 kernel。aten/src/ATen/core/VariableFallbackKernel.cpp 就这么干:给 AutogradCUDA 等所有 autograd key 注册一个"如果某个 op 没有显式 autograd kernel,就当作 differentiable through composition 处理"的 fallback。

3.5 m.def vs m.impl

m.defm.impl
在哪用仅 TORCH_LIBRARY / TORCH_LIBRARY_FRAGMENT 块内仅 TORCH_LIBRARY_IMPL 块内(或带 explicit key 的 def)
作用声明 schema(给一个新的 op-name 注册类型签名),可选附带一个"无 key" kernel给已声明的 op,在当前块的 DispatchKey 上挂一个具体 kernel
schema 校验第一次 def 就锁定签名;后续不能改kernel 签名必须和 schema 解析后的类型兼容(否则注册时 abort)

两种 m.impl 的写法:

m.impl("op_name", &cpu_kernel);                    // raw fn ptr,框架推断签名
m.impl("op_name", TORCH_FN(cpu_kernel));            // 把函数指针包成 compile-time 类型,
                                                    // 启用更快的 unboxed 调用路径
m.impl("op_name", torch::CppFunction::makeFromBoxedFunction<&boxed_fn>()); // 强制走 boxed

3.6 boxed vs unboxed kernel

每个 kernel 在 dispatcher 内部都用 KernelFunction 表示。它可能存两份函数指针:

  • Unboxed:原生 C++ 签名 Tensor (*)(const Tensor&, const Tensor&, Scalar)。Dispatcher 用模板直接把参数推进去,零拷贝、无类型擦除。所有热路径 op 都走这条。
  • Boxed:统一签名 void (*)(const OperatorHandle&, DispatchKeySet, Stack*)。参数被打包成 IValue 压在 Stack 上。慢一点,但不依赖具体类型 —— functorch interpreter、JIT runtime、profiling hook、fallback kernel 都必须走 boxed。

TORCH_FN(fn) 的作用是用模板提取 fn 的字面类型签名,从而让 dispatcher同时保留 unboxed 调用路径 —— 默认 m.impl("name", &fn) 也能走 unboxed,但 TORCH_FN 写法更明确、能让编译器在签名错的时候立刻报错。

m.impl_UNBOXED 是已删除的老 API,只允许 boxed 调用 —— 现在用 m.impl(..., torch::CppFunction::makeFromBoxedFunction<...>()) 表达同样语义。

3.7 fallback / fallthrough

m.fallback(...) 给当前 IMPL 块的整个 DispatchKey 注册一个 op-agnostic 的 kernel,在该 key 下"没单独 impl 的 op"都走它。常见两种:

m.fallback(torch::CppFunction::makeFallthrough());
    // "我这层不做事,把我的 bit 从位图清掉再走下一层"
    // O(1) 实现:dispatcher 看到这是 fallthrough,直接在 fallthrough_mask 里把这个 key 清零

m.fallback(torch::CppFunction::makeFromBoxedFunction<&my_boxed_fallback>());
    // 真的写个 boxed kernel 兜底,比如 functorch 的 vmap interpreter

典型 fallthrough 用户:BackendSelect(只在 factory function 上有 kernel,别的 op 自动跳过)、Named(没开 named tensor 时全跳过)、Conjugate(没用复数共轭标记时全跳过)。这是 dispatcher 能"零开销支持几十个 functionality"的关键。

3.8 完整自定义 op 模板

把上面的零件拼起来 —— 一个最小但功能完整的自定义 op,在三个 .cpp 文件里:

// ============ myops_schema.cpp ============
// 1) 给 namespace `myops` 开 DEF,声明 schema 一次。
#include <torch/library.h>

TORCH_LIBRARY(myops, m) {
  m.def("custom_relu(Tensor self) -> Tensor");
}

// ============ myops_cpu.cpp ============
// 2) CPU 实现 + 注册到 (myops, CPU)
#include <ATen/ATen.h>
#include <torch/library.h>

at::Tensor custom_relu_cpu(const at::Tensor& self) {
  return at::clamp_min(self, 0);   // 借用现成 aten op 实现
}

TORCH_LIBRARY_IMPL(myops, CPU, m) {
  m.impl("custom_relu", TORCH_FN(custom_relu_cpu));
}

// ============ myops_cuda.cu ============
// 3) CUDA 实现 + 注册到 (myops, CUDA)
at::Tensor custom_relu_cuda(const at::Tensor& self) {
  return at::clamp_min(self, 0);
}

TORCH_LIBRARY_IMPL(myops, CUDA, m) {
  m.impl("custom_relu", TORCH_FN(custom_relu_cuda));
}

// ============ myops_autograd.cpp ============
// 4) Autograd 实现:接住 forward, 建 grad_fn, 把 forward 数学透传到 backend kernel
#include <torch/autograd.h>
#include <torch/library.h>

class CustomReluFn : public torch::autograd::Function<CustomReluFn> {
 public:
  static at::Tensor forward(torch::autograd::AutogradContext* ctx,
                            const at::Tensor& self) {
    // 进 forward 时手动排除"AutogradXxx + ADInplaceOrView" —— 否则下面 dispatcher 调用
    // 会再次落回到本 Function,死循环。
    at::AutoDispatchBelowADInplaceOrView guard;
    // 直接调注册到 myops::custom_relu 的 dispatcher 入口 —— guard 在生效期内
    // AutogradCUDA 已被剔除,会落到 CUDA / CPU 那两个 IMPL 之一(由 self.device() 决定)。
    static auto op = c10::Dispatcher::singleton()
        .findSchemaOrThrow("myops::custom_relu", "")
        .typed<at::Tensor(const at::Tensor&)>();
    auto result = op.call(self);
    ctx->save_for_backward({result});
    return result;
  }
  static torch::autograd::variable_list backward(
      torch::autograd::AutogradContext* ctx,
      torch::autograd::variable_list grad_outputs) {
    auto saved = ctx->get_saved_variables();
    auto& result = saved[0];
    // d/dx relu(x) = (x > 0)
    return { grad_outputs[0] * (result > 0).to(grad_outputs[0].dtype()) };
  }
};

at::Tensor custom_relu_autograd(const at::Tensor& self) {
  return CustomReluFn::apply(self);
}

TORCH_LIBRARY_IMPL(myops, Autograd, m) {
  m.impl("custom_relu", &custom_relu_autograd);
}

Python 端:

>>> import torch, my_extension
>>> x = torch.randn(3, requires_grad=True, device='cuda')
>>> y = torch.ops.myops.custom_relu(x)         # 自动找到 AutogradCUDA → CUDA
>>> y.sum().backward()                          # 自动调 CustomReluFn::backward
更省事的做法:如果你的实现完全是 at::* op 的组合(像上面 clamp_min 那样),可以直接注册到 CompositeImplicitAutograd alias:TORCH_LIBRARY_IMPL(myops, CompositeImplicitAutograd, m) —— 一次注册同时覆盖所有 backend、所有 autograd,因为里头的 sub-op 自己会再走一遍 dispatcher,正确性 + autograd 全免费。

5. aten::add 全栈追踪

设定:x, y 是两个 CUDA tensor,requires_grad=True,我们跑 z = torch.add(x, y)。完整调用栈从 Python 进入,经 autograd、CUDA 后端、ufunc 循环,反向时再回到 autograd engine。下面逐层。

5.1 yaml 入口

schema(native_functions.yaml):

- func: add.Tensor(Tensor self, Tensor other, *, Scalar alpha=1) -> Tensor
  device_check: NoCheck
  structured_delegate: add.out      # functional 走 out 委托
  variants: function, method
  ...
- func: add.out(Tensor self, Tensor other, *, Scalar alpha=1, Tensor(a!) out) -> Tensor(a!)
  structured: True                  # 出口
  structured_inherits: TensorIteratorBase
  ufunc_inner_loop:                 # 触发 ufunc codegen
    Generic: add (AllAndComplex, BFloat16, Half, ComplexHalf)
    ScalarOnly: add (Bool)

反向(derivatives.yaml):

- name: add.Tensor(Tensor self, Tensor other, *, Scalar alpha=1) -> Tensor
  self:  handle_r_to_c(self.scalar_type(),  grad)
  other: handle_r_to_c(other.scalar_type(), maybe_multiply(grad, alpha.conj()))

5.2 Python → C 绑定

torch.add(x, y) 在 Python 解析后跳到 THPVariable_add(生成在 build/.../torch/csrc/autograd/generated/python_torch_functions_2.cpp)。它的工作:

  1. 用 PythonArgParser 按 schema 解析参数 → 拿到 Tensor self、Tensor other、Scalar alpha
  2. 从 THPVariable Python wrapper 中拆出底下的 at::Tensor
  3. 调 C++ API:at::add(self, other, alpha)
  4. 把返回的 at::Tensor 包成 THPVariable 还给 Python

5.3 C++ API → Dispatcher::call

at::add 是 codegen 进 Functions.h 的 inline 转发(签名见 Functions.h ~第 700 行):

// build/aten/src/ATen/Functions.h
inline at::Tensor add(const at::Tensor& self, const at::Tensor& other, const at::Scalar& alpha) {
  return at::_ops::add_Tensor::call(self, other, alpha);
}

_ops::add_Tensor::call 在 Operators.cpp(~第 612 行):

at::Tensor _ops::add_Tensor::call(const at::Tensor& self, const at::Tensor& other, const at::Scalar& alpha) {
  static auto op = create_add_Tensor_typed_handle();       // 进程内缓存 TypedOperatorHandle
  return op.call(self, other, alpha);
}

op.call 最终调到 Dispatcher::call。

5.4 DispatchKeySet 计算

// Dispatcher.h:774-828 — Dispatcher::call (核心两步)
auto dispatchKeySet =
    op.operatorDef_->op.dispatchKeyExtractor()
        .template getDispatchKeySetUnboxed<Args...>(args...);
const KernelFunction& kernel = op.operatorDef_->op.lookup(dispatchKeySet);
return kernel.template call<Return, Args...>(op, dispatchKeySet, ...);

对我们的场景:

  1. 每个输入 tensor 的 TensorImpl::key_set_ = {AutogradCUDA, CUDA} —— 两个 tensor or 起来还是 {AutogradCUDA, CUDA}。
  2. TLS local 默认空(没开 autocast、没在 functorch 里)。
  3. fallthrough mask 把 BackendSelect、Named、Conjugate 等位清零。
  4. highestPriorityTypeId() 取 AutogradCUDA —— 因为 functionality 位里 autograd 排在 Dense 上面。

lookup 返回 AutogradCUDA 槽位下的 kernel。

5.5 AutogradCUDA kernel:建 grad_fn,redispatch

kernel 注册在 torch/csrc/autograd/generated/VariableType_2.cpp(具体分片号会随版本变):

TORCH_LIBRARY_IMPL(aten, AutogradCUDA, m) {
  ...
  m.impl("add.Tensor", TORCH_FN(VariableType::add_Tensor));
  ...
}

kernel 体是 codegen 出来的,典型骨架:

at::Tensor VariableType::add_Tensor(
    c10::DispatchKeySet ks,
    const at::Tensor& self, const at::Tensor& other, const at::Scalar& alpha) {
  // === 1) 若任何输入 requires_grad,准备一个新的 grad_fn 节点 ===
  std::shared_ptr<AddBackward0> grad_fn;
  if (compute_requires_grad(self, other)) {
    grad_fn = std::shared_ptr<AddBackward0>(new AddBackward0(), deleteNode);
    grad_fn->set_next_edges(collect_next_edges(self, other));
    grad_fn->alpha = alpha;
    grad_fn->other_scalar_type = other.scalar_type();
    grad_fn->self_scalar_type  = self.scalar_type();
  }

  // === 2) 把"AutogradCUDA + ADInplaceOrView"从位图剔掉,redispatch ===
  auto _tmp = ([&]() {
    at::AutoDispatchBelowADInplaceOrView guard;
    return at::redispatch::add(
        ks & c10::after_autograd_keyset,    // 留下 CUDA 等更低位
        self, other, alpha);
  })();
  auto result = std::move(_tmp);

  // === 3) 把新算出的 result 挂到 grad_fn,加进 history ===
  if (grad_fn) {
    set_history(flatten_tensor_args(result), grad_fn);
  }
  return result;
}

三步:建节点 → 排除自己 + redispatch 到下一层 → 把输出绑到节点。AddBackward0 是 codegen 出来的 Node 子类,定义在 torch/csrc/autograd/generated/Functions.h,5.8 节会看 apply() 体。

5.6 redispatch 到 CUDA backend

at::redispatch::add 在 RedispatchFunctions.h(codegen),做的事和 at::add 几乎一样,只是多带一个 DispatchKeySet 参数透传给 Dispatcher::redispatch。这次位图里 AutogradCUDA 已经清掉,highestPriorityTypeId 返回 CUDA。

CUDA 槽位上注册的 kernel 在 build/.../RegisterCUDA.cpp:

TORCH_LIBRARY_IMPL(aten, CUDA, m) {
  ...
  m.impl("add.Tensor", TORCH_FN(wrapper_CUDA_add_Tensor));
  m.impl("add.out",    TORCH_FN(wrapper_CUDA_add_out_out));
  ...
}

wrapper_CUDA_add_Tensor 是 functional → out 的转发(因为 yaml 里 structured_delegate: add.out):

at::Tensor wrapper_CUDA_add_Tensor(
    const at::Tensor& self, const at::Tensor& other, const at::Scalar& alpha) {
  // 1) 分配输出
  auto common_dtype = at::native::result_type(self, other);
  auto out = at::detail::empty_cuda({0}, common_dtype, self.device());
  // 2) 转给 out 变体
  return wrapper_CUDA_add_out_out(self, other, alpha, out);
}

5.7 structured kernel + ufunc

wrapper_CUDA_add_out_out 实例化 structured kernel base(继承 TensorIteratorBase):

at::Tensor& wrapper_CUDA_add_out_out(
    const at::Tensor& self, const at::Tensor& other, const at::Scalar& alpha,
    at::Tensor& out) {
  structured_add_out_out op(out);
  op.meta(self, other, alpha);   // 算输出 shape/dtype, 设 TensorIterator
  op.impl(self, other, alpha, op.maybe_get_output(0));   // 真的算
  return out;
}

impl 体由 UfuncCUDA_add.cu 生成 —— ufunc 模板的标准形态:

// build/.../UfuncCUDA_add.cu (示意)
void structured_add_out_out::impl(
    const Tensor& self, const Tensor& other, const Scalar& alpha, const Tensor& out) {
  AT_DISPATCH_ALL_TYPES_AND_COMPLEX_AND4(
      kBFloat16, kHalf, kComplexHalf, kBool,
      iter.common_dtype(), "add_cuda",
      [&] {
        scalar_t alpha_val = alpha.to<scalar_t>();
        gpu_kernel(iter, [alpha_val] GPU_LAMBDA (scalar_t a, scalar_t b) -> scalar_t {
          return at::native::ufunc::add(a, b, alpha_val);
        });
      });
}

AT_DISPATCH_ALL_TYPES_AND_COMPLEX_AND4(aten/src/ATen/Dispatch.h) 展开成一个 switch (scalar_type),对每个 dtype 实例化 lambda;gpu_kernel 接 TensorIterator + lambda,展开成 vectorized_elementwise_kernel<...><<<blocks, threads>>>(...)(aten/src/ATen/native/cuda/CUDALoops.cuh)。

真正的标量数学只有一行,在 ATen/native/ufunc/add.h:

template <typename T>
C10_HOST_DEVICE C10_ALWAYS_INLINE T add(T self, T other, T alpha) {
  return self + alpha * other;
}

所以 CPU 和 CUDA 共享同一个 ufunc::add 标量函数,只是各自的循环 wrapper 不同(CPU 是 SIMD 矢量化的 vectorized_loop2d,CUDA 是 launch 的 kernel)。

5.8 backward:AddBackward0

反向阶段:用户调 z.sum().backward(),autograd engine(torch/csrc/autograd/engine.cpp)从 loss 拓扑排序整张图,对每个 Node 调 apply(grads)。AddBackward0::apply(codegen 出在 generated/Functions.cpp)体大致是:

variable_list AddBackward0::apply(variable_list&& grads) {
  auto& grad = grads[0];
  variable_list grad_inputs(2);
  if (should_compute_output(0)) {
    grad_inputs[0] = handle_r_to_c(self_scalar_type, grad);
  }
  if (should_compute_output(1)) {
    grad_inputs[1] = handle_r_to_c(other_scalar_type, maybe_multiply(grad, alpha.conj()));
  }
  return grad_inputs;
}

对照 derivatives.yaml:

self:  handle_r_to_c(self.scalar_type(),  grad)
other: handle_r_to_c(other.scalar_type(), maybe_multiply(grad, alpha.conj()))

一字不差地翻译。返回的 grad_inputs 通过 next_edges 继续传递给上游节点。maybe_multiply 是个小优化:alpha == 1 时直接返回 grad,跳过乘法。

7. 与老博文的 drift

读旧资料时容易踩的几个坑:

  • 2019 年的 PyTorch internals 博文把 dispatch 描述为"先 device/layout 再 dtype 两轮"。今天的 dispatcher 第一轮已经是 30+ 个 functionality 的多层结构 —— device/layout 只是其中一层,dtype 完全不在 dispatcher 里(由 AT_DISPATCH_* 在 kernel 内部 switch)。
  • 2020 年的 dispatcher 文章还在用 AutoNonVariableTypeMode 这个名字,现在的代码用 AutoDispatchBelowAutograd 和 AutoDispatchBelowADInplaceOrView。老名字以 alias 保留。
  • 官方 tutorial(dispatcher 那篇)的 class MyAddFunction : public torch::autograd::Function<MyAddFunction> 例子里也还在用旧 AutoNonVariableTypeMode guard。
  • BackendComponent 拆分是 2022 年才做的。任何更早的资料把 DispatchKey 当一维大枚举看,理解机制没问题,但"为什么 AutogradCUDA 不是单独枚举值,是 AutogradFunctionality + CUDABit 组出来的"这种细节不能直接套。
  • 一些 key 名变过:VariableTensorId → Autograd / AutogradCPU 等;DefaultBackend → CompositeExplicitAutograd(alias 还在);Autocast → AutocastCUDA(alias 还在)。
  • m.impl_UNBOXED 已删,改用 m.impl(..., torch::CppFunction::makeFromBoxedFunction<...>())。
