用 AOTI 把 PyTorch 模型部署到 CUDA C++ runtime

1. 为什么需要 AOTI

生产环境部署 PyTorch 模型常见的三个痛点:

  • 不想带 Python runtime:推理服务、模拟器、嵌入式环境往往只允许 C++/Rust binary,带 CPython + 整个 PyTorch wheel 既臃肿又增加依赖面。
  • 不想要 JIT 编译开销:torch.compile 第一次跑会做 dynamo trace + Inductor codegen + Triton 编 cubin,延迟几秒到几十秒。 生产场景要冷启动友好。
  • 不想绕 TorchScript / ONNX 中转:TorchScript 早就停更,ONNX 需要双工具链对齐,custom op 难落地。 想要一条"纯 PyTorch"的部署路径。

AOT Inductor(AOTI) 是 PyTorch 2.4+ 对应的方案:

  • 在 Python 端 一次性 把 torch.export 出来的图喂给 Inductor,提前完成所有 codegen(包括 Triton kernel → cubin)和链接
  • 产物是一个 独立的 .pt2 包(本质是 zip),里头打包了模型权重、glue C++ 编出的 .so、每个 Triton kernel 的 cubin、元数据
  • C++ 端只链 libtorch_cpu.so + libtorch_cuda.so,用 torch::inductor::AOTIModelPackageLoader 加载 .pt2 跑推理 —— 不需要 Python interpreter

2. 端到端流程

核心观察:

  1. Python 端只跑一次:跑 export + aoti_compile_and_package,得到 .pt2。 这一步要 GPU(因为 Inductor 用 GPU 编 Triton kernel),但产物 .pt2 可以拷到任何同架构 GPU 的机器上跑。
  2. C++ 端零 Python:只需要 libtorch + CUDA driver/runtime。 加载 .pt2 就拿到一个跟原 Module 等价的可调用对象。
  3. 权重打在 .pt2 内:不需要单独的 weights 文件,部署只移动一个 artifact。

3. 例子模型

构造一个混合 native op 与 custom op 的模型:

# model.py
import torch
from torch import nn

# --- 1) 注册一个 custom op ---
# 语义: y = x * scale + bias_per_channel
# 用 torch.library.custom_op 注册(PyTorch 2.4+ 公开稳定 API)
@torch.library.custom_op("mylib::scale_add", mutates_args=())
def scale_add(x: torch.Tensor, scale: float, bias: torch.Tensor) -> torch.Tensor:
    return x * scale + bias

# Inductor / FakeTensor 用的 meta 实现 (shape / dtype 推导)
@scale_add.register_fake
def _(x, scale, bias):
    return torch.empty_like(x)

# --- 2) 一个用到 native + custom op 的 Module ---
class ToyModel(nn.Module):
    def __init__(self):
        super().__init__()
        self.fc1 = nn.Linear(128, 256)        # native: aten::linear → cuBLAS
        self.fc2 = nn.Linear(256, 64)
        self.bias = nn.Parameter(torch.randn(64))

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        h = self.fc1(x)                       # native, fused by Inductor
        h = torch.nn.functional.gelu(h)       # native, fusible elementwise
        h = self.fc2(h)                       # native
        h = torch.ops.mylib.scale_add(h, 2.0, self.bias)  # custom op
        return h.relu()                       # native, fused with scale_add output

model = ToyModel().eval().cuda()
example_inputs = (torch.randn(4, 128, device="cuda"),)

算子选择的考量:

  • nn.Linear → Inductor 走 ATen 的 aten::linear / aten::mm,实际调用 cuBLAS;Inductor 不为 GEMM 单独生成 Triton
  • F.gelu + scale_add + .relu():都是 elementwise,Inductor 通常把这一连串融合进一个 Triton kernel
  • scale_add:custom op,正好检验 AOTI 怎么处理用户自定义算子

4. 阶段 1:torch.export

4.1 API 用法

# export.py
import torch
from torch.export import export, Dim
from model import ToyModel, scale_add, example_inputs

model = ToyModel().eval().cuda()

# dynamic batch 维度声明(可选;不写就是固定 shape)
batch = Dim("batch", min=1, max=64)
dynamic_shapes = {"x": {0: batch}}

ep = export(model, example_inputs, dynamic_shapes=dynamic_shapes)
# ep: ExportedProgram

print(ep.graph_module.code)         # 可读的 FX graph 源
print(ep.graph_signature)            # inputs/outputs 的 schema

ep.graph_module.code 会打印类似:

def forward(self, p_fc1_weight, p_fc1_bias, p_fc2_weight, p_fc2_bias, p_bias, x):
    linear = torch.ops.aten.linear.default(x, p_fc1_weight, p_fc1_bias)
    gelu = torch.ops.aten.gelu.default(linear)
    linear_1 = torch.ops.aten.linear.default(gelu, p_fc2_weight, p_fc2_bias)
    scale_add = torch.ops.mylib.scale_add.default(linear_1, 2.0, p_bias)
    relu = torch.ops.aten.relu.default(scale_add)
    return (relu,)

注意几件事:

  • 所有 op 已经 normalize 成 torch.ops.{namespace}.{op}.{overload} 形式。 没有 Python 控制流的痕迹,没有 nn.Module 的状态属性。
  • 权重被提升为 graph input(p_fc1_weight 等),通过 ep.state_dict / ep.constants 关联回原来的 Parameter 名字。
  • custom op mylib::scale_add 出现在图里,跟 ATen op 平级 —— 这就是 Inductor 后面能识别它的前提。

4.2 内部流程

torch.export 内部分四步:

  1. Dynamo trace:用 TorchDynamo 的 frame-eval hook 拦截 Python 字节码执行,把所有 torch.* 调用记录成 FX node。 用户代码里的 if / for 在 trace 时被具体 input shape 选定一条路径(symbolic shape 的情况下走 torch._dynamo.guards 注册条件守卫)。
  2. FakeTensor propagation:每个 FX node 用 FakeTensor 推一遍,确定输出 shape / dtype。 custom op 在这一步必须有 register_fake 注册的 meta 实现,否则 export 会失败。
  3. Decomposition:某些"复合"算子(aten::linear 在某些 dispatch 下会拆成 aten::mm + aten::add)被分解。 export 默认走 core ATen decomposition table,可以通过参数控制是否分解到更底层。
  4. Graph signature 抽取:把 graph 的输入分类为(user inputs / parameters / buffers / tokens),输出分类为(user outputs / mutated buffers),形成 ExportSignature。

涉及的关键 PyTorch 源码:

路径角色
torch/export/__init__.py公开 API 入口
torch/export/_trace.pytrace 主流程
torch/_dynamo/eval_frame.pyDynamo Python 端
torch/csrc/dynamo/eval_frame.cframe eval C hook(编进 libtorch_python.so)

5. 阶段 2:Inductor lowering

5.1 API 用法

从 ExportedProgram 编出 .pt2 的最小调用:

import torch._inductor

package_path = torch._inductor.aoti_compile_and_package(
    ep,                                  # 上一步的 ExportedProgram
    package_path="model.pt2",         # 输出文件
    inductor_configs={
        "max_autotune": True,         # 在编译时枚举 Triton config 选最快
        "aot_inductor.embed_kernel_binary": True,
            # 把 cubin 直接嵌进 .so 而不是单独存放(单文件部署更方便)
    },
)
# package_path == "model.pt2"

这一调用内部完成三件事:

  1. 把 ExportedProgram 的 FX graph 喂给 Inductor compiler frontend
  2. Inductor 把图 lower 成调用 ATen + 生成的 Triton kernel 的 C++ glue 代码
  3. 编出 .so,跟 cubin、权重、元数据一起打成 .pt2

5.2 Inductor 内部:Native op vs Custom op vs Fused kernel

Inductor 把每个 FX node 按下面三条路径之一处理:

类型例子Inductor 做什么调用方式
Native heavyweight aten::linear, aten::matmul, aten::conv2d 不 codegen,直接调 ATen API C++ 里 aoti_torch_call_dispatcher("aten::linear", ...) → cuBLAS / cuDNN
Fusible elementwise / reduction aten::gelu, aten::relu, pointwise add / mul 合并相邻的 elementwise + 部分 reduction 进一个 Triton kernel C++ 里 triton_kernel_fused_(...) → cuLaunchKernel(cubin)
Custom op mylib::scale_add 默认 fallback 成"调原 C++/Python 实现",**也可手动注册 lowering 把它融进 Triton** 看下面

对应本例,Inductor 把 graph lower 成下面形态的伪 C++:

// AOTI 生成的 model.cpp (示意,真名是 aoti_kernel.cpp / model.so 主体)
extern "C" AOTIRuntimeError model_main(
    AOTITensorHandle* input_handles,
    AOTITensorHandle* output_handles, ...)
{
    // 1) aten::linear → cuBLAS
    auto buf0 = call_aten_op("aten::linear", x, fc1_weight, fc1_bias);

    // 2) Inductor fused triton kernel: gelu + linear_1 是分开两次调用
    //    因为 linear_1 又是 cuBLAS,夹在两个 elementwise 之间。
    auto buf1 = empty_strided_cuda({4, 256}, ...);
    triton_poi_fused_gelu_0(buf0, buf1, ...);     // elementwise gelu

    auto buf2 = call_aten_op("aten::linear", buf1, fc2_weight, fc2_bias);

    // 3) custom op scale_add + relu 融成一个 Triton kernel
    auto buf3 = empty_strided_cuda({4, 64}, ...);
    triton_poi_fused_scale_add_relu_1(buf2, bias, buf3, scale=2.0);

    output_handles[0] = buf3;
    return AOTI_SUCCESS;
}

关键点:custom op + relu 被 Inductor 融合成一个 Triton kernel。 原因是 torch.library.custom_op 注册时声明了 functional + fake impl,Inductor 据此把它视作 pointwise 处理。 若需要更精细的控制(custom op 实现含非简单运算),需要手动注册 lowering rule(详见 §9)。

6. 阶段 3:Triton codegen + cubin

Inductor 决定生成 Triton kernel 后,实际的 Python → cubin 编译走 triton 这个外部 pip 包(不在 libtorch_*.so 里,详见 [[pytorch-shared-libs]] §3.6 的 OpenAI Triton 基建说明)。 流程是:

  1. Inductor 把生成的 Triton kernel 源(Python 文本)写到 /tmp/torchinductor_$USER/<hash>/triton_poi_fused_scale_add_relu_1.py 之类的位置
  2. 调 triton.jit 触发 Triton compiler:Python AST → Triton IR → MLIR → PTX → cubin
  3. cubin 写到 ~/.triton/cache/,同时 Inductor 在自己的 ~/.cache/torch/inductor/ 里记一份指向
  4. cubin 的字节流被嵌入到 AOTI .so 的 .rodata section(embed_kernel_binary=True)或者单独存放在 .pt2 的 data/cubins/ 子目录里

生成的典型 Triton kernel 形如:

# /tmp/torchinductor_user/abcd/triton_poi_fused_scale_add_relu_1.py
import triton
import triton.language as tl

@triton.jit
def triton_poi_fused_scale_add_relu_1(
    in_ptr0,        # buf2: 上游 linear_1 输出, [4, 64]
    in_ptr1,        # bias: [64]
    out_ptr0,       # buf3: [4, 64]
    xnumel,         # 总 element 数 = 256
    XBLOCK: tl.constexpr):
    xoffset = tl.program_id(0) * XBLOCK
    xindex = xoffset + tl.arange(0, XBLOCK)[:]
    xmask = xindex < xnumel
    x0 = xindex % 64                        # bias 的 broadcast 索引
    tmp0 = tl.load(in_ptr0 + xindex, xmask)
    tmp1 = tl.load(in_ptr1 + x0, xmask)     # 加载 bias
    tmp2 = tmp0 * 2.0                       # scale_add 的 scale 部分
    tmp3 = tmp2 + tmp1                      # scale_add 的 add 部分
    tmp4 = tl.maximum(tmp3, 0.0)            # relu
    tl.store(out_ptr0 + xindex, tmp4, xmask)

这个 kernel 把 custom op 的 * scale + bias 与后面的 relu 完全融合。 Inductor 透过 fake impl 识别 scale_add 的运算为 pointwise,从而合并。 fused 之后整个 graph 在 64 维 输出上只有 1 次 GPU 访存(load → compute → store),而不是原生 Python 路径下的 3 次 elementwise 内存 round-trip。

cubin 的产物路径:

~/.triton/cache/<hash>/triton_poi_fused_scale_add_relu_1.cubin   # Triton 自己缓存
~/.cache/torch/inductor/<hash>/triton_poi_fused_scale_add_relu_1.{py,cubin,json}
                                                                # Inductor 缓存 + metadata

JSON 文件记录了 launch parameters(num_warps, num_stages, shared memory bytes, grid 计算公式),AOTI 把它嵌进生成的 C++ 代码里供 cuLaunchKernel 调用。

8. 阶段 5:C++ runtime 加载

8.1 完整 C++ 代码

// runner.cpp
#include <torch/torch.h>
#include <torch/csrc/inductor/aoti_package/model_package_loader.h>

#include <iostream>
#include <vector>

int main(int argc, char** argv) {
    if (argc < 2) {
        std::cerr << "usage: " << argv[0] << " model.pt2\n";
        return 1;
    }

    // 1) 加载 .pt2
    torch::inductor::AOTIModelPackageLoader loader(argv[1]);

    // 2) 准备输入(跟 export 时的 example_inputs shape/dtype 一致)
    auto device = torch::Device(torch::kCUDA, 0);
    auto x = torch::randn({4, 128}, torch::TensorOptions().device(device));
    std::vector<torch::Tensor> inputs = {x};

    // 3) 执行
    auto outputs = loader.run(inputs);

    // 4) 拿结果
    torch::Tensor y = outputs[0];      // CUDA tensor
    std::cout << "output shape: " << y.sizes() << "\n";
    std::cout << "output mean: "  << y.mean().item<float>() << "\n";
    return 0;
}

8.2 编译命令

(生产场景一般不链 libtorch_python,这里链是因为 AOTIModelPackageLoader 的实现目前还在 libtorch_python 里;下一节会说。)

8.3 内部细节

AOTIModelPackageLoader 构造函数的步骤:

  1. 用 zip 库打开 .pt2,把 data/aotinductor/model/c<hash>.so 解压到临时目录
  2. dlopen() 这个 .so,拿到 model_main 符号
  3. 把 data/constants/*.bin 反序列化成 CUDA Tensor(用 cudaMemcpy 拷到 GPU)
  4. 把 data/cubins/*.cubin(或者 .so 内嵌的 cubin)注册给 CUDA driver:cuModuleLoadData(cubin_bytes) → cuModuleGetFunction(name),得到 CUfunction handle
  5. 用 metadata.json 里的 launch parameter 配置每个 kernel handle

loader.run(inputs) 的步骤:

  1. 把 input at::Tensor 包成 AOTITensorHandle(C-ABI wrapper),配上 strides / shape / dtype
  2. 调 model_main(input_handles, output_handles, constants, ...)
  3. model_main 内部:
    • 遇到 native op → 调 aoti_torch_call_dispatcher("aten::linear", ...) → libtorch_cpu 的 dispatcher → libtorch_cuda 的 CUDA backend → cuBLAS
    • 遇到 Triton kernel → cuLaunchKernel(triton_poi_fused_*_handle, grid, ...)
  4. 把 output handle 转回 at::Tensor,返回

关键 PyTorch 源码(对应 [[pytorch-shared-libs]] §3.6 列的 libtorch_python.so 里 Inductor / Triton 基建那一块):

路径角色
torch/csrc/inductor/aoti_package/model_package_loader.{h,cpp}AOTIModelPackageLoader 实现
torch/csrc/inductor/aoti_runner/model_container_runner.cpp底层 AOTIModelContainerRunner
torch/csrc/inductor/aoti_runner/model_container_runner_cuda.cppCUDA 特化 runner(在 libtorch_cuda.so)
torch/csrc/inductor/aoti_torch/shim_common.cppC-ABI shim 的 host-side 实现
torch/csrc/inductor/aoti_torch/shim_cuda.cppCUDA 版 shim(在 libtorch_cuda.so)
torch/csrc/inductor/static_launcher/cuda.cppTriton kernel 静态 launcher,在 libtorch_python.so

所以一次 loader.run() 跨了这几个 .so:libtorch_python(loader 入口)→ libtorch_cpu(C-ABI shim 主体 + ATen dispatcher)→ libtorch_cuda(CUDA dispatcher + cuLaunchKernel)→ .pt2 解出的临时 .so(model_main 主逻辑)→ 内嵌的 cubin。

9. Custom op 的处理细节

例子里 scale_add 能被 Inductor 融进 Triton kernel 是因为它语义简单(elementwise)+ 有 fake impl。 生产场景中 custom op 通常更复杂,有三种处理策略:

9.1 策略 A:让 Inductor 把它当 fallback(默认)

不做任何额外注册时,Inductor 看到 mylib::scale_add 默认会:

  • 不内联到 Triton
  • 在 AOTI 生成的 C++ 里插一行 aoti_torch_call_dispatcher("mylib::scale_add", ...)
  • 运行时由 dispatcher 走到注册了这个 op 的 backend 实现

但 custom op 的实现是 Python 函数,C++ runtime 加载不到 Python。 这条路径要么:

  • 把 op 用 C++ 重写并通过 TORCH_LIBRARY_IMPL 注册给对应 DispatchKey,然后让 C++ runner 链接你的实现 .so
  • 或者用 torch.library.custom_op 注册时同时提供 device_types=["cuda"] 的 C++ 实现路径(详见官方文档)

9.2 策略 B:注册 lowering 让 Inductor inline 它(推荐用于 elementwise / 简单 reduction)

from torch._inductor import lowering

@lowering.register_lowering(torch.ops.mylib.scale_add)
def _scale_add_lowering(x, scale, bias):
    # 用 Inductor 的内部 IR 表达式拼出来,会被自动 fuse 进相邻 elementwise
    return lowering.add(lowering.mul(x, scale), bias)

Inductor 把 scale_add 视作两个原语的组合,跟周围 elementwise 一起 fuse。 本文例子默认就走这条路径,torch.library.custom_op 内部完成了等价登记。

9.3 策略 C:写一个完整的 Triton 实现并注册

复杂 op(例如自定义 attention kernel)的稳妥做法是把 Triton kernel 也跟着 op 注册:

import triton
import triton.language as tl

@triton.jit
def _scale_add_triton_kernel(x_ptr, bias_ptr, out_ptr, scale, n_elements, BLOCK: tl.constexpr):
    pid = tl.program_id(0)
    offsets = pid * BLOCK + tl.arange(0, BLOCK)
    mask = offsets < n_elements
    x = tl.load(x_ptr + offsets, mask)
    b = tl.load(bias_ptr + offsets % bias_dim, mask)
    tl.store(out_ptr + offsets, x * scale + b, mask)

@scale_add.register_kernel("cuda")
def _(x, scale, bias):
    out = torch.empty_like(x)
    n = x.numel()
    grid = (triton.cdiv(n, 1024),)
    _scale_add_triton_kernel[grid](x, bias, out, scale, n, BLOCK=1024)
    return out

Inductor 看到 custom op 已有 Triton kernel 实现时直接调用,跳过 fallback。 这条路径适合已有手工优化 kernel 的场景。

9.4 第三方 op 只暴露 pybind、没注册到 dispatcher 怎么办

实战里 flash-attn / xformers / 早期 apex 这类第三方库的算子常常只通过 pybind 暴露 Python 函数,既没用 TORCH_LIBRARY 注册 schema,也没用 TORCH_LIBRARY_IMPL 注册 backend。 这种调用 Dynamo 看不见(直接 graph break),自然也走不进 AOTI。

缺什么一目了然:

能力pybind 函数TORCH_LIBRARY op
进入 torch.ops.ns.op 命名空间✗✓
Dynamo 能 trace✗(graph break)✓
FakeTensor / Meta 推 shape✗✓
torch.export 进 FX graph✗✓

对 AOTI 部署有意义的解法是两步并行:

第一步:Python 端用 custom_op 把 pybind 函数包成 op

import torch
import third_party_lib   # 只有 pybind binding 的第三方库

@torch.library.custom_op("mylib::fa_fwd", mutates_args=())
def fa_fwd(q: torch.Tensor, k: torch.Tensor, v: torch.Tensor,
           softmax_scale: float) -> torch.Tensor:
    # 内部直接调 pybind,语义对 PyTorch 是 opaque
    return third_party_lib.flash_attn_forward(q, k, v, softmax_scale)

@fa_fwd.register_fake
def _(q, k, v, softmax_scale):
    # Meta 推断:只产生 empty tensor,不实际算
    return torch.empty_like(q)

这一步做完,Dynamo / export 在 FX graph 里看到的就是 torch.ops.mylib.fa_fwd。 AOTI lowering 时把它当成不可分解的原子调用,生成形如 aoti_torch_call_dispatcher("mylib::fa_fwd", ...) 的 C++。

但:实现还在 Python 里,C++ runtime 加载 .pt2 时这个符号根本不存在,运行会在 dispatcher 查找处崩。 所以还要第二步。

第二步:用 TORCH_LIBRARY 写一个 C++ shim,链进 runner

第三方库一般自带 C++ 入口(pybind 只是它的薄包装)。 写一个独立的 shim .so 注册同名 op,在 C++ runtime 启动时 dlopen 它:

// my_third_party_shim.cpp,编成 libfa_shim.so
#include <torch/library.h>

// 第三方库的真实 C++ 入口(头文件或 dlsym 拿到)
extern at::Tensor third_party_flash_attn_forward(
    const at::Tensor& q, const at::Tensor& k, const at::Tensor& v,
    double softmax_scale);

at::Tensor fa_fwd_cuda_impl(const at::Tensor& q, const at::Tensor& k,
                            const at::Tensor& v, double softmax_scale) {
    return third_party_flash_attn_forward(q, k, v, softmax_scale);
}

// schema 必须跟 Python 端 custom_op 完全一致
TORCH_LIBRARY(mylib, m) {
    m.def("fa_fwd(Tensor q, Tensor k, Tensor v, float softmax_scale) -> Tensor");
}
TORCH_LIBRARY_IMPL(mylib, CUDA, m) {
    m.impl("fa_fwd", fa_fwd_cuda_impl);
}

编 shim:

g++ -shared -fPIC -std=c++17 -O3 \
    -I$TORCH_DIR/include -I$TORCH_DIR/include/torch/csrc/api/include \
    -I/path/to/third_party_headers \
    my_third_party_shim.cpp \
    -L$TORCH_DIR/lib -ltorch -ltorch_cpu -ltorch_cuda -lc10 -lc10_cuda \
    -L/path/to/third_party -lflash_attn \
    -o libfa_shim.so

C++ runner 里在构造 AOTIModelPackageLoader 之前 dlopen 它,让 TORCH_LIBRARY 的全局构造函数把 fa_fwd 注册进 dispatcher:

#include <dlfcn.h>
// ...

torch::inductor::AOTIModelPackageLoader loader("model.pt2");
// ... 后续 loader.run(inputs) 就能正常 dispatch 到 fa_fwd_cuda_impl

具体踩过的几个坑

  1. schema 必须前后一致:Python 端 custom_op 与 C++ 端 TORCH_LIBRARY 注册的参数类型、数量、返回值要完全对齐,否则 aoti_torch_call_dispatcher 查不到对应 overload。
  2. in-place 改输入要声明 mutates_args:flash-attn varlen 之类的 op 经常带 out 输出参数原地写,要写 mutates_args=("out",),否则 functionalization 会乱序。
  3. autograd 不自动接:custom_op 默认无 backward,推理场景可以省略,训练要 fa_fwd.register_autograd(setup_context=..., backward=...)。
  4. shim 必须先 dlopen:libfa_shim.so 的 dlopen 必须在 AOTIModelPackageLoader 构造之前,且用 RTLD_GLOBAL(否则 dispatcher 注册不到全局符号表),否则 AOTI runner 启动时找不到 op,报 "Could not run 'mylib::fa_fwd' with arguments from the 'CUDA' backend"。
  5. 不要在 Python 端只写 pybind 调用就 export:第三方函数对 Dynamo 是不可符号化的外部调用,直接 export 100% 失败。 加 custom_op 包装是绕不开的一步。

10. 坑与边界

  1. SM 兼容性:cubin 是 SM-specific 的(sm_80 / sm_90 / sm_90a)。 export 机器和部署机器的 GPU 必须 SM 等级一致或部署机器更新。 想跨 SM,要么编 PTX(aot_inductor.embed_cubin=False 用 ptx 由 driver JIT),要么按目标 SM 多打几份包。
  2. PyTorch 版本绑定:.pt2 跟编译它的 PyTorch 版本绑定(C-ABI shim 名字 / signature 可能有 minor break),最好部署机器装相同 minor 版本的 libtorch。
  3. Dynamic shape:export 时用 Dim 声明动态维度。 不声明的话部署时输入 shape 必须跟 example 完全一致,否则 dispatch 会失败。 max-autotune 在 dynamic shape 下会扩展 cubin 数量(每个 size bucket 一份)。
  4. Custom op 必须有 fake impl:没注册 register_fake 的 custom op export 时直接报错。
  5. State / 控制流:模型里有 Python 循环(被 Dynamo unroll 处理)、可变状态(list.append)等会让 export 失败或行为意外。 用 torch.cond / torch.while_loop 显式表达。
  6. 首次编译耗时:export + aoti_compile_and_package 在大模型上整体耗时 1 到 10 分钟(Inductor lowering 与 Triton autotune 占主要时间)。 此步骤只跑一次,部署阶段加载为秒级。
  7. 不要链 libtorch_python(理想状态):目前 AOTIModelPackageLoader 实现的部分还在 libtorch_python.so,要在不带 Python 的环境里跑得纯净,需要拷一份不依赖 Python 的 libtorch(BUILD_LIBTORCHLESS 模式)。 这条路径仍在 PyTorch 社区演进,PyTorch 2.5+ 已基本可行,具体细节依装的版本而定。

Deploying a PyTorch Model to a CUDA C++ Runtime via AOT Inductor

1. Why AOTI

Three production-deployment pain points usually push PyTorch users away from a pure Python runtime:

  • You do not want a Python runtime in production. Inference services, simulators, embedded environments often constrain you to C++ or Rust binaries; shipping CPython plus the whole PyTorch wheel is bulky and grows the dependency surface.
  • You do not want JIT compilation latency. torch.compile on first run does Dynamo trace + Inductor codegen + Triton cubin compilation, which can add seconds to tens of seconds. Production wants cold-start to be cheap.
  • You do not want to round-trip through TorchScript or ONNX. TorchScript is in maintenance mode, ONNX requires two toolchains to agree, and custom ops are hard to push through either. A "pure PyTorch" deployment path is more appealing.

AOT Inductor (AOTI) is PyTorch 2.4+'s answer:

  • On the Python side, you run torch.export once and feed the result into Inductor, which produces all the codegen ahead of time (including Triton-kernel-to-cubin compilation) and links it.
  • The output is a self-contained .pt2 package (a zip under the hood) containing the model weights, the C++ glue library, every Triton kernel's cubin, and metadata.
  • On the C++ side, you only link libtorch_cpu.so and libtorch_cuda.so and use torch::inductor::AOTIModelPackageLoader to load the .pt2 and run inference. No Python interpreter is involved at runtime.

2. End-to-end flow

See the diagram in the ZH section above. Key takeaways:

  1. The Python side runs once. You run export and aoti_compile_and_package to produce a .pt2. A GPU is required at this stage because Inductor uses it to compile Triton kernels, but the resulting .pt2 can be copied to any same-architecture GPU machine.
  2. The C++ side needs zero Python. All you need is libtorch plus the CUDA driver/runtime. Loading the .pt2 gives you a callable that is equivalent to the original module.
  3. Weights are packed inside the .pt2. No separate weights file is needed; deployment moves a single artifact.

3. The example model

A toy module that deliberately mixes native ops and a custom op:

# model.py
import torch
from torch import nn

# --- 1) Register a custom op ---
# Semantics: y = x * scale + bias_per_channel
# Registered via torch.library.custom_op (the stable public API since PyTorch 2.4).
@torch.library.custom_op("mylib::scale_add", mutates_args=())
def scale_add(x: torch.Tensor, scale: float, bias: torch.Tensor) -> torch.Tensor:
    return x * scale + bias

# Meta implementation for FakeTensor / Inductor (shape / dtype inference).
@scale_add.register_fake
def _(x, scale, bias):
    return torch.empty_like(x)

# --- 2) A Module that uses both native and custom ops ---
class ToyModel(nn.Module):
    def __init__(self):
        super().__init__()
        self.fc1 = nn.Linear(128, 256)        # native: aten::linear → cuBLAS
        self.fc2 = nn.Linear(256, 64)
        self.bias = nn.Parameter(torch.randn(64))

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        h = self.fc1(x)
        h = torch.nn.functional.gelu(h)
        h = self.fc2(h)
        h = torch.ops.mylib.scale_add(h, 2.0, self.bias)   # custom op
        return h.relu()

The op choices are intentional:

  • nn.Linear: Inductor lowers it to aten::linear / aten::mm, which dispatches to cuBLAS. Inductor does not generate a Triton kernel for GEMM.
  • F.gelu + scale_add + .relu(): all elementwise; Inductor will almost certainly fuse them into a single Triton kernel.
  • scale_add: a custom op that lets us check how AOTI handles user-defined operators.

4. Phase 1: torch.export

4.1 API

import torch
from torch.export import export, Dim
from model import ToyModel, example_inputs

model = ToyModel().eval().cuda()
batch = Dim("batch", min=1, max=64)
ep = export(model, example_inputs, dynamic_shapes={"x": {0: batch}})

print(ep.graph_module.code)         # readable FX graph

The printed graph looks like:

def forward(self, p_fc1_weight, p_fc1_bias, p_fc2_weight, p_fc2_bias, p_bias, x):
    linear = torch.ops.aten.linear.default(x, p_fc1_weight, p_fc1_bias)
    gelu = torch.ops.aten.gelu.default(linear)
    linear_1 = torch.ops.aten.linear.default(gelu, p_fc2_weight, p_fc2_bias)
    scale_add = torch.ops.mylib.scale_add.default(linear_1, 2.0, p_bias)
    relu = torch.ops.aten.relu.default(scale_add)
    return (relu,)

Observations:

  • Every op is normalised into torch.ops.{namespace}.{op}.{overload} form. No Python control flow, no nn.Module attribute access.
  • Weights are lifted into graph inputs (p_fc1_weight etc.), with the mapping back to the original Parameter names tracked in ep.state_dict / ep.constants.
  • The custom op mylib::scale_add sits in the graph alongside ATen ops; this is what lets Inductor recognise it later.

4.2 Internals

torch.export is a single function but it does four things internally:

  1. Dynamo trace: TorchDynamo's frame-eval hook intercepts Python bytecode, recording every torch.* call as an FX node. if and for in user code are specialised against the concrete input shapes at trace time (with symbolic shape mode adding guards via torch._dynamo.guards).
  2. FakeTensor propagation: each FX node is run with FakeTensor to determine output shape and dtype. Custom ops MUST have a register_fake meta implementation here or export fails.
  3. Decomposition: certain composite ops (aten::linear in some dispatch paths) decompose into more primitive ones (aten::mm + aten::add). Export uses the core-ATen decomposition table by default; you can control how far decomposition goes.
  4. Graph signature: graph inputs are classified into (user inputs / parameters / buffers / tokens) and outputs into (user outputs / mutated buffers), producing an ExportSignature.

5. Phase 2: Inductor lowering

5.1 API

import torch._inductor

package_path = torch._inductor.aoti_compile_and_package(
    ep,
    package_path="model.pt2",
    inductor_configs={
        "max_autotune": True,
        "aot_inductor.embed_kernel_binary": True,

This single call does three things: feed the ExportedProgram's FX graph into the Inductor compiler frontend, lower it into a mix of ATen calls and generated Triton kernels with C++ glue, then build a .so and package it with the cubins, weights, and metadata into a .pt2.

5.2 Native op vs custom op vs fused kernel

Inductor routes each FX node through one of three paths:

KindExamplesInductor's actionCall form
Heavyweight native aten::linear, aten::matmul, aten::conv2d No codegen; call ATen directly C++ aoti_torch_call_dispatcher("aten::linear", ...) → cuBLAS / cuDNN
Fusible elementwise / reduction aten::gelu, aten::relu, pointwise add / mul Merge adjacent elementwise (and some reduction) ops into a single Triton kernel C++ triton_kernel_fused_(...) → cuLaunchKernel(cubin)
Custom op mylib::scale_add Default fallback: call the registered backend impl. Can also be inlined into Triton via a lowering rule See below

For this example, Inductor produces roughly the following pseudo-C++:

// AOTI-generated model.cpp (sketch; actual file is aoti_kernel.cpp / model.so main body)
extern "C" AOTIRuntimeError model_main(
    AOTITensorHandle* input_handles,
    AOTITensorHandle* output_handles, ...)
{
    auto buf0 = call_aten_op("aten::linear", x, fc1_weight, fc1_bias);   // cuBLAS

    auto buf1 = empty_strided_cuda({4, 256}, ...);
    triton_poi_fused_gelu_0(buf0, buf1, ...);                             // elementwise gelu

    auto buf2 = call_aten_op("aten::linear", buf1, fc2_weight, fc2_bias); // cuBLAS

    auto buf3 = empty_strided_cuda({4, 64}, ...);
    triton_poi_fused_scale_add_relu_1(buf2, bias, buf3, scale=2.0);       // fused custom op + relu

    output_handles[0] = buf3;
    return AOTI_SUCCESS;
}

Key observation: the custom op fuses with relu into one Triton kernel. This works because torch.library.custom_op registers the op as functional with a fake impl, and Inductor treats it as pointwise. For more complex custom ops you need an explicit lowering rule (see §9).

6. Phase 3: Triton codegen + cubin

Once Inductor decides to generate a Triton kernel, the Python-to-cubin compilation goes through the external triton pip package (not part of libtorch_*.so; see [[pytorch-shared-libs]] §3.6 on OpenAI Triton infrastructure). The flow:

  1. Inductor writes the generated Triton source (Python text) to a path like /tmp/torchinductor_$USER/<hash>/triton_poi_fused_scale_add_relu_1.py.
  2. It invokes triton.jit to trigger compilation: Python AST → Triton IR → MLIR → PTX → cubin.
  3. The cubin lands in ~/.triton/cache/, with Inductor keeping a pointer to it under ~/.cache/torch/inductor/.
  4. The cubin bytes are either embedded in the AOTI .so's .rodata section (embed_kernel_binary=True) or kept as separate files under data/cubins/ inside the .pt2.

A typical generated kernel looks like:

# /tmp/torchinductor_user/abcd/triton_poi_fused_scale_add_relu_1.py
import triton
import triton.language as tl

@triton.jit
def triton_poi_fused_scale_add_relu_1(
    in_ptr0, in_ptr1, out_ptr0, xnumel, XBLOCK: tl.constexpr):
    xoffset = tl.program_id(0) * XBLOCK
    xindex = xoffset + tl.arange(0, XBLOCK)[:]
    xmask = xindex < xnumel
    x0 = xindex % 64
    tmp0 = tl.load(in_ptr0 + xindex, xmask)
    tmp1 = tl.load(in_ptr1 + x0, xmask)        # load bias
    tmp2 = tmp0 * 2.0                          # scale_add scale part
    tmp3 = tmp2 + tmp1                         # scale_add add part
    tmp4 = tl.maximum(tmp3, 0.0)               # relu
    tl.store(out_ptr0 + xindex, tmp4, xmask)

This kernel completely fuses the custom op's * scale + bias with the trailing relu. Inductor sees through the custom op's fake impl and treats it as elementwise. Across the 64-wide output the entire fused chain touches GPU memory exactly once (load, compute, store) instead of three round-trips in the native Python path.

Cubin artifact locations:

~/.triton/cache/<hash>/triton_poi_fused_scale_add_relu_1.cubin
~/.cache/torch/inductor/<hash>/triton_poi_fused_scale_add_relu_1.{py,cubin,json}

The JSON records launch parameters (num_warps, num_stages, shared memory bytes, the grid formula). AOTI embeds these into the generated C++ for the cuLaunchKernel call.

(Production deployments typically avoid linking libtorch_python; the dependency is shown here because the AOTIModelPackageLoader implementation still lives in libtorch_python today. See the pitfalls.)

8.3 What actually happens

AOTIModelPackageLoader constructor:

  1. Opens the .pt2 with a zip library and extracts data/aotinductor/model/c<hash>.so to a temp directory.
  2. dlopen()s that .so and resolves the model_main symbol.
  3. Deserialises data/constants/*.bin into CUDA tensors (via cudaMemcpy).
  4. Registers each cubin (either from data/cubins/*.cubin or embedded in the .so) with the CUDA driver: cuModuleLoadData(cubin_bytes) → cuModuleGetFunction(name), yielding CUfunction handles.
  5. Configures launch parameters per handle from metadata.json.

loader.run(inputs):

  1. Wraps each input at::Tensor as an AOTITensorHandle (C-ABI) carrying strides, shape, and dtype.
  2. Calls model_main(input_handles, output_handles, constants, ...).
  3. Inside model_main:
    • Each native op uses aoti_torch_call_dispatcher("aten::linear", ...) → libtorch_cpu dispatcher → libtorch_cuda CUDA backend → cuBLAS.
    • Each Triton kernel uses cuLaunchKernel(triton_poi_fused_*_handle, grid, ...).
  4. Converts output handles back to at::Tensor and returns them.

Relevant PyTorch source (the Inductor / Triton infrastructure inside libtorch_python.so covered in [[pytorch-shared-libs]] §3.6):

PathRole
torch/csrc/inductor/aoti_package/model_package_loader.{h,cpp}AOTIModelPackageLoader implementation
torch/csrc/inductor/aoti_runner/model_container_runner.cppUnderlying AOTIModelContainerRunner
torch/csrc/inductor/aoti_runner/model_container_runner_cuda.cppCUDA-specialised runner (in libtorch_cuda.so)

So a single loader.run() crosses several .so boundaries: libtorch_python (loader entry) → libtorch_cpu (C-ABI shim plus ATen dispatcher) → libtorch_cuda (CUDA dispatcher and cuLaunchKernel) → the temp .so extracted from the .pt2 (the model_main body) → the embedded cubins.

9. Custom op handling details

The scale_add example fuses cleanly because its semantics are simple (elementwise) and it has a fake impl. Real-world custom ops are usually heavier, and there are three handling strategies.

9.1 Strategy A: let Inductor fall back (default)

Without any extra registration, Inductor sees mylib::scale_add and:

  • Does not inline it into Triton.
  • Emits a literal aoti_torch_call_dispatcher("mylib::scale_add", ...) in the AOTI-generated C++.
  • At runtime the dispatcher routes to whichever backend the op is registered against.

But the op's implementation here is a Python function, which the C++ runtime cannot reach. You have to either rewrite the op in C++ and register it via TORCH_LIBRARY_IMPL for the right DispatchKey (and link your impl .so into the runner), or provide a C++ implementation through the device_types=["cuda"] path of torch.library.custom_op.

9.2 Strategy B: register a lowering so Inductor inlines it (best for elementwise / simple reduction)

from torch._inductor import lowering

@lowering.register_lowering(torch.ops.mylib.scale_add)
def _scale_add_lowering(x, scale, bias):
    return lowering.add(lowering.mul(x, scale), bias)

Now Inductor treats scale_add as a composition of primitives, fusing it with surrounding elementwise ops. This example walks that path: torch.library.custom_op arranges the equivalent registration internally.

9.3 Strategy C: ship a Triton implementation along with the op

For heavier ops (e.g. a custom attention kernel), the cleanest path is to register a Triton kernel alongside the op:

import triton
import triton.language as tl

@triton.jit
def _scale_add_triton_kernel(x_ptr, bias_ptr, out_ptr, scale, n_elements, BLOCK: tl.constexpr):
    pid = tl.program_id(0)
    offsets = pid * BLOCK + tl.arange(0, BLOCK)
    mask = offsets < n_elements
    x = tl.load(x_ptr + offsets, mask)
    b = tl.load(bias_ptr + offsets % bias_dim, mask)
    tl.store(out_ptr + offsets, x * scale + b, mask)

@scale_add.register_kernel("cuda")
def _(x, scale, bias):
    out = torch.empty_like(x)
    n = x.numel()
    grid = (triton.cdiv(n, 1024),)
    _scale_add_triton_kernel[grid](x, bias, out, scale, n, BLOCK=1024)
    return out

Inductor sees the registered Triton kernel and calls it directly, bypassing any fallback. This path suits "already hand-optimised Triton kernels."

9.4 Third-party ops exposed only through pybind, never registered with the dispatcher

In practice, libraries like flash-attn, xformers, and early apex frequently expose their CUDA ops only as pybind functions: no TORCH_LIBRARY schema, no TORCH_LIBRARY_IMPL backend. Dynamo cannot see these calls (it graph-breaks at them), and they cannot enter AOTI.

What is missing:

Capabilitypybind functionTORCH_LIBRARY op
Accessible as torch.ops.ns.op✗✓
Dynamo traces it✗ (graph break)✓

The AOTI-ready fix is a two-step solution.

Step 1: wrap the pybind call in custom_op on the Python side

import torch
import third_party_lib   # pybind-only third-party module

@torch.library.custom_op("mylib::fa_fwd", mutates_args=())
def fa_fwd(q: torch.Tensor, k: torch.Tensor, v: torch.Tensor,
           softmax_scale: float) -> torch.Tensor:
    # Call into pybind directly; the semantics are opaque to PyTorch.
    return third_party_lib.flash_attn_forward(q, k, v, softmax_scale)

@fa_fwd.register_fake
def _(q, k, v, softmax_scale):
    # Meta inference: produce an empty tensor of the right shape / dtype.
    return torch.empty_like(q)

After this step, Dynamo and export see torch.ops.mylib.fa_fwd in the FX graph. AOTI lowering treats it as an atomic, non-decomposable call and emits aoti_torch_call_dispatcher("mylib::fa_fwd", ...) in the generated C++.

But: the implementation is still in Python. When the C++ runtime loads the .pt2, the symbol does not exist; execution crashes at the dispatcher lookup. Hence step 2.

Step 2: write a TORCH_LIBRARY C++ shim and link it into the runner

Third-party libraries usually expose a real C++ entry point underneath the pybind wrapper. Build a separate shim .so that registers the same op against the dispatcher, then dlopen it at C++ runtime startup:

// my_third_party_shim.cpp — built into libfa_shim.so
#include <torch/library.h>

// Real C++ entry from the third-party library (header or dlsym).
extern at::Tensor third_party_flash_attn_forward(
    const at::Tensor& q, const at::Tensor& k, const at::Tensor& v,
    double softmax_scale);

at::Tensor fa_fwd_cuda_impl(const at::Tensor& q, const at::Tensor& k,
                            const at::Tensor& v, double softmax_scale) {
    return third_party_flash_attn_forward(q, k, v, softmax_scale);
}

// Schema must match the Python-side custom_op registration exactly.
TORCH_LIBRARY(mylib, m) {
    m.def("fa_fwd(Tensor q, Tensor k, Tensor v, float softmax_scale) -> Tensor");
}
TORCH_LIBRARY_IMPL(mylib, CUDA, m) {
    m.impl("fa_fwd", fa_fwd_cuda_impl);
}

Build the shim:

g++ -shared -fPIC -std=c++17 -O3 \
    -I$TORCH_DIR/include -I$TORCH_DIR/include/torch/csrc/api/include \
    -I/path/to/third_party_headers \
    my_third_party_shim.cpp \
    -L$TORCH_DIR/lib -ltorch -ltorch_cpu -ltorch_cuda -lc10 -lc10_cuda \
    -L/path/to/third_party -lflash_attn \
    -o libfa_shim.so

In the C++ runner, dlopen the shim before constructing AOTIModelPackageLoader so the TORCH_LIBRARY global constructors register fa_fwd into the dispatcher first:

#include <dlfcn.h>
// ...
void* shim = dlopen("./libfa_shim.so", RTLD_LAZY | RTLD_GLOBAL);
if (!shim) { std::cerr << dlerror() << "\n"; return 1; }

torch::inductor::AOTIModelPackageLoader loader("model.pt2");
// loader.run(inputs) now dispatches correctly into fa_fwd_cuda_impl.

Pitfalls observed in practice

  1. Schema must match on both sides. The argument types, count, and return signature of the Python custom_op and the C++ TORCH_LIBRARY registration must be identical, otherwise aoti_torch_call_dispatcher cannot resolve the overload.
  2. Declare mutates_args for in-place outputs. Ops like flash-attn varlen write into an out parameter; missing mutates_args=("out",) confuses functionalization and produces out-of-order memory effects.
  3. Autograd is not automatic. custom_op has no backward by default; for training, register it with fa_fwd.register_autograd(setup_context=..., backward=...). Inference-only deployment can ignore this.
  4. Shim load order matters. libfa_shim.so must be dlopened before AOTIModelPackageLoader's constructor and with RTLD_GLOBAL (otherwise dispatcher registration does not reach the global symbol table). If the shim loads too late, AOTI startup fails with "Could not run 'mylib::fa_fwd' with arguments from the 'CUDA' backend".
  5. Do not skip the wrapper. Calling the pybind function directly and then running torch.export always fails: Dynamo cannot symbolise an external function. The custom_op wrapper is non-optional.

10. Pitfalls

  1. SM compatibility. Cubins are SM-specific (sm_80 / sm_90 / sm_90a). The export machine and the deployment machine must share an SM level (or the deployment machine must be newer). For cross-SM portability, emit PTX (aot_inductor.embed_cubin=False) and let the driver JIT it, or build separate packages per target SM.
  2. PyTorch version pinning. A .pt2 is bound to the PyTorch version that built it (the C-ABI shim signatures can break on minor versions). Match libtorch minor versions between build and deployment.
  3. Dynamic shapes. Declare dynamic dims via Dim at export. Otherwise deployment inputs must match the example shapes exactly. With max_autotune in dynamic-shape mode, expect the cubin count to grow (one bucket per size class).
  4. Custom op must have a fake impl. A custom op without register_fake fails export outright.
  5. State / control flow. Python loops (Dynamo unrolls them), mutable state (list.append) etc. can fail export or change behaviour. Use torch.cond / torch.while_loop for explicit control flow.
  6. First-run cost. Export plus aoti_compile_and_package can take 1 to 10 minutes for sizable models (Inductor lowering and Triton autotuning dominate). It runs once; deployment loads in seconds.
  7. Avoid linking libtorch_python (ideally). The AOTIModelPackageLoader implementation still resides in libtorch_python.so at the time of writing, so a truly Python-free environment needs a Python-less libtorch build (the BUILD_LIBTORCHLESS path). The community is still shaking this out; PyTorch 2.5+ has the bones in place, but details vary by exact version.
