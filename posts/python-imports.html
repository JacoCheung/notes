Python Import 机制笔记：绝对导入 vs 相对导入#

1. 基础概念#

Module（模块）#

一个 .py 文件就是一个 module。被 import 之后，会变成一个 module 对象，加进 sys.modules 缓存里，避免重复执行。

Package（包）#

一个 包含 __init__.py 的目录 就是一个 package（"regular package"）。它本身也是一个 module 对象，但额外有 __path__ 属性，能继续装下层子 module / 子 package。

PEP 420 之后还有一种 namespace package：目录里没有 __init__.py，可以跨多个物理目录拼接。它是真实生产场景里会踩到的坑，单独成节展开（见下面"Namespace package"小节）。

__init__.py 的作用#

  1. 标记：让目录被当作 regular package 识别。
  2. 初始化代码：第一次 import pkg 时执行一次（即使只是 from pkg.a import x 也会先执行 pkg/__init__.py）。
  3. re-export 门面：常见用法是在里面 from .a import Foo，让外面可以直接写 from pkg import Foo。
  4. 定义 __all__：控制 from pkg import * 暴露哪些名字。

顶级 package（top-level package）#

sys.path 里某条目录的直接子项 —— 如果它是个带 __init__.py 的目录，那它就是一个顶级 package。

举例：sys.path 里有 /home/me/proj，而 /home/me/proj/app/__init__.py 存在，那 app 就是顶级 package。app.pkg、app.pkg.sub 是它的子 package，但不是顶级。

判断"谁是顶级"完全取决于 sys.path，跟磁盘上的目录嵌套没有必然关系。

Namespace package（PEP 420）#

目录里没有 __init__.py，但 Python 仍然把它当成一个 package。它和 regular package 的核心区别：

regular package namespace package
__init__.py 必须有 必须没有
__file__ 有 没有
__path__ 普通 list _NamespacePath（可由多个物理目录拼成）
用途 单一 wheel 拥有整个 package 多个 wheel 拼接同一个 import 命名空间

典型场景：fbgemm_gpu/experimental/ 这种 hub-style 目录故意不放 __init__.py——这样 fbgemm_gpu_experimental_hstu、fbgemm_gpu_experimental_gen_ai 等独立 wheel 可以各自往 site-packages/fbgemm_gpu/experimental/<name>/ 装自己的子 package，运行时 Python 把它们合并成同一个 fbgemm_gpu.experimental 命名空间。

关键约束：只要任何一个 wheel 在这个目录里塞进 __init__.py，它立刻退化成 regular package，从此别的 wheel 再往里装的子目录就不再被识别了（见反例 6）。

2. import 是怎么工作的#

sys.path 决定能找到什么#

import X 的本质是：解释器拿 X 这个点分名字，去 sys.modules 找；找不到就让 finder 沿 sys.path 找顶级 package，再顺着子 package 的 __path__ 一层一层往下找。

sys.path 的常见来源： - 启动脚本所在目录（python a.py → a.py 的目录被插到 sys.path[0]） - python -m pkg.mod → 当前工作目录被插到 sys.path[0]（这是 -m 和直接跑文件最大的区别之一） - PYTHONPATH 环境变量 - 安装的 site-packages

__name__ 和 __package__#

每个 module 都有这两个属性，决定相对导入怎么解析。

启动方式 __name__ __package__
python app/pkg/a.py（直接跑文件） "__main__" None 或 ""
python -m app.pkg.a（按模块跑） "__main__" "app.pkg"
被别处 import app.pkg.a "app.pkg.a" "app.pkg"

相对导入（from . import x）只看 __package__：以它为起点，每多一个点退一层。如果 __package__ 是 None/""，相对导入直接失败。

这就是为什么直接 python a.py 跑一个包里的文件，里面的 from . import b 会立刻报错——它根本不知道自己属于哪个 package。

绝对导入 vs 相对导入#

  • 绝对导入：from app.pkg import b，从 sys.path 根开始找完整路径。可读性好，跨 package 移动文件后基本不用改。
  • 相对导入：from . import b、from ..sib import e，基于 __package__ 算。重命名顶级 package 时不用改，但不能跨出顶级，并且不能在脚本入口里用。

PEP 8 推荐绝对导入；相对导入用于 package 内部 同辈/近亲互引。

dotted name 是怎么一段段解析的#

from a.b.c import d 并不是一步到位，而是按 . 一段段走，每段独立决定它是 regular / namespace package 还是普通 module：

  1. 找 a：沿 sys.path 找叫 a 的目录 / 文件，写进 sys.modules['a']。
  2. 找 a.b：在 a.__path__ 里找 b，再走一遍 finder。如果 a 是 namespace package，a.__path__ 可能是多个物理目录的合集，b 可能来自其中任何一个。
  3. 找 a.b.c：在 a.b.__path__ 里找 c。
  4. 从 a.b.c 这个 module 对象上取属性 d。

每一步加载的 module 都写进 sys.modules，下次重复 import 直接命中缓存、不再走 finder。所以中间任何一层只要被 import 过一次，后面再换 sys.path 也不会让它"重新解析"。

import 的两个隐藏副作用#

这两条副作用解释了大量"为什么空的 __init__.py 也能这样用"的困惑。

(a) import a.b 会把 b 设为 a 的属性。 这是 import 系统在加载子模块时主动做的赋值，不是 __init__.py 里必须写 from . import b 才有的效果：

import a.b           # 哪怕 a/__init__.py 完全为空
a.b.foo()            # 也能 work：import 系统已经把 b 绑到 a 上了

(b) from a import b 在 a 上找不到属性 b 时，会兜底去加载子模块 a.b。 这就是为什么 __init__.py 里没写 from . import b，外面仍然能 from a import b 拿到子模块。注意这条兜底只对单层 attribute 生效——from a import b.c 是语法错误。

同名子模块 / 同名顶级 package 的歧义#

import 系统只看名字，不区分"逻辑上是同一个包"。两种常见歧义：

  • 同名顶级 package 撞车：sys.path 里有两条目录都包含 foo/，import foo 只会拿到 sys.path 中第一条命中的那个，另一个永远见不到。靠 foo.__file__ 排查。
  • 同名子模块不同含义：例如 fbgemm_gpu.experimental.hstu.hstu_ops_gpu（C++/CUDA 绑定）和 fbgemm_gpu.experimental.hstu.hstu_blackwell.hstu_ops_gpu（Triton 实现）是两个完全不同的 module，只是末尾文件名重合。靠绝对路径区分，常见做法是在调用方 import ... as _sm90 / as _sm100 用别名 dispatch。

3. 典型例子#

假设项目结构（sys.path 里有 /proj，所以 app 是顶级 package）：

/proj/
  app/                  <- 顶级 package
    __init__.py
    main.py             <- 入口：python -m app.main
    pkg/
      __init__.py
      a.py
      b.py
      sub/
        __init__.py
        c.py
    sib/
      __init__.py
      e.py
      sub2/
        __init__.py
        f.py

例 1：同目录导入兄弟 module#

app/pkg/a.py 想用 app/pkg/b.py 里的 foo：

# 相对（推荐用于 package 内部）
from . import b
from .b import foo

# 绝对（也合法）
from app.pkg import b
from app.pkg.b import foo

. 表示"当前 package"，也就是 __package__ == "app.pkg"。

例 2：从兄弟 package 导入#

app/pkg/a.py 想用 app/sib/e.py 里的 bar：

# 相对：两个点退到 app，再进 sib
from ..sib import e
from ..sib.e import bar

# 绝对
from app.sib import e
from app.sib.e import bar

注意 .. 不能越过顶级 package。这里 pkg 和 sib 都是 app 的子 package，.. 退到 app 是合法的。如果app 下面没有__init__.py, 就是非法的.

例 3：从兄弟 package 的下一级导入#

app/pkg/a.py 想用 app/sib/sub2/f.py 里的 baz：

# 相对
from ..sib.sub2 import f
from ..sib.sub2.f import baz

# 绝对
from app.sib.sub2 import f
from app.sib.sub2.f import baz

例 4：脚本入口#

app/main.py 作为入口，引用其它模块：

# main.py —— 推荐用 -m 启动：python -m app.main
from app.pkg.a import something    # 绝对，最稳
from .pkg.a import something       # 相对，要求 __package__ == "app"，即用 -m 启动

4. 反例（常见报错）#

反例 1：ImportError: attempted relative import with no known parent package#

$ cd /proj/app/pkg
$ python a.py
a.py 里有 from . import b。直接跑文件，a.py 的 __name__ == "__main__"，__package__ 是 None——它压根不知道自己属于 app.pkg，相对导入起点丢了。

修法： - 改用 python -m app.pkg.a（在 /proj 下跑），让解释器把它当 module 加载，__package__ 会被设成 "app.pkg"。 - 或者把 a.py 写成纯脚本，里面只用绝对导入 from app.pkg import b（前提是 /proj 在 sys.path 里）。

反例 2：ModuleNotFoundError: No module named 'app'#

$ cd /home/me
$ python -m app.main
当前目录是 /home/me，sys.path[0] 是 /home/me，里面根本没有 app/ 目录。解释器找不到顶级 package。

修法： - cd /proj && python -m app.main，或者 - PYTHONPATH=/proj python -m app.main，或者 - 把项目装成可编辑包（pip install -e . + pyproject.toml），让 app 进 site-packages。

变体：from app.pgk.a import x（拼错 pkg → pgk）也会报同样错——错误信息只告诉你"没这个名字"，不会告诉你拼错了。

反例 3：ImportError: attempted relative import beyond top-level package#

# app/__init__.py 里写：
from ..util import helper
app 已经是顶级 package，.. 想退到 app 的"父 package"——根本不存在。Python 不允许相对导入跨出顶级 package 的边界。

注意：错误信息里的 "top-level package" 指的是 相对导入起点所在的顶级 package（这里是 app），不是说"我已经在最外层目录了"。即使磁盘上 app/ 外面还有别的目录，也无关——sys.path 决定的"顶"就是顶。

修法：用绝对导入 from util import helper（前提是 util 也是顶级 package），或者把 util 挪进 app 里变成 app.util。

反例 4：__init__.py 里循环 import → ImportError: cannot import name 'X'#

# app/pkg/__init__.py
from .a import Foo

# app/pkg/a.py
from . import b           # 触发 pkg/__init__.py 重入

# app/pkg/b.py
from .a import Foo        # 此时 a 还没执行完，Foo 尚未定义
报错形式是 ImportError: cannot import name 'Foo' from partially initialized module 'app.pkg.a'。

修法：把交叉依赖打散（按需 import 到函数体内，或拆出共享底层 module），不要在 __init__.py 顶层就 re-export 出会引发回环的符号。

反例 5：被 sys.path[0] 偷换的"假顶级"#

$ cd /proj/app
$ python pkg/a.py
a.py 里 from app.pkg import b 会报 ModuleNotFoundError: No module named 'app'。原因：python pkg/a.py 把 /proj/app/pkg 插到 sys.path[0]——这下 pkg 看起来像顶级了，但 app 反而不在 sys.path 任何一项的直接子项里。

修法：永远用 python -m app.pkg.a 在项目根目录跑，别直接 python xxx.py 跑包里的文件。

反例 6：给 namespace package 补了 __init__.py#

某天有人觉得 fbgemm_gpu/experimental/ 里没 __init__.py "看着不规范"，手贱补了一个。结果：

  • fbgemm_gpu.experimental 从 namespace 退化成 regular package。
  • 它的 __path__ 从可拼接的 _NamespacePath 变成单一目录的普通 list。
  • 别的 wheel（fbgemm_gpu_experimental_gen_ai 等）往别处的 experimental/gen_ai/ 装的子目录不再被识别——只有这个被"占坑"了的 __init__.py 所在目录下的子目录算数。

报错形式：ModuleNotFoundError: No module named 'fbgemm_gpu.experimental.gen_ai'，但磁盘上 gen_ai/ 明明存在。

修法：删掉那个不该存在的 __init__.py。判断一个目录"该不该有 __init__.py"的标准：这个目录是单个 wheel 拥有的，还是要给多个 wheel 拼接命名空间的？ 前者必须有，后者绝对不能有。

5. 检查清单#

碰到 import 报错时，按顺序排查：

  1. python -c "import sys; print(sys.path)" —— 顶级 package 的父目录在不在里面？
  2. python -c "import app; print(app.__file__)" —— 找到的是不是你期望的那个 app？（小心同名 package 撞车）
  3. 出错的文件里打 print(__name__, __package__) —— 相对导入的起点对不对？
  4. 启动命令是 python file.py 还是 python -m pkg.file？前者几乎一定会让 __package__ 丢失。
  5. __init__.py 是不是在导入时就触发了循环？

Python Import Mechanism: Absolute vs Relative#

1. Foundations#

Module#

A single .py file is a module. Once imported, it becomes a module object and is cached in sys.modules so it never executes twice.

Package#

A directory containing an __init__.py is a package (a "regular package"). It is itself a module object, but additionally carries a __path__ attribute that holds child modules and child packages.

Since PEP 420 there is also the namespace package: a directory with no __init__.py that can be stitched together from multiple physical directories. It is a real production-grade footgun, so it gets its own section below ("Namespace package").

What __init__.py does#

  1. Marker: tells Python to treat the directory as a regular package.
  2. Initialization code: runs once on the first import pkg (even from pkg.a import x first executes pkg/__init__.py).
  3. Re-export façade: a typical idiom is to write from .a import Foo inside it, so that callers can use from pkg import Foo directly.
  4. Defining __all__: controls which names from pkg import * exposes.

Top-level package#

A direct child of some entry in sys.path: if that child is a directory carrying an __init__.py, it is a top-level package.

Example: sys.path contains /home/me/proj, and /home/me/proj/app/__init__.py exists, therefore app is a top-level package. app.pkg and app.pkg.sub are its sub-packages, but they are not top-level.

Who counts as "top-level" is decided entirely by sys.path; it has no necessary relationship to how deeply directories nest on disk.

Namespace package (PEP 420)#

A directory with no __init__.py that Python nonetheless treats as a package. The core differences from a regular package:

regular package namespace package
__init__.py required must be absent
__file__ present absent
__path__ plain list _NamespacePath (stitched from several physical directories)
Purpose one wheel owns the whole package multiple wheels splice into the same import namespace

Canonical use case: a hub-style directory like fbgemm_gpu/experimental/ deliberately omits __init__.py so that independent wheels (fbgemm_gpu_experimental_hstu, fbgemm_gpu_experimental_gen_ai, etc.) can each install their own sub-package into site-packages/fbgemm_gpu/experimental/<name>/, and at runtime Python merges them all into a single fbgemm_gpu.experimental namespace.

Critical constraint: the moment any wheel drops an __init__.py into that directory, it instantly degrades into a regular package and the sub-directories that other wheels install there are no longer discoverable (see Pitfall 6).

2. How import actually works#

sys.path decides what can be found#

What import X really does: the interpreter takes the dotted name X, looks it up in sys.modules; on a miss, it asks the finders to walk sys.path for a top-level package, then descends through each sub-package's __path__ one segment at a time.

Where sys.path entries come from: - The directory of the startup script (python a.py → the directory containing a.py is inserted at sys.path[0]). - python -m pkg.mod → the current working directory is inserted at sys.path[0] (this is one of the biggest differences between -m and running a file directly). - The PYTHONPATH environment variable. - The installed site-packages.

__name__ and __package__#

Every module carries these two attributes, and they decide how relative imports resolve.

How it was launched __name__ __package__
python app/pkg/a.py (run as a file) "__main__" None or ""
python -m app.pkg.a (run as a module) "__main__" "app.pkg"
Imported elsewhere as import app.pkg.a "app.pkg.a" "app.pkg"

Relative imports (from . import x) only consult __package__: they take it as the starting point, and each extra dot strips off one level. If __package__ is None/"", the relative import fails outright.

That is exactly why running a file inside a package with python a.py immediately breaks on from . import b: the file has no idea which package it belongs to.

Absolute imports vs. relative imports#

  • Absolute import: from app.pkg import b. Looks up the full path starting at a sys.path root. Readable, and stays valid when files move between packages.
  • Relative import: from . import b, from ..sib import e. Computed from __package__. Survives renaming the top-level package, but cannot cross past the top level, and cannot be used at a script entry point.

PEP 8 recommends absolute imports; relative imports are appropriate for intra-package sibling/close-relative references.

How a dotted name is resolved one segment at a time#

from a.b.c import d is not resolved in one step. It walks the .s segment by segment, and each segment is independently decided to be a regular package, a namespace package, or a plain module:

  1. Find a: walk sys.path looking for a directory or file named a, write it into sys.modules['a'].
  2. Find a.b: look for b in a.__path__, running the finder once more. If a is a namespace package, a.__path__ may be a union of several physical directories, and b may come from any of them.
  3. Find a.b.c: look for c in a.b.__path__.
  4. From the module object a.b.c, fetch the attribute d.

Every module loaded along the way is written into sys.modules, so a repeat import hits the cache directly and never runs the finder again. Once any intermediate layer has been imported, changing sys.path afterwards will not cause it to be "re-resolved".

Two hidden side effects of import#

These two side effects explain a great deal of the confusion around "why does an empty __init__.py still let me do this?"

(a) import a.b sets b as an attribute on a. The import system makes this assignment proactively when it loads a submodule; it is not something that only happens if __init__.py contains from . import b:

import a.b           # even if a/__init__.py is completely empty
a.b.foo()            # this still works: the import system already bound b onto a

(b) from a import b, when it cannot find an attribute b on a, falls back to loading the submodule a.b. This is why from a import b still pulls in the submodule even when __init__.py has no from . import b. Note that this fallback only applies to a single attribute hop: from a import b.c is a syntax error.

Ambiguity from same-named submodules / top-level packages#

The import system only looks at names; it does not understand "logically the same package". Two common ambiguities:

  • Two top-level packages with the same name collide: if sys.path contains two directories that both hold a foo/, import foo only gets the one in the first sys.path entry that matches; the other is invisible forever. Diagnose with foo.__file__.
  • Same-named submodules meaning different things: for example, fbgemm_gpu.experimental.hstu.hstu_ops_gpu (the C++/CUDA binding) and fbgemm_gpu.experimental.hstu.hstu_blackwell.hstu_ops_gpu (the Triton implementation) are two entirely different modules that happen to share the same file name at the end. They are distinguished by the absolute path, and the usual practice is to bring them in with aliases at the call site (import ... as _sm90 / as _sm100) and dispatch through those aliases.

3. Worked examples#

Assume the following project layout (sys.path contains /proj, so app is the top-level package):

/proj/
  app/                  <- top-level package
    __init__.py
    main.py             <- entry point: python -m app.main
    pkg/
      __init__.py
      a.py
      b.py
      sub/
        __init__.py
        c.py
    sib/
      __init__.py
      e.py
      sub2/
        __init__.py
        f.py

Example 1: importing a sibling module in the same directory#

app/pkg/a.py wants foo from app/pkg/b.py:

# relative (recommended for intra-package use)
from . import b
from .b import foo

# absolute (also valid)
from app.pkg import b
from app.pkg.b import foo

The single . means "the current package", that is, __package__ == "app.pkg".

Example 2: importing from a sibling package#

app/pkg/a.py wants bar from app/sib/e.py:

# relative: two dots back up to app, then descend into sib
from ..sib import e
from ..sib.e import bar

# absolute
from app.sib import e
from app.sib.e import bar

Note that .. cannot cross past the top-level package. Here both pkg and sib are sub-packages of app, so .. reaching app is legal. If app itself had no __init__.py, this would be illegal.

Example 3: importing from one level down inside a sibling package#

app/pkg/a.py wants baz from app/sib/sub2/f.py:

# relative
from ..sib.sub2 import f
from ..sib.sub2.f import baz

# absolute
from app.sib.sub2 import f
from app.sib.sub2.f import baz

Example 4: the script entry point#

app/main.py is the entry point and pulls in other modules:

# main.py — recommended launch: python -m app.main
from app.pkg.a import something    # absolute, most robust
from .pkg.a import something       # relative, requires __package__ == "app", i.e. launched with -m

4. Pitfalls (common error cases)#

Pitfall 1: ImportError: attempted relative import with no known parent package#

$ cd /proj/app/pkg
$ python a.py
a.py contains from . import b. When run as a file, a.py's __name__ == "__main__" and __package__ is None. It does not know it belongs to app.pkg, so the relative import has no starting point.

Fix: - Switch to python -m app.pkg.a (run from /proj), so the interpreter loads it as a module and __package__ is set to "app.pkg". - Or write a.py as a pure script that uses only absolute imports, e.g. from app.pkg import b (assuming /proj is on sys.path).

Pitfall 2: ModuleNotFoundError: No module named 'app'#

$ cd /home/me
$ python -m app.main
The current directory is /home/me, so sys.path[0] is /home/me, which contains no app/ directory at all. The interpreter cannot locate the top-level package.

Fix: - cd /proj && python -m app.main, or - PYTHONPATH=/proj python -m app.main, or - Install the project as an editable package (pip install -e . with pyproject.toml) so that app ends up in site-packages.

Variant: from app.pgk.a import x (typo, pkg → pgk) raises the same error. The message only tells you "no such name", not that you misspelled it.

Pitfall 3: ImportError: attempted relative import beyond top-level package#

# Inside app/__init__.py:
from ..util import helper
app is already the top-level package, and .. tries to back out to app's "parent package", which does not exist. Python forbids relative imports from crossing the boundary of the top-level package.

Note: "top-level package" in this error message means the top-level package that the relative import started from (here, app). It does not mean "I am already at the outermost directory". Even if there are other directories on disk outside app/, that is irrelevant: the "top" is whatever sys.path defines as the top.

Fix: use an absolute import, from util import helper (assuming util is also a top-level package), or relocate util under app so it becomes app.util.

Pitfall 4: a circular import in __init__.py → ImportError: cannot import name 'X'#

# app/pkg/__init__.py
from .a import Foo

# app/pkg/a.py
from . import b           # triggers re-entry into pkg/__init__.py

# app/pkg/b.py
from .a import Foo        # at this point a has not finished executing, Foo is not yet defined
The error shows up as ImportError: cannot import name 'Foo' from partially initialized module 'app.pkg.a'.

Fix: break the cross-dependency apart (lazily import inside function bodies, or split the shared core into its own module). Do not re-export at the top of __init__.py any symbol whose construction will loop back through the package.

Pitfall 5: the "fake top-level" caused by sys.path[0]#

$ cd /proj/app
$ python pkg/a.py
a.py's from app.pkg import b raises ModuleNotFoundError: No module named 'app'. The reason: python pkg/a.py inserts /proj/app/pkg at sys.path[0]. Now pkg looks like a top-level package, but app is no longer a direct child of any sys.path entry.

Fix: always launch with python -m app.pkg.a from the project root; never run a file inside a package with python xxx.py.

Pitfall 6: someone "helpfully" adds an __init__.py to a namespace package#

Suppose someone decides that fbgemm_gpu/experimental/ "looks irregular" without an __init__.py and drops one in. Consequences:

  • fbgemm_gpu.experimental degrades from a namespace package into a regular package.
  • Its __path__ changes from the splice-capable _NamespacePath into a plain single-directory list.
  • Sub-directories that other wheels (fbgemm_gpu_experimental_gen_ai, etc.) installed into experimental/gen_ai/ at other locations are no longer discoverable. Only the sub-directories under the one directory holding the squatted __init__.py count.

The error looks like ModuleNotFoundError: No module named 'fbgemm_gpu.experimental.gen_ai', even though gen_ai/ visibly exists on disk.

Fix: delete that rogue __init__.py. The rule for deciding whether a directory should have an __init__.py: is this directory owned by a single wheel, or is it a namespace meant to be stitched from multiple wheels? The former must have one; the latter absolutely must not.

5. Diagnostic checklist#

When you hit an import error, work through these in order:

  1. python -c "import sys; print(sys.path)": is the parent directory of the top-level package present?
  2. python -c "import app; print(app.__file__)": is the app that gets found the one you expected? (Watch out for same-name collisions.)
  3. In the failing file, print(__name__, __package__): is the relative-import starting point what you thought?
  4. Is the launch command python file.py or python -m pkg.file? The former almost guarantees __package__ is lost.
  5. Does __init__.py trigger a cycle at import time?
