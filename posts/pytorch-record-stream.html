record_stream 与 CUDA storage 复用

1. 作用范围

tensor.record_stream(stream) 作用于 CUDA storage 的复用安全,不建立 stream 间的执行依赖。它告诉 PyTorch CUDA caching allocator:这个 Tensor 的底层 storage 也被 stream 使用过。Tensor 释放时,allocator 不能只根据创建 stream 的进度判断这块显存是否可复用。

API管什么不管什么
s2.wait_stream(s1)执行顺序。让 s2 等 s1 已经排队的工作。不延长 Tensor storage 的生命周期。
x.record_stream(s2)内存生命周期。告诉 allocator:释放 x 时也要考虑 s2。不让 s2 自动等待 s1。
跨 stream 使用同一个 Tensor 时,需要同时处理两类约束:wait_stream 负责读写顺序,record_stream 或手动 event / wait 负责 storage 生命周期。

2. Caching allocator 与 CUDA event

PyTorch 的 CUDA allocator 是 caching allocator。Tensor 的 Python 引用消失时,显存通常不会真的 cudaFree,而是回到 allocator 的缓存池。问题在于 CUDA kernel 是异步的:Python 对象已经没引用了,不代表 GPU 真的不用这块显存了。

简化模型如下:allocator 用 CUDA event 判断一块缓存 block 是否可以复用。

释放时:不是真释放,而是记录 event

# 伪代码: allocator 内部逻辑,用于理解机制

def deallocate(block):
    for stream in block.associated_streams:
        event = cudaEventCreate()
        cudaEventRecord(event, stream)  # 在 stream 当前队列位置插一个 event
        block.pending_events.append(event)

    block.state = "PENDING"  # 还不能复用

这个 event 的含义是:当 GPU 执行到这个 event 时,说明这个 stream 上在 event 之前排队的工作都已经完成。allocator 不需要同步整个 GPU,只需要以后查询这个 event 的状态。

分配时:检查 event 是否完成

# 伪代码: allocator 尝试复用缓存 block

def try_reuse(block):
    for event in block.pending_events:
        if cudaEventQuery(event) == cudaErrorNotReady:
            return False  # GPU 还没跑到那个 event 点,不能复用

    block.state = "FREE"
    return True

cudaEventQuery 是非阻塞查询。它不会等待 GPU 完成,只返回 event 当前是否已经触发。若 event 尚未完成,allocator 跳过这块 block,继续查找其他可用内存。

3. record_stream 的语义

x.record_stream(s2) 可以理解成把 s2 加入 x 的 storage 关联 stream 集合:

# 语义近似,不是 PyTorch 的真实 Python 实现
x.storage.associated_streams.add(s2)

这样 x 释放时,allocator 会把 s2 上已经排队的工作纳入判断。相关 event 全部完成后,这块显存才会回到可复用状态。

注意:这里说的是 allocator 视角的生命周期。record_stream 不会插入计算依赖,也不会让一个 stream 等另一个 stream。

4. 跨 stream 复用风险

x 在 s1 上产生,随后在 s2 上被 g(x) 异步使用:

s1 = torch.cuda.Stream()
s2 = torch.cuda.Stream()

with torch.cuda.stream(s1):
    x = f(a, b, c)      # x 的 storage 在 s1 上写入

# 计算依赖: s2 必须等 s1 写完 x
s2.wait_stream(s1)

with torch.cuda.stream(s2):
    y = g(x)            # g(x) 在 s2 上异步读 x

如果后面 x 的 Python 引用消失,例如 del x 或变量被覆盖,allocator 可能只知道 x 的创建 stream 是 s1。如果没有把 s2 记录进去,后续 s1 上的新分配就可能复用 x 原来的 storage:

# 假设这里 x 已经没有 Python 引用了

with torch.cuda.stream(s1):
    z = torch.rand(shape, device="cuda")  # 可能复用原来 x 的显存

风险来自这里:s2 上的 g(x) 可能尚未执行完。s1 上的新 rand 不会自动等待 s2,因此新写入可能覆盖 g(x) 仍在读取的显存。

为什么同一个 stream 就没问题

如果 f、g 和后续新分配都在 s1 上,CUDA stream 的 FIFO 顺序天然保证:

f 写 x  ->  g 读 x  ->  rand 写新内存

即使 allocator 复用了同一块显存,新的写入也会排在旧的读取之后。问题只在跨 stream 时出现,因为 s1 和 s2 是两条可以并行推进的队列。

5. 生命周期管理方式

写法 A:用 record_stream

s1 = torch.cuda.Stream()
s2 = torch.cuda.Stream()

with torch.cuda.stream(s1):
    x = f(a, b, c)

s2.wait_stream(s1)

with torch.cuda.stream(s2):
    y = g(x)

# 内存生命周期: 告诉 allocator s2 也使用了 x 的 storage
x.record_stream(s2)

这样即使 Python 侧 x 被释放,allocator 也会等 s2 上相关工作执行到释放时记录的 event 之后,才复用这块显存。

写法 B:不用 record_stream,手动同步回创建 stream

with torch.cuda.stream(s1):
    x = f(a, b, c)

s2.wait_stream(s1)

with torch.cuda.stream(s2):
    y = g(x)

# 在释放 x 或允许 s1 复用它的 storage 前,
# 让创建 stream s1 等待 side stream s2。
s1.wait_stream(s2)

del x

这种写法由调用方显式管理生命周期。它更可控,但同步点放得过早会损失 overlap;放得过晚或遗漏同步,仍会留下数据竞争。

6. 与 wait_stream 的关系

仅调用 record_stream 不会建立计算依赖:

with torch.cuda.stream(s1):
    x = f(a, b, c)

with torch.cuda.stream(s2):
    y = g(x)

x.record_stream(s2)

这段代码仍然不安全。record_stream 不会让 s2 等 s1。如果 g(x) 开始读的时候 f 还没写完,结果仍然可能错误。

两类约束分别对应不同 API:

  • s2.wait_stream(s1):保证 s2 读 x 之前,s1 已经写完。
  • x.record_stream(s2):保证 x 的 storage 不会在 s2 用完前被 allocator 复用。
wait_stream 约束执行顺序;record_stream 约束 storage 生命周期。

record_stream and CUDA storage reuse

1. Scope

tensor.record_stream(stream) concerns CUDA storage reuse, not execution dependency between streams. It tells the PyTorch CUDA caching allocator that the tensor's storage was also used by stream. When the tensor is freed, the allocator cannot decide reuse from the creation stream alone.

APIHandlesDoes not handle
s2.wait_stream(s1)Execution order. Makes s2 wait for work already queued on s1.Does not extend the tensor storage lifetime.
x.record_stream(s2)Memory lifetime. Tells the allocator to consider s2 when x is freed.Does not make s2 wait for s1.
Cross-stream tensor use has two separate constraints: wait_stream orders reads and writes; record_stream or manual events / waits protect the storage lifetime.

2. Caching allocator and CUDA events

PyTorch uses a CUDA caching allocator. When a Python tensor reference disappears, the GPU memory is usually not returned by cudaFree; it goes back to the allocator's cache. CUDA kernels are asynchronous, so a dead Python object does not necessarily mean the GPU is done with that memory.

A simplified model: the allocator uses CUDA events to decide whether a cached block can be reused.

On deallocation: record events instead of freeing immediately

# Pseudocode: allocator-side model

def deallocate(block):
    for stream in block.associated_streams:
        event = cudaEventCreate()
        cudaEventRecord(event, stream)  # insert event at the current stream position
        block.pending_events.append(event)

    block.state = "PENDING"  # not reusable yet

Once the GPU reaches the event, all earlier work on that stream has completed. The allocator does not need to synchronize the whole GPU. It can query the event later.

On allocation: query whether events are complete

# Pseudocode: allocator tries to reuse a cached block

def try_reuse(block):
    for event in block.pending_events:
        if cudaEventQuery(event) == cudaErrorNotReady:
            return False  # GPU has not reached this event yet

    block.state = "FREE"
    return True

cudaEventQuery is non-blocking. It does not wait for the GPU to finish; it only reports whether the event has fired. If not, the allocator can skip this block and look for another one.

3. record_stream semantics

You can think of x.record_stream(s2) as adding s2 to the set of streams associated with x's storage:

# Semantic approximation, not PyTorch's real Python implementation
x.storage.associated_streams.add(s2)

When x is freed, the allocator now checks s2 as well as the creation stream. The memory can be reused only after the relevant events on those streams have completed.

This is a memory-lifetime rule from the allocator's perspective. record_stream does not insert compute dependencies or make one stream wait for another.

4. Cross-stream reuse hazard

Here x is produced on s1, then consumed asynchronously by g(x) on s2:

s1 = torch.cuda.Stream()
s2 = torch.cuda.Stream()

with torch.cuda.stream(s1):
    x = f(a, b, c)      # x's storage is written on s1

# Compute dependency: s2 must wait until s1 has written x
s2.wait_stream(s1)

with torch.cuda.stream(s2):
    y = g(x)            # g(x) reads x asynchronously on s2

If the Python reference to x later disappears, for example through del x or assignment overwrite, the allocator may only know that x was created on s1. Without recording s2, a later allocation on s1 may reuse x's old storage:

# Suppose x has no Python references now.

with torch.cuda.stream(s1):
    z = torch.rand(shape, device="cuda")  # may reuse x's old storage

The hazard is that g(x) on s2 may still be running. The new rand on s1 does not automatically wait for s2, so its writes may overwrite memory that g(x) is still reading.

Why the same stream is fine

If f, g, and the later allocation all run on s1, the CUDA stream FIFO order guarantees:

f writes x  ->  g reads x  ->  rand writes new data

Even if the allocator reuses the same memory, the new write is queued after the old read. The problem appears with multiple streams, because s1 and s2 are independent queues that can make progress concurrently.

5. Lifetime management options

Pattern A: use record_stream

s1 = torch.cuda.Stream()
s2 = torch.cuda.Stream()

with torch.cuda.stream(s1):
    x = f(a, b, c)

s2.wait_stream(s1)

with torch.cuda.stream(s2):
    y = g(x)

# Memory lifetime: tell the allocator that s2 also used x's storage.
x.record_stream(s2)

Now, even if Python releases x, the allocator waits until the relevant work on s2 reaches the recorded event before reusing the storage.

Pattern B: no record_stream, but manually sync back to the creation stream

with torch.cuda.stream(s1):
    x = f(a, b, c)

s2.wait_stream(s1)

with torch.cuda.stream(s2):
    y = g(x)

# Before releasing x, or allowing s1 to reuse its storage,
# make the creation stream wait for the side stream.
s1.wait_stream(s2)

del x

This puts lifetime management in the caller's hands. It is more explicit, but the synchronization point matters: too early loses overlap; too late, or missing, leaves the race in place.

6. Relationship with wait_stream

record_stream by itself does not establish a compute dependency:

with torch.cuda.stream(s1):
    x = f(a, b, c)

with torch.cuda.stream(s2):
    y = g(x)

x.record_stream(s2)

record_stream does not make s2 wait for s1. If g(x) starts reading before f finishes writing, the result can still be wrong.

The two constraints map to different APIs:

  • s2.wait_stream(s1): before s2 reads x, make sure s1 has finished writing it.
  • x.record_stream(s2): before allocator reuses x's storage, make sure s2 has finished using it.
wait_stream constrains execution order; record_stream constrains storage lifetime.
