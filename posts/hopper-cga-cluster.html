Hopper Cluster (CGA): DSMEM、集群 barrier 与 TMA 跨 CTA

Hopper 把 SM 之上多加了一层硬件感知的执行层级 Thread Block Cluster(Cooperative Grid Array,CGA),它把 GPC 内的若干 CTA 绑成一个组,硬件保证它们同时驻留并提供跨 CTA 的直连原语:

能力对应 CUDA / PTX 原语本质
Distributed Shared Memorycluster.map_shared_rank() / __cluster_shared_*跨 CTA 直读/写 smem,走 SM-to-SM 网络
Cluster barriercluster.sync() / barrier.cluster.arrive整个 cluster 所有线程对齐
Cluster mbarriermbarrier.arrive.shared::cluster把 mbarrier 暴露成集群可见,跨 CTA arrive/wait
TMA → shared::clustercp.async.bulk.tensor.*.shared::cluster.globalbulk async copy 直接落到 cluster 内另一个 CTA 的 smem

这四个能力是 Hopper GEMM(WGMMA)、FlashAttention v3、CUTLASS 3.x producer-consumer 流水线背后的硬件基础。

1. 为什么引入 cluster

Hopper 之前,跨 CTA 唯一的合法通信路径是 global memory(走 L2)。即使两个 CTA 物理上落在同一个 SM 邻居上,语言层面也必须当成"远端"对待:

  • 跨 CTA 的 smem 是不可见的,只能通过 global 中转;
  • 跨 CTA 同步只能用 __threadfence() + atomic + spin,粒度粗、开销大;
  • 当 GEMM 的 producer-consumer 流水线被切到多 CTA 时,中间张量必须落 L2/global,等同于失去了 smem 复用。

Hopper 硬件层级在 SM 和 GPU 之间多了一层 GPC(Graphics Processing Cluster,H100 上有 8 个 GPC,每个 GPC 含 ~18 个 SM)。GPC 内部 SM 之间有直连的 SM-to-SM 网络,带宽远高于 L2 出口。Cluster 就是把这个硬件结构暴露到编程模型:把若干 CTA 同时调度到同一个 GPC,允许它们直读对方 smem、共享 barrier。

2. 硬件如何保证 cluster 内 CTA 同时驻留

§1 说了 cluster 提供的能力,但要理解这些能力的硬件基础,需要看 Hopper 在分派器上做了什么。Hopper 之前 CWD(Compute Work Distributor)是 CTA 粒度的流式分发器,完全不保证"同时性"。Hopper 把 cluster 升级成原子调度单位,分四层落地。

2.1 CWD 原子分派

  • CWD 知道 SM ↔ GPC 的拓扑,内部维护 per-GPC 的空闲 SM 计数器。
  • 待分派的 cluster($N$ 个 CTA)在 CWD 队列里等,直到某一个 GPC 单独空出 $\ge N$ 个 SM 才被取出。任何 GPC 都凑不齐 $N$ 个 slot 时,cluster 继续排队。
  • 派发是原子的:cluster 的 $N$ 个 CTA 同一拍全部进入 running,要么 $N$ 个一起,要么一个都不上。不存在"先派 2 个、剩 2 个排队"的中间态。

这一条直接保证了"开始时同时驻留"——同时性不是事后凑齐的,是派发那一拍就同时。

2.2 GPC 是硬约束(不可跨 GPC)

CWD 的可用性检查不是"全 GPU 一共有 $N$ 个空 SM",而是"某一个 GPC 单独有 $N$ 个空 SM"。理由:cluster 的所有跨 CTA 原语(DSMEM、cluster mbarrier、TMA → shared::cluster)都走 GPC 内部的 SM-to-SM 网络,这个网络物理上不跨 GPC。

H100 一个 GPC 约 18 个 SM,portable 上限 8 既是历史保守值,也是给同 GPC 内其它 cluster 留出 occupancy 的余量。

2.3 已驻留就 gang-lock,不会被单独抽出

cluster 进入 SM 后,Hopper 调度器还有一条 gang-scheduling 锁:

  • 不允许把 cluster 的某个 CTA 单独 evict 而保留其它的(普通 CTA 没这层保护)。
  • 不允许 cluster 内 CTA 在 SM 间迁移。
  • 退出是整体退出:所有 CTA 都 exit 后才把这一组 SM 资源归还给 CWD。

这条保证"运行期间持续同时驻留",不只是"刚开始时同时"。

2.4 资源预留与地址翻译表

派发的那一拍硬件同时完成几件事:

  • 每个 CTA 在它的 SM 上拿到完整的 RF / smem / barrier slot 配额(与普通 CTA 相同)。
  • CWD 给这个 cluster 分配一个 cluster ID,并配置 GPC 内 SM-to-SM 网络的 peer 路由表:某个 SM 上的 CTA 发起 shared::cluster 访存时,硬件根据 cluster ID + target block_rank 把请求路由到正确对端 SM 的 smem。
  • 每个 CTA 拿到一个硬件分配的 cluster-local rank(cluster.block_rank() 看到的就是它),不是 blockIdx 的函数。
  • DSMEM 的地址翻译表填好:cluster.map_shared_rank(local_smem, k) 返回的指针被 SM MMU 翻译成"目标 SM 的 smem 基址 + 同 offset"。表的内容是派发瞬间一次性写入,运行期间不再变。

任意一步资源凑不齐(RF / smem 超 SM 上限、路由表 slot 用完等),CWD 在派发阶段就拒掉,整个 cluster 不会进 running。

2.5 准入失败的两种情形

  • 永远不可能凑齐:cluster size 超过单个 GPC 的 SM 数,或者 kernel 的 RF / smem 需求让单 GPC 容不下 $N$ 个 CTA。这种情况 driver 在 cudaLaunchKernelEx 阶段就返回错误(典型是 cudaErrorLaunchOutOfResources 或 cudaErrorInvalidConfiguration)。launch 一旦成功,硬件就承诺了至少这一组 cluster 能落地。
  • 暂时凑不齐:其它 kernel 正在用 SM。cluster 在 CWD 队列里等,这是正常的流水线 backpressure,语义跟普通 CTA 排队一样。

cudaOccupancyMaxActiveClusters() 把"GPC 内同一个 cluster 配置能并发驻留多少个"算出来,数值直接对应 CWD 这个准入逻辑。返回 0 说明这个 cluster shape 在当前 kernel 资源用量下不可行。

一句话总结。CWD 升级成 cluster-granularity 的原子调度器;GPC 拓扑成为硬约束;一旦驻留就 gang-lock 不可分割;SM-to-SM 网络的路由表在派发瞬间一次性配置。你从 PTX 层只看到"这 $N$ 个 CTA 一定同时在跑、彼此 smem 互通",但这是 CWD + GPC fabric + 资源管理器 + MMU 协作的结果。

3. cluster 几何与约束

3.1 shape 与层级

Hopper 编程模型现在的层级:

grid  -->  cluster  -->  block (CTA)  -->  warp  -->  thread

cluster 的 shape 是三维的 (cluster_dim.x, cluster_dim.y, cluster_dim.z),描述一个 cluster 包含多少 CTA、按什么形状排列。常见配置:

  • (2, 1, 1):1D,2 个 CTA: Hopper GEMM 的典型 producer / consumer 配对
  • (2, 2, 1):2D,4 个 CTA: 多消费者扇出
  • (4, 2, 1):2D,8 个 CTA: portable size 上限

3.2 size 上限

  • Portable 上限:8 CTA,任何 sm_90 设备都支持。
  • Non-portable 上限:16 CTA,需要在 launch 时显式打开 cudaLaunchAttributeNonPortableClusterSizeAllowed,且只在部分 SKU 上可用。生产代码默认走 8。
  • cluster 不能跨 GPC;cluster size 越大,可同时驻留的 cluster 越少。

3.3 important gridDim 与 clusterDim 的整除约束

Hopper 硬件要求 grid 的每一维都必须能被 cluster 的对应维整除:

$$\text{gridDim.x} \bmod \text{clusterDim.x} \;=\; 0,\quad \text{gridDim.y} \bmod \text{clusterDim.y} \;=\; 0,\quad \text{gridDim.z} \bmod \text{clusterDim.z} \;=\; 0$$

违反任意一维都会导致 cudaLaunchKernelEx 返回 cudaErrorInvalidConfiguration 或 cudaErrorInvalidValue。常见违规场景:

gridDimclusterDim结果
(8, 1, 1)(2, 1, 1)OK,4 个 cluster
(8, 4, 1)(2, 2, 1)OK,8 个 cluster
(7, 1, 1)(2, 1, 1)报错,7 % 2 ≠ 0
(1, 1, 1)(2, 1, 1)报错,gridDim.x < clusterDim.x
(2, 1, 1)(2, 2, 1)报错,gridDim.y < clusterDim.y

实战中处理方式:

  • 静态 GEMM(已知 M, N):host 端按 tile shape 算出 grid,确保对齐到 cluster shape。比如 cluster=(2,2,1), tile=128x128,要求 M, N 都对齐到 256(2 × 128),否则 padding 或者切回 cluster=(1,1,1)。
  • 动态 shape:host 端检查 gridDim,如果不整除就动态降级 cluster shape(2x2 → 2x1 → 1x1),或者使用 persistent kernel + cluster size = 1 fallback。CUTLASS 3.x 的 TileScheduler 内置了这个 fallback 逻辑。
  • 用 cudaOccupancyMaxActiveClusters() 探测:运行时拿到当前 cluster shape 下能同时驻留的 cluster 数,如果是 0 说明这个 shape 不可行。

4. cluster size 怎么定:编译期 vs 运行期

两种互斥的方式,二选一。CUTLASS 在不同模板路径下会走不同的方式。

4.1 编译期:__cluster_dims__(X, Y, Z) kernel 属性

在 kernel 定义上直接打属性,nvcc 会把它编进 PTX 的 .reqnctapercluster:

// kernel.cu — cluster size 写死成 (2, 2, 1)
#include <cooperative_groups.h>
namespace cg = cooperative_groups;

__global__ __cluster_dims__(2, 2, 1)
void my_kernel(int* x) {
    auto cluster = cg::this_cluster();
    int cta_rank = cluster.block_rank();    // 0..3
    cluster.sync();
    // ...
}

这种方式下,kernel 自带 cluster 形状,launch 时不需要额外指定 cluster 属性(实际上指定了反而会冲突)。Driver 看到 kernel 的 .reqnctapercluster,直接按这个 shape 调度。

适用场景:cluster shape 是算法本身的常量(比如 CUTLASS 内核的 ClusterShape 模板参数),不会运行时变化。

4.2 运行期:cudaLaunchAttributeClusterDimension 通过 cudaLaunchKernelEx

把 kernel 写成没有 __cluster_dims__ 属性,launch 时通过 attribute 传 cluster shape:

// kernel.cu — 不带 __cluster_dims__
__global__ void my_kernel(int* x) {
    auto cluster = cg::this_cluster();
    int n_cta = cluster.dim_blocks().x * cluster.dim_blocks().y;
    cluster.sync();
}

// host.cpp — launch 时传 cluster shape
cudaLaunchAttribute attrs[1];
attrs[0].id                  = cudaLaunchAttributeClusterDimension;
attrs[0].val.clusterDim.x    = 2;
attrs[0].val.clusterDim.y    = 2;
attrs[0].val.clusterDim.z    = 1;

cudaLaunchConfig_t config = {};
config.gridDim       = dim3(4, 4, 1);   // 必须被 cluster shape 整除
config.blockDim      = dim3(128, 1, 1);
config.dynamicSmemBytes = 0;
config.stream        = stream;
config.attrs         = attrs;
config.numAttrs      = 1;

cudaLaunchKernelEx(&config, my_kernel, x);

适用场景:cluster shape 在运行时根据问题尺寸动态选(比如 GEMM 里根据 M, N 决定走 1x1 / 2x1 / 2x2)。

4.3 CUTLASS:ClusterShape_MNK 模板参数

CUTLASS 3.x 的 Hopper 内核(cutlass::gemm::collective::CollectiveBuilder / cutlass::gemm::kernel::GemmUniversal)把 cluster shape 当成模板参数:

using ClusterShape_MNK = cute::Shape<cute::_2, cute::_1, cute::_1>;  // 2x1x1 cluster
using TileShape_MNK    = cute::Shape<cute::_128, cute::_256, cute::_64>;

using CollectiveMainloop = cutlass::gemm::collective::CollectiveBuilder<
    cutlass::arch::Sm90,
    cutlass::arch::OpClassTensorOp,
    ElementA, LayoutA, AlignA,
    ElementB, LayoutB, AlignB,
    ElementAccum,
    TileShape_MNK, ClusterShape_MNK,           // 这里
    cutlass::gemm::collective::StageCountAutoCarveout<sizeof(SharedStorage)>,
    cutlass::gemm::collective::KernelTmaWarpSpecialized>::CollectiveOp;

using GemmKernel = cutlass::gemm::kernel::GemmUniversal<
    cute::Shape<int, int, int, int>,    // ProblemShape MNKL
    CollectiveMainloop,
    CollectiveEpilogue>;

CUTLASS 内部根据这个 ClusterShape_MNK 同时做两件事:

  1. 在 kernel 实现里使用对应 cluster size 的 cluster_* 原语(WGMMA producer-consumer split、DSMEM 跨 CTA TMA 的目标 CTA rank 计算等);
  2. 在启动时通过 GemmUniversalAdapter::run() 自动设置 cudaLaunchAttributeClusterDimension。换句话说 CUTLASS 走的是运行期 attribute路径(原因是同一个编译产物可能在不同 problem shape 下用不同 cluster size,需要 host 端动态决定)。

用户层面只看到 ClusterShape_MNK 这个模板参数,但底层走的就是 3.2 节那套 cudaLaunchKernelEx 流程。CUTLASS 的 device_kernel launch helper 还会自动处理 cudaOccupancyMaxActiveClusters 检查和不可整除时的报错。

额外:CUTLASS 还支持动态 cluster size(在 GemmUniversal::Arguments 里运行时填),走的也是 cudaLaunchAttributeClusterDimension。

5. 运行时 launch API:cudaLaunchKernelEx 完整签名

传统 kernel<<<grid, block, smem, stream>>>(args) 不支持传递 attribute(它在 CUDA C++ 里被翻译成 cudaLaunchKernel),所以 cluster 必须走新的 cudaLaunchKernelEx 接口。

cudaError_t cudaLaunchKernelEx(
    const cudaLaunchConfig_t* config,
    void* kernel_func,
    Args... args);                // 可变参数,与 kernel 签名一致

struct cudaLaunchConfig_t {
    dim3 gridDim;
    dim3 blockDim;
    size_t dynamicSmemBytes;
    cudaStream_t stream;
    cudaLaunchAttribute* attrs;
    unsigned int numAttrs;
};

struct cudaLaunchAttribute {
    cudaLaunchAttributeID id;
    cudaLaunchAttributeValue val;     // union
};

cluster 相关的 attribute:

  • cudaLaunchAttributeClusterDimension: 指定 cluster shape
  • cudaLaunchAttributeNonPortableClusterSizeAllowed: 允许 size > 8
  • cudaLaunchAttributeClusterSchedulingPolicyPreference: 选 spread(分散到不同 GPC)或 load_balancing(默认),只影响调度倾向、不影响正确性

完整 host 侧 launch 模板(2x2x1 cluster,gridDim 必须能被 2 × 2 整除):

void launch_with_cluster(int* d_data, int grid_m, int grid_n, cudaStream_t stream) {
    // 1) 检查整除约束
    constexpr int CX = 2, CY = 2;
    if (grid_m % CX != 0 || grid_n % CY != 0) {
        std::abort();                         // 或者降级到 cluster=(1,1,1)
    }

    // 2) 准备 cluster attribute
    cudaLaunchAttribute attrs[1];
    attrs[0].id                = cudaLaunchAttributeClusterDimension;
    attrs[0].val.clusterDim.x  = CX;
    attrs[0].val.clusterDim.y  = CY;
    attrs[0].val.clusterDim.z  = 1;

    // 3) 组装 config
    cudaLaunchConfig_t cfg = {};
    cfg.gridDim          = dim3(grid_m, grid_n, 1);
    cfg.blockDim         = dim3(128, 1, 1);
    cfg.dynamicSmemBytes = 0;
    cfg.stream           = stream;
    cfg.attrs            = attrs;
    cfg.numAttrs         = 1;

    // 4) launch
    cudaLaunchKernelEx(&cfg, my_kernel, d_data);
}

如果 kernel 自带 __cluster_dims__(2, 2, 1),把上面的 attribute 部分整段去掉就行;driver 会从 kernel 自带的 cluster shape 推 cluster grid。

6. 四个核心能力

6.1 Cluster barrier

整个 cluster 里所有 CTA 的所有线程对齐:

auto cluster = cg::this_cluster();
cluster.sync();                                   // 整 cluster barrier

底层是 PTX 的 barrier.cluster.arrive + barrier.cluster.wait,实现上是 cluster 内 SM-to-SM 网络的硬件 fence,延迟比走 global 的 atomic 同步低一个量级。

常见 idiom:让所有 CTA 同时完成 GEMM 第一阶段、再进入第二阶段(producer→consumer 切换):

// 阶段 1: 各 CTA 把自己负责的 A_tile / B_tile 加载到本地 smem
load_tile_to_smem();
cluster.sync();                                   // 全 cluster 都到位了

// 阶段 2: 各 CTA 现在可以通过 DSMEM 直读对方的 smem 段
cross_cta_use_dsmem();
cluster.sync();                                   // 必要时再 barrier 一次

6.2 Distributed Shared Memory (DSMEM)

一个 CTA 可以拿到其它同 cluster CTA 的 smem 指针:

__shared__ float local_buf[256];

auto cluster = cg::this_cluster();
int peer_rank = (cluster.block_rank() + 1) % cluster.num_blocks();

float* peer_buf = cluster.map_shared_rank(local_buf, peer_rank);
// peer_buf 现在指向 rank=peer_rank 这个 CTA 的 local_buf 起始
float v = peer_buf[threadIdx.x];                 // 直接跨 CTA 读 smem

语义和访存:

  • 语言层面是 generic 指针;底层 nvcc 编成 PTX 里带 shared::cluster address space 的 ld/st 指令。
  • 访问延迟 ≈ L2 命中(~30 cycle),但带宽显著高于走 L2 的 global 间接访问。
  • 同步:从 peer smem 读之前,必须保证 peer 已经完成对那块 smem 的写。所以一般的写法是: peer 写完 → cluster.sync() → 本 CTA 用 map_shared_rank 读。
  • 对应的 PTX 是 ld.shared::cluster.b32 / st.shared::cluster.b32 等。

6.3 Cluster scope mbarrier

mbarrier(memory barrier)是 sm_80 引入的 smem 内一个 64-bit object,支持 arrive / wait 语义,Hopper 把它的作用域扩展到 cluster scope:

// 在某个 CTA(比如 rank=0)的 smem 上放一个 mbarrier
__shared__ uint64_t mbar;

if (cluster.block_rank() == 0 && threadIdx.x == 0) {
    asm volatile("mbarrier.init.shared::cluster.b64 [%0], %1;"
                 :: "r"(__cvta_generic_to_shared(&mbar)),
                    "r"(expected_arrivals));
}
cluster.sync();

// 其它 CTA arrive 到 rank=0 这个 mbarrier
if (threadIdx.x == 0) {
    uint64_t* remote_mbar = cluster.map_shared_rank(&mbar, 0);
    asm volatile("mbarrier.arrive.shared::cluster.b64 _, [%0];"
                 :: "r"(__cvta_generic_to_shared(remote_mbar)));
}

// rank=0 等够 expected_arrivals 个 arrive
if (cluster.block_rank() == 0 && threadIdx.x == 0) {
    asm volatile("mbarrier.try_wait.shared::cluster.b64 ...;" : ...);
}

用途:把 producer / consumer 流水线的状态 token 放在 mbarrier 上,producer 完成一个阶段就 arrive,consumer wait 到 N 个 arrive 后开始消费。Hopper GEMM 的 TMA 流水线(包括 cutlass::PipelineTmaAsync)就建立在这套语义上。

6.4 TMA → shared::cluster

TMA(Tensor Memory Accelerator,Hopper 引入的硬件 DMA 引擎)的 bulk async copy 既可以目的是本地 smem,也可以是cluster 内其它 CTA 的 smem:

// PTX 伪码:bulk async copy 一个 2D tile,目的是 cluster 内 dst_cta_rank 的 smem
cp.async.bulk.tensor.2d.shared::cluster.global.tile.mbarrier::complete_tx::bytes
    [dst_smem_ptr_in_peer_cta], [tensor_map], {coord_x, coord_y}, [mbar_in_peer_cta];

关键点:

  • 目的 smem 地址是另一个 CTA 的 smem 起始 + 偏移。一个 CTA 可以发起 TMA,把数据落到 cluster 内任何 peer 的 smem。
  • complete_tx 信号会更新到peer CTA 的 mbarrier,所以 peer CTA 自己用 mbarrier.wait 就能知道"我自己的 smem 已经被某 producer CTA 写好了"。
  • 这是 Hopper GEMM "single TMA → multi consumer" 模式的硬件支撑:一个 producer CTA 发起 TMA,把 A_tile 同时落到 N 个 consumer CTA 的 smem,N 个 consumer 并行做 wgmma。

7. CUDA C++ 例子

所有例子都假设 H100(sm_90)+ CUDA 12.x。下面 4 个例子从最简单的 hello cluster 逐步过渡到 TMA 跨 CTA。

7.1 例 1:Hello cluster: 用 cluster.sync() 同步

// hello_cluster.cu
#include <cstdio>
#include <cooperative_groups.h>
namespace cg = cooperative_groups;

__global__ __cluster_dims__(2, 2, 1)
void hello_cluster() {
    auto cluster = cg::this_cluster();
    int rank_in_cluster = cluster.block_rank();
    int cluster_size   = cluster.num_blocks();

    if (threadIdx.x == 0) {
        printf("[block %d in cluster (%d/%d)] before sync\n",
               blockIdx.x, rank_in_cluster, cluster_size);
    }
    cluster.sync();                                // 全 cluster barrier
    if (threadIdx.x == 0) {
        printf("[block %d in cluster (%d/%d)] after sync\n",
               blockIdx.x, rank_in_cluster, cluster_size);
    }
}

int main() {
    hello_cluster<<<dim3(4, 4, 1), 32>>>();        // 4 cluster, 每个 cluster 4 CTA
    cudaDeviceSynchronize();
}

7.3 例 3:Cluster mbarrier 同步 producer-consumer

2 个 CTA 一组,rank=0 当 producer,rank=1 当 consumer。Producer 把数据写到 consumer 的 smem,通过 cluster scope mbarrier 通知 consumer:

// mbarrier_pc.cu
#include <cstdio>
#include <cooperative_groups.h>
#include <cuda/barrier>
namespace cg = cooperative_groups;

__global__ __cluster_dims__(2, 1, 1)
void producer_consumer(int* out) {
    __shared__ int shared_buf[32];
    __shared__ cuda::barrier<cuda::thread_scope_block> mbar;     // 等价于 mbarrier

    auto cluster = cg::this_cluster();
    int rank = cluster.block_rank();

    // 全 cluster 一次性 init 一个 mbarrier(只有 rank=1 的实例会被远端 arrive)
    if (threadIdx.x == 0) {
        init(&mbar, 32 + 1);          // 1 个 producer + 32 个本地 consumer thread
    }
    cluster.sync();                  // 所有 mbarrier 已 init

    if (rank == 0) {
        // producer: 把数据写到 consumer 的 shared_buf,然后远端 arrive 它的 mbarrier
        int* peer_buf  = cluster.map_shared_rank(shared_buf, 1);
        auto* peer_mbar = cluster.map_shared_rank(&mbar, 1);
        peer_buf[threadIdx.x] = threadIdx.x * 7;

        __syncthreads();                                  // 本 CTA 内对齐
        if (threadIdx.x == 0) {
            peer_mbar->arrive();                          // 远端 arrive,等价于 mbarrier.arrive.shared::cluster
        }
    } else {
        // consumer: 等 mbarrier,然后读自己的 shared_buf
        auto tok = mbar.arrive();                          // 本地 thread 也 arrive 一次
        mbar.wait(std::move(tok));                        // 等够 32+1 次 arrive
        out[threadIdx.x] = shared_buf[threadIdx.x];
    }
}

int main() {
    int* d_out; cudaMalloc(&d_out, 32 * sizeof(int));
    producer_consumer<<<dim3(2, 1, 1), 32>>>(d_out);    // 1 cluster, 2 CTA
    cudaDeviceSynchronize();
    int h[32]; cudaMemcpy(h, d_out, sizeof h, cudaMemcpyDeviceToHost);
    for (int i = 0; i < 32; ++i) printf("%d ", h[i]);
}

编译:

nvcc -arch=sm_90a -std=c++17 -O3 mbarrier_pc.cu -o mbarrier_pc
./mbarrier_pc

要点:

  • cuda::barrier<cuda::thread_scope_block> 是 <cuda/barrier> 提供的 C++ 包装,底层就是 PTX 的 mbarrier。arrive() / wait() 实际编出 mbarrier.arrive / mbarrier.try_wait 指令。
  • 通过 cluster.map_shared_rank(&mbar, 1) 拿到 peer 的 mbarrier 指针,producer 在上面 arrive,nvcc 自动选 mbarrier.arrive.shared::cluster 变体(目标是 cluster 内非本 CTA 的 smem)。

7.4 例 4:TMA 把 global 数据落到 cluster 内 peer 的 smem

这是最贴近 Hopper GEMM 的模式:一个 CTA 发起 TMA bulk copy,把数据同时落到自己 + cluster 内其它 CTA 的 smem。下面给一个化简版:

// tma_to_cluster.cu — 化简版,只展示 cp.async.bulk 落到 peer smem 的形态
#include <cstdio>
#include <cooperative_groups.h>
#include <cuda/barrier>
namespace cg = cooperative_groups;

__global__ __cluster_dims__(2, 1, 1)
void tma_demo(const __grid_constant__ CUtensorMap tmap, int* out) {
    __shared__ alignas(128) float tile[256];                 // 16×16 tile
    __shared__ cuda::barrier<cuda::thread_scope_block> mbar;

    auto cluster = cg::this_cluster();
    int rank = cluster.block_rank();

    if (threadIdx.x == 0) init(&mbar, 1);
    cluster.sync();

    if (rank == 0 && threadIdx.x == 0) {
        // 计算 peer CTA(rank=1)的 tile 在 cluster smem space 里的地址
        float* peer_tile = cluster.map_shared_rank(tile, 1);
        auto*  peer_mbar = cluster.map_shared_rank(&mbar, 1);

        cuda::memcpy_async(peer_tile,
                           &tmap,
                           {0 /* x */, 0 /* y */},
                           sizeof(float) * 256,
                           *peer_mbar);                // TMA → peer smem, 完成时 signal peer mbar
    }

    if (rank == 1) {
        auto tok = mbar.arrive();
        mbar.wait(std::move(tok));                    // 等 TMA 写完
        if (threadIdx.x < 16) out[threadIdx.x] = (int)tile[threadIdx.x];
    }
}

// host 端用 cuTensorMapEncodeTiled / cuTensorMapEncodeIm2col 构造 tmap,
// 然后用 cudaLaunchKernelEx 启动(略,见 CUDA samples bulk_tensor)

编译(含 CUDA driver API 头):

     -lcuda \
     tma_to_cluster.cu -o tma_to_cluster

注意点:

  • cuda::memcpy_async 配合 CUtensorMap 会生成 cp.async.bulk.tensor.* PTX。当目标 smem 来自 map_shared_rank,nvcc 自动选 shared::cluster 变体。
  • CUtensorMap 是 host 端用 cuTensorMapEncodeTiled()(driver API)预先构造的描述符,之后作为 __grid_constant__ 参数传入 kernel。这一步在所有 TMA 例子里都要做。

9. 坑与边界

  1. sm_90a vs sm_90:cluster / TMA / wgmma 所有 Hopper 新原语都需要 architecture-specific PTX,编译时务必 -arch=sm_90a。如果不带 a 后缀,nvcc 会让你用编译能过但运行时 illegal instruction 或者 PTX assembler error。
  2. gridDim 不整除 → launch 报错:见 §3.3。host 端处理 dynamic shape 时一定要先做整除检查并准备 fallback(降到 cluster=(1,1,1))。
  3. cluster size 与 occupancy 倒挂:cluster size 越大,可同时驻留的 cluster 越少。H100 一个 GPC 约 18 SM,cluster=8 时同 GPC 内顶多 2 个 cluster 并发,跨 GPC 也只能放 16 个 cluster(8 GPC × 2)。cudaOccupancyMaxActiveClusters 给出实际数。
  4. cluster 不跨 GPC:这是硬件约束,不能突破。即便你显式打开 NonPortableClusterSizeAllowed 把 size 拉到 16,16 个 CTA 也必须落在同一个 GPC,而一个 GPC SM 数有限,会让某些 SKU(尤其是阉割版)直接拒绝 launch。
  5. DSMEM 不能跨 cluster:map_shared_rank(ptr, k) 的 k 只能在当前 cluster 内取值。跨 cluster 沟通仍然只能走 global。
  6. cluster mbarrier 的 arrive 必须用 cluster scope 变体:对远端 mbarrier 直接用普通 mbarrier.arrive 是未定义行为(可能编出指令但不会通过 SM-to-SM 网络发出)。cuda::barrier 包装层会按指针 address space 自动选,但写 inline PTX 时要自己写对 .shared::cluster 后缀。
  7. cluster 内 CTA rank 编号:用 cluster.block_rank() 而不是 blockIdx。blockIdx 是全 grid 的 rank,跨 cluster 间会跳变。
  8. CUTLASS 的 cluster size 动态选取:GemmUniversal::Arguments 里可以填运行时 cluster shape,但实际可选的 shape 受 ClusterShape_MNK 这个模板参数约束(CUTLASS 在 collective builder 阶段会按这个 shape 生成对应的 PipelineTmaAsync stage 数和 DSMEM 路由)。运行时只能在不变 stage 数的前提下变 cluster shape。

Hopper Cluster (CGA): DSMEM, Cluster-scope Barriers and Cross-CTA TMA

Hopper adds a new hardware-aware execution layer above the SM called Thread Block Cluster (Cooperative Grid Array, CGA). It groups several CTAs on the same GPC into a unit that is guaranteed to be co-resident and exposes cross-CTA primitives:

CapabilityCUDA / PTX primitiveWhat it really is
Distributed Shared Memorycluster.map_shared_rank() / __cluster_shared_*Direct read/write of another CTA's smem over the SM-to-SM fabric
Cluster barriercluster.sync() / barrier.cluster.arriveAll threads in all CTAs of the cluster align
Cluster mbarriermbarrier.arrive.shared::clusterExposes an mbarrier across the cluster for cross-CTA arrive/wait
TMA → shared::clustercp.async.bulk.tensor.*.shared::cluster.globalBulk async copy that lands data into another CTA's smem

Together they form the hardware backbone for Hopper GEMM (WGMMA), FlashAttention v3, and the CUTLASS 3.x producer-consumer pipeline.

1. Why clusters

Before Hopper, the only legal cross-CTA communication path was global memory through L2. Even when two CTAs happened to land on neighbouring SMs, the language model treated them as remote:

  • One CTA's smem was invisible to another; data had to round-trip through global memory.
  • Cross-CTA sync used __threadfence() + atomic + spin: coarse and expensive.
  • When a GEMM producer-consumer pipeline was split across multiple CTAs, the intermediate tile had to land in L2 or global, which essentially threw away smem reuse.

Hopper adds a GPC (Graphics Processing Cluster) layer between the SM and the GPU (H100 has 8 GPCs, ~18 SMs each). Within a GPC, SMs are linked by a direct SM-to-SM network with much higher bandwidth than the L2 exit. The cluster construct exposes that hardware to the programming model: it schedules a group of CTAs onto the same GPC and lets them read each other's smem and share barriers.

2. How the hardware guarantees CTA co-residency

§1 listed the capabilities clusters expose; understanding their hardware basis means looking at what Hopper changed in the work distributor. Before Hopper, the CWD (Compute Work Distributor) streamed CTAs one at a time to any free SM and offered no "simultaneity" guarantee. Hopper promotes the cluster into an atomic scheduling unit, with four pieces that make it real.

2.1 Atomic dispatch by the CWD

  • The CWD knows the SM ↔ GPC topology and maintains a per-GPC free-SM counter.
  • A pending cluster of $N$ CTAs sits in the CWD queue until some single GPC has $\ge N$ free SMs. While no GPC can muster $N$ slots, the cluster waits.
  • Dispatch is atomic: all $N$ CTAs enter the running state in the same cycle. Either all $N$ go in together, or none. There is no "ship 2 now, queue the other 2" intermediate state.

This is what makes co-residency true at launch time, not stitched together post hoc.

2.2 GPC is a hard constraint (no cross-GPC clusters)

The CWD's admission check is not "the GPU as a whole has $N$ free SMs" but "some single GPC has $N$ free SMs." The reason: every cross-CTA primitive (DSMEM, cluster mbarrier, TMA into shared::cluster) runs over the GPC-local SM-to-SM fabric, which physically does not cross GPCs.

A H100 GPC has roughly 18 SMs, so the portable cap of 8 is both a historical safety margin and headroom for other clusters to be co-resident within the same GPC.

2.3 Once resident, gang-locked and not evicted in isolation

After a cluster is dispatched onto SMs, the Hopper scheduler enforces a gang-scheduling lock:

  • A cluster CTA cannot be evicted while its peers keep running (ordinary CTAs have no such protection).
  • A cluster CTA cannot migrate between SMs.
  • Exit is collective: only after every CTA of the cluster exits does the resource bundle return to the CWD.

This guarantees "continuous co-residency across the cluster's lifetime," not just "co-resident at launch."

2.4 Resource reservation and the address-translation table

The dispatch cycle simultaneously does several things:

  • Each CTA gets its full RF / smem / barrier-slot allocation on its SM (identical to a non-cluster CTA).
  • The CWD assigns the cluster a cluster ID and configures the GPC's SM-to-SM peer routing table: when a CTA issues a shared::cluster access, the fabric uses (cluster ID, target block_rank) to route the request to the correct peer SM's smem.
  • Each CTA receives a hardware-assigned cluster-local rank (what cluster.block_rank() returns); it is not a function of blockIdx.
  • The DSMEM address-translation table is populated: pointers returned by cluster.map_shared_rank(local_smem, k) are translated by the SM's MMU into "the target SM's smem base + the same offset". The table is written once at dispatch time and stays put for the cluster's lifetime.

If any of these resources cannot be satisfied (RF / smem exceeds the SM cap, routing-table slots are exhausted, etc.), the CWD rejects the dispatch and the cluster never enters running.

2.5 Two admission failure modes

  • Permanently infeasible: cluster size exceeds the SM count of a single GPC, or the kernel's RF / smem demand prevents any GPC from holding $N$ CTAs at once. The driver fails at cudaLaunchKernelEx time with cudaErrorLaunchOutOfResources or cudaErrorInvalidConfiguration. Once launch succeeds, the hardware has committed that at least one cluster of this configuration can be resident.
  • Temporarily infeasible: another kernel is using the SMs. The cluster waits in the CWD queue. This is ordinary pipeline backpressure, the same as a regular CTA queue.

cudaOccupancyMaxActiveClusters() exposes "how many clusters of this configuration can be co-resident inside a GPC," which is the exact quantity the CWD's admission logic computes. A return value of 0 means the chosen cluster shape is infeasible for the current kernel's resource usage.

In one line. The CWD became an atomic cluster-granularity scheduler; the GPC became a hard topology constraint; once resident the cluster is gang-locked and indivisible; the SM-to-SM routing table is configured once at dispatch time. From PTX you only see "these $N$ CTAs definitely run concurrently and can share smem," but it is delivered by the CWD, the GPC fabric, the resource manager, and the MMU working together.

3. Cluster geometry and constraints

3.1 Shape and hierarchy

The Hopper hierarchy:

grid  -->  cluster  -->  block (CTA)  -->  warp  -->  thread

A cluster's shape is a 3D tuple (cluster_dim.x, cluster_dim.y, cluster_dim.z) describing how many CTAs the cluster contains and how they are arranged. Common configurations:

  • (2, 1, 1): 1D, 2 CTAs. The typical Hopper GEMM producer/consumer pair.
  • (2, 2, 1): 2D, 4 CTAs. A small consumer fan-out.
  • (4, 2, 1): 2D, 8 CTAs. The portable upper bound.

3.2 Size limits

  • Portable cap: 8 CTAs, supported on every sm_90 device.
  • Non-portable cap: 16 CTAs, requires the cudaLaunchAttributeNonPortableClusterSizeAllowed attribute at launch and is only available on some SKUs. Production code defaults to 8.
  • A cluster cannot cross GPC boundaries; larger clusters mean fewer resident clusters in flight.

3.3 important gridDim divisibility constraint

Hopper requires each grid dimension to be divisible by the corresponding cluster dimension:

$$\text{gridDim.x} \bmod \text{clusterDim.x} \;=\; 0,\quad \text{gridDim.y} \bmod \text{clusterDim.y} \;=\; 0,\quad \text{gridDim.z} \bmod \text{clusterDim.z} \;=\; 0$$

Violating any axis makes cudaLaunchKernelEx return cudaErrorInvalidConfiguration or cudaErrorInvalidValue. Common failure cases:

gridDimclusterDimResult
(8, 1, 1)(2, 1, 1)OK, 4 clusters
(8, 4, 1)(2, 2, 1)OK, 8 clusters
(7, 1, 1)(2, 1, 1)error, 7 % 2 ≠ 0
(1, 1, 1)(2, 1, 1)error, gridDim.x < clusterDim.x
(2, 1, 1)(2, 2, 1)error, gridDim.y < clusterDim.y

How to handle this in production:

  • Static GEMM (known M, N): compute the grid on the host from the tile shape and make sure it aligns to the cluster. For cluster=(2,2,1), tile=128x128, that means M and N must be multiples of 256 (= 2 × 128); otherwise either pad or fall back to cluster=(1,1,1).
  • Dynamic shape: check gridDim on the host, then downgrade the cluster shape (2x2 → 2x1 → 1x1) until everything is divisible, or use a persistent kernel with cluster size 1 as the fallback. CUTLASS 3.x's TileScheduler bakes this fallback in.
  • Probe with cudaOccupancyMaxActiveClusters(): returns the max clusters resident for a given cluster shape. A return value of 0 means the chosen shape is infeasible.

4. Cluster size: compile-time vs runtime

Two mutually exclusive ways. CUTLASS uses one or the other depending on the template path.

4.1 Compile-time: __cluster_dims__(X, Y, Z) kernel attribute

Annotate the kernel directly; nvcc bakes the value into PTX's .reqnctapercluster:

// kernel.cu — cluster size is fixed at (2, 2, 1)
#include <cooperative_groups.h>
namespace cg = cooperative_groups;

__global__ __cluster_dims__(2, 2, 1)
void my_kernel(int* x) {
    auto cluster = cg::this_cluster();
    int cta_rank = cluster.block_rank();    // 0..3
    cluster.sync();
    // ...
}

With this style the kernel carries its cluster shape; the launch site does not need a separate cluster attribute (and supplying one would actually conflict). The driver reads the kernel's .reqnctapercluster and schedules accordingly.

Good fit when the cluster shape is an algorithmic constant (e.g. the ClusterShape template parameter of a CUTLASS kernel) and never changes at runtime.

4.2 Runtime: cudaLaunchAttributeClusterDimension via cudaLaunchKernelEx

Leave the kernel free of __cluster_dims__, supply the shape at launch as an attribute:

// kernel.cu — no __cluster_dims__
__global__ void my_kernel(int* x) {
    auto cluster = cg::this_cluster();
    int n_cta = cluster.dim_blocks().x * cluster.dim_blocks().y;
    cluster.sync();
}

// host.cpp — pass cluster shape at launch
cudaLaunchAttribute attrs[1];
attrs[0].id                  = cudaLaunchAttributeClusterDimension;
attrs[0].val.clusterDim.x    = 2;
attrs[0].val.clusterDim.y    = 2;
attrs[0].val.clusterDim.z    = 1;

cudaLaunchConfig_t config = {};
config.gridDim       = dim3(4, 4, 1);   // must divide by cluster shape
config.blockDim      = dim3(128, 1, 1);
config.dynamicSmemBytes = 0;
config.stream        = stream;
config.attrs         = attrs;
config.numAttrs      = 1;

cudaLaunchKernelEx(&config, my_kernel, x);

Good fit when the cluster shape is chosen at runtime based on the problem (a GEMM picking 1x1 / 2x1 / 2x2 from M, N).

4.3 CUTLASS: ClusterShape_MNK template parameter

CUTLASS 3.x Hopper kernels (cutlass::gemm::collective::CollectiveBuilder / cutlass::gemm::kernel::GemmUniversal) take cluster shape as a template parameter:

using ClusterShape_MNK = cute::Shape<cute::_2, cute::_1, cute::_1>;  // 2x1x1 cluster
using TileShape_MNK    = cute::Shape<cute::_128, cute::_256, cute::_64>;

using CollectiveMainloop = cutlass::gemm::collective::CollectiveBuilder<
    cutlass::arch::Sm90,
    cutlass::arch::OpClassTensorOp,
    ElementA, LayoutA, AlignA,
    ElementB, LayoutB, AlignB,
    ElementAccum,
    TileShape_MNK, ClusterShape_MNK,           // here
    cutlass::gemm::collective::StageCountAutoCarveout<sizeof(SharedStorage)>,
    cutlass::gemm::collective::KernelTmaWarpSpecialized>::CollectiveOp;

using GemmKernel = cutlass::gemm::kernel::GemmUniversal<
    cute::Shape<int, int, int, int>,    // ProblemShape MNKL
    CollectiveMainloop,
    CollectiveEpilogue>;

From this single ClusterShape_MNK CUTLASS does two things at once:

  1. Inside the kernel implementation it uses the matching cluster_* primitives (the WGMMA producer-consumer split, the target CTA rank for cross-CTA TMA, etc.);
  2. At launch time GemmUniversalAdapter::run() automatically sets cudaLaunchAttributeClusterDimension. CUTLASS goes through the runtime attribute path (because one compiled binary may need different cluster sizes across problem shapes, which is a host-side decision).

At the user level only the ClusterShape_MNK template parameter is visible, but under the hood it is exactly the cudaLaunchKernelEx flow from §4.2. CUTLASS's device_kernel launch helper also calls cudaOccupancyMaxActiveClusters and surfaces the divisibility error if the configuration is infeasible.

Extra: CUTLASS also supports a dynamic cluster size (filled at runtime through GemmUniversal::Arguments) and routes it through cudaLaunchAttributeClusterDimension.

5. Runtime launch API: full cudaLaunchKernelEx signature

The classic kernel<<<grid, block, smem, stream>>>(args) does not let you pass launch attributes (it lowers to cudaLaunchKernel), so cluster work has to go through cudaLaunchKernelEx.

cudaError_t cudaLaunchKernelEx(
    const cudaLaunchConfig_t* config,
    void* kernel_func,
    Args... args);                // variadic, matches the kernel signature

struct cudaLaunchConfig_t {
    dim3 gridDim;
    dim3 blockDim;
    size_t dynamicSmemBytes;
    cudaStream_t stream;
    cudaLaunchAttribute* attrs;
    unsigned int numAttrs;
};

struct cudaLaunchAttribute {
    cudaLaunchAttributeID id;
    cudaLaunchAttributeValue val;     // union
};

Cluster-related attributes:

  • cudaLaunchAttributeClusterDimension: pass the cluster shape
  • cudaLaunchAttributeNonPortableClusterSizeAllowed: allow sizes > 8
  • cudaLaunchAttributeClusterSchedulingPolicyPreference: pick spread (across GPCs) or load_balancing (default); affects scheduling preference, not correctness

A complete host-side launch template (2x2x1 cluster, gridDim must divide by 2 × 2):

void launch_with_cluster(int* d_data, int grid_m, int grid_n, cudaStream_t stream) {
    // 1) Check the divisibility constraint
    constexpr int CX = 2, CY = 2;
    if (grid_m % CX != 0 || grid_n % CY != 0) {
        std::abort();                         // or fall back to cluster=(1,1,1)
    }

    // 2) Build the cluster attribute
    cudaLaunchAttribute attrs[1];
    attrs[0].id                = cudaLaunchAttributeClusterDimension;
    attrs[0].val.clusterDim.x  = CX;
    attrs[0].val.clusterDim.y  = CY;
    attrs[0].val.clusterDim.z  = 1;

    // 3) Assemble the config
    cudaLaunchConfig_t cfg = {};
    cfg.gridDim          = dim3(grid_m, grid_n, 1);
    cfg.blockDim         = dim3(128, 1, 1);
    cfg.dynamicSmemBytes = 0;
    cfg.stream           = stream;
    cfg.attrs            = attrs;
    cfg.numAttrs         = 1;

    // 4) Launch
    cudaLaunchKernelEx(&cfg, my_kernel, d_data);
}

If the kernel already carries __cluster_dims__(2, 2, 1), drop the attribute block entirely; the driver derives the cluster grid from the kernel's own cluster shape.

6. Four core capabilities

6.1 Cluster barrier

Aligns every thread of every CTA in the cluster:

auto cluster = cg::this_cluster();
cluster.sync();                                   // cluster-wide barrier

Underneath it lowers to PTX's barrier.cluster.arrive + barrier.cluster.wait: a hardware fence on the SM-to-SM network, an order of magnitude cheaper than atomic-based cross-CTA sync through global.

Common idiom: let every CTA finish phase 1 of a GEMM, then advance to phase 2 (the producer-to-consumer hand-off):

// Phase 1: each CTA stages its own A_tile / B_tile into local smem
load_tile_to_smem();
cluster.sync();                                   // every CTA is in place

// Phase 2: each CTA can now read peer smem through DSMEM
cross_cta_use_dsmem();
cluster.sync();                                   // barrier again if needed

6.2 Distributed Shared Memory (DSMEM)

A CTA can obtain a pointer to another in-cluster CTA's smem:

__shared__ float local_buf[256];

auto cluster = cg::this_cluster();
int peer_rank = (cluster.block_rank() + 1) % cluster.num_blocks();

float* peer_buf = cluster.map_shared_rank(local_buf, peer_rank);
// peer_buf now points to local_buf at the CTA with rank=peer_rank
float v = peer_buf[threadIdx.x];                 // cross-CTA smem read

Semantics and access:

  • At the language level it is a generic pointer; nvcc emits PTX with the shared::cluster address space (ld.shared::cluster.b32, st.shared::cluster.b32).
  • Latency is comparable to an L2 hit (~30 cycles), but the bandwidth is far higher than going through L2.
  • Synchronisation: before reading peer smem, the peer must have finished writing it. The canonical pattern is: peer writes → cluster.sync() → this CTA reads via map_shared_rank.

6.3 Cluster-scope mbarrier

An mbarrier (memory barrier) is a 64-bit smem object introduced in sm_80 that supports arrive/wait. Hopper extends its scope to the whole cluster:

// Put an mbarrier in some CTA's smem, e.g. rank=0
__shared__ uint64_t mbar;

if (cluster.block_rank() == 0 && threadIdx.x == 0) {
    asm volatile("mbarrier.init.shared::cluster.b64 [%0], %1;"
                 :: "r"(__cvta_generic_to_shared(&mbar)),
                    "r"(expected_arrivals));
}
cluster.sync();

// Other CTAs arrive on the mbarrier hosted by rank=0
if (threadIdx.x == 0) {
    uint64_t* remote_mbar = cluster.map_shared_rank(&mbar, 0);
    asm volatile("mbarrier.arrive.shared::cluster.b64 _, [%0];"
                 :: "r"(__cvta_generic_to_shared(remote_mbar)));
}

// rank=0 waits for expected_arrivals arrivals
if (cluster.block_rank() == 0 && threadIdx.x == 0) {
    asm volatile("mbarrier.try_wait.shared::cluster.b64 ...;" : ...);
}

Use case: park the pipeline state token on an mbarrier; the producer arrives at each stage boundary, the consumer waits until N arrivals are collected and then consumes. Hopper's TMA pipeline (including cutlass::PipelineTmaAsync) is built on this model.

6.4 TMA → shared::cluster

The TMA (Tensor Memory Accelerator) is Hopper's bulk-async DMA engine. Its target smem can be the local CTA's smem or another CTA's smem in the same cluster:

// PTX (sketch): bulk-async copy a 2D tile to a peer CTA's smem in the cluster
cp.async.bulk.tensor.2d.shared::cluster.global.tile.mbarrier::complete_tx::bytes
    [dst_smem_ptr_in_peer_cta], [tensor_map], {coord_x, coord_y}, [mbar_in_peer_cta];

Key points:

  • The destination smem address is another CTA's smem base plus offset. One CTA can issue a TMA copy whose destination is any peer CTA in the cluster.
  • The complete_tx signal lands on the peer CTA's mbarrier, so the peer can use mbarrier.wait to learn that "my smem has just been filled by some producer CTA."
  • This is the hardware basis of Hopper's "single TMA → multi-consumer" pattern: one producer CTA issues a TMA copy that fills N consumer CTAs' smem in parallel, and the N consumers run wgmma concurrently.

7. CUDA C++ examples

All examples assume H100 (sm_90) plus CUDA 12.x. The four examples progress from a minimal hello-cluster to a cross-CTA TMA copy.

7.1 Example 1: Hello cluster, synchronising with cluster.sync()

// hello_cluster.cu
#include <cstdio>
#include <cooperative_groups.h>
namespace cg = cooperative_groups;

__global__ __cluster_dims__(2, 2, 1)
void hello_cluster() {
    auto cluster = cg::this_cluster();
    int rank_in_cluster = cluster.block_rank();
    int cluster_size   = cluster.num_blocks();

    if (threadIdx.x == 0) {
        printf("[block %d in cluster (%d/%d)] before sync\n",
               blockIdx.x, rank_in_cluster, cluster_size);
    }
    cluster.sync();                                // cluster-wide barrier
    if (threadIdx.x == 0) {
        printf("[block %d in cluster (%d/%d)] after sync\n",
               blockIdx.x, rank_in_cluster, cluster_size);
    }
}

int main() {
    hello_cluster<<<dim3(4, 4, 1), 32>>>();        // 4 clusters, 4 CTAs each
    cudaDeviceSynchronize();
}

  • The kernel carries __cluster_dims__(2, 2, 1), so the launch site is plain <<<dim3(4,4,1), 32>>>. gridDim is 4 × 4 = 16 CTAs = 4 clusters; 4 % 2 = 0 satisfies the divisibility constraint.

7.2 Example 2: DSMEM cross-CTA read with a runtime cluster size

// dsmem_read.cu — each CTA writes its rank into smem, reads the peer's smem to verify
#include <cstdio>
#include <cooperative_groups.h>
namespace cg = cooperative_groups;

__global__ void dsmem_demo(int* out) {
    __shared__ int local_buf[1];
    auto cluster = cg::this_cluster();

    if (threadIdx.x == 0) local_buf[0] = cluster.block_rank() * 100 + 42;
    cluster.sync();                                // wait for peer write

    int peer_rank = (cluster.block_rank() + 1) % cluster.num_blocks();
    int* peer_buf = cluster.map_shared_rank(local_buf, peer_rank);

    if (threadIdx.x == 0) {
        int v = peer_buf[0];                     // cross-CTA smem read
        out[blockIdx.x] = v;
    }
    cluster.sync();                                // ensure reads complete before exit
}

    attrs[0].val.clusterDim.z = 1;

    cfg.numAttrs         = 1;

7.3 Example 3: Cluster mbarrier producer-consumer sync

Two CTAs per cluster, rank=0 as producer and rank=1 as consumer. The producer writes to the consumer's smem and signals it via a cluster-scope mbarrier:

// mbarrier_pc.cu
#include <cstdio>
#include <cooperative_groups.h>
#include <cuda/barrier>
namespace cg = cooperative_groups;

__global__ __cluster_dims__(2, 1, 1)
void producer_consumer(int* out) {
    __shared__ int shared_buf[32];
    __shared__ cuda::barrier<cuda::thread_scope_block> mbar;     // mbarrier wrapper

    auto cluster = cg::this_cluster();
    int rank = cluster.block_rank();

    // Every CTA inits its own mbarrier instance; rank=1's instance is the one signaled remotely
    if (threadIdx.x == 0) {
        init(&mbar, 32 + 1);          // 1 remote producer arrival + 32 local consumer arrivals
    }
    cluster.sync();                  // all mbarriers initialised

    if (rank == 0) {
        // Producer: write to the consumer's shared_buf, then remote-arrive its mbarrier
        int* peer_buf  = cluster.map_shared_rank(shared_buf, 1);
        auto* peer_mbar = cluster.map_shared_rank(&mbar, 1);
        peer_buf[threadIdx.x] = threadIdx.x * 7;

        __syncthreads();                                  // local-CTA sync
        if (threadIdx.x == 0) {
            peer_mbar->arrive();                          // mbarrier.arrive.shared::cluster
        }
    } else {
        // Consumer: every local thread also arrives once, then waits, then reads its own smem
        auto tok = mbar.arrive();
        mbar.wait(std::move(tok));                        // wait for 32+1 arrivals
        out[threadIdx.x] = shared_buf[threadIdx.x];
    }
}

int main() {
    int* d_out; cudaMalloc(&d_out, 32 * sizeof(int));
    producer_consumer<<<dim3(2, 1, 1), 32>>>(d_out);    // 1 cluster, 2 CTAs
    cudaDeviceSynchronize();
    int h[32]; cudaMemcpy(h, d_out, sizeof h, cudaMemcpyDeviceToHost);
    for (int i = 0; i < 32; ++i) printf("%d ", h[i]);
}

Compile:

nvcc -arch=sm_90a -std=c++17 -O3 mbarrier_pc.cu -o mbarrier_pc
./mbarrier_pc

Notes:

  • cuda::barrier<cuda::thread_scope_block> from <cuda/barrier> is the C++ wrapper that compiles down to mbarrier PTX. arrive() / wait() emit mbarrier.arrive / mbarrier.try_wait.
  • The peer mbarrier pointer comes from cluster.map_shared_rank(&mbar, 1). When the producer arrives on that pointer, nvcc automatically picks the mbarrier.arrive.shared::cluster variant (the destination is non-local smem inside the same cluster).

7.4 Example 4: TMA lands global data into a peer CTA's smem

The closest model to Hopper GEMM: one CTA issues a TMA bulk copy that fills its own and other in-cluster CTAs' smem at once. A trimmed version:

// tma_to_cluster.cu — trimmed version showing cp.async.bulk into peer smem
#include <cstdio>
#include <cooperative_groups.h>
#include <cuda/barrier>
namespace cg = cooperative_groups;

__global__ __cluster_dims__(2, 1, 1)
void tma_demo(const __grid_constant__ CUtensorMap tmap, int* out) {
    __shared__ alignas(128) float tile[256];                 // 16×16 tile
    __shared__ cuda::barrier<cuda::thread_scope_block> mbar;

    auto cluster = cg::this_cluster();
    int rank = cluster.block_rank();

    if (threadIdx.x == 0) init(&mbar, 1);
    cluster.sync();

    if (rank == 0 && threadIdx.x == 0) {
        // Address of peer (rank=1) tile inside the cluster smem space
        float* peer_tile = cluster.map_shared_rank(tile, 1);
        auto*  peer_mbar = cluster.map_shared_rank(&mbar, 1);

        cuda::memcpy_async(peer_tile,
                           &tmap,
                           {0 /* x */, 0 /* y */},
                           sizeof(float) * 256,
                           *peer_mbar);                // TMA → peer smem, signals peer mbar
    }

    if (rank == 1) {
        auto tok = mbar.arrive();
        mbar.wait(std::move(tok));                    // wait for TMA write to land
        if (threadIdx.x < 16) out[threadIdx.x] = (int)tile[threadIdx.x];
    }
}

// On the host, build tmap with cuTensorMapEncodeTiled / cuTensorMapEncodeIm2col
// and launch with cudaLaunchKernelEx (omitted; see the bulk_tensor CUDA sample).

Compile (with the CUDA driver API):

nvcc -arch=sm_90a -std=c++17 -O3 \
     -lcuda \
     tma_to_cluster.cu -o tma_to_cluster

Notes:

  • cuda::memcpy_async combined with CUtensorMap emits cp.async.bulk.tensor.* PTX. When the destination smem comes from map_shared_rank, nvcc picks the shared::cluster variant.
  • CUtensorMap is constructed on the host with cuTensorMapEncodeTiled() (driver API), then passed in as a __grid_constant__ kernel argument. Every TMA example needs this step.
  • Compile with -lcuda to link the driver API (the cuTensorMap* entry points live in the driver lib, not the runtime lib).

9. Pitfalls and boundaries

  1. sm_90a vs sm_90: cluster, TMA and wgmma all need architecture-specific PTX. Always compile with -arch=sm_90a; the plain sm_90 target only emits generic PTX and will either fail to assemble or produce an illegal instruction at runtime.
  2. gridDim divisibility: see §3.3. Always check on the host before launching dynamic shapes and prepare a fallback (e.g. drop to cluster=(1,1,1)).
  3. Cluster size vs occupancy trade-off: larger clusters mean fewer resident clusters. A H100 GPC has ~18 SMs, so cluster=8 caps the in-GPC concurrency at about 2 clusters, and the device-wide limit is ~16 clusters (8 GPCs × 2). Use cudaOccupancyMaxActiveClusters to query the real number.
  4. Clusters do not cross GPCs: a hardware constraint, not configurable. Even with NonPortableClusterSizeAllowed enabled, 16 CTAs still have to live on a single GPC, and on the smaller SKUs the launch will be rejected outright.
  5. DSMEM does not cross clusters: map_shared_rank(ptr, k) requires k to be inside the current cluster. Cross-cluster communication still has to go through global.
  6. Cluster mbarrier arrivals must use the cluster-scope variant: hitting a remote mbarrier with a vanilla mbarrier.arrive is undefined behaviour (it may even assemble but the signal will not cross the SM-to-SM network). The cuda::barrier wrapper selects the right variant based on pointer address space; when writing inline PTX, supply the .shared::cluster qualifier explicitly.
  7. Use cluster.block_rank(), not blockIdx, for the in-cluster rank. blockIdx is grid-wide and jumps across cluster boundaries.
  8. Dynamic cluster size in CUTLASS: GemmUniversal::Arguments takes a runtime cluster shape, but the set of feasible shapes is bounded by the ClusterShape_MNK template parameter (the collective builder picks the PipelineTmaAsync stage count and DSMEM routing based on it). At runtime you can only vary cluster shape without changing stage count.
