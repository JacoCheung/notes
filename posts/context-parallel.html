Context Parallel 全景解析:从 RingAttention 到 MagiAttention

本文系统整理 LLM 长序列训练中的所有主流公开 Context Parallel(CP)方案:AllGather KV(TransformerEngine 默认)、Ring Attention(P2P)、Striped/Zigzag 因果负载均衡、DeepSpeed Ulysses(A2A)、USP/yunchang、LoongTrain Double-Ring、MagiAttention,以及若干次要变体。每个方案放在统一的"通信 + 计算"建模框架下分析:它解决什么瓶颈、在哪种规模下失效、在 Megatron-LM/TransformerEngine 等开源项目里落到哪个文件哪个类。文末有一张全方法对比总表和一棵选型决策树。

所有引用均来自公开源:arXiv 论文、开源 GitHub 仓库、官方文档。本文是「LLM 训练系统」系列第 1 篇。

0. 为什么需要 Context Parallel

从 GPT-3 (2K context) 到 GPT-4 (128K) 到 Gemini 1.5 (1M+),LLM 的上下文窗口在 5 年里扩大了 500 倍以上。Sora、视频生成、长文档 RAG、Agent 长任务、基因序列 ...... 几乎每个新场景都在推这条曲线。但训练长 context 的模型,会撞上两堵墙:

  • 显存墙 —— 激活显存按 $O(S)$ 线性增长。一个 LLaMA2-7B 在 8K context 时激活已占 ~10 GB;128K context 直接 OOM,即使开 selective recomputation 也只能勉强撑住。
  • 算力墙 —— attention 的 FLOPs 按 $O(S^2)$ 增长。32K → 128K(4×) 时 attention 时间增加 16×,主导整个 step 时间。

已有的并行方式都救不了:

已有并行为什么不够
Data Parallel (DP)每个 rank 都要持有完整序列的激活。S 增大,每个 rank 显存照样炸。
Tensor Parallel (TP)沿 hidden 维度切分;通信(AllReduce / AllGather)发生在每一层,且必须在 critical path 上。TP > 8 时通信无法 overlap,扩展性差。
Pipeline Parallel (PP)沿 layer 维度切分。仍然每个 stage 持有完整 seq 激活,显存问题不变,只是把批切小。
Expert Parallel (EP)只对 MoE 层,跟 dense attention 完全正交。

Context Parallel 是这套体系里专门为长 seq 设计的第五种正交并行:把长序列在 $S$ 这一维切成 cp 块,每块给一张 GPU。模型的所有非 attention 层(LayerNorm、FFN、QKV projection)天然可以在序列方向独立计算,完全不需要通信。唯独 attention 例外 —— 因为 $\text{Attention}(Q, K, V) = \text{softmax}(QK^T/\sqrt{d})\,V$ 中,每个 query token 需要和所有 key/value token 算积。一个 rank 只持有 $S/\text{cp}$ 个 token 的 KV,但每个 token 都要看到全部 K V。

核心问题:Context Parallel 的所有挑战和方法变体,归根到底都是 attention 这一层在做"KV 的跨 rank 通信"这一件事 —— 用什么通信原语?多少冗余?怎么和 attention 计算 overlap?怎么处理 causal mask 带来的不均衡?上层论文五花八门,但都在回答这同一个问题。

本文按这套思路展开:先建立一个能横向对比的通用建模框架(§1),然后从最朴素的 AllGather KV(§2)开始,沿"通信原语演进"主线逐步引入 Ring Attention(§3)→ 解决 causal mask 的 Striped/Zigzag(§4)→ Ulysses 的换轴(§5)→ USP/LoongTrain 的二维(§6)→ MagiAttention 的灵活 mask + 零冗余通信(§7)。最后给一张八方法总表 + 选型决策树(§9)。

1. 通用建模:通信 vs 计算

横向对比各 CP 方案,需要把它们放进同一组等式里。这一节建立一个简化但准确的建模框架,后面每节都会用到。

1.1 attention 的 FLOPs 与通信量

记号(全文统一):

符号含义
$S$序列长度(per sample)
$H$注意力头数(MHA 时 $H_{kv} = H$)
$H_{kv}$KV 头数(GQA 时 $< H$)
$d$单头隐藏维度
$\text{cp}$context parallel size
$\text{tp}$tensor parallel size(沿 head 维切)
$\text{flops}$单卡实际 GPU 算力 (TFLOPS)
$BW$实际通信带宽 (GB/s)

一次 forward attention 的计算量(忽略 softmax / LayerNorm,只算 $QK^T$ 和 $\text{attn} \cdot V$ 两次 matmul):

$$ V_{\text{compute}} = 4 \times \frac{H}{\text{tp}} \times S^2 \times d $$

其中 4 = 2 个 matmul × 2 FMA ops/element。因果 mask 让大约一半 entry 不算,除以 2:

$$ V_{\text{compute, causal}} = 2 \times \frac{H}{\text{tp}} \times S^2 \times d $$

引入 cp 后,每张卡只算 $1/\text{cp}$ 份(理想均衡):

$$ V_{\text{compute, per GPU}} = \frac{V_{\text{compute}}}{\text{cp}} $$

不同 CP 方法的核心差别在通信量 $V_{\text{comm}}$:它取决于通信什么(KV / Q+KV+O)、通信多少次(1 次 AG vs cp-1 次 P2P)、通信粒度(整个 KV vs 分块)。我们用 $V_{\text{comm}}^{\text{method}}$ 在各节具体推导。

一次 attention 的总时间(假设通信和计算不重叠):

$$ T_{\text{attn}} = \frac{V_{\text{compute, per GPU}}}{\text{flops}} + \frac{V_{\text{comm}}}{BW} $$

能 overlap 的话取 $\max$。这是后续比较的统一基准。

1.2 两个评估场景

同样的 CP 方法,放在不同问题规模下会显示不同性质。我们需要两套场景做对比:

场景固定什么变化什么评估什么
场景 1:强扩展$S$ 固定 + $\text{tp}$ 固定$\text{cp}$ 增大(用更多 GPU 算同一个 seq)能不能继续线性加速?
场景 2:弱扩展$S$ 固定 + $\text{tp} \times \text{cp}$ 固定(总 GPU 数固定)$\text{cp}$ vs $\text{tp}$ 的权衡给定算力,怎么分给 TP 和 CP?
小例 1.2 LLaMA2-7B 的 attention 参数:$H = 32, d = 128$。一个 $S = 128\text{K}$、$\text{tp} = 8$ 的训练:
compute(causal) = $2 \times 32/8 \times (128\text{K})^2 \times 128 = 2 \times 4 \times 1.7 \times 10^{10} \times 128 \approx 1.7 \times 10^{13}$ FLOPs/sample。
一张 H100 算力 ~989 TFLOPS(BF16),efficiency 0.55 → 实际 550 TFLOPS。每张卡 ($\text{cp} = 4$): $\frac{1.7 \times 10^{13}}{4 \times 5.5 \times 10^{14}} \approx 7.7$ ms。

attention 单卡占了 ~ms 级别 —— 通信预算就是几个 ms。注意这数字会随 $S^2/\text{cp}$ 标度。

1.3 通信优先级:TP > CP > PP > DP

一个 4D 并行场景下($DP \times PP \times CP \times TP$),不同维度的通信频次和带宽需求差别很大,所以在 GPU 网格上分配位置时,要遵循优先级:

  1. TP 通信最重(每层都有 AllReduce),必须放在节点内 NVLink/NVSwitch 上(intra-node)。
  2. CP 通信次之(每层 attention 有 KV 通信),应优先 intra-node。如 intra-node 不够再延伸到 inter-node。
  3. PP 通信小(每 micro-batch 一次 send/recv),可以跨节点。
  4. DP 通信少且可异步(only reduce gradient),最适合跨节点。

对应到 DGX-H100 (8 卡 NVLink + 8 张 NDR 400G InfiniBand) 这种典型机器:

  • NVLink 4.0 全连接:~900 GB/s (理论),~540 GB/s (实测) 单卡聚合带宽
  • InfiniBand NDR 8×50 GB/s:~50 GB/s (理论),~43 GB/s (实测) per NIC

intra-node vs inter-node 带宽差 10×+。这个差距决定了:CP 一旦跨出节点边界,通信代价会跳变。后面 §6 LoongTrain 的 Double-Ring、§7 MagiAttention 的 GroupCast 都是在专门解决这个跨节点的痛点。

2. 方法一:AllGather KV

最朴素的方案:既然每张卡只有 $S/\text{cp}$ 段 KV,但 attention 需要全部 KV ——那就先 AllGather 把 KV 拼齐,再做 local attention。这正是 TransformerEngine 的默认 CP 实现(cp_comm_type="all_gather")。

2.1 思路与张量形状

每张卡持有的 KV:[H, S/cp, d]。AllGather 后:[H, S, d]。Q 不变,仍是 [H, S/cp, d]。然后跑标准 FlashAttention,得 [H, S/cp, d] 的 output。

# transformer_engine/pytorch/attention/dot_product_attention/context_parallel.py
# 简化自 AttnFuncWithCPAndKVAllGather (line ~2948)
def forward(ctx, q, k, v, cp_group, ...):
    cp_size = get_world_size(cp_group)

    # Step 1: AllGather K, V 跨 cp_group
    k_ag, _ = gather_along_first_dim(k, cp_group)   # [cp*S/cp, ..] -> [S, ..]
    v_ag, _ = gather_along_first_dim(v, cp_group)

    # Step 2: 因果 mask 下做 chunk 重排(§4 zigzag)
    chunk_ids = get_seq_chunk_ids_for_reordering(cp_size, k.device, True)
    k_ag = torch.index_select(k_ag, dim=0, index=chunk_ids)
    v_ag = torch.index_select(v_ag, dim=0, index=chunk_ids)

    # Step 3: 本地 FA(每卡处理自己的 Q chunk,但 KV 全有)
    out = flash_attn_func(q, k_ag, v_ag, causal=is_causal)
    return out

2.2 通信量与建模

每张卡需要的 KV 通信量(bf16,2 bytes):

$$ V_{\text{comm}}^{\text{AG}} = 2 \times 2 \times \frac{H_{kv}}{\text{tp}} \times S \times d $$

这里两个 2:一个是 K + V,一个是 bf16 字节数。注意是 $\frac{H_{kv}}{\text{tp}}$,GQA 下 $H_{kv} \ll H$,通信省得很多。

时间(忽略 reordering 开销):

$$ T_{\text{AG}} = \frac{V_{\text{compute}}}{\text{flops} \cdot \text{cp}} + \frac{V_{\text{comm}}^{\text{AG}}}{BW_{\text{AG}}} $$

"AllGather 算法带宽" $BW_{\text{AG}}$ 在固定 tp 时基本和 cp 无关(NCCL 的 AG 实现是 ring 类拓扑,bus BW 几乎不随 rank 数变),所以:

  • 场景 1(tp 固定,cp 增大):$T_{\text{compute}}$ 随 cp 线性下降,$T_{\text{comm}}$ 不变(因为 $V_{\text{comm}}$ 跟 cp 无关 —— 每卡都 gather 完整的 S)。cp 大了 attention 就变成 communication-bound,继续加 GPU 性价比迅速下降。
  • 场景 2(tp × cp 固定):cp 增大 → tp 缩小 → $\frac{H_{kv}}{\text{tp}}$ 增大 → $V_{\text{comm}}$ 增大。所以这个场景下 cp 越大 attention 反而越慢。

2.3 优点与局限

优点 ✓缺点 ✗
实现极其简单(AllGather + 标准 FA)整个 AG 暴露在 critical path,无 overlap
支持任意 attention mask(sliding window、block-causal、随机 mask 全行)每卡显存峰值 = 完整 KV,部分抵消了 CP 省显存的初衷
GQA 友好(只 gather $H_{kv}$,不是 $H$)大 cp 时 attention 100% 通信瓶颈
什么时候选 AllGather KV?cp 小(2-4)、用 GQA、且需要 sliding-window 等非标准 mask 时。TE 源码也确认:cp_comm_type ∈ {"a2a", "all_gather"} 才支持 sliding-window(context_parallel.py:4638)。

3. 方法二:Ring Attention (P2P)

AllGather 的痛点是"通信暴露在 critical path"。Ring Attention(Li et al. 2021,Liu et al. 2023)的核心思路:把 AllGather 拆成 cp 步 P2P send/recv,每步收到一块 KV 就跟本地 Q 算一次 attention,通信和计算同步进行。

3.1 思路与执行流

每张卡持有自己的 Q chunk,$K_i$ 在 ring 上转一圈。每一步:

  1. 计算:当前手上的 $(Q_{\text{local}}, K_j, V_j)$ 算 partial attention,用 online softmax 累加。
  2. 通信:把当前的 $(K_j, V_j)$ 通过 P2P send 给 ring 上下一个 rank;同时 recv 上一个 rank 的 $(K_{j-1}, V_{j-1})$。
  3. 循环:重复 cp 次直到每块 KV 都和本地 Q 配过一次。

关键是步骤 1 和 2 并行。如果通信时间 ≤ 计算时间,通信被完全隐藏。下图示意 cp=4 的情况:

  • Triton/PyTorch 社区版:zhuzilin/ring-flash-attention

3.2 通信-计算 overlap 边界

每步(per chunk)的通信量与计算量:

$$ V_{\text{comm}}^{\text{step}} = 4 \times \frac{H_{kv}}{\text{tp}} \times \frac{S}{\text{cp}} \times d $$

$$ V_{\text{compute}}^{\text{step}} = 4 \times \frac{H}{\text{tp}} \times \frac{S^2}{\text{cp}^2} \times d $$

4 = K + V × 2 bytes(bf16);计算分 $\frac{1}{\text{cp}^2}$ 是因为 Q chunk 是 $S/\text{cp}$、KV chunk 也是 $S/\text{cp}$。

通信能被计算完全 hide 的条件:

$$ \frac{V_{\text{comm}}^{\text{step}}}{BW} \leq \frac{V_{\text{compute}}^{\text{step}}}{\text{flops}} \quad\Longrightarrow\quad \text{cp} \leq \frac{BW \times S}{\text{flops}} \times \frac{H}{H_{kv}} $$

这就是"通信-计算重叠边界" $\text{cp}_{\text{bor}}$。$\text{cp} \leq \text{cp}_{\text{bor}}$ 时通信被隐藏 → attention 时间 ≈ 计算时间 ≈ $V_{\text{compute}} / \text{flops}$,与 cp 成反比。$\text{cp} > \text{cp}_{\text{bor}}$ 时通信暴露 → attention 时间反过来正比于 cp(因为还得跑 cp 步,每步通信暴露)。

小例 3.2 DGX-H100 + IB:实际 NVLink 带宽 540 GB/s,实际 IB 带宽 43 GB/s(单 NIC),H100 BF16 实际算力 550 TFLOPS。$S = 128\text{K}$,MHA(无 GQA):
intra-node: $\text{cp}_{\text{bor}} \approx \frac{540 \times 10^9 \times 128 \times 1024}{550 \times 10^{12}} \approx 128$。intra-node 几乎不被通信束缚。
inter-node: $\text{cp}_{\text{bor}} \approx \frac{43 \times 10^9 \times 128 \times 1024}{550 \times 10^{12}} \approx 10$。一旦 cp 超过 ~10 跨出 NVLink,attention 立即进入通信瓶颈。

这就是为什么所有"P2P 类"方案在多节点扩展(strong scaling)时性能急剧下降。

3.3 反向传播

反向 Ring 比正向更复杂 —— 不仅要发 KV,还要发 KV 的梯度。每步通信量翻倍。同时反向 attention kernel 本身 FLOPs 也更高(约 2.5× forward)。TE 源码里这两条路径分别在 cp_p2p_fwd_flash_attn 和 cp_p2p_bwd_flash_attn 两个 helper 里。

3.4 优缺点

优点 ✓缺点 ✗
通信和计算可重叠 → critical path 上只剩计算cp > cp_bor 时通信暴露,扩展性骤降
每卡显存只需当前 KV chunk + 1 buffer因果 mask 下负载不均(见 §4)
支持任意非 sliding-window 的 mask跨节点 ring 只用一条 NIC 路径,其它 NIC 闲置

4. 方法三:因果 mask 的负载均衡

Ring Attention 有个"隐形负载不均"问题,只在 causal mask 下出现 —— 也就是 GPT 类模型的标准 attention。本节专门讲怎么解决。

4.1 问题:三角形负载

用 $S = 8, \text{cp} = 2$ 举例,Causal mask 下哪些 $(i, j)$ 要算?$j \leq i$ 的下三角:

              KV chunk
              0,1,2,3  4,5,6,7
       Q  0  [  ■        ░     ]    GPU 0 (持 Q0..Q3)
       c  1  [  ■        ░     ]
       h  2  [  ■        ░     ]
       u  3  [  ■        ░     ]
       n  ─────────────────────
       k  4  [  ■        ■     ]    GPU 1 (持 Q4..Q7)
          5  [  ■        ■     ]
          6  [  ■        ■     ]
          7  [  ■        ■     ]

       ■ = 要算的位置;░ = 不算(j > i)

朴素切分:GPU 0 持有 Q[0..3]、KV[0..3];GPU 1 持有 Q[4..7]、KV[4..7]。但因果 mask 下,GPU 0 只算左上角 4×4 一半 = 8 个 cell;GPU 1 要算 4×4 左半 + 4×4 右半的一半 = 16+8=24 个 cell。GPU 1 的工作量是 GPU 0 的 3 倍 —— 严重不均。Ring 中每一步都要等最慢的卡,所以整体被拖慢。

4.2 Striped Attention

Striped Attention(Brandon et al., arXiv:2311.09431)的解法:不让 token 连续切块,而是按 stride = cp 交错分配。每张卡分到的 token 是均匀散布在整个 sequence 里的"条带":

       cp = 2 的 Striped 切分:
       GPU 0 持有 token 0, 2, 4, 6  (偶数位置)
       GPU 1 持有 token 1, 3, 5, 7  (奇数位置)

       这样每张卡的 (Q_i, K_j) 配对中,causal mask
       的"几何形状"几乎一致,负载完全均衡。

公开性能(论文 §5):A100 上 256K seq 提速 1.45×,16× TPUv4 上 786K seq 提速 1.65×。

4.3 Zigzag / DualChunkSwap

Striped 之外,实践上更常用的方案叫 Zigzag(社区命名)或 DualChunkSwap(TransformerEngine 用语)。思路:把 sequence 切成 2×cp 个 chunk,而不是 cp 个;每张卡分到一前一后两个 chunk。

       cp = 2,seq 切成 4 个 chunk:
       chunk_0  chunk_1  chunk_2  chunk_3

       GPU 0 持有:chunk_0 + chunk_3  (头 + 尾)
       GPU 1 持有:chunk_1 + chunk_2  (中间两块)

       Causal mask 下:
       - GPU 0 在 chunk_0 处工作量小 + chunk_3 处工作量大
       - GPU 1 在 chunk_1 处工作量中 + chunk_2 处工作量中

       两个之和几乎一致 → 负载均衡。

TransformerEngine 实现(context_parallel.py:236):

def get_seq_chunk_ids_for_reordering_before_attn(cp_size, device):
    """把 2*cp 个 chunk 重新排列,使第 rank 张卡拿到
       chunk_{2*rank} 和 chunk_{2*cp - 2*rank - 1}。"""
    chunk_ids = torch.empty(2 * cp_size, dtype=torch.int32, device=device)
    for rank in range(cp_size):
        chunk_ids[rank]           = 2 * rank
        chunk_ids[rank + cp_size] = 2 * cp_size - 2 * rank - 1
    return chunk_ids

这个 reorder 在 dataloader 阶段(megatron/core/utils.py 的 get_batch_on_this_cp_rank)做一次,所以 attention kernel 内部跟普通 ring attention 完全一样,不增加 kernel 复杂度。

命名混乱提醒:同一个技术在不同地方有不同名字 —— TransformerEngine 叫 DualChunkSwap,Megatron-LM 文档叫 load-balanced,社区(zhuzilin/ring-flash-attention、USP)叫 zigzag。Striped Attention 是另一个方案(交错按 token,粒度 = 1),不是同一个东西。论文里"zigzag"这名字没有正式 paper 起源,大致与 Striped(2023.11)同期在社区独立涌现。

4.4 实践提示

  • 这个 reorder必须在 dataloader 做,否则 attention mask 索引会错。
  • require sequence length 能被 2 * cp_size 整除(必要时 padding)。
  • Zigzag 是 TransformerEngine / Megatron-LM 默认开启的,你直接用 CP 就自动获得这个优化。

5. 方法四:Ulysses (A2A)

Ring/Striped/Zigzag 都是同一条路 —— 沿序列维 S 切,通过 P2P 转 KV。DeepSpeed 的 Ulysses(Jacobs et al., arXiv:2309.14509)走了完全不同的思路:用 AllToAll 在 attention 前换轴,attention 时不分割 sequence 而是分割 head。

5.1 思路:换轴分发

每个 transformer block 进 attention 之前的张量:[H, S/cp, d](沿 S 分布)。Ulysses 在 attention 前后各插一次 AllToAll:

进 attention 前:                       attention 中:                       attention 后:
─────────────────                       ─────────────                       ─────────────────
GPU 0: [H, S/cp, d]    AlltoAll    GPU 0: [H/cp, S, d]    本地 FA      GPU 0: [H/cp, S, d]    AlltoAll    GPU 0: [H, S/cp, d]
GPU 1: [H, S/cp, d]   ─────────→   GPU 1: [H/cp, S, d]    ──────→      GPU 1: [H/cp, S, d]   ─────────→   GPU 1: [H, S/cp, d]
GPU 2: [H, S/cp, d]                GPU 2: [H/cp, S, d]                 GPU 2: [H/cp, S, d]                GPU 2: [H, S/cp, d]
GPU 3: [H, S/cp, d]                GPU 3: [H/cp, S, d]                 GPU 3: [H/cp, S, d]                GPU 3: [H, S/cp, d]

     ↑                                  ↑                                    ↑                                ↑
   沿 S 切                          沿 H 切                              沿 H 切                          沿 S 切
                                  每卡持完整 S                        每卡持完整 S
                                  跑 H/cp 头 attention

开源参考:

  • 原版:microsoft/DeepSpeed,deepspeed/sequence/layer.py 的 DistributedAttention
  • TE 版:AttnFuncWithCPAndQKVOA2A(context_parallel.py:3807),通过 cp_comm_type="a2a" 启用

5.2 通信复杂度

Ulysses 一次 attention 做 4 次 AllToAll:正向 attention 前后各一次(QKV → output reshape),反向同样两次。总通信量(全部 token 都参与 A2A):

$$ V_{\text{comm}}^{\text{Ulysses, per pass}} = 4 \times 2 \times \frac{H + 2 H_{kv}}{\text{tp}} \times \frac{S}{\text{cp}} \times d \times \frac{\text{cp} - 1}{\text{cp}} $$

关键性质:通信量 per GPU 跟 cp 几乎无关 —— A2A 的本质就是 $\frac{n-1}{n}$ 的 bus 带宽利用率。这是 Ulysses 区别于 Ring 的最大优势:cp 翻倍时,Ring 通信总量翻倍,Ulysses 通信几乎不变。

论文原文表述(Jacobs et al. §3):"maintains constant communication volume when sequence length and compute devices are increased proportionally"。

5.3 致命约束:cp ≤ num_kv_heads

Ulysses 沿 head 维度切分,所以 cp 必须能整除 H(MHA)或 $H_{kv}$(GQA)。而且 TP 也是沿 head 切的,所以联合约束:

$$ \text{tp} \times \text{cp} \leq H_{kv} $$

这就是 Ulysses 的死穴。LLaMA3-8B 用 GQA-8(num_kv_heads = 8),所以 $\text{tp} \times \text{cp} \leq 8$。如果你想训 1M context,需要 cp = 256,完全做不到。

DeepSpeed-Ulysses不适合现代 GQA 大模型在长序列下的多节点扩展。这是单独使用它的根本限制。

解决思路:把它和 Ring 组合,Ulysses 在小范围内(cp_a2a ≤ H_kv)用,Ring 把剩下的 cp 维度撑起来 —— 这正是 §6 USP / LoongTrain 的核心。

6. 方法五:USP / LoongTrain 2D

Ring 强在能任意扩展但通信量大;Ulysses 强在通信量小但被 head 数卡死。一个自然的想法:把两者乘起来,做二维并行。这正是 USP(Fang & Zhao, arXiv:2405.07719)和 LoongTrain(Gu et al., arXiv:2406.18485)的共同思路。两篇论文发布只差几周(USP 5 月,LoongTrain 6 月),核心想法一致,LoongTrain 多了一个 Double-Ring 的工程优化。

6.1 思路:Ulysses × Ring 二维网格

把 cp 拆成两个维度:

$$ \text{cp} = \text{cp}_{a2a} \times \text{cp}_{p2p} $$

GPU 拓扑组织成二维网格:

  • 同一行(cp_a2a 个 rank):用 Ulysses A2A 切 head。Intra-node NVLink 上,A2A 极快。约束 cp_a2a ≤ H_kv / tp。
  • 同一列(cp_p2p 个 rank):用 Ring P2P 切 sequence。每个"a2a 组"内独立跑 Ring Attention。可以跨节点。

所以一次 attention 的流程是:

  1. 第 1 个 A2A:在 a2a 组内换轴,每个 rank 拿到 H_kv/cp_a2a 个 head 的完整 sequence。
  2. Ring Attention:在 p2p 组内对这部分 head 跑 ring attention(cp_p2p 步轮转)。
  3. 第 2 个 A2A:换轴回来,output 还是 [H/tp, S/cp, d] 形状。
小例 6.1 — LLaMA3-8B 在 16 卡上 $H_{kv} = 8$,假设 tp=2,要 cp=16:
- 纯 Ulysses:cp ≤ $H_{kv} / \text{tp} = 4$ → 做不到 cp=16。
- 纯 Ring:cp=16 一路 P2P,跨节点单 NIC 瓶颈严重。
- USP/LoongTrain:cp_a2a=4(=$H_{kv}/\text{tp}$,顶到 head 上限),cp_p2p=4 → 总 cp=16。Intra-node 4 卡跑 A2A(NVLink 540 GB/s),inter-node 4 节点跑 Ring(IB 43 GB/s)。组合后能用上所有维度。

实现:feifeibear/long-context-attention(yunchang)统一封装两个维度。TE 也实现了同样思路,通过 cp_comm_type="a2a+p2p" 启用,内部把 cp_group 拆成 [a2a_group, p2p_group]。

USP 公开 benchmark(arXiv:2405.07719):LLAMA3-8B 208K seq,2 × 8 × A800 RDMA,269.8 TFLOPS,MFU 0.86。

6.2 LoongTrain 的 Double-Ring

纯 Ring 在 inter-node 通信时有个隐藏问题:每张 GPU 只能用一条 NIC 路径(因为 ring 是一个环)。一个 DGX-H100 有 8 张 IB NIC,但 ring 上每张卡只用一张,剩下 7 张 NIC 闲置。

LoongTrain 的 Double-Ring Attention 把单一 ring 拆成双层 ring:

  • 内环(inner ring):intra-node 几张 GPU 组成,走 NVLink。多个内环并行。
  • 外环(outer ring):跨节点,内环之间用 P2P。所有 NIC 同时用。

公开 benchmark(arXiv:2406.18485 abstract):"improves Model FLOPs Utilization (MFU) by up to 2.88×",在 1M sequence 上对比 DeepSpeed-Ulysses 和 Megatron-CP。

6.3 USP / LoongTrain 的局限

优点 ✓缺点 ✗

7. 方法六:MagiAttention

USP/LoongTrain 把 Ring 和 Ulysses 拼起来,缓解了大 cp 时的扩展性。但它们都没碰一个更深的问题:对于 sparse / varlen / block-causal / 视频 AR / MoD 等"非标准"mask,所有上述方法都不友好。

MagiAttention(SandAI 2025,GitHub+官方博客)从这个角度切入,提出了一套完全不同的设计。它有四个核心组件,我们一个个看。

7.1 Flexible Flash Attention 内核(FFA)

传统 FlashAttention 把 mask 写死成 causal / full / sliding_window 几种枚举。FFA 用一组通用结构表达任意 mask:

AttnSlice = (QRange, KRange, MaskType)
         ↑       ↑         ↑
       Q 段     KV 段    mask 类型: FULL / CAUSAL / INV-CAUSAL / BI-CAUSAL

一个 mask = AttnSlice 的集合 = 多个矩形块的并集

例如一个 document-causal mask(每个文档内部 causal,跨文档不可见)就被表达为多个 (Qrange, Krange, CAUSAL) 的拼接;视频 AR 的 block-causal 也类似分解。FFA 内核能直接消化这个表达,不需要为每种 mask 手写一个 kernel。

性能(README):"performance comparable to Flash-Attention 3 on Hopper GPUs" —— 也就是不损失 attention kernel 本身的速度。

7.2 Dispatch Solver:任意 mask 的负载均衡

这是 zigzag/striped 思路的泛化。Zigzag 假定 mask 是规整的因果三角形,所以闭式解(前 + 后)就能均衡。MagiAttention 面对的是任意 mask,需要正式建模成一个最优化问题:

$$ f^* = \arg\min_f \max_j \text{SumArea}(B_j) $$

其中 $f$ 是把 chunks 分到 cp 个 bucket 的分配方案,$B_j$ 是 bucket $j$ 的负载(mask 内的 entry 数)。这是 NP-hard,实际用贪心 min-heap 近似,$O(n \log n)$ 一次迭代。

意义:任意 mask 都能做到几乎均衡的 sharding,不再受限于 zigzag 那种"只在 causal mask 上有效"的局部技巧。

7.3 GroupCast/GroupReduce:零冗余通信

这是 MagiAttention 区别于所有先驱的最大创新。Ring/Ulysses 都基于统一的 P2P 或 A2A 拓扑,但实际通信中有大量冗余:

  • Ring 在 causal mask 下转的 KV,其中很多 entry 根本不会被用到(因为被 mask 掉)—— 论文里说 "causal mask causes 25% redundant communication"。
  • varlen block-causal 下冗余更高,"can exceed 33% redundancy"。

MagiAttention 用两个新通信原语取代 P2P/A2A 的固定模式:

原语语义底层
GroupCast(正向)给定 ranks → (起始 offset, 长度) 的 mapping,把需要的 KV 段发给需要它的 rank,不冗余AllToAll-v + Range-Gather kernel
GroupReduce(反向)类似,但 reduce 梯度,不分发Range-Scatter-Reduce kernel

v1.0.5+ 用 DeepEP(DeepSeek 开源)做底层 hierarchical AllToAll-v,在 RDMA 上自动去重,直击 LoongTrain Double-Ring 解决的同一痛点 —— 但通过更通用的 AllToAll-v 原语,而不是 ring overlay。

7.4 Multi-Stage Overlap

把 GroupCast/GroupReduce 切成 stage,前一 stage 通信和后一 stage 的 FFA 计算 pipeline 并行,只有最后一个 stage 的计算不能被通信隐藏 —— 经典的 prologue/epilogue 重叠思路,但应用在 attention 这一层。

7.5 公开 benchmark

README 和官方博客里和 Ulysses、Ring P2P、Ring AllGather、USP、LoongTrain、Megatron HybridCP 逐一对比,测试规模:H100 集群 8K seq/device,B200 集群 16K seq/device,cp 从 8 到 64。Mask 类型:full / causal / varlen full / varlen causal。

博客图(Figs. 44-59):"near-linear scalability on varlen causal masks" on H100 + B200。具体 TFLOPS 数字只在图里给(没在文本里直接 quote),所以本文不引用具体值。

7.6 局限性

  • FFA 当前 Hopper 优先(B200 通过 fork 的 FA4 v1.1.0+ 支持,Ampere v1.1.1+ 支持)。
  • GroupCast/GroupReduce 原生实现依赖 DeepEP,目前主要针对 NVLink + RDMA 的 Hopper 级拓扑。

A Survey of Context Parallel Methods: From RingAttention to MagiAttention

This article systematically organizes all the major public Context Parallel (CP) methods used in long-sequence LLM training: AllGather KV (the TransformerEngine default), Ring Attention (P2P), Striped/Zigzag causal load balancing, DeepSpeed Ulysses (A2A), USP/yunchang, LoongTrain Double-Ring, MagiAttention, plus a few secondary variants. Each method is analyzed in a unified "communication + compute" modeling framework: what bottleneck it solves, at what scale it breaks, and where it lands in open-source projects like Megatron-LM / TransformerEngine. The article ends with a comprehensive comparison table and a selection decision tree.

All references are drawn from public sources: arXiv papers, open-source GitHub repositories, and official documentation. This is Part 1 of the "LLM Training Systems" series.

0. Why Context Parallel is needed

From GPT-3 (2K context) to GPT-4 (128K) to Gemini 1.5 (1M+), the LLM context window has expanded more than 500× in five years. Sora, video generation, long-document RAG, long-horizon agent tasks, genomic sequences. Almost every new scenario pushes this curve. But training a long-context model runs into two walls:

  • Memory wall: activation memory grows linearly as $O(S)$. A LLaMA2-7B at 8K context already uses ~10 GB for activations; at 128K it OOMs outright, and even selective recomputation barely keeps it afloat.
  • Compute wall: attention FLOPs grow as $O(S^2)$. Going from 32K to 128K (4×) makes attention 16× more expensive, and it ends up dominating the entire step time.

None of the existing parallelism schemes can save you:

Existing parallelismWhy it's not enough
Data Parallel (DP)Every rank holds the activations for the full sequence. As S grows, every rank's memory still explodes.
Tensor Parallel (TP)Splits along the hidden dimension; communication (AllReduce / AllGather) happens at every layer and must be on the critical path. Beyond TP > 8 the communication can't be overlapped, hurting scalability.
Pipeline Parallel (PP)Splits along the layer dimension. Each stage still holds full-sequence activations, so the memory problem is unchanged; it only chops up the batch.
Expert Parallel (EP)Applies only to MoE layers, completely orthogonal to dense attention.

Context Parallel is the fifth orthogonal parallelism in this lineup, specifically designed for long sequences: split the long sequence into cp chunks along the $S$ dimension, one chunk per GPU. All non-attention layers of the model (LayerNorm, FFN, QKV projection) compute independently along the sequence direction and need no communication. Attention is the only exception, because in $\text{Attention}(Q, K, V) = \text{softmax}(QK^T/\sqrt{d})\,V$, every query token must take an inner product against all key/value tokens. A rank only holds $S/\text{cp}$ tokens of KV, but every token needs to see all KV.

The core problem: all the challenges and method variants of Context Parallel ultimately boil down to a single thing, the attention layer's "cross-rank KV communication". Which communication primitive to use? How much redundancy is there? How do you overlap it with attention computation? How do you handle the imbalance introduced by the causal mask? The papers come in all shapes, but they're all answering this same question.

The article unfolds along this line: first build a unified modeling framework for side-by-side comparison (§1), then start from the most naive AllGather KV (§2) and follow the "evolution of the communication primitive" through Ring Attention (§3) → Striped/Zigzag for causal masks (§4) → Ulysses's axis swap (§5) → USP/LoongTrain's 2D grid (§6) → MagiAttention's flexible mask + zero-redundant communication (§7). At the end we give a comprehensive eight-method table and a selection decision tree (§9).

1. Unified model: communication vs compute

Comparing CP methods side by side requires putting them into a single set of equations. This section builds a simplified but accurate modeling framework that every later section will lean on.

1.1 FLOPs and communication volume of attention

Notation (used throughout the article):

SymbolMeaning
$S$Sequence length (per sample)
$H$Number of attention heads (with MHA, $H_{kv} = H$)
$H_{kv}$Number of KV heads (with GQA, $< H$)
$d$Per-head hidden dimension
$\text{cp}$Context parallel size
$\text{tp}$Tensor parallel size (split along the head dimension)
$\text{flops}$Achievable per-GPU FLOPS (TFLOPS)
$BW$Achievable communication bandwidth (GB/s)

The compute for one forward attention (ignoring softmax / LayerNorm; only the two matmuls $QK^T$ and $\text{attn} \cdot V$):

$$ V_{\text{compute}} = 4 \times \frac{H}{\text{tp}} \times S^2 \times d $$

where 4 = 2 matmuls × 2 FMA ops/element. The causal mask drops about half the entries, so divide by 2:

$$ V_{\text{compute, causal}} = 2 \times \frac{H}{\text{tp}} \times S^2 \times d $$

Introducing cp, each GPU only computes $1/\text{cp}$ of the work (ideal balance):

$$ V_{\text{compute, per GPU}} = \frac{V_{\text{compute}}}{\text{cp}} $$

The core difference between CP methods lies in the communication volume $V_{\text{comm}}$: it depends on what is communicated (KV / Q+KV+O), how many times (1 AG vs cp-1 P2Ps), and at what granularity (whole KV vs chunks). We derive $V_{\text{comm}}^{\text{method}}$ in each section.

The total time for one attention (assuming communication and compute do not overlap):

$$ T_{\text{attn}} = \frac{V_{\text{compute, per GPU}}}{\text{flops}} + \frac{V_{\text{comm}}}{BW} $$

If they can overlap, take $\max$. This is the unified baseline for all later comparisons.

1.2 Two evaluation scenarios

The same CP method exhibits different properties at different problem scales. We need two scenarios for comparison:

ScenarioHeld fixedVariesMeasures
Scenario 1: strong scaling$S$ fixed + $\text{tp}$ fixed$\text{cp}$ grows (more GPUs on the same sequence)Can we keep scaling linearly?
Scenario 2: weak scaling$S$ fixed + $\text{tp} \times \text{cp}$ fixed (total GPU count fixed)$\text{cp}$ vs $\text{tp}$ tradeoffGiven a fixed compute budget, how to split between TP and CP?
Example 1.2 LLaMA2-7B attention parameters: $H = 32, d = 128$. A training run with $S = 128\text{K}$, $\text{tp} = 8$:
compute (causal) = $2 \times 32/8 \times (128\text{K})^2 \times 128 = 2 \times 4 \times 1.7 \times 10^{10} \times 128 \approx 1.7 \times 10^{13}$ FLOPs/sample.
One H100 at ~989 TFLOPS (BF16) with 0.55 efficiency → effective 550 TFLOPS. Per GPU ($\text{cp} = 4$): $\frac{1.7 \times 10^{13}}{4 \times 5.5 \times 10^{14}} \approx 7.7$ ms.

Per-GPU attention sits at the millisecond scale, so the communication budget is a few ms. Note this number scales with $S^2/\text{cp}$.

1.3 Communication priority: TP > CP > PP > DP

In a 4D parallelism scenario ($DP \times PP \times CP \times TP$), the different dimensions have very different communication frequencies and bandwidth requirements, so when laying them out on the GPU mesh you should respect this priority:

  1. TP has the heaviest communication (an AllReduce at every layer), so it must sit on intra-node NVLink/NVSwitch.
  2. CP is next (per-layer KV communication in attention); it should be intra-node first, and spill to inter-node only when intra-node isn't enough.
  3. PP is light (one send/recv per micro-batch), so it can cross nodes.
  4. DP communicates little and can be asynchronous (reduce gradients only), making it the best fit for cross-node.

Mapping this onto a typical machine like DGX-H100 (8 GPUs on NVLink + 8 NDR 400G InfiniBand NICs):

  • NVLink 4.0 full mesh: ~900 GB/s (theoretical), ~540 GB/s (measured) aggregate per GPU
  • InfiniBand NDR 8×50 GB/s: ~50 GB/s (theoretical), ~43 GB/s (measured) per NIC

Intra-node vs inter-node bandwidth differ by 10×+. This gap decides everything: once CP crosses the node boundary, the communication cost takes a step change. LoongTrain's Double-Ring in §6 and MagiAttention's GroupCast in §7 are both specifically attacking this inter-node pain point.

2. Method 1: AllGather KV

The most naive scheme: since each GPU has only $S/\text{cp}$ of KV but attention needs all of KV, just AllGather the KV first, then do a local attention. This is exactly the default CP implementation in TransformerEngine (cp_comm_type="all_gather").

2.1 Idea and tensor shapes

Each GPU holds KV of shape [H, S/cp, d]. After AllGather: [H, S, d]. Q is unchanged, still [H, S/cp, d]. Then run a standard FlashAttention to get the output of shape [H, S/cp, d].

# transformer_engine/pytorch/attention/dot_product_attention/context_parallel.py
# Simplified from AttnFuncWithCPAndKVAllGather (line ~2948)
def forward(ctx, q, k, v, cp_group, ...):
    cp_size = get_world_size(cp_group)

    # Step 1: AllGather K, V across cp_group
    k_ag, _ = gather_along_first_dim(k, cp_group)   # [cp*S/cp, ..] -> [S, ..]
    v_ag, _ = gather_along_first_dim(v, cp_group)

    # Step 2: chunk reorder for causal mask (§4 zigzag)
    chunk_ids = get_seq_chunk_ids_for_reordering(cp_size, k.device, True)
    k_ag = torch.index_select(k_ag, dim=0, index=chunk_ids)
    v_ag = torch.index_select(v_ag, dim=0, index=chunk_ids)

    # Step 3: local FA (each GPU processes its own Q chunk, but has full KV)
    out = flash_attn_func(q, k_ag, v_ag, causal=is_causal)
    return out

2.2 Communication volume and model

The KV communication volume per GPU (bf16, 2 bytes):

$$ V_{\text{comm}}^{\text{AG}} = 2 \times 2 \times \frac{H_{kv}}{\text{tp}} \times S \times d $$

The two 2s: one for K + V, one for bf16 byte size. Note the $\frac{H_{kv}}{\text{tp}}$: under GQA, $H_{kv} \ll H$, so the savings are large.

Time (ignoring reorder overhead):

$$ T_{\text{AG}} = \frac{V_{\text{compute}}}{\text{flops} \cdot \text{cp}} + \frac{V_{\text{comm}}^{\text{AG}}}{BW_{\text{AG}}} $$

The "AllGather algorithm bandwidth" $BW_{\text{AG}}$ is essentially independent of cp at fixed tp (NCCL's AG uses a ring-class topology, so bus BW barely varies with rank count), so:

  • Scenario 1 (tp fixed, cp grows): $T_{\text{compute}}$ drops linearly with cp, $T_{\text{comm}}$ is unchanged (because $V_{\text{comm}}$ is independent of cp, since every GPU gathers the full $S$). Once cp is large, attention becomes communication-bound, and adding more GPUs has rapidly diminishing returns.
  • Scenario 2 (tp × cp fixed): cp grows → tp shrinks → $\frac{H_{kv}}{\text{tp}}$ grows → $V_{\text{comm}}$ grows. So in this scenario, larger cp actually makes attention slower.

3. Method 2: Ring Attention (P2P)

The pain point of AllGather is that "communication is on the critical path". The core idea of Ring Attention (Li et al. 2021, Liu et al. 2023): split the AllGather into cp P2P send/recv steps; each step, take in a new KV chunk and immediately attention it against the local Q, so communication runs concurrently with compute.

3.1 Idea and execution flow

Each GPU holds its own Q chunk, and $K_i$ rotates around the ring. At each step:

  1. Compute: compute partial attention on the currently held $(Q_{\text{local}}, K_j, V_j)$, accumulating via online softmax.
  2. Communicate: P2P-send $(K_j, V_j)$ to the next rank in the ring; simultaneously recv $(K_{j-1}, V_{j-1})$ from the previous rank.
  3. Loop: repeat cp times, until each KV chunk has met every local Q chunk once.

3.2 Communication-computation overlap boundary

Per-step (per chunk) communication and compute:

$$ V_{\text{comm}}^{\text{step}} = 4 \times \frac{H_{kv}}{\text{tp}} \times \frac{S}{\text{cp}} \times d $$

$$ V_{\text{compute}}^{\text{step}} = 4 \times \frac{H}{\text{tp}} \times \frac{S^2}{\text{cp}^2} \times d $$

4 = K + V × 2 bytes (bf16); compute scales as $\frac{1}{\text{cp}^2}$ because both the Q chunk and the KV chunk are of size $S/\text{cp}$.

The condition for communication to be fully hidden by compute:

$$ \frac{V_{\text{comm}}^{\text{step}}}{BW} \leq \frac{V_{\text{compute}}^{\text{step}}}{\text{flops}} \quad\Longrightarrow\quad \text{cp} \leq \frac{BW \times S}{\text{flops}} \times \frac{H}{H_{kv}} $$

This is the "communication-computation overlap boundary" $\text{cp}_{\text{bor}}$. When $\text{cp} \leq \text{cp}_{\text{bor}}$, communication is hidden → attention time ≈ compute time ≈ $V_{\text{compute}} / \text{flops}$, which is inversely proportional to cp. When $\text{cp} > \text{cp}_{\text{bor}}$, communication is exposed → attention time becomes proportional to cp (because you still run cp steps and the communication of every step is exposed).

Example 3.2 DGX-H100 + IB: measured NVLink bandwidth 540 GB/s, measured IB bandwidth 43 GB/s (per NIC), H100 effective BF16 compute 550 TFLOPS. $S = 128\text{K}$, MHA (no GQA):
intra-node: $\text{cp}_{\text{bor}} \approx \frac{540 \times 10^9 \times 128 \times 1024}{550 \times 10^{12}} \approx 128$. Intra-node, communication is hardly a bottleneck at all.
inter-node: $\text{cp}_{\text{bor}} \approx \frac{43 \times 10^9 \times 128 \times 1024}{550 \times 10^{12}} \approx 10$. The moment cp exceeds ~10 and crosses out of NVLink, attention immediately becomes communication-bound.

This is exactly why every "P2P-class" method's performance crashes when strong-scaling across nodes.

3.3 Backward pass

Backward Ring is more complex than forward: you have to send not just KV but also the KV gradients. The per-step communication volume doubles. The backward attention kernel itself is also more compute-heavy (about 2.5× forward). In the TE source, the two paths live in the helpers cp_p2p_fwd_flash_attn and cp_p2p_bwd_flash_attn.

4. Method 3: Load balancing under causal mask

Ring Attention has a "hidden load imbalance" that only shows up under the causal mask, i.e., the standard attention for GPT-class models. This section is dedicated to how to solve it.

4.1 The problem: triangular load

Take $S = 8, \text{cp} = 2$ as an example. Under a causal mask, which $(i, j)$ have to be computed? The lower triangle with $j \leq i$:

              KV chunk
              0,1,2,3  4,5,6,7
       Q  0  [  ■        ░     ]    GPU 0 (holds Q0..Q3)
       c  1  [  ■        ░     ]
       h  2  [  ■        ░     ]
       u  3  [  ■        ░     ]
       n  ─────────────────────
       k  4  [  ■        ■     ]    GPU 1 (holds Q4..Q7)
          5  [  ■        ■     ]
          6  [  ■        ■     ]
          7  [  ■        ■     ]

       ■ = compute;  ░ = skip (j > i)

Naive partitioning: GPU 0 holds Q[0..3], KV[0..3]; GPU 1 holds Q[4..7], KV[4..7]. But under the causal mask, GPU 0 only computes half of the upper-left 4×4 = 8 cells; GPU 1 computes the full left 4×4 + half of the right 4×4 = 16+8 = 24 cells. GPU 1 has 3× the work of GPU 0, a severe imbalance. Every Ring step has to wait on the slowest GPU, so the whole thing is dragged down.

4.2 Striped Attention

Striped Attention (Brandon et al., arXiv:2311.09431) solves this by not chunking tokens contiguously, but instead interleaving them with stride = cp. Each GPU's tokens are uniformly scattered across the whole sequence as a "stripe":

       Striped partition with cp = 2:
       GPU 0 holds tokens 0, 2, 4, 6  (even positions)
       GPU 1 holds tokens 1, 3, 5, 7  (odd  positions)

       This way every (Q_i, K_j) pair on every GPU sees
       a nearly identical "geometry" of the causal mask,
       so the load is fully balanced.

Published results (paper §5): 1.45× speedup at 256K sequence on A100, 1.65× speedup at 786K sequence on 16× TPUv4.

4.3 Zigzag / DualChunkSwap

Beyond Striped, the scheme more often used in practice is called Zigzag (community name) or DualChunkSwap (the TransformerEngine term). The idea: split the sequence into 2×cp chunks instead of cp; each GPU gets one front and one back chunk.

       cp = 2, seq split into 4 chunks:
       chunk_0  chunk_1  chunk_2  chunk_3

       GPU 0 holds: chunk_0 + chunk_3  (front + back)
       GPU 1 holds: chunk_1 + chunk_2  (middle two)

       Under the causal mask:
       - GPU 0: small work on chunk_0 + large work on chunk_3
       - GPU 1: medium work on chunk_1 + medium work on chunk_2

       The two sums are nearly equal → load-balanced.

TransformerEngine implementation (context_parallel.py:236):

def get_seq_chunk_ids_for_reordering_before_attn(cp_size, device):
    """Reorder 2*cp chunks so that the rank-th GPU gets
       chunk_{2*rank} and chunk_{2*cp - 2*rank - 1}."""
    chunk_ids = torch.empty(2 * cp_size, dtype=torch.int32, device=device)
    for rank in range(cp_size):
        chunk_ids[rank]           = 2 * rank
        chunk_ids[rank + cp_size] = 2 * cp_size - 2 * rank - 1
    return chunk_ids

This reorder runs once in the dataloader (in megatron/core/utils.py's get_batch_on_this_cp_rank), so the attention kernel itself is identical to plain ring attention, with no extra kernel complexity.

Naming confusion warning: the same technique goes by different names in different places. TransformerEngine calls it DualChunkSwap, the Megatron-LM docs call it load-balanced, and the community (zhuzilin/ring-flash-attention, USP) calls it zigzag. Striped Attention is a different scheme (token-level interleaving with granularity = 1), not the same thing. The name "zigzag" has no formal paper origin; it emerged in the community around the same time as Striped (2023.11), independently.

4.4 Practical notes

  • This reorder must happen in the dataloader, otherwise the attention mask indices will be wrong.
  • The sequence length must be divisible by 2 * cp_size (pad if necessary).
  • Zigzag is on by default in TransformerEngine / Megatron-LM, so you get this optimization for free as soon as you turn on CP.

5. Method 4: Ulysses (A2A)

Ring/Striped/Zigzag all walk the same road: split along the sequence dimension S and rotate KV via P2P. DeepSpeed's Ulysses (Jacobs et al., arXiv:2309.14509) takes a completely different path: swap axes with AllToAll before attention, so that within attention you split along heads instead of the sequence.

5.1 Idea: axis swap via dispatch

The tensor entering attention in each transformer block: [H, S/cp, d] (distributed along S). Ulysses inserts one AllToAll before attention and one after:

Before attention:                      Inside attention:                    After attention:
─────────────────                      ─────────────                       ─────────────────
GPU 0: [H, S/cp, d]    AlltoAll    GPU 0: [H/cp, S, d]    local FA      GPU 0: [H/cp, S, d]    AlltoAll    GPU 0: [H, S/cp, d]
GPU 1: [H, S/cp, d]   ─────────→   GPU 1: [H/cp, S, d]    ──────→      GPU 1: [H/cp, S, d]   ─────────→   GPU 1: [H, S/cp, d]
GPU 2: [H, S/cp, d]                GPU 2: [H/cp, S, d]                 GPU 2: [H/cp, S, d]                GPU 2: [H, S/cp, d]
GPU 3: [H, S/cp, d]                GPU 3: [H/cp, S, d]                 GPU 3: [H/cp, S, d]                GPU 3: [H, S/cp, d]

     ↑                                  ↑                                    ↑                                ↑
  split S                            split H                              split H                          split S
                                  each GPU has full S                  each GPU has full S
                                  runs attention on H/cp heads

  • Original: microsoft/DeepSpeed, DistributedAttention in deepspeed/sequence/layer.py
  • TE version: AttnFuncWithCPAndQKVOA2A (context_parallel.py:3807), enabled via cp_comm_type="a2a"

5.2 Communication complexity

Ulysses runs 4 AllToAlls per attention: one before and one after in forward (QKV → output reshape), and the same two in backward. Total communication (all tokens participate in A2A):

$$ V_{\text{comm}}^{\text{Ulysses, per pass}} = 4 \times 2 \times \frac{H + 2 H_{kv}}{\text{tp}} \times \frac{S}{\text{cp}} \times d \times \frac{\text{cp} - 1}{\text{cp}} $$

Key property: per-GPU communication volume is essentially independent of cp. That's the essence of A2A, with a $\frac{n-1}{n}$ bus bandwidth utilization. This is Ulysses's biggest advantage over Ring: when cp doubles, Ring's total communication doubles while Ulysses's stays almost the same.

The original wording in the paper (Jacobs et al. §3): "maintains constant communication volume when sequence length and compute devices are increased proportionally".

5.3 Fatal constraint: cp ≤ num_kv_heads

Ulysses splits along the head dimension, so cp must divide H (MHA) or $H_{kv}$ (GQA). TP also splits along heads, giving the joint constraint:

$$ \text{tp} \times \text{cp} \leq H_{kv} $$

This is Ulysses's kill switch. LLaMA3-8B uses GQA-8 (num_kv_heads = 8), so $\text{tp} \times \text{cp} \leq 8$. If you want to train on 1M context, you need cp = 256, which is flat-out impossible.

DeepSpeed-Ulysses by itself is not suitable for multi-node scaling of modern GQA large models on long sequences. That's its fundamental limitation as a standalone method.

The fix: combine it with Ring. Use Ulysses in a small range (cp_a2a ≤ H_kv) and let Ring carry the rest of the cp dimension. This is the heart of USP / LoongTrain in §6.

6. Method 5: USP / LoongTrain 2D

Ring's strength is arbitrary scalability but with large communication; Ulysses's strength is small communication but capped by the head count. The natural idea: multiply them together and do 2D parallelism. This is the shared idea behind USP (Fang & Zhao, arXiv:2405.07719) and LoongTrain (Gu et al., arXiv:2406.18485). The two papers are only weeks apart (USP in May, LoongTrain in June), with the same core idea; LoongTrain adds the Double-Ring engineering optimization on top.

6.1 Idea: Ulysses × Ring 2D grid

Split cp into two dimensions:

$$ \text{cp} = \text{cp}_{a2a} \times \text{cp}_{p2p} $$

The GPU topology is organized as a 2D grid:

  • Same row (cp_a2a ranks): Ulysses A2A splits heads. On intra-node NVLink, A2A is extremely fast. Constraint: cp_a2a ≤ H_kv / tp.
  • Same column (cp_p2p ranks): Ring P2P splits the sequence. Each "a2a group" runs Ring Attention independently. Can cross nodes.

So one attention does the following:

  1. First A2A: swap axes within the a2a group; each rank ends up with H_kv/cp_a2a heads of the full sequence.
  2. Ring Attention: within the p2p group, run ring attention on these heads (cp_p2p rotations).
  3. Second A2A: swap axes back; output is still shaped [H/tp, S/cp, d].
Example 6.1: LLaMA3-8B on 16 GPUs $H_{kv} = 8$; assume tp=2 and want cp=16:
- Pure Ulysses: cp ≤ $H_{kv} / \text{tp} = 4$ → can't reach cp=16.
- Pure Ring: cp=16 in one P2P chain, severe single-NIC bottleneck across nodes.
- USP/LoongTrain: cp_a2a=4 (= $H_{kv}/\text{tp}$, maxed out on heads), cp_p2p=4 → total cp=16. Intra-node 4 GPUs run A2A (NVLink 540 GB/s), inter-node 4 nodes run Ring (IB 43 GB/s). The combined scheme uses every dimension available.

Implementation: feifeibear/long-context-attention (yunchang) wraps both dimensions uniformly. TE also implements the same idea, enabled via cp_comm_type="a2a+p2p", splitting the cp_group internally into [a2a_group, p2p_group].

USP's public benchmark (arXiv:2405.07719): LLAMA3-8B at 208K seq on 2 × 8 × A800 RDMA, 269.8 TFLOPS, MFU 0.86.

6.2 LoongTrain's Double-Ring

Pure Ring has a hidden issue during inter-node communication: each GPU can only use one NIC path (because a ring is a single loop). A DGX-H100 has 8 IB NICs, but a ring uses only one of them per GPU, leaving 7 NICs idle.

LoongTrain's Double-Ring Attention splits the single ring into a two-level ring:

  • Inner ring: a few intra-node GPUs forming a ring over NVLink. Multiple inner rings run in parallel.
  • Outer ring: cross-node P2P between inner rings. All NICs are used simultaneously.

Public benchmark (arXiv:2406.18485 abstract): "improves Model FLOPs Utilization (MFU) by up to 2.88×", on 1M sequence vs DeepSpeed-Ulysses and Megatron-CP.

7. Method 6: MagiAttention

USP/LoongTrain glue Ring and Ulysses together and ease the scalability problem at large cp. But neither tackles a deeper issue: for "non-standard" masks like sparse / varlen / block-causal / video AR / MoD, none of the methods above are friendly.

MagiAttention (SandAI 2025, GitHub + official blog) cuts in from this angle and proposes a completely different design. It has four core components, which we walk through one by one.

7.1 Flexible Flash Attention kernel (FFA)

Traditional FlashAttention hard-codes the mask as one of a few enums: causal / full / sliding_window. FFA expresses any mask via a generic structure:

AttnSlice = (QRange, KRange, MaskType)
         ↑       ↑         ↑
       Q range  KV range  mask type: FULL / CAUSAL / INV-CAUSAL / BI-CAUSAL

A mask = a set of AttnSlices = the union of rectangular blocks

For example, a document-causal mask (causal within each document, invisible across documents) is expressed as the concatenation of multiple (Qrange, Krange, CAUSAL) slices; video AR's block-causal decomposes similarly. The FFA kernel directly consumes this representation, with no need to write a kernel per mask.

Performance (README): "performance comparable to Flash-Attention 3 on Hopper GPUs", i.e., no degradation in the attention kernel speed itself.

7.2 Dispatch Solver: load balance for arbitrary masks

This is the generalization of zigzag/striped. Zigzag assumes the mask is a regular causal triangle, so a closed-form solution (front + back) gives balance. MagiAttention faces arbitrary masks and has to formalize this as an optimization problem:

$$ f^* = \arg\min_f \max_j \text{SumArea}(B_j) $$

where $f$ is the assignment of chunks to cp buckets and $B_j$ is the load (entries inside the mask) of bucket $j$. This is NP-hard; in practice they use a greedy min-heap approximation, $O(n \log n)$ per iteration.

Significance: any mask can be sharded with near-perfect balance, no longer limited to a local trick like zigzag that only works on causal masks.

7.3 GroupCast/GroupReduce: zero-redundant communication

This is MagiAttention's biggest innovation relative to all its predecessors. Ring/Ulysses both rest on a uniform P2P or A2A topology, but the actual communication carries a lot of redundancy:

  • The KV that Ring rotates under a causal mask contains many entries that are never used (they get masked out); the paper notes "causal mask causes 25% redundant communication".
  • Under varlen block-causal the redundancy is higher, "can exceed 33% redundancy".

MagiAttention replaces the fixed P2P/A2A patterns with two new communication primitives:

PrimitiveSemanticsBacking
GroupCast (forward)Given a ranks → (start offset, length) mapping, send the needed KV ranges to the ranks that need them, with no redundancyAllToAll-v + Range-Gather kernel
GroupReduce (backward)Same shape, but reduces gradients instead of dispatchingRange-Scatter-Reduce kernel

v1.0.5+ uses DeepEP (open-sourced by DeepSeek) as the underlying hierarchical AllToAll-v, which auto-dedupes on RDMA. It hits the same pain point that LoongTrain's Double-Ring addresses, but through a more general AllToAll-v primitive instead of a ring overlay.

7.4 Multi-Stage Overlap

Slice GroupCast/GroupReduce into stages and pipeline the communication of one stage against the FFA compute of the next, so only the last stage's compute can't be hidden behind communication. This is the classic prologue/epilogue overlap idea, applied at the attention layer.

7.5 Public benchmarks

The README and the official blog compare head-to-head against Ulysses, Ring P2P, Ring AllGather, USP, LoongTrain, and Megatron HybridCP. Test scales: H100 cluster at 8K seq/device, B200 cluster at 16K seq/device, cp from 8 to 64. Mask types: full / causal / varlen full / varlen causal.

Blog figures (Figs. 44-59): "near-linear scalability on varlen causal masks" on H100 + B200. Specific TFLOPS numbers appear only in the figures (not quoted in text), so we don't reproduce values here.

7.6 Limitations

  • FFA currently prioritizes Hopper (B200 supported via a forked FA4 in v1.1.0+, Ampere in v1.1.1+).
  • Native GroupCast/GroupReduce depends on DeepEP and currently targets the Hopper-class topology with NVLink + RDMA.

9.2 Selection decision tree

The table above compressed into an actionable flow. Given: cp size, causal or not, GQA or not, single-node or multi-node.

START
  │
  ├─ cp == 1
  │      └→ no CP needed, skip this series
  │
  ├─ mask is non-standard (varlen / sparse / block-causal / video AR)
  │      └→ MagiAttention (the only one that expresses arbitrary masks cleanly)
  │
  ├─ sliding-window mask required
  │      └→ AllGather KV (TE: cp_comm_type="all_gather") or Ulysses
  │
  ├─ cp ≤ 2 and intra-node
  │      └→ RingAttention + Zigzag (simplest; communication overlaps)
  │
  ├─ cp ≤ H_kv / tp and intra-node
  │      └→ Ulysses (A2A is extremely fast on NVLink; constant communication volume)
  │
  ├─ cp moderate (8 ~ 32) and cp > H_kv / tp
  │      └→ USP / LoongTrain 2D
  │             ├─ cp_a2a = min(H_kv / tp, intra-node size)
  │             └─ cp_p2p = cp / cp_a2a
  │      Prefer LoongTrain's Double-Ring on multi-node
  │
  └─ cp very large (> 64), or pursuing peak MFU, or needing full mask flexibility
         └→ MagiAttention (zero-redundant communication)
Practical tips:
① Always turn on zigzag/DualChunkSwap under causal masks. It's a free lunch and TE/Megatron enable it by default.
② The product of tp and cp must fit the head count. With LLaMA3-8B GQA-8, when tp + cp exceeds 8, Ulysses must be combined with Ring.
③ Cross-node cp is always a bottleneck. Keep cp intra-node (NVLink) as a priority; let PP/DP cross node boundaries.
④ On new architectures (B200 / GB200), NVLink is upgraded, so intra-node 8 GPUs can handle cp=8 on very long sequences, weakening the "2D" value of USP/LoongTrain. MagiAttention's "flexible mask" value remains unchanged (video generation, MoD).
