GEMM 计算/访存比（Arithmetic Intensity）分析

方阵基线 → 零复用下界 → 分块 → 任意形状 tile 的完整推导

元素字节数记作 $b$（FP32: $b=4$；FP16/BF16: $b=2$；FP8: $b=1$）。

📌 建模前提：下面 Section 一、二 采用 单层 blocking 模型 —— 每个输出 tile 的输入 A / B 都假设从 DRAM 独立加载，不考虑相邻 C tile 之间通过 cache 产生的复用。这是为什么 $K_t$ 在渐近 AI 中"消掉"。Section 三 放开这个假设，用 2 级 blocking（DRAM ↔ L2 ↔ SMEM）重新审视，$K_t$ 会通过 cache 预算间接入场。

一、方阵基线（$N \times N$）

$C = A \cdot B$，三者都是 $N \times N$。总 FLOPs $= 2N^3$（每个 $C$ 元素 $N$ 次乘加）。

1. 理论下界（完美复用）

每个输入 / 输出元素只从主存进出一次，总访存 $3N^2$ 元素：

$$\text{AI}_{\text{ideal}} \;=\; \frac{2N^3}{3N^2 \cdot b} \;=\; \frac{2N}{3b}\ \text{FLOP/byte}$$

随 $N$ 线性增长。FP32、$N = 4096$ 时 $\approx 683$ FLOP/byte——远超 H100（$\approx 7$）、B200（$\approx 20$）的平衡点，所以大 GEMM 理论上是 compute-bound。

2. 完全无复用（naive 三重循环）

$C$ 在寄存器 acc 里累加，K 走完后一次写回。整个 GEMM：

  • FLOPs：$2 N^3$
  • 读 $A$：$N^3$（每个 $A[i,k]$ 跨 $j$ 重复加载 $N$ 次）
  • 读 $B$：$N^3$（每个 $B[k,j]$ 跨 $i$ 重复加载 $N$ 次）
  • 写 $C$：$N^2$（每个 $(i,j)$ 累加完写回一次）
$$\text{AI}_{\text{no-reuse}} \;=\; \frac{2 N^3}{(2 N^3 + N^2)\cdot b} \;\xrightarrow{N\to\infty}\; \frac{1}{b}\ \text{FLOP/byte}$$

写 $C$ 的 $N^2$ 项被 $2 N^3$ 压住，渐近 与 $N$ 无关。FP32 下仅 $0.25$ FLOP/byte——彻底 memory-bound，H100 只能发挥不到 4% 峰值。

朴素三重循环伪代码：

# Naive GEMM: C = A · B, 三者均为 N×N
for i in 0..N:
    for j in 0..N:
        acc = 0
        for k in 0..N:
            acc += A[i, k] * B[k, j]      # 每次迭代 2 loads + 2 FLOPs
        C[i, j] = acc                       # C 全程留在寄存器

内层每次 FMA 都从 DRAM 拉 A[i,k] 和 B[k,j]，没有任何跨迭代复用，故 $\text{AI} = 1/b$。

3. 分块（块大小 $T \times T$，cache 零延迟）

一个 $T \times T$ 的 $C$ 块沿 $k$ 方向扫过 $N/T$ 次：

  • FLOPs: $(N/T) \cdot 2T^3 = 2NT^2$
  • 主存访问：$A$ tiles $(N/T)\cdot T^2 = NT$，$B$ tiles $NT$，$C$ 写回 $T^2$（K-loop 累加完一次）
$$\text{AI}_{\text{blocked}} \;=\; \frac{2NT^2}{(2NT + T^2)\cdot b} \;\xrightarrow{N\to\infty}\; \frac{T}{b}\ \text{FLOP/byte}$$

当 $N \gg T/2$ 时写 $C$ 可忽略，全图总访存 $\approx 2N^3/T$，正是 Hong–Kung 下界 $\Omega(N^3/\sqrt{M})$ 的具象形式。

分块 GEMM 伪代码（方形 tile，$T \times T$）：

# Blocked GEMM, 假设 3·T² 元素能同时放进 cache
for i0 in 0..N step T:              # 遍历 C 的行块
    for j0 in 0..N step T:          # 遍历 C 的列块
        C_tile = 0                       # T×T 累加器（寄存器 / shared mem）
        for k0 in 0..N step T:      # 沿 K 方向扫块
            # —— 一次性把两个输入 tile 搬进 fast storage ——
            A_tile = A[i0:i0+T, k0:k0+T]
            B_tile = B[k0:k0+T, j0:j0+T]
            # —— tile 内做稠密小 GEMM（在 cache 里，零 DRAM 访问）——
            for i in 0..T:
                for j in 0..T:
                    for k in 0..T:
                        C_tile[i, j] += A_tile[i, k] * B_tile[k, j]
        C[i0:i0+T, j0:j0+T] = C_tile     # 一次写回

关键差异：每对 A_tile、B_tile 只从 DRAM 加载 一次，在 tile 内被复用 $T$ 次——AI 从 $1/b$ 放大到 $T/b$。

4. cache 容量约束

三个 tile 同时在 cache：$3T^2 b \le M$，故

$$T_{\max} \;=\; \sqrt{\frac{M}{3b}}, \qquad \text{AI}_{\max} \;=\; \sqrt{\frac{M}{3b^{3}}}$$

要 AI 翻倍，cache 必须变 4 倍。

二、任意形状 tile generalized

一般 GEMM：$C\,(M{\times}N)\mathrel{+}= A\,(M{\times}K)\cdot B\,(K{\times}N)$，tile 尺寸 $M_t \times N_t$（$C$ 块）、$M_t \times K_t$（$A$ 块）、$K_t \times N_t$（$B$ 块）。

1. 核心公式

算一个 $M_t \times N_t$ 的 $C$ tile：

项量（元素）
FLOPs$2\, M_t N_t K$
读 $A$$(K/K_t)\cdot M_t K_t = M_t K$
读 $B$$(K/K_t)\cdot K_t N_t = K N_t$
写 $C$$M_t N_t$（K-loop 累加完一次写回）
$$\text{AI} \;=\; \frac{2 M_t N_t K}{\big((M_t + N_t)\,K + M_t N_t\big)\cdot b}$$

分母里 $(M_t + N_t) K$ 来自 $A$、$B$ 在 K 方向的反复加载（随 $K$ 线性增长），$M_t N_t$ 是 C tile 在累加结束后的一次写回（与 $K$ 无关）。当 $K \gg \tfrac{M_t N_t}{M_t + N_t}$ 时，写 C 可忽略，渐近退化为：

$$\boxed{\;\text{AI} \;\xrightarrow{K \to \infty}\; \frac{2 M_t N_t K}{(M_t + N_t)\, K \cdot b} \;=\; \frac{2}{b} \cdot \frac{M_t N_t}{M_t + N_t} \;=\; \frac{H(M_t, N_t)}{b}\;}$$

其中 $H(M_t, N_t) \;=\; \dfrac{2 M_t N_t}{M_t + N_t}$ 是 调和平均。下文分析都基于这个渐近形式；但 Split-K 把单个 C tile 的 $K$ 切短、又把写回放大 $S$ 倍（见 Section 4），那里就必须回到严格式子。

2. Takeaway 1：AI 由较短边主导

调和平均性质——被短板拉死。直观理解：$A$ 每个元素复用 $N_t$ 次，$B$ 每个元素复用 $M_t$ 次。

$M_t \times N_t$$H(M_t, N_t)$解释
$128 \times 128$128方阵基准
$256 \times 64$$102$元素个数 $\times 1$，AI 降到 80%
$512 \times 32$$60$元素个数 $\times 1$，AI 只剩 47%
$1024 \times 16$$31.5$接近 $2 N_t$
$M_t \to \infty,\; N_t$ 固定$2 N_t$极限等于短边两倍
$1 \times N_t$（GEMV）$\approx 2$常数，算力发挥不出
这就是 GEMV、skinny GEMM、LLM decode 阶段 $Q\cdot K^\top$（当一侧很小）变 memory-bound 的根源。

3. Takeaway 2：$K_t$ 在渐近 AI 中消掉（仅限单层 blocking）

  • $K_t$ 只是 $K$ 循环的切片，不改变 $A$、$B$ 的复用倍数（仍是 $N_t$、$M_t$）。
  • $K_t$ 实际影响：working set $= M_t K_t + K_t N_t + M_t N_t$，流水线深度，accumulator 初始化摊销。
  • 工程选型：够 Tensor Core 指令宽度（HMMA $k{=}16$、WGMMA $k{=}16/32$）加一点流水化即可，不追求大。
  • 注意：放开"tile 间无复用"假设后，$K_t$ 会通过 cache 预算间接影响 AI —— 详见 Section 三。

4. cache 预算下的最优形状

固定 $K_t$，约束 $M_t N_t + K_t (M_t + N_t) \le M/b$，最大化 $H(M_t, N_t)$：

$$\arg\max_{M_t,\,N_t} H(M_t, N_t)\ \Longrightarrow\ M_t = N_t,\quad M_t \approx \sqrt{M/b} - K_t$$

在容量预算内，越接近正方形 AI 越高。这也是用方阵推导就能说明问题的根因。

5. 两种退化极端：行条 / 列条 tile

$M_t$ 或 $N_t$ 退化到 1 的两种边界——对应 $H(1, N) \to 2$ 和 $H(M, 1) \to 2$，AI 都落到 $2/b$。它们是 Berkeley slide 里 "naive" 给出 $q \approx 2$ 的本质——隐式把 tile 设成了行条或列条，比 $1{\times}1$ 多了一维复用。

$M_t = 1,\ N_t = N$（行条）：A 的第 $i$ 行加载一次、被 $N$ 次复用：

# 行条 tile: 一次扫完 C 的一整行, A 行在 cache 里被复用 N 次
for i in 0..M:
    # 把 A[i, :] (K 个元素) 搬进 fast storage, 整个 j 循环都用它
    for j in 0..N:                     # 扫 C 的整行
        acc = 0                             # C[i, j] 在寄存器
        for k in 0..K:
            acc += A[i, k] * B[k, j]        # A[i,k] 从 cache, B[k,j] 从 DRAM
        C[i, j] = acc

访存：读 A 用 $M K$（每行一次），读 B 用 $M N K$（跨 $i$ 无复用），FLOPs $2 M N K$：

$$\text{AI}_{\text{row-strip}} \;=\; \frac{2 M N K}{(M K + M N K)\, b} \;\xrightarrow{N \to \infty}\; \frac{2}{b}$$

$M_t = M,\ N_t = 1$（列条）：对称情形，B 的第 $j$ 列被 $M$ 次复用：

# 列条 tile: 一次扫完 C 的一整列, B 列在 cache 里被复用 M 次
for j in 0..N:
    # 把 B[:, j] (K 个元素) 搬进 fast storage, 整个 i 循环都用它
    for i in 0..M:                     # 扫 C 的整列
        acc = 0
        for k in 0..K:
            acc += A[i, k] * B[k, j]        # B[k,j] 从 cache, A[i,k] 从 DRAM
        C[i, j] = acc

访存对称：读 B 用 $N K$，读 A 用 $M N K$，FLOPs $2 M N K$：

$$\text{AI}_{\text{col-strip}} \;=\; \frac{2 M N K}{(N K + M N K)\, b} \;\xrightarrow{M \to \infty}\; \frac{2}{b}$$

要把 AI 从 $2/b$ 再往上推，就必须 同时 复用 $A$ 和 $B$ —— 即 $M_t, N_t$ 都大于 1 的方形 tile，这就回到了 Section 1.3 的分块 GEMM，AI 可以放大到 $T/b$。

三、2 级 blocking 与跨 tile 复用

Section 一、二 只看了"一个 tile 独立从 DRAM 取数据"的最保守情形。真实硬件是多级 cache（HBM ↔ L2 ↔ SMEM ↔ 寄存器），相邻 C tile 之间会自动跨级复用。放开"tile 间无复用"假设后，$K_t$ 会通过 cache 预算间接回到 AI 公式里。

1. 跨 tile 的可复用数据总量 —— 与 $K_t$ 无关

处理 $C[0, j_1]$ 和 $C[0, j_2]$ 时，都需要 $A$ 的第 0 行块全部 $K$ 列 —— 即 $M_t \times K$ 个元素。切成 $K/K_t$ 个 $M_t \times K_t$ 小块还是整条一次加载，共享量始终是 $M_t K$。$K_t$ 只决定分块粒度，不决定复用潜力。B 端对称：$C[i_1, 0]$ 和 $C[i_2, 0]$ 共享 $N_t K$ 元素。

2. 两层 tile 结构

层级tile容量带宽决定什么
外层（DRAM ↔ L2）$M_o \times N_o \times K$$M_{L2}$（~几十 MB）$BW_{\text{DRAM}}$DRAM 流量、跨 CTA 复用
内层（L2 ↔ SMEM / 寄存器）$M_t \times N_t \times K_t$SMEM（~200 KB）+ 寄存器$BW_{L2}$CTA 内部复用、L2 流量

每一层各自有 $H$ 型的 AI：

$$\text{AI}_{\text{outer}} = \frac{H(M_o, N_o)}{b}, \qquad \text{AI}_{\text{inner}} = \frac{H(M_t, N_t)}{b}$$

3. 多级 Roofline：有效吞吐由最慢一层决定

算力峰值 $P$ 要同时被两层 cache 的带宽喂饱：

$$\text{Throughput}_{\text{eff}} = \min\!\left(P,\; BW_{\text{DRAM}} \cdot \text{AI}_{\text{outer}},\; BW_{L2} \cdot \text{AI}_{\text{inner}}\right)$$

想让算力不被任何一层卡死，需要

$$\text{AI}_{\text{outer}} \;\ge\; \frac{P}{BW_{\text{DRAM}}}, \qquad \text{AI}_{\text{inner}} \;\ge\; \frac{P}{BW_{L2}}$$

两个不等式各自决定该层 tile 至少多大。

4. 两种带宽比例下的 tile 选型

Case A：$BW_{L2} \approx BW_{\text{DRAM}}$（L2 近乎没用，或内外层带宽接近）

两层对 AI 的要求一样大：$H(M_t, N_t) \approx H(M_o, N_o)$，也就是 $M_t$ 必须和 $M_o$ 同量级。内层 tile 已经要做到和外层一样大，2 级 blocking 退化 成单层——分两层没带宽收益。

Case B：$BW_{L2} = 10 \times BW_{\text{DRAM}}$（典型 GPU：Hopper L2 对 HBM 约 $5\times\!\sim\!20\times$）

内层只需满足

$$\text{AI}_{\text{inner}} \;\ge\; \frac{P}{BW_{L2}} \;=\; \frac{1}{10}\cdot\frac{P}{BW_{\text{DRAM}}}$$

即 $H(M_t, N_t) \gtrsim \tfrac{1}{10} H(M_o, N_o)$。方形近似下 $M_t \approx M_o / 10$ 就够了。

这正是 GPU CTA tile（$128\times 128$）+ cluster / pipeline stage 级别大 panel 这种配置的根据：内外两层带宽差拉开后，内层 tile 可以缩小很多，换来更高 occupancy 和更灵活的调度，而不牺牲整机算力。

数值对比（$P = 1000$ TFLOPS，FP16 $b=2$，方形最优）：

Case A（同速）3 TB/s3 TB/s$\gtrsim 333$$\gtrsim 333$分两层无意义

L2 越快，内层 tile 可以越小，2 级 blocking 才有收益。这是为什么 CPU 上多级 cache 能帮到 GEMM（L1 可能 30–50× DRAM）、而移动端或弱 cache 层级的架构常常只做单层就够了。

5. $K_t$ 通过 cache 预算间接影响 $M_o \cdot N_o$

外层 tile 的 working set 占 L2：

$$\underbrace{M_o K_t}_{\text{A slab}} + \underbrace{K_t N_o}_{\text{B slab}} + \underbrace{M_o N_o}_{\text{C accum}} \;\le\; M_{L2}$$

固定 $M_{L2}$，$K_t$ 越大，留给 $M_o N_o$ 的预算越小。在 $M_o = N_o$ 的最优下：

$$M_o \;\approx\; \sqrt{M_{L2}} - K_t, \qquad \text{AI}_{\text{DRAM}} \;\approx\; \frac{\sqrt{M_{L2}} - K_t}{b}$$

所以 $K_t$ 小 → 外层 tile 可以做大 → 跨 tile 复用更多 → DRAM AI 更高。

六、调度算法与 tile 选型（工程实务）

前面讨论的都是 "tile 的形状如何影响 AI"；现在讨论"怎么把 CTA 铺到这些 tile 上"，以及 $K_t$ 和 L2 在工程里实际怎么处理。

1. 调度：把 CTA 铺到工作上

核心矛盾：当 $M \cdot N$ 小到 tile 数 $\lt$ SM 数时，默认的 Data-Parallel 调度会让大量 SM 闲置。下表按递进思路排列：

3. L2 cache 在实战里是怎么考虑的

真实 GEMM 库不把 L2 当黑盒；三个层面的优化让 L2 中的 A/B 面板最大限度被跨 CTA 复用。

(a) Threadblock Swizzling——L2 的首要优化手段

默认 row-major 的 CTA 编号会让"调度上相邻的 CTA"跨得很远（典型 $\Delta_i{=}0, \Delta_j{=}1$，两个 CTA 共享一条 A 行但要求完全不同的 B 列）。改成以 $g \times g$ 小方块为调度单位（group raster）：$g^2$ 个 CTA 共享同一条 A 的 $g$ 行 + 同一条 B 的 $g$ 列，L2 中的 A/B 块能被 $g$ 次复用。

CUTLASS 的 ThreadblockSwizzle 选项：

  • IdentitySwizzle：row-major，无 L2 优化
  • GemmIdentityThreadblockSwizzle<log_tile>：group raster，$g = 2^{\text{log\_tile}}$，典型 $g{=}4$ 或 $8$
  • Hilbert / Morton swizzle：更精细的空间局部性，实测比 group raster 只高一两个百分点，很少用

实测：$g = 8$ group raster 能把 L2 对 A/B 的命中率从 ~50% 拉到 ~90%，对应 DRAM 带宽压力减半。

(b) 硬件级 L2 控制

  • Ampere+ persisting access：cudaAccessPolicyWindow 把某段地址标为 persisting，提高 L2 留存优先级（典型用于 weights）
  • Hopper TMA：Tensor Memory Accelerator 从 L2 直接搬 2D/3D tile 到 SMEM，路径绕开 L1，释放 L1 带宽给其他 kernel 用
  • Hopper Cluster + DSMEM：$n$ 个 CTA 组成 cluster 通过 DSMEM 共享同一份 A 或 B 面板 —— 把"L2 跨 CTA 复用"升级成"cluster 内 SMEM 多播"，少一级 cache

(c) 选 $M_o, N_o$（swizzle group 大小）时拟合 L2

经验法则：外层 tile 的 A + B 面板占 L2 约半，剩下给 C accumulator、驱逐缓冲和其他 kernel 碎数据：

$$K \cdot (M_o + N_o) \cdot b \;\lesssim\; 0.5 \cdot M_{L2}$$

H100 的 L2 $\approx 50$ MB，FP16 $b=2$。$K = 4096$ 时 $(M_o + N_o) \lesssim 3000$，取 $M_o = N_o = 1500$，对应约 $12 \times 12$ 个 $128\times 128$ CTA tile 的 swizzle group。Stream-K 会自动调到接近这个尺寸。

4. 一条汇总经验

  • $K_t$：硬件指令宽度 × 几个 pipeline stage，受 SMEM 限制。实战靠穷举选最佳 = atom × {2, 3, 4, 6, 8} 里找。
  • $M_t, N_t$：方形优先；受寄存器文件限制（$M_t N_t$ 是累加器大小）；Hopper 上 $128 \times 128$ 或 $128 \times 256$。
  • $M_o, N_o$（outer / swizzle group）：拟合 L2 的一半容量；优先靠 group raster 或 Stream-K 自动逼近。
  • 调度：默认 DP；tile 数 $\lt$ SM 数时切 Stream-K；小 batched 走 Persistent；Hopper+ 叠 Cluster。

七、Back-to-back GEMM（B2B）

$D = (A \cdot B) \cdot C$ 这种串联两个 GEMM 的模式（注意力的 $Q K^\top V$ 是典型例子，先忽略 softmax）有一个独立于前面所有讨论的新优化维度：中间矩阵 $P = A \cdot B$ 不必落回 DRAM。设 $A \in \mathbb{R}^{M \times K}$、$B \in \mathbb{R}^{K \times N}$、$C \in \mathbb{R}^{N \times L}$，则 $P \in \mathbb{R}^{M \times N}$、$D \in \mathbb{R}^{M \times L}$，两个 GEMM 的总 FLOPs：

$$\text{FLOPs}_\text{total} \;=\; 2MNK + 2MNL \;=\; 2MN(K+L)$$

1. 理论下界（完美融合）

沿用 §一 的口径：所有输入只从主存读一次，输出只写一次，中间 $P$ 全程留在 on-chip。

  • 读 $A$：$MK$ 元素
  • 读 $B$：$KN$ 元素
  • 读 $C$：$NL$ 元素
  • 写 $D$：$ML$ 元素
$$\text{AI}_\text{ideal}^\text{B2B} \;=\; \frac{2MN(K+L)}{(MK + KN + NL + ML) \cdot b}$$

注意 $P$ 完全不进分母。B2B 融合的全部价值都来自消除中间张量的 DRAM 读写。

2. 朴素双 kernel（中间张量物化到 DRAM）

把 GEMM1、GEMM2 当两个独立 kernel 调度：GEMM1 写出 $P$，GEMM2 再读回 $P$。相比融合下界，分母多出 $2MN$：

$$\text{AI}_\text{naive}^\text{B2B} \;=\; \frac{2MN(K+L)}{(MK + KN + \mathbf{2MN} + NL + ML) \cdot b}$$

提升倍数：

$$\frac{\text{AI}_\text{ideal}^\text{B2B}}{\text{AI}_\text{naive}^\text{B2B}} \;=\; 1 + \frac{2MN}{MK + KN + NL + ML}$$

当 $N$ 在 $\{M, K, L\}$ 中占主导（典型如 attention：$M = N = $ seq_len，而 $K = L = $ head_dim 小很多），这个比值会非常大；反之 $N$ 远小于 $\{M, K, L\}$ 时融合近乎无收益。

3. 分块布局：$M$ 联动、$N$ 决定融合方式

B2B 的核心约束是 GEMM1 的 $N$ 维就是 GEMM2 的归约维。这导致：

  • $M$ 联动：同一个 CTA 必须包揽 $D$ 的 $M_t \times L_t$ 块的全部计算 —— 因为 GEMM1 和 GEMM2 用的是同一组 $M_t$ 行的 $A$。
  • $N$ 决定布局类型：GEMM1 产出的 $M_t \times N$ 这块 "行片" 必须留在 on-chip（RF / smem），否则就退化回朴素双 kernel。
  • $L$ 自由：GEMM2 的 $L$ 维与 GEMM1 解耦，可独立选 $L_t$。

布局 A：整行 $P$ 物化（$N$ 小）

适用条件：$N$ 不大（典型如注意力的 head_dim $\le 128$）。一个 CTA 把整行 $M_t \times N$ 的 $P$ 算完留在 RF / smem，再用 GEMM2 沿 $N$ 累加：

# 布局 A：整 N 物化
for i0 in 0..M step Mt:
    for l0 in 0..L step Lt:
        # —— GEMM1：算出 Mt × N 的 P_tile，留在 RF/smem ——
        P_tile = 0                                    # Mt × N 累加器
        for k0 in 0..K step Kt:
            P_tile += A[i0:i0+Mt, k0:k0+Kt] @ B[k0:k0+Kt, :]
        # —— GEMM2：用整行 P_tile 沿 N 归约 ——
        D_tile = 0                                    # Mt × Lt
        for n0 in 0..N step Nt:
            D_tile += P_tile[:, n0:n0+Nt] @ C[n0:n0+Nt, l0:l0+Lt]
        D[i0:i0+Mt, l0:l0+Lt] = D_tile

这是 CUTLASS 的 B2B GEMM 例子默认采用的布局。

布局 B：$N$ 切片流式（$N$ 大，FlashAttention 模式）

当 $N$ 大到塞不下整行 $P$（如 attention 里 $N = $ seq_len $= $ 8K～128K）时，沿 $N$ 切块流式，让 $P$ 永远只是个 $M_t \times N_t$ 小块：

# 布局 B：N 维切片流式
for i0 in 0..M step Mt:
    for l0 in 0..L step Lt:
        D_tile = 0                                    # Mt × Lt，留在 RF
        for n0 in 0..N step Nt:        # 中间维切片
            # —— GEMM1 子块：Mt × Nt，留在 RF ——
            P_sub = A[i0:i0+Mt, :] @ B[:, n0:n0+Nt]
            # —— GEMM2 增量累加 ——
            D_tile += P_sub @ C[n0:n0+Nt, l0:l0+Lt]
        D[i0:i0+Mt, l0:l0+Lt] = D_tile

这个模式下 $P$ 完全不需要整行存在 on-chip，代价是 $A$ 可能被多次加载（每个 $N_t$ 步都要把当前 $M_t$ 行重新读一遍 $B$ 的一列条带），除非 $K$ 很小且 $A$ 行能驻留 smem。FlashAttention 在这个基础上还把 softmax 的归一化也做成在线累加。

布局 C：smem 中转

GEMM1 把 $P$ 写到 smem 而非寄存器，GEMM2 从 smem 重读。寄存器压力小、smem 压力大；Hopper 上还能配合 wgmma 用 smem 操作数。CUTLASS B2B GEMM 实现可切换 RF / smem 两种中转模式。

4. on-chip 容量约束与融合的临界条件

融合的可行性受 on-chip 存储制约（类比 §一 §4 的 cache 容量约束）：

  • 布局 A（整 $N$ 物化）：累加器 $\sim M_t \cdot N \cdot b$ 字节，再加 GEMM2 累加器 $M_t \cdot L_t \cdot b$ 和 $A / B / C$ 的 smem 缓冲。Hopper 上 RF $\le 256$ KB/CTA，典型 $M_t \cdot N$ 上限 $\sim 64$K 元素。所以只有 $N \le 128$ 量级时才走得通。
  • 布局 B（$N$ 切片）：累加器 $\sim M_t \cdot (N_t + L_t) \cdot b$ 字节，$N_t$ 通常取 32–128。
  • 布局 C（smem 中转）：smem 占用 $\sim M_t \cdot N_t \cdot b$ 字节加双缓冲；smem 总量 $\sim 228$ KB/SM 是约束上限。

融合何时真正带来访存比的显著提升（即朴素 AI 远低于融合 AI），等价于：

$$2MN \;\gtrsim\; MK + KN + NL + ML$$

即中间张量比输入加输出还大。下面是几种典型场景的数值对照（FP16，$b = 2$）：

场景$M, N, K, L$$\text{AI}_\text{naive}$$\text{AI}_\text{B2B}$提升
方阵 B2B$4096, 4096, 4096, 4096$$\sim 1365$$\sim 2048$$1.5\times$

几个直觉：

  • Attention 是 B2B 融合的核心受益场景：$M = N = $ seq，$2MN$ 是 seq 的平方，占分母主导，导致 $\text{AI}_\text{naive} \to (K+L)/b$（仅由 head_dim 决定，与 seq 无关），而 $\text{AI}_\text{B2B} \approx N/2$（随 seq 线性增长）。seq 越长，融合的收益越夸张。
  • 方阵 B2B 收益约 $1.5\times$：$2MN \approx 2 \cdot M^2$，分母 $4 M^2$，提升固定在 $1.5\times$ 上下。普通方阵 B2B 不是首要优化目标。
  • "瘦中间" 情形几乎没有收益：$N \ll M, K, L$ 时 $2MN$ 项对分母贡献极小。这种 shape 不需要专门走 B2B 优化。

TL;DR

  • 方阵 tile：$\text{AI} = T / b$，随 $T$ 线性增长，受 cache 容量的平方根限制。
  • 任意形状：$\text{AI} = H(M_t, N_t) / b$，由较小的那一维主导。
  • $K_t$ 不进入单层 blocking 的渐近 AI；2 级 blocking 下通过 cache 预算间接影响 $M_o \cdot N_o$。
  • 两级 Roofline：$BW_{L2}$ 越高于 $BW_\text{DRAM}$，内层 $M_t$ 可以越小于外层 $M_o$（$10\times$ 带宽差 $\Rightarrow$ $M_t \approx M_o/10$）。
  • cache 预算内，越接近正方形 AI 越高。
  • $MN$ 小到 tile 数 $\lt$ SM 数时必须切 $K$ 维：Stream-K > Split-K，辅以 Persistent / Cluster 调度。
  • 工程选型：$K_t$ = MMA atom × 3–6 stages；$M_t \times N_t$ 方形优先受寄存器约束；$M_o \times N_o$（swizzle group）拟合 $\sim 0.5 \cdot M_{L2}$。
  • 任何一侧被问题固有地压小（GEMV、batched MoE、LLM decode），都会从根本上变 memory-bound——这是算法层面（拼 $M$、FlashDecoding）而非 kernel 层面的问题。
  • B2B GEMM：$D = (A B) C$ 融合可消除中间张量 $2MN$ 的 DRAM 读写，提升 $= 1 + 2MN / (MK+KN+NL+ML)$。Attention 是最大受益场景（$\text{AI}_\text{naive}$ 卡在 $(K+L)/b$，融合后 $\text{AI} \approx N / 2$ 随 seq 线性增长）；方阵 B2B 收益固定 $1.5\times$；瘦中间近乎无收益。

GEMM Arithmetic Intensity Analysis

Square-matrix baseline → zero-reuse lower bound → blocking → arbitrary tile shapes → full derivation

Element byte size is denoted $b$ (FP32: $b=4$; FP16/BF16: $b=2$; FP8: $b=1$).

Modeling premise. Sections 1 and 2 below adopt a single-level blocking model: the A / B inputs of each output tile are assumed to be loaded from DRAM independently, with no reuse via cache between adjacent C tiles. This is why $K_t$ "cancels" in the asymptotic AI. Section 3 lifts this assumption and revisits the analysis with 2-level blocking (DRAM ↔ L2 ↔ SMEM); $K_t$ then re-enters indirectly through the cache budget.

1. Square-matrix baseline ($N \times N$)

$C = A \cdot B$, all three matrices $N \times N$. Total FLOPs $= 2N^3$ (each $C$ element costs $N$ multiply-adds).

1. Theoretical lower bound (perfect reuse)

Each input / output element enters and leaves main memory exactly once, totalling $3N^2$ elements:

$$\text{AI}_{\text{ideal}} \;=\; \frac{2N^3}{3N^2 \cdot b} \;=\; \frac{2N}{3b}\ \text{FLOP/byte}$$

Linear in $N$. For FP32 at $N = 4096$, this is $\approx 683$ FLOP/byte, far above the balance points of H100 ($\approx 7$) and B200 ($\approx 20$), so a large GEMM is theoretically compute-bound.

2. Zero reuse (naive triple loop)

$C$ is accumulated in register acc and written back once after the K loop. For the whole GEMM:

  • FLOPs: $2 N^3$
  • Reads of $A$: $N^3$ (each $A[i,k]$ is reloaded $N$ times across $j$)
  • Reads of $B$: $N^3$ (each $B[k,j]$ is reloaded $N$ times across $i$)
  • Writes of $C$: $N^2$ (one write-back per $(i,j)$ after accumulation)
$$\text{AI}_{\text{no-reuse}} \;=\; \frac{2 N^3}{(2 N^3 + N^2)\cdot b} \;\xrightarrow{N\to\infty}\; \frac{1}{b}\ \text{FLOP/byte}$$

The $N^2$ from writing $C$ is dwarfed by $2N^3$, so the asymptote is independent of $N$. At FP32 this is just $0.25$ FLOP/byte, thoroughly memory-bound; H100 can deliver less than 4% of peak.

Naive triple-loop pseudocode:

# Naive GEMM: C = A · B, all three N×N
for i in 0..N:
    for j in 0..N:
        acc = 0
        for k in 0..N:
            acc += A[i, k] * B[k, j]      # 2 loads + 2 FLOPs per iteration
        C[i, j] = acc                       # C stays in a register throughout

Every inner-loop FMA pulls A[i,k] and B[k,j] from DRAM with no cross-iteration reuse, hence $\text{AI} = 1/b$.

3. Blocking (block size $T \times T$, zero-latency cache)

A $T \times T$ $C$ block sweeps along $k$ for $N/T$ iterations:

  • FLOPs: $(N/T) \cdot 2T^3 = 2NT^2$
  • Main-memory traffic: $A$ tiles $(N/T)\cdot T^2 = NT$, $B$ tiles $NT$, $C$ write-back $T^2$ (a single write at the end of the K loop)
$$\text{AI}_{\text{blocked}} \;=\; \frac{2NT^2}{(2NT + T^2)\cdot b} \;\xrightarrow{N\to\infty}\; \frac{T}{b}\ \text{FLOP/byte}$$

When $N \gg T/2$, the $C$ write is negligible and total memory traffic over the whole GEMM is $\approx 2N^3/T$: the concrete form of the Hong-Kung lower bound $\Omega(N^3/\sqrt{M})$.

Blocked-GEMM pseudocode (square tile, $T \times T$):

# Blocked GEMM, assuming 3·T² elements fit in cache simultaneously
for i0 in 0..N step T:              # iterate over row-blocks of C
    for j0 in 0..N step T:          # iterate over col-blocks of C
        C_tile = 0                       # T×T accumulator (registers / shared mem)
        for k0 in 0..N step T:      # sweep along K
            # --- load both input tiles into fast storage once ---
            A_tile = A[i0:i0+T, k0:k0+T]
            B_tile = B[k0:k0+T, j0:j0+T]
            # --- dense small GEMM inside the tile (cache-resident, zero DRAM) ---
            for i in 0..T:
                for j in 0..T:
                    for k in 0..T:
                        C_tile[i, j] += A_tile[i, k] * B_tile[k, j]
        C[i0:i0+T, j0:j0+T] = C_tile     # single write-back

The key difference: each pair of A_tile, B_tile is loaded from DRAM once and reused $T$ times inside the tile, amplifying AI from $1/b$ to $T/b$.

4. Cache-capacity constraint

Three tiles must coexist in cache: $3T^2 b \le M$, so

$$T_{\max} \;=\; \sqrt{\frac{M}{3b}}, \qquad \text{AI}_{\max} \;=\; \sqrt{\frac{M}{3b^{3}}}$$

Doubling AI requires quadrupling the cache.

2. Arbitrary tile shape generalized

General GEMM: $C\,(M{\times}N)\mathrel{+}= A\,(M{\times}K)\cdot B\,(K{\times}N)$, with tile sizes $M_t \times N_t$ ($C$ block), $M_t \times K_t$ ($A$ block), $K_t \times N_t$ ($B$ block).

1. Core formula

Computing one $M_t \times N_t$ $C$ tile:

QuantityElements
FLOPs$2\, M_t N_t K$
Read $A$$(K/K_t)\cdot M_t K_t = M_t K$
Read $B$$(K/K_t)\cdot K_t N_t = K N_t$
Write $C$$M_t N_t$ (one write-back after the K loop finishes)
$$\text{AI} \;=\; \frac{2 M_t N_t K}{\big((M_t + N_t)\,K + M_t N_t\big)\cdot b}$$

The $(M_t + N_t) K$ term in the denominator comes from repeatedly loading $A$, $B$ along K (linear in $K$); $M_t N_t$ is the single C-tile write-back after the accumulation finishes (independent of $K$). When $K \gg \tfrac{M_t N_t}{M_t + N_t}$, the $C$ write becomes negligible and the asymptote is:

$$\boxed{\;\text{AI} \;\xrightarrow{K \to \infty}\; \frac{2 M_t N_t K}{(M_t + N_t)\, K \cdot b} \;=\; \frac{2}{b} \cdot \frac{M_t N_t}{M_t + N_t} \;=\; \frac{H(M_t, N_t)}{b}\;}$$

where $H(M_t, N_t) \;=\; \dfrac{2 M_t N_t}{M_t + N_t}$ is the harmonic mean. The rest of the analysis is based on this asymptotic form; but Split-K shortens the $K$ of each C tile and inflates the write-back by a factor of $S$ (see Section 4), so there we must return to the exact formula.

2. Takeaway 1: AI is dominated by the shorter side

Property of the harmonic mean: pinned down by the smaller term. Intuition: each $A$ element is reused $N_t$ times, each $B$ element is reused $M_t$ times.

$M_t \times N_t$$H(M_t, N_t)$Comment
$128 \times 128$128square baseline
$256 \times 64$$102$same element count, AI drops to 80%
$512 \times 32$$60$same element count, AI down to 47%
$1024 \times 16$$31.5$close to $2 N_t$
$M_t \to \infty,\; N_t$ fixed$2 N_t$limit equals twice the short side
$1 \times N_t$ (GEMV)$\approx 2$constant, compute throughput cannot be realized
This is the root cause for why GEMV, skinny GEMM, and the decode-phase $Q\cdot K^\top$ in LLM inference (when one side is small) become memory-bound.

3. Takeaway 2: $K_t$ cancels in the asymptotic AI (single-level blocking only)

  • $K_t$ is merely a slice of the $K$ loop; it does not change the reuse factors of $A$ and $B$ (still $N_t$ and $M_t$).
  • What $K_t$ actually controls: working set $= M_t K_t + K_t N_t + M_t N_t$, pipeline depth, accumulator-initialization amortization.
  • Engineering choice: large enough to feed Tensor Core instruction width (HMMA $k{=}16$, WGMMA $k{=}16/32$) plus a bit of pipelining. No need to chase a large value.
  • Caveat: once the "no reuse between tiles" assumption is lifted, $K_t$ does affect AI indirectly through the cache budget (see Section 3).

4. Optimal shape under a cache budget

Fix $K_t$, subject to $M_t N_t + K_t (M_t + N_t) \le M/b$, maximize $H(M_t, N_t)$:

$$\arg\max_{M_t,\,N_t} H(M_t, N_t)\ \Longrightarrow\ M_t = N_t,\quad M_t \approx \sqrt{M/b} - K_t$$

Within the capacity budget, the closer to square, the higher the AI. This is also the fundamental reason the square-matrix derivation already captures the problem.

5. Two degenerate extremes: row-strip / column-strip tiles

At the two boundaries where $M_t$ or $N_t$ collapse to 1, we get $H(1, N) \to 2$ and $H(M, 1) \to 2$, and AI bottoms out at $2/b$. This is the essence of the "naive" $q \approx 2$ in the Berkeley slides: those slides implicitly set the tile to a row strip or a column strip, which already adds one dimension of reuse beyond $1{\times}1$.

$M_t = 1,\ N_t = N$ (row strip): row $i$ of $A$ is loaded once and reused $N$ times:

# Row-strip tile: sweep one full row of C, A row reused N times in cache
for i in 0..M:
    # Bring A[i, :] (K elements) into fast storage; the whole j loop uses it
    for j in 0..N:                     # sweep one row of C
        acc = 0                             # C[i, j] lives in a register
        for k in 0..K:
            acc += A[i, k] * B[k, j]        # A[i,k] from cache, B[k,j] from DRAM
        C[i, j] = acc

Memory traffic: $A$ reads $M K$ (once per row), $B$ reads $M N K$ (no reuse across $i$), FLOPs $2 M N K$:

$$\text{AI}_{\text{row-strip}} \;=\; \frac{2 M N K}{(M K + M N K)\, b} \;\xrightarrow{N \to \infty}\; \frac{2}{b}$$

$M_t = M,\ N_t = 1$ (column strip): the symmetric case; column $j$ of $B$ is reused $M$ times:

# Column-strip tile: sweep one full column of C, B column reused M times in cache
for j in 0..N:
    # Bring B[:, j] (K elements) into fast storage; the whole i loop uses it
    for i in 0..M:                     # sweep one column of C
        acc = 0
        for k in 0..K:
            acc += A[i, k] * B[k, j]        # B[k,j] from cache, A[i,k] from DRAM
        C[i, j] = acc

Memory traffic is symmetric: $B$ reads $N K$, $A$ reads $M N K$, FLOPs $2 M N K$:

$$\text{AI}_{\text{col-strip}} \;=\; \frac{2 M N K}{(N K + M N K)\, b} \;\xrightarrow{M \to \infty}\; \frac{2}{b}$$

To push AI above $2/b$, we must reuse $A$ and $B$ simultaneously, i.e. a square tile with both $M_t, N_t > 1$, which lands us back at the blocked GEMM of Section 1.3 with AI amplified up to $T/b$.

3. Two-level blocking and cross-tile reuse

Sections 1 and 2 only considered the most conservative case: "each tile fetches its data from DRAM independently". Real hardware has a multi-level cache hierarchy (HBM ↔ L2 ↔ SMEM ↔ registers), and adjacent C tiles automatically reuse data across levels. Once the "no reuse between tiles" assumption is lifted, $K_t$ returns to the AI formula indirectly through the cache budget.

1. Total cross-tile reusable data: independent of $K_t$

When processing $C[0, j_1]$ and $C[0, j_2]$, both need the full 0th row-block of $A$ over all $K$ columns, i.e. $M_t \times K$ elements. Whether this slab is split into $K/K_t$ pieces of $M_t \times K_t$ or loaded as one long stripe, the shared amount is always $M_t K$. $K_t$ only determines blocking granularity, not the reuse potential. Symmetric on the B side: $C[i_1, 0]$ and $C[i_2, 0]$ share $N_t K$ elements.

2. Two-level tile structure

Leveltilecapacitybandwidthwhat it determines
Outer (DRAM ↔ L2)$M_o \times N_o \times K$$M_{L2}$ (tens of MB)$BW_{\text{DRAM}}$DRAM traffic, cross-CTA reuse
Inner (L2 ↔ SMEM / registers)$M_t \times N_t \times K_t$SMEM (~200 KB) + registers$BW_{L2}$intra-CTA reuse, L2 traffic

Each level has its own $H$-shaped AI:

$$\text{AI}_{\text{outer}} = \frac{H(M_o, N_o)}{b}, \qquad \text{AI}_{\text{inner}} = \frac{H(M_t, N_t)}{b}$$

3. Multi-level Roofline: effective throughput is set by the slowest level

Peak compute $P$ must be fed by both cache levels' bandwidths simultaneously:

$$\text{Throughput}_{\text{eff}} = \min\!\left(P,\; BW_{\text{DRAM}} \cdot \text{AI}_{\text{outer}},\; BW_{L2} \cdot \text{AI}_{\text{inner}}\right)$$

To keep compute from being throttled by either level:

$$\text{AI}_{\text{outer}} \;\ge\; \frac{P}{BW_{\text{DRAM}}}, \qquad \text{AI}_{\text{inner}} \;\ge\; \frac{P}{BW_{L2}}$$

The two inequalities each set a lower bound on the tile size at that level.

4. Tile sizing under two bandwidth regimes

Case A: $BW_{L2} \approx BW_{\text{DRAM}}$ (L2 is essentially useless, or inner/outer bandwidths are similar)

Both levels demand the same AI: $H(M_t, N_t) \approx H(M_o, N_o)$, i.e. $M_t$ must be of the same order as $M_o$. The inner tile already has to be as large as the outer one, so 2-level blocking degenerates into single-level. Splitting into two levels yields no bandwidth benefit.

Case B: $BW_{L2} = 10 \times BW_{\text{DRAM}}$ (typical GPU: Hopper L2 vs. HBM ratio is about $5\times\!\sim\!20\times$)

The inner level only needs:

$$\text{AI}_{\text{inner}} \;\ge\; \frac{P}{BW_{L2}} \;=\; \frac{1}{10}\cdot\frac{P}{BW_{\text{DRAM}}}$$

i.e. $H(M_t, N_t) \gtrsim \tfrac{1}{10} H(M_o, N_o)$. Under the square approximation, $M_t \approx M_o / 10$ suffices.

This is the rationale behind the GPU configuration "CTA tile ($128\times 128$) + cluster / pipeline-stage panel": once the inner/outer bandwidth gap opens up, the inner tile can shrink, trading for higher occupancy and more flexible scheduling without giving up peak compute.

Numerical comparison ($P = 1000$ TFLOPS, FP16 $b=2$, square optimum):

5. $K_t$ indirectly affects $M_o \cdot N_o$ via the cache budget

The outer-tile working set in L2:

$$\underbrace{M_o K_t}_{\text{A slab}} + \underbrace{K_t N_o}_{\text{B slab}} + \underbrace{M_o N_o}_{\text{C accum}} \;\le\; M_{L2}$$

With $M_{L2}$ fixed, the larger $K_t$ is, the less budget remains for $M_o N_o$. Under the $M_o = N_o$ optimum:

$$M_o \;\approx\; \sqrt{M_{L2}} - K_t, \qquad \text{AI}_{\text{DRAM}} \;\approx\; \frac{\sqrt{M_{L2}} - K_t}{b}$$

So smaller $K_t$ → larger outer tile → more cross-tile reuse → higher DRAM AI.

6. Scheduling algorithms and tile sizing (engineering practice)

So far we've discussed "how the tile shape affects AI"; now we turn to "how to lay CTAs onto these tiles", and how $K_t$ and L2 are actually handled in practice.

1. Scheduling: spreading CTAs over the work

3. How L2 cache is treated in practice

Real GEMM libraries do not treat L2 as a black box; three layers of optimization maximize cross-CTA reuse of the A/B panels in L2.

(a) Threadblock swizzling: the primary L2 optimization

Default row-major CTA numbering makes "scheduling-adjacent" CTAs land far apart in the work grid (typical $\Delta_i{=}0, \Delta_j{=}1$: two CTAs share one A row but demand completely different B columns). Switch the scheduling unit to a $g \times g$ small block (group raster): $g^2$ CTAs share the same $g$ rows of A and $g$ columns of B, so A/B blocks in L2 are reused $g$ times.

CUTLASS's ThreadblockSwizzle options:

  • IdentitySwizzle: row-major, no L2 optimization
  • GemmIdentityThreadblockSwizzle<log_tile>: group raster, $g = 2^{\text{log\_tile}}$, typically $g{=}4$ or $8$
  • Hilbert / Morton swizzle: finer spatial locality; in measurements typically only 1 to 2 percentage points better than group raster, rarely used

Measurement: $g = 8$ group raster pushes the L2 hit rate on A/B from ~50% to ~90%, halving DRAM bandwidth pressure.

(b) Hardware-level L2 controls

  • Ampere+ persisting access: cudaAccessPolicyWindow marks an address range as persisting, raising its L2 retention priority (typical use: weights)
  • Hopper TMA: the Tensor Memory Accelerator moves 2D/3D tiles directly from L2 into SMEM, bypassing L1 and freeing L1 bandwidth for other kernels
  • Hopper Cluster + DSMEM: $n$ CTAs form a cluster sharing a single A or B panel through DSMEM, promoting "L2 cross-CTA reuse" to "intra-cluster SMEM multicast", one cache level less

(c) Sizing $M_o, N_o$ (swizzle-group size) to fit L2

Rule of thumb: outer-tile A + B panels take roughly half of L2, leaving the rest for the C accumulator, eviction buffering, and stray data from other kernels:

$$K \cdot (M_o + N_o) \cdot b \;\lesssim\; 0.5 \cdot M_{L2}$$

H100's L2 is $\approx 50$ MB, FP16 has $b=2$. At $K = 4096$ we get $(M_o + N_o) \lesssim 3000$, so $M_o = N_o = 1500$, corresponding to a swizzle group of roughly $12 \times 12$ tiles of $128\times 128$. Stream-K converges to this size automatically.

4. One-line summary

  • $K_t$: hardware instruction width × a few pipeline stages, capped by SMEM. In practice, sweep atom × {2, 3, 4, 6, 8} and pick the best.
  • $M_t, N_t$: prefer square; capped by register file ($M_t N_t$ is the accumulator size); on Hopper, $128 \times 128$ or $128 \times 256$.
  • $M_o, N_o$ (outer / swizzle group): fit roughly half of L2 capacity; prefer letting group raster or Stream-K converge to it automatically.
  • Scheduling: default DP; switch to Stream-K when tile count $\lt$ SM count; small batched workloads go Persistent; on Hopper+ layer on Cluster.

7. Back-to-back GEMM (B2B)

The chained pattern $D = (A \cdot B) \cdot C$ (attention's $Q K^\top V$ is the canonical example, ignoring softmax for now) opens a new optimization axis on top of everything discussed so far: the intermediate matrix $P = A \cdot B$ does not have to land back in DRAM. Let $A \in \mathbb{R}^{M \times K}$, $B \in \mathbb{R}^{K \times N}$, $C \in \mathbb{R}^{N \times L}$, so $P \in \mathbb{R}^{M \times N}$ and $D \in \mathbb{R}^{M \times L}$. The total FLOPs of both GEMMs:

$$\text{FLOPs}_\text{total} \;=\; 2MNK + 2MNL \;=\; 2MN(K+L)$$

1. Theoretical lower bound (perfect fusion)

Following the discipline of §1: every input is read from main memory once, every output is written once, and $P$ stays on-chip throughout.

  • Read $A$: $MK$ elements
  • Read $B$: $KN$ elements
  • Read $C$: $NL$ elements
  • Write $D$: $ML$ elements
$$\text{AI}_\text{ideal}^\text{B2B} \;=\; \frac{2MN(K+L)}{(MK + KN + NL + ML) \cdot b}$$

Notice that $P$ does not appear in the denominator at all. The entire payoff of B2B fusion comes from eliminating the DRAM read and write of the intermediate tensor.

2. Naive two-kernel (intermediate materialized to DRAM)

If GEMM1 and GEMM2 are dispatched as two independent kernels, GEMM1 writes $P$ out and GEMM2 reads it back. The denominator gains an extra $2MN$:

$$\text{AI}_\text{naive}^\text{B2B} \;=\; \frac{2MN(K+L)}{(MK + KN + \mathbf{2MN} + NL + ML) \cdot b}$$

Improvement factor:

$$\frac{\text{AI}_\text{ideal}^\text{B2B}}{\text{AI}_\text{naive}^\text{B2B}} \;=\; 1 + \frac{2MN}{MK + KN + NL + ML}$$

When $N$ dominates $\{M, K, L\}$ (the attention case: $M = N = $ seq_len while $K = L = $ head_dim are much smaller) the ratio gets very large. Conversely, if $N$ is much smaller than $\{M, K, L\}$ fusion brings almost nothing.

3. Tiling layout: $M$ is coupled, $N$ picks the fusion mode

The core constraint of B2B is that GEMM1's $N$ dimension is GEMM2's reduction dimension. Consequences:

  • $M$ is coupled: a single CTA must own the entire computation of the $M_t \times L_t$ output tile of $D$, because GEMM1 and GEMM2 both consume the same group of $M_t$ rows of $A$.
  • $N$ determines the layout flavor: the $M_t \times N$ "row slice" produced by GEMM1 must stay on-chip (RF or smem), otherwise the kernel degrades into the naive two-kernel case.
  • $L$ is free: GEMM2's $L$ dimension is decoupled from GEMM1; $L_t$ can be picked independently.

Layout A: materialize the entire $P$ row ($N$ small)

Applicable when $N$ is small (typical: head_dim $\le 128$ in attention). A CTA computes the entire $M_t \times N$ tile of $P$, keeps it in RF or smem, then runs GEMM2 accumulating along $N$:

# Layout A: materialize the whole N row
for i0 in 0..M step Mt:
    for l0 in 0..L step Lt:
        # —— GEMM1: produce Mt × N P_tile, keep in RF/smem ——
        P_tile = 0                                    # Mt × N accumulator
        for k0 in 0..K step Kt:
            P_tile += A[i0:i0+Mt, k0:k0+Kt] @ B[k0:k0+Kt, :]
        # —— GEMM2: reduce along N using the whole P_tile ——
        D_tile = 0                                    # Mt × Lt
        for n0 in 0..N step Nt:
            D_tile += P_tile[:, n0:n0+Nt] @ C[n0:n0+Nt, l0:l0+Lt]
        D[i0:i0+Mt, l0:l0+Lt] = D_tile

This is the default layout in the CUTLASS B2B GEMM example.

Layout B: stream over $N$ ($N$ large, the FlashAttention mode)

When $N$ is too large to fit a whole row of $P$ on-chip (e.g. attention with $N = $ seq_len $= $ 8K to 128K), tile along $N$ and keep $P$ as a small $M_t \times N_t$ block forever:

# Layout B: stream over the N dimension
for i0 in 0..M step Mt:
    for l0 in 0..L step Lt:
        D_tile = 0                                    # Mt × Lt, stays in RF
        for n0 in 0..N step Nt:        # slice the inner dim
            # —— GEMM1 sub-block: Mt × Nt, stays in RF ——
            P_sub = A[i0:i0+Mt, :] @ B[:, n0:n0+Nt]
            # —— GEMM2 accumulates incrementally ——
            D_tile += P_sub @ C[n0:n0+Nt, l0:l0+Lt]
        D[i0:i0+Mt, l0:l0+Lt] = D_tile

In this mode $P$ never needs a full row on-chip. The cost is that $A$ may be loaded multiple times (each $N_t$ step re-reads the current $M_t$ rows against a column stripe of $B$), unless $K$ is small enough for the $A$ rows to live in smem. FlashAttention adds an online softmax normalization on top.

Layout C: smem as the intermediate

GEMM1 writes $P$ into smem rather than RF; GEMM2 re-reads from smem. Register pressure drops at the cost of smem pressure. On Hopper this also pairs with wgmma's smem operands. The CUTLASS B2B GEMM kernel exposes both RF and smem variants.

4. On-chip capacity and when fusion actually pays

Feasibility is bounded by on-chip storage (the analog of §1.4's cache-capacity constraint):

  • Layout A (whole-$N$ materialization): accumulator $\sim M_t \cdot N \cdot b$ bytes, plus the GEMM2 accumulator $M_t \cdot L_t \cdot b$ and smem buffers for $A / B / C$. On Hopper the RF cap is 256 KB/CTA, so $M_t \cdot N$ tops out around 64K elements, which means $N \le \sim 128$ in practice.
  • Layout B ($N$ streaming): accumulator $\sim M_t \cdot (N_t + L_t) \cdot b$ bytes, $N_t$ typically 32 to 128.
  • Layout C (smem intermediate): smem usage $\sim M_t \cdot N_t \cdot b$ bytes plus double buffering, capped by the $\sim 228$ KB/SM smem budget.

Fusion actually delivers a meaningful AI improvement (i.e. naive AI is much lower than fused AI) when:

$$2MN \;\gtrsim\; MK + KN + NL + ML$$

That is, when the intermediate tensor is larger than inputs plus outputs combined. Concrete numbers under FP16 ($b = 2$):

Scenario$M, N, K, L$$\text{AI}_\text{naive}$$\text{AI}_\text{B2B}$Speedup
Square B2B$4096, 4096, 4096, 4096$$\sim 1365$$\sim 2048$$1.5\times$
Attention (short)seq $= 4096$, $d = 128$$\sim 120$$\sim 2048$$\mathbf{17\times}$

Intuitions:

  • Attention is the headline beneficiary of B2B fusion: with $M = N = $ seq, the $2MN$ term is quadratic in seq and dominates the denominator, so $\text{AI}_\text{naive} \to (K+L) / b$ (determined only by head_dim, independent of seq), while $\text{AI}_\text{B2B} \approx N / 2$ (linear in seq). The longer the sequence, the more dramatic the fusion payoff.
  • Square B2B caps out at $\sim 1.5\times$: with $2MN \approx 2 M^2$ and the denominator at $4 M^2$, the ratio sits at $1.5\times$ regardless of $M$. Plain square B2B is not the primary optimization target.
  • "Skinny middle" sees no benefit: when $N \ll M, K, L$, the $2MN$ term contributes almost nothing to the denominator. Shapes like this do not need a dedicated B2B kernel.

TL;DR

  • Square tile: $\text{AI} = T / b$, linear in $T$, capped by the square root of the cache capacity.
  • Arbitrary shape: $\text{AI} = H(M_t, N_t) / b$, dominated by the shorter side.
  • $K_t$ does not enter the asymptotic AI of single-level blocking; under 2-level blocking it affects $M_o \cdot N_o$ indirectly through the cache budget.
  • Two-level Roofline: the higher $BW_{L2}$ is above $BW_\text{DRAM}$, the smaller the inner $M_t$ can be relative to the outer $M_o$ ($10\times$ bandwidth gap $\Rightarrow$ $M_t \approx M_o/10$).
  • Within the cache budget, the closer to square, the higher the AI.
  • When $MN$ is small enough that tile count $\lt$ SM count, $K$ must be split: Stream-K > Split-K, complemented by Persistent / Cluster scheduling.
  • Engineering picks: $K_t$ = MMA atom × 3 to 6 stages; $M_t \times N_t$ prefers square subject to register file; $M_o \times N_o$ (swizzle group) fits $\sim 0.5 \cdot M_{L2}$.
  • If either side is intrinsically small (GEMV, batched MoE, LLM decode), the workload becomes fundamentally memory-bound. That is an algorithm-level problem (stitch $M$, FlashDecoding), not a kernel-level one.
  • B2B GEMM: fusing $D = (A B) C$ removes the $2MN$ DRAM round-trip of the intermediate tensor; speedup $= 1 + 2MN / (MK+KN+NL+ML)$. Attention is the marquee beneficiary ($\text{AI}_\text{naive}$ caps at $(K+L)/b$ regardless of seq, fused $\text{AI} \approx N / 2$ grows linearly with seq). Square B2B is fixed at $1.5\times$. Skinny middle gives essentially no benefit.
