FP8 scaling 与训练 recipe

FP8 训练不是把模型参数改成 torch.float8 后开跑。它是一组系统约定:哪些 GEMM operand 用 FP8,scale 怎么算,amax 是否跨 rank 同步,哪些层和状态保留在 BF16/FP32。recipe 讨论的就是这些约定。

0. 范围

这篇只讨论 LLM 训练里的 FP8 mixed precision,重点是 Transformer block 中的线性层、MLP 和部分 attention 计算。optimizer state、master weights、loss、softmax、normalization、reduction 这类位置通常继续用更高精度。这里把 FP8 当作训练系统里的局部加速路径,不是整张计算图的默认数值类型。

先把两个层面拆开:

层面问题例子
format一个 FP8 值本身怎么编码E4M3, E5M2
recipe训练系统怎么使用这些值DelayedScaling, current scaling, MXFP8 block scaling, HYBRID format

把这两层混在一起,很多描述会变形。E4M3 只是一个 dtype。HYBRID 不是第三种 FP8 编码,而是训练时把 forward 和 backward 分别映射到不同格式的一种约定。MXFP8 也不是简单的 "更小 dtype",它把 scale 粒度推到 block 级别,还改变了 scale 的表示方式。

1. FP8 数值格式

FP8 仍然是浮点数:1 bit 符号位,再把剩下的 bit 分给 exponent 和 mantissa。E4M3 给 exponent 4 bit,给 mantissa 3 bit。E5M2 给 exponent 5 bit,给 mantissa 2 bit。

格式取舍常见位置备注
E4M3范围较窄,有效精度较高weights, activations, forward GEMMNVIDIA 文档给出的最大有限值约为 448
E5M2范围更宽,有效精度较低activation gradients, backward GEMMNVIDIA 文档给出的最大有限值约为 57344,支持 inf
HYBRIDforward 用 E4M3,backward 用 E5M2Transformer Engine 默认 FP8 训练格式之一这是 format 选择策略,不是新的 bit layout

这样分配 bit 的理由很朴素。forward 里的权重和激活通常更需要 mantissa 精度,梯度更容易出现大范围波动,backward 往往更需要 exponent 范围。不是所有模型都必须这样配,但这是许多 FP8 训练栈的默认起点。

2. scale 的位置

FP8 有 exponent,但它仍然需要外部 scaling。真正麻烦的不是单个大数,而是同一个 LLM 里不同 tensor 的分布差异太大。一个 tensor 的 amax 可能在 0.02,另一个可能在 2000。用同一套 FP8 网格直接表示,小值会浪费码点,大值会溢出或饱和。

实际系统一般把高精度 tensor x 映射成 FP8 raw data q,同时保存一个 scale。常见有两种等价写法。Transformer Engine 文档采用乘法 scale 习惯:

scale = FP8_MAX / amax(x)
q = cast_fp8(x * scale)
x_hat = q / scale

也有人把反方向的量叫 scale:

s = amax(x) / FP8_MAX
q = cast_fp8(x / s)
x_hat = q * s

两种写法只差一个倒数。看代码和论文时,先确认 scale 的方向,否则公式会看起来互相矛盾。更严格地说,q 不是原始数值域里的数,而是一个 FP8 code。真实参与后续计算的近似值由 (q, scale) 或 (q, scale_inv) 共同决定。

# TE 习惯:保存 scale 和 scale_inv = 1 / scale
q_i = cast_fp8(x_i * scale_i)
r_i = decode_fp8(q_i)
x_hat_i = descale(r_i, scale_i) = r_i / scale_i

# 若代码里保存的是 scale_inv
x_hat_i = r_i * scale_inv_i

相同的 FP8 bit pattern 不一定表示相同的原始域数值。它先 decode 成 FP8 网格里的 r_i,再通过 scale 或 scale inverse 回到 tensor 自己的数值域。只有 raw FP8 data 而没有 scale metadata,通常还原不出训练系统想表达的数。

MXFP8 只是把这个公式从 tensor 级 scale 换成 block 级 scale。若每 32 个值一个 block,元素 i 属于 b(i)=floor(i/32),则:

q_i = cast_fp8(x_i * scale_b(i))
r_i = decode_fp8(q_i)
x_hat_i = r_i * scale_inv_b(i)

在 GEMM 里,这个 block 通常沿那次 matmul 的 reduction 维度放。这样每个 dot product 的 32-element chunk 都能用自己的 scale,而不是被整行、整列或整个 tensor 的 outlier 牵着走。

scale 主要处理两件事:

  • 把 tensor 的最大值压到 FP8 可表示范围内,减少 overflow 或 saturation。
  • 让 tensor 中的大多数值落在更有分辨率的区域,少把 mantissa 浪费在空区间上。

BF16 的 exponent 足够宽,通常不需要为每个 tensor 单独维护 scale。FP8 的 bit 数不够,这件事必须显式做。

FP8 recipe 可以按几条轴来看。图里每个分支都是一个决策点,不是互斥产品名。一个实际配置可能同时是 HYBRID format、delayed scaling、per-tensor FP32 scale、跨 TP group reduce amax。

FP8 scaling and training recipes

FP8 training is not "change the model to torch.float8 and run". It is a set of systems decisions: which GEMM operands use FP8, how scales are computed, whether amax values are synchronized across ranks, and which layers or states remain in BF16/FP32. A recipe is the bundle of those choices.

0. Scope

This article focuses on FP8 mixed precision for LLM training, mainly the linear layers, MLPs, and selected attention paths inside Transformer blocks. Optimizer state, master weights, loss computation, softmax, normalization, and reductions usually stay in higher precision. FP8 is a local acceleration path inside the training system, not the default dtype for the whole graph.

Two layers should stay separate:

LayerQuestionExamples
formatHow one FP8 value is encodedE4M3, E5M2
recipeHow the training stack uses those valuesDelayedScaling, current scaling, MXFP8 block scaling, HYBRID format

Mixing these two layers creates confusion. E4M3 is a dtype. HYBRID is not a third FP8 encoding, it is a convention that maps forward and backward tensors to different formats. MXFP8 is not just a smaller dtype either; it changes scale granularity and the representation of the scales.

1. FP8 formats

FP8 is still floating point: 1 sign bit, with the remaining bits split between exponent and mantissa. E4M3 uses 4 exponent bits and 3 mantissa bits. E5M2 uses 5 exponent bits and 2 mantissa bits.

FormatTradeoffCommon placementNote
E4M3Narrower range, more useful precisionweights, activations, forward GEMMNVIDIA docs list the maximum finite value at about 448
E5M2Wider range, less useful precisionactivation gradients, backward GEMMNVIDIA docs list the maximum finite value at about 57344, with inf support
HYBRIDE4M3 for forward, E5M2 for backwardOne default FP8 training format in Transformer EngineA format policy, not a new bit layout

The split is practical. Weights and activations in the forward pass tend to benefit from mantissa precision. Gradients often need more exponent range. Not every model must use that split, but it is a common starting point.

2. Where scale lives

FP8 has an exponent, but it still needs external scaling. The issue is not merely that FP8 cannot represent large values. Tensor distributions inside one LLM vary too much. One tensor may have an amax near 0.02, another near 2000. A single FP8 grid wastes codes for small tensors and clips large tensors.

A training stack usually maps a high precision tensor x into FP8 raw data q, plus a scale. Transformer Engine documents the multiplicative convention:

scale = FP8_MAX / amax(x)
q = cast_fp8(x * scale)
x_hat = q / scale

Some systems name the inverse quantity as the scale:

s = amax(x) / FP8_MAX
q = cast_fp8(x / s)
x_hat = q * s

The two forms differ by a reciprocal. When reading code or papers, check the scale direction first. Here, q is not a number in the original tensor domain. It is an FP8 code. Later computation recovers an approximate value from (q, scale) or (q, scale_inv) together.

# TE convention: keep scale and scale_inv = 1 / scale
q_i = cast_fp8(x_i * scale_i)
r_i = decode_fp8(q_i)
x_hat_i = descale(r_i, scale_i) = r_i / scale_i

# If the code stores scale_inv
x_hat_i = r_i * scale_inv_i

The same FP8 bit pattern does not necessarily represent the same value in the original domain. It first decodes to r_i on the FP8 grid, then scale or scale inverse maps it back into that tensor's value domain. Raw FP8 data without scale metadata usually cannot reconstruct the value the training system meant.

MXFP8 applies the same rule at block granularity. If one block covers 32 values and element i belongs to b(i)=floor(i/32), then:

q_i = cast_fp8(x_i * scale_b(i))
r_i = decode_fp8(q_i)
x_hat_i = r_i * scale_inv_b(i)

In GEMM, the block usually follows the reduction dimension of that matmul. Each 32-element dot-product chunk can then use its own scale instead of being dominated by an outlier from a full row, full column, or full tensor.

Scaling mainly does two things:

  • It keeps the largest values inside the representable range, reducing overflow or saturation.
  • It places common values into regions with useful resolution, instead of wasting mantissa codes on empty space.

BF16 has enough exponent range that per-tensor scaling is usually unnecessary. FP8 does not.

4. scale 粒度

scale 粒度决定一个 scale 覆盖多少元素。粒度越粗,metadata 越少,kernel 越简单,但 outlier 会影响更多元素。粒度越细,数值更稳,但 scale 的存储、加载、同步和 layout 都会变复杂。

粒度做法适合场景代价
per tensor整个 tensor 共用一个 FP32 scaleHopper 上常见 FP8 训练路径一个 outlier 会拉大整个 tensor 的范围
per axis按 row 或 column 维护 scale权重或 activation 分布沿某一维差异明显scale load 和 layout 约束增加
software block按小块维护 scale,例如 1x128 或 128x128需要处理 outlier,硬件未提供原生 microscalingkernel 和转置路径更复杂
MXFP8沿 scaling 方向每 32 个逻辑连续值一个 E8M0 scaleBlackwell Tensor Core 原生支持的 microscalingscale 方向和转置副本需要特别处理

per-tensor scaling 解释了为什么早期 FP8 recipe 常用 HYBRID。每个 tensor 只有一个 scale 时,梯度的动态范围压力仍然很大,E5M2 的宽 exponent 有用。MXFP8 把 block 内的动态范围压小,因此很多 tensor 可以统一用 E4M3,把更多 bit 留给 mantissa。

这里说的 "连续" 是沿 scaling 方向的逻辑连续值,不要求原始 tensor 地址连续。比如 X[M,K] row-major 里沿 K 的 32 个值通常是 memory contiguous;但 W[K,N] row-major 里沿 K 的 32 个值是 stride-N 访问。二者在数学上都沿 reduction 维度 K 分 block,只是存储连续性不同。实际 kernel 可能会生成 columnwise copy、scale layout、padding 或 swizzle,避免直接按原始内存地址每 32 个元素切 block。

MXFP8 的 block scale 最容易误解的地方是方向。对矩阵乘来说,block scale 不是随便沿物理 row 或 column 放,而是要服务那一次 matmul 的 reduction 维度。以 Y = X[M,K] @ W[K,N] 为例,两个 operand 的 scale 在数学语义上都沿 K 放,因为每个输出元素都在 K 上做 dot product。若 W 真的按 [K,N] row-major 存,那沿 K 是 columnwise;若 Linear 权重按常见的 [N,K] 存,并在计算里用 W.T,那沿 K 又变成 rowwise。TE 代码里的 rowwise/columnwise 是存储视角,不是数学轴名。

# 数学语义
Y[m, n] = sum_k X[m, k] * W[k, n]

# block scale 要贴着 reduction chunk
X[m, k:k+32]      # 沿 K
W[k:k+32, n]      # 也沿 K

5. MXFP8 GEMM 的数学语义

把一个普通 GEMM 写成 C[M,N] = A[M,K] @ B[K,N]。MXFP8 block-scaled MMA 不是把乘积结果直接变成 MXFP8,而是在输入 operand 上带 scale。每个 32-value K chunk 共享一个 scale,硬件语义可以写成:

a_hat[m, k] = decode_fp8(qA[m, k]) * scale_inv_A[m, floor(k / 32)]
b_hat[k, n] = decode_fp8(qB[k, n]) * scale_inv_B[floor(k / 32), n]

acc_fp32[m, n] = sum_k a_hat[m, k] * b_hat[k, n]

CUTLASS 的 tcgen05 文档把它写成 D = (SFA * A) * (SFB * B) + C,scale factor 沿 GEMM-K 放。这个公式里的 SFA 和 SFB 是 A/B operand 的 scale,不是 output C 的 scale。

Tensor Core 的输出也要分清。MXFP8 Tensor Core operand 是 MXFP8 payload 加 block scale;MMA accumulator 是 FP32 tile。最后写回什么 dtype,由 epilogue 决定。

MXFP8 A/B + E8M0 scale
  -> block-scaled MMA
  -> FP32 accumulator tile
  -> epilogue store as BF16, FP16, FP32, or re-quantized FP8/MXFP8

如果 epilogue 要把 C 存成 MXFP8,它必须给 C[M,N] 自己选 block 并产生 output scale。这个 scale 跟 K 没有继承关系,因为 output tensor 已经没有 K 维。

qC[m, n] = cast_fp8(acc_fp32[m, n] / scale_inv_C[block(m, n)])
C_hat[m, n] = decode_fp8(qC[m, n]) * scale_inv_C[block(m, n)]

TE 的 Quantizer.set_usage(rowwise=..., columnwise=...) 用来指定这个 output tensor 将来按哪种布局被消费。对 MXFP8 来说,它不是任意轴选择器,而是两种 1D block layout:

usagescale shape含义
rowwise[prod(shape[:-1]), shape[-1] / 32]沿最后一维每 32 个值一个 scale。
columnwise[prod(shape[:-1]) / 32, shape[-1]]沿 flattened leading dimension 每 32 个值一个 scale。

表里的 shape 是逻辑形状。TE 源码还会按 kernel 要求把 scale tensor padding 到指定倍数,并要求参与 MXFP8 quantization 的维度满足 block 对齐。

TE Linear 的 forward epilogue 会把 output_quantizer 传给 general_gemm。如果调用方传入 fp8_output=True,Linear 取 GEMM1_OUTPUT quantizer,并把它设成 rowwise=True, columnwise=False。普通 C[M,N] 的 MXFP8 Linear output 会沿 N 方向量化:

C[m, n:n+32] 共享一个 output scale

QKV projection 不能把 MXFP8 output 直接交给 attention。QKV projection 的 output quantizer 沿 packed feature 维量化,也就是 [M, 3*H*D] 的最后一维。attention 需要的是 Q/K 沿 head dim D,V 沿 sequence 维 Sk。FP8 payload 可以 reshape,但 (payload, scale) 的 block membership 已经不再对应 attention 的 reduction 维度。

6. amax 更新方式

scale 通常来自 amax,也就是 tensor 的最大绝对值。recipe 需要决定什么时候拿到这个 amax,以及用它怎样更新下一次的 scale。

策略怎么做优点风险
current scaling用当前 tensor 的 amax 计算当前 scalescale 跟分布同步amax reduction 和 cast 可能进入 critical path
just-in-time output scaling先产生高精度输出,再统计 amax 并量化数值上直接多 pass 代价高,NVIDIA 文档认为训练中通常不可取
delayed scaling记录历史 amax,用前几次的统计决定本次 scale性能路径干净,Transformer Engine 常用分布突变时 scale 会滞后
static scalingscale 来自校准或离线规则推理路径简单预训练分布持续变化,不适合作为默认训练策略

DelayedScaling 的状态一般包含 amax history、history 长度、取 history 最大值还是最近值、margin、format、以及分布式 reduce 规则。从实现角度看,它是一套 operator state。每个参与 FP8 的 tensor 都要能找到自己的 scale 和 amax 记录。

7. 训练系统里的 recipe

把前面的轴组合起来,常见 recipe 大致有下面几类。

recipe组成使用动机排查点
FP8 delayed HYBRIDper-tensor FP32 scale,forward E4M3,backward E5M2,历史 amaxHopper 时代的稳妥默认项amax history 是否过长,梯度 outlier 是否频繁
FP8 current scaling当前 amax,通常仍是 per-tensor 或 per-axis scalescale 跟随更快,便于实验amax reduction 是否拖慢 step time
FP8 block scaling小块 scale,format 多为 E4M3 或 HYBRID减轻 outlier 对整块 tensor 的影响转置、scale layout、kernel 支持
MXFP8每 32 个值一个 E8M0 scale,Blackwell 原生路径更细粒度 scaling,减少对 E5M2 的依赖scale rounding、转置副本、rowwise/columnwise 方向
calibrated FP8 inference离线 scale,常用于 W8A8 或 KV cache 量化部署简单,吞吐和显存收益明确校准集是否覆盖真实分布

训练 recipe 还会规定哪些张量不进 FP8。很多系统只让 GEMM operand 走 FP8,accumulation、bias、residual add、LayerNorm、RMSNorm、softmax、loss 和 optimizer 继续留在高精度。原因很实际:这些位置的数值误差更容易累积,并且它们未必是 step time 的主耗时。

分布式训练还要处理 amax 同步。若 TP 或 FSDP group 内不同 rank 用不同 scale,数学上不一定错,但 checkpoint、recompute、通信融合和调试会变复杂。Transformer Engine 的 recipe 里有 reduce_amax 这类选项,就是在性能、确定性和局部统计之间取舍。

8. TE MHA 的实现例子

Transformer Engine 里 attention 的 FP8 recipe 不能只看 recipe object 上有没有 fp8_dpa 或 fp8_mha 字段,还要看 backend selector 最后是否接受这个组合。以 TE v2.16 为例,MXFP8 attention 的实现会把 DPA 当成自己的 consumer,重新按 attention 需要的方向量化 Q/K/V。

这三个判断的职责不同。fp8_dpa 控制 DPA 是否进入 FP8 路径:TE 的 backend selector 只有在 fp8 autocast 已开启且 fp8_recipe.fp8_dpa 为 true 时,才进入 FP8 attention 分支。fp8_recipe.mxfp8() 不是 DPA 开关,它只说明当前 recipe 类型是 MXFP8BlockScaling。fp8_mha 管 MHA 边界,也就是是否尝试去掉 QKV projection 和 DPA 之间、DPA 和 output projection 之间的 BF16/FP16 cast。实践中 fp8_mha 要和 fp8_dpa=True 一起看,否则只讨论边界 dtype 没有意义。

开关或判断控制范围对 MHA 数据流的影响
fp8_dpaDPA 内部允许 DPA 把输入量化到 FP8,在 FP8 attention backend 中计算,再按边界需要返回高精度或 FP8。
fp8_mhaMHA 边界尝试让 QKV projection 直接输出 FP8 给 DPA,并让 DPA 输出 FP8 给 output projection。
mxfp8()recipe 类型判断当前 recipe 是否是 MXFP8 block scaling。它不等价于 fp8_dpa=True。

这个依赖关系是单向的:正常语义上 fp8_mha=True 依赖 fp8_dpa=True,但 fp8_dpa=True 不要求 fp8_mha=True。TE API 文档描述的也是这两条数据流:fp8_dpa=True, fp8_mha=False 时,QKV projection 输出高精度,DPA 边界做 cast;两个都为 true 时,TE 才尝试让 QKV projection 输出 FP8,并把 DPA 的 FP8 输出直接交给 output projection。源码层面,recipe object 上这两个字段只是两个 bool,不要把它理解成 dataclass 构造时一定会拦截的 invariant。真正决定 FP8 attention backend 入口的仍然是 fp8_dpa。

attention 有两次 matmul:

S = Q[B,H,Sq,D] @ K[B,H,Sk,D]^T
O = P[B,H,Sq,Sk] @ V[B,H,Sk,Dv]

QK^T 的 reduction 维度是 D,Q 和 K 的 MXFP8 block scale 沿 D。P@V 的 reduction 维度是 Sk,P 和 V 的 scale 沿 Sk。如果 V 按 [B,H,Sk,Dv] 存,沿 Sk 就不是最后一维,TE 术语里会落到 columnwise。

attention matmuloperandreduction 维度TE v2.16 MXFP8 forward 用法
Q @ K^TQDrowwise
Q @ K^TKDrowwise
P @ VVSkcolumnwise
 
# TE v2.16 combine_and_quantize() 的 forward MXFP8 方向,概念化写法
Q: rowwise=True,  columnwise=False   # 沿 D
K: rowwise=True,  columnwise=False   # 沿 D
V: rowwise=False, columnwise=True    # 沿 Sk

这和 QKV projection 的 output scale 方向不是同一个问题。projection 是普通 GEMM:输入 X 和权重 W 的 scale 跟那次 Linear GEMM 的 reduction 维度走。DPA 是两次 attention matmul:Q/K 要服务 QK^T,V 要服务 P@V。

QKV projection 的 output dtype 由 qkv_fp8_output 决定,不是由 fp8_dpa 单独决定。TE v2.16 的条件是:FP8 autocast 已开启,fp8_mha=True,没有 RoPE,不是 current scaling,也不是 MXFP8。这个布尔值随后以 fp8_output=qkv_fp8_output 传给 QKV 的 LayerNormLinear 或 Linear。

qkv_fp8_output = (
    fp8
    and fp8_mha
    and rotary_pos_emb is None
    and not float8_current_scaling
    and not mxfp8_scaling
)

MXFP8 被排除在这个条件外,原因是 projection 输出的形状和 attention 需要的 scaling 方向不一致。QKV GEMM 先产出 bs(hd)、sb(hd) 或 t(hd) 这类布局;MXFP8 attention 需要按 s 和 d 方向重新量化成 bhsd、sbhd 或 thd 可消费的形状。在 MXFP8 路径下,QKV projection 仍按 module 的高精度 dtype 返回,后面的 DPA 准备逻辑再按 attention 自己的 reduction 方向量化 Q/K/V。

DPA output dtype 也有单独 gate。TE v2.16 写成 dpa_fp8_output = fp8 and (fp8_dpa or fp8_mha) and not mxfp8_scaling。MXFP8 DPA 在这个版本不会把 O 作为 FP8 边界输出给 output projection。_update_output_quantizer_roles 会把这些边界关系交给 quantizer role:QKV linear 到 DPA、DPA 到 output projection,以及对应 backward 边。

TE v2.16 可落地的 MXFP8 attention 数据流是:

QKV projection:
  MXFP8 GEMM 内部使用 projection 自己的 scale 方向
  输出回 BF16/FP16 Q, K, V

DPA:
  按 attention reduction 维度重新量化 Q/K/V
  Q/K 沿 D,V 沿 Sk
  FusedAttention 执行 FP8 DPA

MXFP8 下的 fp8_mha 限制来自 backend selector。TE v2.16 先用 fp8_dpa 决定是否进入 FP8 attention 分支;进入之后,如果 fp8_recipe.mxfp8() 为 true 且 fp8_mha=True,就明确禁用 FusedAttention。下面这个 recipe 可以构造,但不代表 v2.16 已经有端到端 MXFP8 MHA backend:

MXFP8BlockScaling(
    fp8_dpa=True,
    fp8_mha=True,
)

可用配置通常保留 DPA 边界:

MXFP8BlockScaling(
    fp8_dpa=True,
    fp8_mha=False,
)

即便如此,还要满足 selector 的其它条件,例如 sm100+、cuDNN 版本足够新、qkv_format != "thd"。如果这些条件不满足,TE 会把 FusedAttention 关掉,最后可能没有可用的 FP8 attention backend。

9. 用 BF16 GEMM 模拟 MXFP8

如果手上只有 BF16 GEMM,仍然可以模拟 MXFP8 的数值影响。做法是 fake quantization:先按 MXFP8 的 block scale 规则把输入 cast 到 FP8,再 dequant 回 BF16,最后调用原来的 BF16 GEMM。这个方法只模拟量化误差,不模拟 MXFP8 Tensor Core 的吞吐、寄存器布局或 scale load 开销。

BF16 A, B
  -> MXFP8 fake quantize
  -> dequantize back to BF16
  -> BF16 GEMM

对一个 32-value reduction block,可以按下面的语义写 reference。这里 scale_inv 对应 TE 里常见的反 scale,MXFP8 的 E8M0 scale 约束用 power-of-2 rounding 模拟:

scale_inv_b = pow2ceil(amax(block) / FP8_MAX_E4M3)
q_i = cast_fp8_e4m3(x_i / scale_inv_b)
x_hat_i = decode_fp8(q_i) * scale_inv_b

下面这段代码假设数学 GEMM 是 C = A[M,K] @ B[K,N]。A 沿最后一维 K 分 block,B 沿第一维 K 分 block。若你的权重是 PyTorch Linear 常见的 weight[N,K],见后面的 linear_mxfp8_sim。

import torch
import torch.nn.functional as F

FP8_MAX_E4M3 = 448.0
MXFP8_BLOCK = 32

def pow2ceil(x):
    x = torch.clamp(x, min=2.0 ** -126)
    return torch.pow(2.0, torch.ceil(torch.log2(x)))

def fake_mxfp8_along_last_dim(x, block=MXFP8_BLOCK):
    # x: [..., K], block 沿最后一维 K
    k = x.shape[-1]
    pad = (-k) % block

    x_fp32 = x.float()
    if pad:
        x_fp32 = F.pad(x_fp32, (0, pad))

    k_pad = x_fp32.shape[-1]
    xb = x_fp32.reshape(*x_fp32.shape[:-1], k_pad // block, block)

    amax = xb.abs().amax(dim=-1, keepdim=True)
    scale_inv = pow2ceil(amax / FP8_MAX_E4M3)
    scale_inv = torch.where(amax == 0, torch.ones_like(scale_inv), scale_inv)

    q = (xb / scale_inv).to(torch.float8_e4m3fn)
    x_hat = q.to(torch.bfloat16) * scale_inv.to(torch.bfloat16)
    x_hat = x_hat.reshape(*x_fp32.shape)
    return x_hat[..., :k]

def fake_mxfp8_along_first_dim(x, block=MXFP8_BLOCK):
    # x: [K, N], block 沿第一维 K
    k, n = x.shape
    pad = (-k) % block

    x_fp32 = x.float()
    if pad:
        x_fp32 = F.pad(x_fp32, (0, 0, 0, pad))

    k_pad = x_fp32.shape[0]
    xb = x_fp32.reshape(k_pad // block, block, n)

    amax = xb.abs().amax(dim=1, keepdim=True)
    scale_inv = pow2ceil(amax / FP8_MAX_E4M3)
    scale_inv = torch.where(amax == 0, torch.ones_like(scale_inv), scale_inv)

    q = (xb / scale_inv).to(torch.float8_e4m3fn)
    x_hat = q.to(torch.bfloat16) * scale_inv.to(torch.bfloat16)
    x_hat = x_hat.reshape(k_pad, n)
    return x_hat[:k, :]

def mxfp8_gemm_sim(A, B, fp32_accum=False):
    # A: [M, K], B: [K, N]
    A_hat = fake_mxfp8_along_last_dim(A)
    B_hat = fake_mxfp8_along_first_dim(B)

    if fp32_accum:
        return A_hat.float() @ B_hat.float()
    return A_hat.to(torch.bfloat16) @ B_hat.to(torch.bfloat16)

Linear 权重常见存储是 weight[N,K],计算写成 X @ weight.T。这时 X 和 weight 的 reduction 维度 K 都在最后一维,所以 fake quantize 都可以沿 last dim 做:

def linear_mxfp8_sim(x, weight, bias=None, fp32_accum=False):
    # x: [M, K], weight: [N, K]
    x_hat = fake_mxfp8_along_last_dim(x)
    w_hat = fake_mxfp8_along_last_dim(weight)

    if fp32_accum:
        y = x_hat.float() @ w_hat.float().T
    else:
        y = x_hat.to(torch.bfloat16) @ w_hat.to(torch.bfloat16).T

    if bias is not None:
        y = y + bias
    return y

这段模拟有两个用途。若目标是看 MXFP8 cast 对你现有 BF16 GEMM 输出的影响,保留 BF16 GEMM。若目标是构造更稳定的数值 reference,可以打开 fp32_accum=True,把最后的 matmul 改成 FP32 accumulate。

10. PyTorch 视角

PyTorch 的 float8 dtype 是 unscaled 的。torch.float8_e4m3fn 和 torch.float8_e5m2 只描述 raw data 的格式,并不自动携带 scale。scale 需要由上层建模代码、tensor subclass 或专门的 kernel API 管理。PyTorch dev-discuss 里提到的 torch.scaled_mm 就是这种分层设计:matrix 是 FP8,scale 作为单独参数传入。

这也是为什么 FP8 训练很少表现为一个普通 autocast dtype。autocast 只负责 dtype promotion 或 cast 还不够。系统还要维护 per-tensor state、amax history、分布式 reduce、module rewrite、保存给 backward 的 tensor 版本,以及编译器能否把这些额外操作融合掉。

调 FP8 训练时,我一般检查四组信号:

  • 每类 tensor 的 amax 曲线是否突然跳变。
  • scale 是否长期被某些 outlier 拉大。
  • E4M3 路径是否出现明显 saturation,梯度是否需要 E5M2 或更细粒度 scale。
  • amax reduce 和 cast 是否抵消了 FP8 GEMM 的收益。

这些信号比单独问 "FP8 有没有开" 更有用。FP8 的精度和性能都藏在 recipe 里。

4. Scale granularity

Scale granularity decides how many values share one scale. Coarser granularity has less metadata and simpler kernels, but one outlier can affect more values. Finer granularity is usually more robust numerically, but scale storage, loads, synchronization, and layout get more complicated.

GranularityMechanismWhere it fitsCost
per tensorOne FP32 scale for the whole tensorCommon Hopper FP8 training pathOne outlier stretches the whole tensor range
per axisScale per row or columnWeights or activations with axis-specific distributionsMore scale loads and layout constraints
software blockScale small blocks, such as 1x128 or 128x128Outlier handling without native microscalingMore complex kernels and transpose paths
MXFP8One E8M0 scale per 32 logically consecutive values along the scaling directionNative Blackwell Tensor Core microscalingScale direction and transpose copies need care

Per-tensor scaling explains why early FP8 recipes often used HYBRID. With only one scale per tensor, gradients still put heavy pressure on dynamic range, so E5M2 is useful. MXFP8 reduces the dynamic range inside each block, allowing many tensors to use E4M3 and spend more bits on mantissa.

"Consecutive" here means logically consecutive along the scaling direction. It does not necessarily mean contiguous addresses in the original tensor. In row-major X[M,K], 32 values along K are usually memory contiguous. In row-major W[K,N], 32 values along K are strided by N. Both are block-scaled along the mathematical reduction dimension K, but their storage contiguity differs. Real kernels may materialize a columnwise copy, scale layout, padding, or swizzle instead of cutting the original memory buffer into 32-element chunks.

The common trap with MXFP8 is direction. For matrix multiplication, the block scale should follow the reduction dimension of that matmul, not a fixed physical row or column. In Y = X[M,K] @ W[K,N], both operands are scaled along K in the mathematical sense, because every output element is a dot product over K. If W is physically stored as row-major [K,N], that K direction is columnwise. If a Linear layer stores weight as [N,K] and computes with W.T, the same K direction is rowwise. TE's rowwise/columnwise names are storage-facing terms, not mathematical axis names.

# Mathematical semantics
Y[m, n] = sum_k X[m, k] * W[k, n]

# Block scale follows the reduction chunk
X[m, k:k+32]      # along K
W[k:k+32, n]      # also along K

5. MXFP8 GEMM semantics

Write a normal GEMM as C[M,N] = A[M,K] @ B[K,N]. MXFP8 block-scaled MMA does not make the multiply result inherently MXFP8. It attaches scales to the input operands. Each 32-value chunk along K shares one scale, so the hardware semantics can be written as:

a_hat[m, k] = decode_fp8(qA[m, k]) * scale_inv_A[m, floor(k / 32)]
b_hat[k, n] = decode_fp8(qB[k, n]) * scale_inv_B[floor(k / 32), n]

acc_fp32[m, n] = sum_k a_hat[m, k] * b_hat[k, n]

The CUTLASS tcgen05 docs write this as D = (SFA * A) * (SFB * B) + C, with scale factors along GEMM-K. SFA and SFB are operand scales for A and B. They are not the output scale for C.

The Tensor Core output needs the same separation. MXFP8 Tensor Core operands are MXFP8 payload plus block scales. The MMA accumulator is an FP32 tile. The epilogue decides the dtype written out.

MXFP8 A/B + E8M0 scale
  -> block-scaled MMA
  -> FP32 accumulator tile
  -> epilogue store as BF16, FP16, FP32, or re-quantized FP8/MXFP8

If the epilogue stores C as MXFP8, it must choose blocks for C[M,N] and produce output scales for C. Those scales are not inherited from K, because the output tensor no longer has a K dimension.

qC[m, n] = cast_fp8(acc_fp32[m, n] / scale_inv_C[block(m, n)])
C_hat[m, n] = decode_fp8(qC[m, n]) * scale_inv_C[block(m, n)]

TE's Quantizer.set_usage(rowwise=..., columnwise=...) says how this output tensor is expected to be consumed. For MXFP8, it is not a free axis selector. It chooses between two 1D block layouts:

usagescale shapemeaning
rowwise[prod(shape[:-1]), shape[-1] / 32]One scale per 32 values along the last dimension.
columnwise[prod(shape[:-1]) / 32, shape[-1]]One scale per 32 values along the flattened leading dimension.

The shapes in the table are logical shapes. TE also pads scale tensors to kernel-friendly multiples and requires MXFP8-quantized dimensions to satisfy block alignment.

TE Linear passes output_quantizer into general_gemm in the forward epilogue. If the caller sets fp8_output=True,Linear takes the GEMM1_OUTPUT quantizer and sets it to rowwise=True, columnwise=False. A normal C[M,N] MXFP8 Linear output is quantized along N:

C[m, n:n+32] shares one output scale

A QKV projection cannot directly hand its MXFP8 output to attention. The QKV projection output quantizer follows the packed feature dimension, the last dimension of [M, 3*H*D]. Attention wants Q/K scaled along head dim D, and V scaled along sequence dim Sk. The FP8 payload can be reshaped, but the (payload, scale) block membership no longer matches attention's reduction dimensions.

6. Amax update policy

The scale usually comes from amax, the maximum absolute value of a tensor. A recipe must decide when that amax is known and how it updates the next scale.

PolicyMechanismBenefitRisk
current scalingUse the current tensor amax for the current scaleScale follows the current distributionAmax reductions and casts may enter the critical path
just-in-time output scalingProduce high precision output, compute amax, then quantizeNumerically directExtra passes are expensive; NVIDIA docs describe it as impractical for training
delayed scalingKeep amax history and use previous statistics for this stepClean performance path, common in Transformer EngineScale can lag sudden distribution changes
static scalingUse calibration or offline rulesSimple inference pathPre-training distributions drift, so it is not a default training strategy

DelayedScaling state usually contains amax history, history length, max versus most recent selection, margin, format, and distributed reduction rules. Implementation-wise, it is operator state. Every tensor that participates in FP8 needs a way to find its scale and amax record.

7. Recipes in training systems

Combining the axes above gives a few common recipe families.

RecipeCompositionMotivationDebug focus
FP8 delayed HYBRIDper-tensor FP32 scale, E4M3 forward, E5M2 backward, historical amaxConservative Hopper-era defaultHistory length, gradient outliers
FP8 current scalingcurrent amax, often per-tensor or per-axis scaleScale follows faster and is easy to experiment withAmax reduction overhead
FP8 block scalingsmall-block scales, usually E4M3 or HYBRIDReduce outlier impact on the whole tensorTranspose, scale layout, kernel support
MXFP8one E8M0 scale per 32 values, native Blackwell pathFiner scaling and less dependence on E5M2Scale rounding, transpose copies, rowwise/columnwise direction
calibrated FP8 inferenceoffline scales, often for W8A8 or KV cache quantizationSimple deployment, clear throughput and memory winsCalibration coverage

A training recipe also states which tensors do not enter FP8. Many systems only run GEMM operands in FP8. Accumulation, bias, residual add, LayerNorm, RMSNorm, softmax, loss, and optimizer state stay in higher precision. The reason is practical: these positions accumulate error more visibly, and many of them are not the dominant step-time cost.

Distributed training adds amax synchronization. If ranks inside a TP or FSDP group keep different scales, the math may still run, but checkpointing, recompute, communication fusion, and debugging get harder. Transformer Engine exposes options such as reduce_amax because this is a tradeoff between performance, determinism, and local statistics.

8. Transformer Engine MHA example

In Transformer Engine, an attention FP8 recipe cannot be judged from the fp8_dpa and fp8_mha fields alone. The backend selector still decides whether the requested combination is available. In TE v2.16, MXFP8 attention treats DPA as its own consumer and quantizes Q/K/V again with attention-specific directions.

The three checks do different work. fp8_dpa controls whether DPA may enter the FP8 path: TE's backend selector only enters the FP8 attention branch when FP8 autocast is active and fp8_recipe.fp8_dpa is true. fp8_recipe.mxfp8() is not a DPA switch; it only says that the current recipe class is MXFP8BlockScaling. fp8_mha controls MHA boundaries, meaning whether TE tries to remove the BF16/FP16 casts between QKV projection and DPA, and between DPA and the output projection. In normal use, fp8_mha should be read together with fp8_dpa=True; otherwise the boundary dtype discussion is detached from FP8 DPA.

switch or checkscopeeffect on MHA data flow
fp8_dpaDPA internalsAllows DPA to quantize inputs to FP8, run in an FP8 attention backend, then return high precision or FP8 at the boundary.
fp8_mhaMHA boundariesTries to make QKV projection output FP8 directly into DPA, and DPA output FP8 directly into the output projection.
mxfp8()recipe typeChecks whether the current recipe uses MXFP8 block scaling. It is not equivalent to fp8_dpa=True.

The dependency is one-way: fp8_mha=True depends on fp8_dpa=True, but fp8_dpa=True does not require fp8_mha=True. The TE API documentation describes those two data flows. With fp8_dpa=True, fp8_mha=False, QKV projection returns high precision and DPA casts at its boundary. When both are true, TE tries to make QKV projection output FP8 and pass FP8 DPA output directly to the output projection. In source code, the recipe object stores these as two bool fields, so this should not be read as a dataclass constructor invariant. The real entry gate for an FP8 attention backend is still fp8_dpa.

Attention has two matmuls:

S = Q[B,H,Sq,D] @ K[B,H,Sk,D]^T
O = P[B,H,Sq,Sk] @ V[B,H,Sk,Dv]

For QK^T, the reduction dimension is D, so Q and K use MXFP8 blocks along D. For P@V, the reduction dimension is Sk, so P and V use blocks along Sk. If V is stored as [B,H,Sk,Dv], Sk is not the last dimension, which maps to columnwise in TE terminology.

attention matmuloperandreduction dimTE v2.16 MXFP8 forward usage
Q @ K^TQDrowwise
Q @ K^TKDrowwise
P @ VVSkcolumnwise
 
# Conceptualized from TE v2.16 combine_and_quantize()
Q: rowwise=True,  columnwise=False   # along D
K: rowwise=True,  columnwise=False   # along D
V: rowwise=False, columnwise=True    # along Sk

This is separate from the output scale direction of the QKV projection. The projection is a normal GEMM: X and W follow the reduction dimension of that Linear GEMM. DPA has its own two matmuls: Q/K serve QK^T, while V serves P@V.

The QKV projection output dtype is controlled by qkv_fp8_output, not by fp8_dpa alone. In TE v2.16, the conditions are: FP8 autocast is active, fp8_mha=True, no RoPE, not current scaling, and not MXFP8. This boolean is then passed as fp8_output=qkv_fp8_output into the QKV LayerNormLinear or Linear.

qkv_fp8_output = (
    fp8
    and fp8_mha
    and rotary_pos_emb is None
    and not float8_current_scaling
    and not mxfp8_scaling
)

MXFP8 is excluded here because the projection output shape and the attention scaling directions do not line up. The QKV GEMM first produces layouts such as bs(hd), sb(hd), or t(hd). MXFP8 attention needs Q/K/V quantized along s and d, in shapes consumable as bhsd, sbhd, or thd. On this path, QKV projection returns in the module's high-precision dtype, and the DPA preparation code re-quantizes Q/K/V along attention's own reduction directions.

DPA output dtype has its own gate. TE v2.16 uses dpa_fp8_output = fp8 and (fp8_dpa or fp8_mha) and not mxfp8_scaling. MXFP8 DPA in this version does not expose O as an FP8 boundary tensor to the output projection. _update_output_quantizer_roles records these boundary relationships in quantizer roles: QKV linear to DPA, DPA to output projection, and the matching backward edges.

TE v2.16 lands on this MXFP8 attention path:

QKV projection:
  MXFP8 GEMM uses projection-specific scale directions
  output returns to BF16/FP16 Q, K, V

DPA:
  re-quantize Q/K/V along attention reduction dimensions
  Q/K along D, V along Sk
  FusedAttention runs FP8 DPA

MXFP8's fp8_mha limitation comes from the backend selector. TE v2.16 first uses fp8_dpa to decide whether it enters the FP8 attention branch. Once inside that branch, if fp8_recipe.mxfp8() is true and fp8_mha=True, FusedAttention is explicitly disabled. The recipe below can be constructed, but that does not mean v2.16 has an end-to-end MXFP8 MHA backend:

MXFP8BlockScaling(
    fp8_dpa=True,
    fp8_mha=True,
)

The more realistic configuration keeps the DPA boundary:

MXFP8BlockScaling(
    fp8_dpa=True,
    fp8_mha=False,
)

Even then, the selector still requires conditions such as sm100+, a recent enough cuDNN, and qkv_format != "thd". If those checks fail, TE disables FusedAttention and may end up with no usable FP8 attention backend.

9. Simulating MXFP8 with BF16 GEMM

If the available GEMM path is BF16, you can still simulate the numerical effect of MXFP8. Use fake quantization: cast inputs to FP8 with MXFP8 block scales, dequantize them back to BF16, then call the existing BF16 GEMM. This simulates quantization error. It does not simulate MXFP8 Tensor Core throughput, register layout, or scale-load overhead.

BF16 A, B
  -> MXFP8 fake quantize
  -> dequantize back to BF16
  -> BF16 GEMM

For one 32-value reduction block, a reference can use the following semantics. scale_inv is the inverse scale commonly stored by TE, and the E8M0 constraint is approximated with power-of-2 rounding:

scale_inv_b = pow2ceil(amax(block) / FP8_MAX_E4M3)
q_i = cast_fp8_e4m3(x_i / scale_inv_b)
x_hat_i = decode_fp8(q_i) * scale_inv_b

The code below assumes mathematical GEMM C = A[M,K] @ B[K,N]. A is blocked along its last dimension K, while B is blocked along its first dimension K. If your weight uses the usual PyTorch Linear storage weight[N,K], use linear_mxfp8_sim below.

import torch
import torch.nn.functional as F

FP8_MAX_E4M3 = 448.0
MXFP8_BLOCK = 32

def pow2ceil(x):
    x = torch.clamp(x, min=2.0 ** -126)
    return torch.pow(2.0, torch.ceil(torch.log2(x)))

def fake_mxfp8_along_last_dim(x, block=MXFP8_BLOCK):
    # x: [..., K], block along the last dimension K
    k = x.shape[-1]
    pad = (-k) % block

    x_fp32 = x.float()
    if pad:
        x_fp32 = F.pad(x_fp32, (0, pad))

    k_pad = x_fp32.shape[-1]
    xb = x_fp32.reshape(*x_fp32.shape[:-1], k_pad // block, block)

    amax = xb.abs().amax(dim=-1, keepdim=True)
    scale_inv = pow2ceil(amax / FP8_MAX_E4M3)
    scale_inv = torch.where(amax == 0, torch.ones_like(scale_inv), scale_inv)

    q = (xb / scale_inv).to(torch.float8_e4m3fn)
    x_hat = q.to(torch.bfloat16) * scale_inv.to(torch.bfloat16)
    x_hat = x_hat.reshape(*x_fp32.shape)
    return x_hat[..., :k]

def fake_mxfp8_along_first_dim(x, block=MXFP8_BLOCK):
    # x: [K, N], block along the first dimension K
    k, n = x.shape
    pad = (-k) % block

    x_fp32 = x.float()
    if pad:
        x_fp32 = F.pad(x_fp32, (0, 0, 0, pad))

    k_pad = x_fp32.shape[0]
    xb = x_fp32.reshape(k_pad // block, block, n)

    amax = xb.abs().amax(dim=1, keepdim=True)
    scale_inv = pow2ceil(amax / FP8_MAX_E4M3)
    scale_inv = torch.where(amax == 0, torch.ones_like(scale_inv), scale_inv)

    q = (xb / scale_inv).to(torch.float8_e4m3fn)
    x_hat = q.to(torch.bfloat16) * scale_inv.to(torch.bfloat16)
    x_hat = x_hat.reshape(k_pad, n)
    return x_hat[:k, :]

def mxfp8_gemm_sim(A, B, fp32_accum=False):
    # A: [M, K], B: [K, N]
    A_hat = fake_mxfp8_along_last_dim(A)
    B_hat = fake_mxfp8_along_first_dim(B)

    if fp32_accum:
        return A_hat.float() @ B_hat.float()
    return A_hat.to(torch.bfloat16) @ B_hat.to(torch.bfloat16)

PyTorch Linear usually stores weights as weight[N,K] and computes X @ weight.T. In that case, the reduction dimension K is the last dimension for both X and weight, so both fake-quantization calls use the last-dimension helper:

def linear_mxfp8_sim(x, weight, bias=None, fp32_accum=False):
    # x: [M, K], weight: [N, K]
    x_hat = fake_mxfp8_along_last_dim(x)
    w_hat = fake_mxfp8_along_last_dim(weight)

    if fp32_accum:
        y = x_hat.float() @ w_hat.float().T
    else:
        y = x_hat.to(torch.bfloat16) @ w_hat.to(torch.bfloat16).T

    if bias is not None:
        y = y + bias
    return y

This simulation has two common uses. To see how MXFP8 casts affect your current BF16 GEMM output, keep the BF16 GEMM. To build a more stable numerical reference, set fp32_accum=True and accumulate the final matmul in FP32.

10. PyTorch view

PyTorch float8 dtypes are unscaled. torch.float8_e4m3fn and torch.float8_e5m2 describe the raw data format; they do not automatically carry a scale. The modeling layer, a tensor subclass, or a specialized kernel API must manage the scale. The PyTorch dev-discuss post uses torch.scaled_mm as the example: matrices are FP8, scales are separate arguments.

FP8 training rarely looks like an ordinary autocast dtype. Autocast handles dtype promotion or casts, but FP8 training also needs per-tensor state, amax history, distributed reduction, module rewrites, saved tensor policy for backward, and compiler fusion for the extra operations.

When debugging FP8 training, I start with four signals:

  • whether amax curves jump suddenly for specific tensor classes;
  • whether scales are held too large by recurring outliers;
  • whether E4M3 paths show saturation, especially for gradients;
  • whether amax reductions and casts erase the FP8 GEMM speedup.

These signals are more useful than asking whether FP8 is simply "on". FP8 precision and speed both live inside the recipe.
