PyTorch C++↔Python 双向调度:TorchDispatchMode 与反向跳转

这是 《PyTorch Dispatcher 全栈剖析》的姐妹篇。前一篇沿"Python → CUDA"自顶向下走完一次 op 调用;本文反向 —— 钻进 dispatcher 的 Python 这一层,讲清"C++ 调用走到一半,怎么跳回 Python 让用户代码拦截每个 op"的完整机制。所有 PyTorch 源码以 pytorch/pytorch main 分支为准(May 2026)。

1. 反向跳转的需求

本章建立动机。前一篇文章把 dispatcher 描述成一条"自顶向下"的链:Python 调用 → autograd 包装 → backend kernel → 真正算数值。但对很多 PyTorch 的现代功能(torch.compile、shape inference、分布式张量),这条链是不够的 —— 必须在 C++ 中途"反向跳"回 Python,让用户/框架代码在 op 调用现场插一脚。本章解释:为什么需要这种反向跳,什么时候发生,以及它和前一篇讲的 dispatcher 是什么关系。读完你应该能在脑子里画出"正向跳"(前文)和"反向跳"(本文)的完整闭环。

1.1 我们已经知道的事

先快速回顾前一篇的关键结论。如果你完全没读过那篇,这一节会让你跟上节奏;读过的话,这是个对齐起点。

PyTorch 的 dispatcher 是一张按 DispatchKey 索引的函数表。每个 ATen 算子(aten::add、aten::matmul 等)拥有自己的一张表,表里每一行对应一个"行为层":CUDA 后端、CPU 后端、Autograd 包装、Autocast 包装等。一次调用做四件事:

  1. 从输入张量的属性和 TLS 状态算出一个 64 位位图(DispatchKeySet),表示当前激活了哪些 key。
  2. 取位图中最高优先级的 key。
  3. 跳到该 key 对应的函数指针。
  4. 该函数体可选择性地把"自己"从位图里 mask 掉再 redispatch,让控制权落到下一层 —— 比如 Autograd kernel 在建好反向图节点后,redispatch 到 CUDA kernel 真的算数值。

优先级阶梯从高到低大致是:模式层(Python / Functionalize / Autocast 等)→ Autograd 层 → Backend 层(CPU / CUDA / MPS 等)。这种排序让"先记反向图再算数值"、"先降精度再 autograd"这些层级约束自动得到保证。

关键句:dispatcher 不是一次跳转,而是一条 redispatch 驱动的链。每层做完自己的事,把自己 mask 掉,让下一层成为新的最高位。

1.2 真正的痛点:在 op 调用现场拦截

知道这套机制,我们可以问一个看似无害的问题:

"我想拦截 PyTorch 的每一次 ATen op 调用,看看输入是什么、把它们改一改、或者根本不让真的去算,而是返回一个我自己定的结果。"

这听起来很抽象,但这正是下面几个核心功能的本质:

功能它要拦截每个 op 来做什么
torch.compile 的 graph capture用户写 def f(x, y): return (x + y).relu(),编译器要把这函数变成一张 fx.Graph。怎么知道里头有哪些 op?答案是真的跑一次,但每次 ATen op 都被截下,记到 graph 里。
FakeTensorMode (shape inference)调 x @ y 但不希望真的分配显存 + 算 matmul,只想知道"输出会是什么 shape/dtype/device"。每个 op 调用要被截下,跑一个"假"实现 —— 只算 meta,不算数值。
DTensor (分布式)张量在多 GPU 上分片(shard)了。一个 add 操作要变成:① 检查输入分片方式;② 必要时插 all-reduce;③ 每个 rank 局部算 add;④ 包装回 DTensor。这些决策必须在 op 调用现场做。
调试 / 日志"我想 print 出整个模型每个 op 的输入 shape" —— 简单需求,但要在 op 调用每次触发。

这些需求都有同一个形状:"在 op 调用现场,插入一段我自己的逻辑"。而且这段逻辑必须是 Python(因为用户、研究员、第三方框架都写 Python,不可能要求每个新工具都用 C++ 重新实现 ATen)。

但 dispatcher 是 C++ 的!它存的是 C++ 函数指针,跳的也是 C++ 函数。我们怎么让一个Python 函数成为 dispatcher 表里的"一行"?

1.3 三个动机场景

抽象解释完了,看三个真实代码段感受一下。这三个就是后面 §5 要展开讲的三个用户。

(a) FakeTensorMode

import torch
from torch._subclasses.fake_tensor import FakeTensorMode

with FakeTensorMode():
    x = torch.randn(1024, 1024, device="cuda")     # 不分配显存
    y = torch.randn(1024, 512,  device="cuda")
    z = x @ y                                       # 不跑 matmul
    print(z.shape, z.dtype, z.device)
    # 输出: torch.Size([1024, 512]) torch.float32 cuda:0

这没有真的占 4 MB 显存,没有真的算 matmul。但 PyTorch 知道:如果你真跑这一段,输出会是 [1024, 512] 的 fp32 CUDA 张量。这个推断怎么做出来的?FakeTensorMode 在 dispatch 链中拦下了 torch.randn 和 @(aten::matmul)的每次调用,跑一个"假"实现 —— 只算 shape 不算数值。

(b) make_fx / ProxyTorchDispatchMode

import torch
from torch.fx.experimental.proxy_tensor import make_fx

def f(x, y):
    return (x + y).relu()

gm = make_fx(f, tracing_mode="fake")(torch.randn(3, 4), torch.randn(3, 4))
gm.graph.print_tabular()
# opcode         name    target              args        kwargs
# placeholder    x       x                   ()          {}
# placeholder    y       y                   ()          {}
# call_function  add     aten.add.Tensor     (x, y)      {}
# call_function  relu    aten.relu.default   (add,)      {}
# output         output  output              ((relu,),)  {}

这把一个 Python 函数编译成了一张图。怎么做到的?跑一次 f(x, y),但用 ProxyTorchDispatchMode 拦下每个 ATen op,把它记录到 fx.Graph 上而不光算结果。这是 torch.compile 内部 AOTAutograd 模块的核心机制。

(c) DTensor

import torch
from torch.distributed.tensor import DTensor, DeviceMesh, Shard, Replicate, distribute_tensor

mesh = DeviceMesh("cuda", torch.arange(4))                          # 4 GPU 网格
x = distribute_tensor(torch.randn(1024, 1024), mesh, [Shard(0)])    # 行分片
w = distribute_tensor(torch.randn(1024, 512),  mesh, [Replicate()]) # 复制
y = x @ w                                                            # 自动决定通信 + 局部算
# y 是 DTensor,placement=[Shard(0)],每个 rank 持有 256x512 的本地结果

表面上就是个 @ 操作,但内部:dispatcher 在算 aten::matmul 时被拦下,DTensor 的代码决定"输入分片方式 + 输出分片方式 + 是否需要 collective communication + 本地算什么",然后才真的让每个 rank 调底层 matmul kernel。

共同的形状:这三个用户都注册了一个 Python 端的"mode"(或者 tensor subclass),激活后 ATen op 调用每次都先经过他们的 Python 代码。我们要搞清楚:dispatcher 怎么把控制权从 C++ 交给 Python?Python 代码怎么决定"我处理完了"还是"让 dispatcher 继续走"?这就是本文的全部任务。

1.4 为什么不能在 Python 层面拦?

你可能会想:"我重写 torch.add、torch.matmul 不就行了?Python 函数我自己掌控,加个钩子嘛。"

这种思路对应的旧机制叫 __torch_function__ 协议,确实存在。但它只能拦截用户直接调的 torch.* API。看不到下面这些情况:

  • Autograd 内部产生的 op。z = x + y 反向时,engine 调 AddBackward0::apply,体内调 at::add / at::mul —— 这些不经过 torch.add,直接在 C++ 里跑,__torch_function__ 看不到。
  • Decomposition 产生的子 op。torch.softmax 在某些 backend 上被分解成 exp + sum + div 三个底层 op,这些子 op 也是在 C++ 里直接调,跳过用户 Python 层。
  • 第三方算子调用其它算子。某个自定义算子内部调 at::clamp_min,这也是 C++ → C++ 直接跳。

简言之:op 的"真实调用次数"远大于"用户写出来的次数"。torch.compile / FakeTensorMode 这些工具要见到每一次真实调用,必须在 dispatcher 内部拦,而不是在 Python 用户层拦。

那解决方案就是:在 dispatcher 内部留一个"跳回 Python"的通道。具体实现要回答几个工程问题:

  • C++ 的 dispatcher 怎么知道"现在该跳 Python"了?
  • 跳过去之后,怎么找到要调用的具体 Python 函数?(可能是 mode 的 __torch_dispatch__,也可能是 tensor subclass 的)
  • Python 函数处理完之后,怎么把结果送回 C++ 的调用方?
  • 如果 Python 函数体内又调了一个 op,怎么不让它再递归回到自己?

下一章拆开这些机制。先剧透一下回答的形状:

问题答案
怎么知道该跳 Python?位图里有 DispatchKey::Python(由 mode 上下文 push 或 tensor subclass 自带)
跳哪里?PythonFallbackKernel —— 注册到 Python key 的 boxed kernel
怎么找到 Python 函数?① TLS 模式栈 → 拿 mode 对象的 __torch_dispatch__;② 输入张量的 pyobj_slot → 拿 subclass 的 __torch_dispatch__
怎么调到 Python?过 PyInterpreterVTable 的虚函数,acquire GIL,把 IValue 参数转 PyObject,Python 调函数,结果反向转
怎么不递归?ExcludeDispatchKeyGuard 把 Python 及"以下"的位 mask 掉 —— Python 内调的 op 会跳过 Python kernel

这张表就是 §2 的提纲。每行展开成一节,从最浅的"为什么 c10 不直接依赖 CPython"开始。

2. C++ → Python 的物理机制

本章把 1.4 节那张机制总表一行一行展开看源码。读完应该能在脑子里画出 dispatcher 看到 Python key 后的完整流程,包括 GIL 获取、参数转换、Python 函数查找、结果回传、防递归。所有源码路径以 PyTorch main 分支为准 —— 这部分代码 2024-2025 期间有过文件重组,如果你看到老博客指向 c10/core/PyInterpreter.h 找不到,那是因为它现在在 c10/core/impl/PyInterpreter.h。

2.1 c10 与 CPython 的零依赖矛盾

要理解 C++→Python 跳转的设计,先得知道一个看似不起眼但贯穿全文的约束:

约束:libc10.so(PyTorch 的核心 C++ 库)不能直接依赖 CPython 的头文件或 libpython.so。libtorch_python.so 才能依赖 CPython。

为什么?因为 libtorch(libc10 的下游)必须能在没有 Python 的进程里用 —— 比如:

  • 移动端推理:把 PyTorch 模型编译成 C++ 可执行文件,运行时没有 Python。
  • JIT/lite-interpreter 部署:Android / iOS 的 PyTorch 应用。
  • 嵌入式系统:Jetson Nano 上跑模型,Python 是可选的。
  • 纯 C++ 服务:像 Triton inference server 这种,只用 libtorch C++ API,完全不加载 Python。

这就和我们想做的事产生矛盾:dispatcher 在 libc10 里(注册表数据结构、key 枚举、KernelFunction 都在这一层),它需要"调 Python"。但 libc10 不能 #include <Python.h>。

PyTorch 用一个经典的反转控制 (IoC) 设计破解这个矛盾:

  1. libc10 定义一个抽象接口(PyInterpreterVTable),里面是一堆纯虚函数 —— "我需要 Python 帮我做这些事,但我不知道怎么做"。
  2. libtorch_python(实际加载 CPython 的层)在静态初始化时填充这个 vtable 的具体实现,把指针塞回 c10 的全局变量。
  3. 运行时 c10 调用 Python 相关功能,走的是 vtable 虚函数 —— 它编译期不知道 CPython 存在,链接期也不需要 libpython.so;但运行期如果 libtorch_python 被加载了,vtable 就是有效的。
  4. 没有 Python 的进程里,vtable 是个 "no-op" 版本 —— 调到这些虚函数会触发友好的 error,而不是 segfault。

这个 IoC 决定了所有C++→Python 调用都必须过 vtable 这一关。下一节看 vtable 长什么样。

2.2 PyInterpreterVTable:抽象的桥

vtable 定义在 c10/core/impl/PyInterpreter.h 第 75-175 行附近。完整结构(裁剪):

// c10/core/impl/PyInterpreter.h
struct C10_API PyInterpreterVTable {
  virtual ~PyInterpreterVTable() = default;

  // 调试用:返回 "cpython" 或 "noop"
  virtual std::string name() const = 0;

  // === 引用计数管理 ===
  virtual void incref(PyObject* pyobj) const = 0;
  virtual void decref(PyObject* pyobj) const = 0;
  virtual bool try_incref(const c10::impl::PyObjectSlot& pyobj_slot) const = 0;
  virtual size_t refcnt(PyObject* pyobj) const = 0;

  // === Tensor 操作 ===
  virtual c10::intrusive_ptr<TensorImpl> detach(const TensorImpl* self) const = 0;

  // === ★ 关键的"分派到 Python"方法 ===
  virtual void dispatch(const c10::OperatorHandle& op,
                        torch::jit::Stack* stack) const = 0;

  virtual void python_op_registration_trampoline(
      const c10::OperatorHandle& op, c10::DispatchKey,
      c10::DispatchKeySet keyset, torch::jit::Stack* stack,
      bool with_keyset, bool with_op) const = 0;

  virtual void python_dispatcher(const c10::OperatorHandle& op,
      c10::DispatchKeySet, torch::jit::Stack* stack) const = 0;

  // === 元数据查询 ===
  virtual c10::IntArrayRef strides(const TensorImpl* self) const = 0;
  virtual c10::Device device(const TensorImpl* self) const = 0;
  // ... 还有 sizes / numel / dim / sym_size 等十几个
};

struct C10_API PyInterpreter {
  const PyInterpreterVTable* vtable_;
  const PyInterpreterVTable& operator*() const noexcept { return *vtable_; }
  const PyInterpreterVTable* operator->() const noexcept { return vtable_; }
  void disarm() noexcept;             // 进程退出时切换到 noop_vtable
};

三组虚函数,各管一类事:

组用途谁会调它?
引用计数(incref/decref/refcnt)管理 c10 持有的 Python 对象的生命周期TensorImpl 析构时、PyObjectSlot 操作时
Dispatch(dispatch、python_op_registration_trampoline、python_dispatcher)把 C++ op 调用转发到 PythonPythonFallbackKernel(2.4 节)和 torch.library 注册 Python 实现的 op
元数据(strides/sizes/device 等)Tensor subclass 重写 shape 信息时,C++ 通过 vtable 反查TensorImpl::sym_sizes_custom() 等

本文重点是第二组 —— dispatch 那行虚函数。它的具体实现(在 libtorch_python 里)就是 §2.6 要展开的 ConcretePyInterpreterVTable::dispatch。

小例 2.2 — disarm 的存在理由 PyTorch 进程退出时,Python 解释器先析构(Py_Finalize),但 libtorch 持有的 C++ 对象可能晚于 Python 析构(比如缓存的张量在 atexit 之后才被销毁)。这些对象的析构函数会调 pyobj_slot.decref() —— 但此时 Python 已经死了,真正调 Py_DECREF 会 segfault。

disarm() 在 Python 析构开始时被调用,把 vtable 指针整体替换成一个 noop 版本。后续 C++ 调 decref(...) 走 noop —— 安静地什么都不做。如果你看到 "attempted to call dispatch on a Tensor with nontrivial PyObject after corresponding interpreter died" 这种诊断信息,就是这套机制在工作。

2.3 TensorImpl::pyobj_slot:反向指针

vtable 解决了"c10 如何调 Python 函数"的问题。还有一个对偶问题:给定一个 C++ TensorImpl,怎么知道它对应的 Python 对象是哪个?这对 tensor subclass 至关重要 —— 调 FakeTensor 的某个方法时,dispatcher 拿到的是 C++ TensorImpl*,但要调的 __torch_dispatch__ 定义在 Python 那个 FakeTensor 类上。

答案是:每个 TensorImpl 内嵌一个 PyObjectSlot,它就是一个原子的 PyObject*(指向 Python 那一侧的 THPVariable 对象,即 torch.Tensor Python 包装)。定义在 c10/core/impl/PyObjectSlot.h:

// c10/core/impl/PyObjectSlot.h (完整,约 60 行,简化)
struct C10_API PyObjectSlot {
 public:
  PyObjectSlot() : pyobj_(nullptr) {}

  PyObject* load_pyobj() const {
    return pyobj_.load(std::memory_order_acquire);
  }
  void store_pyobj(PyObject* obj) {
    pyobj_.store(obj, std::memory_order_release);
  }

  bool has_unique_reference() const {
    PyObject* pyobj = load_pyobj();
    return pyobj != nullptr
        && (*getGlobalPyInterpreter())->refcnt(pyobj) == 1;
  }

  void clear() { pyobj_.store(nullptr, std::memory_order_relaxed); }

  // 引用计数 —— 内部通过 vtable 调到 CPython 的 Py_INCREF / Py_DECREF
  void incref() const noexcept;
  void decref() const noexcept;
  bool try_incref() const noexcept;

 private:
  std::atomic<PyObject*> pyobj_;     // ← 整个反向指针机制就是这 8 字节
};

嵌入到 TensorImpl 里(c10/core/TensorImpl.h 第 2974 行附近):

class TORCH_API TensorImpl : public c10::intrusive_ptr_target {
  ...
  impl::PyObjectSlot pyobj_slot_;       // ← 占用 8 字节(64-bit ptr)

 public:
  impl::PyObjectSlot* pyobj_slot()       { return &pyobj_slot_; }
  const impl::PyObjectSlot* pyobj_slot() const { return &pyobj_slot_; }
};

初始状态 pyobj_ 是 nullptr。两种情况会被设上:

  1. 普通 torch.Tensor 第一次被 Python 包装时:Python C 绑定层(THPVariable_Wrap)创建一个新的 THPVariable 对象,把它指针写进 pyobj_slot。下次 C++ 想"包回 Python"时直接复用,而不是建新包装。
  2. Tensor subclass(如 FakeTensor)被构造时:Python 端构造方法主动调 store_pyobj(self),同时把 DispatchKey::Python 设进这个 TensorImpl 的 key_set_ —— 这是触发 PythonFallbackKernel 的关键标记。
记住这一点:tensor subclass 的整套机制建立在两件事上:① TensorImpl 的 key_set_ 里有 Python 位,② pyobj_slot_ 指向对应的 Python subclass 实例。两件事都必须有,任何一个缺失这个 subclass 都不会被正确识别。

2.4 PythonFallbackKernel:正式跳过去

vtable 和 pyobj_slot 准备好了基础设施。现在看真正干"跳"这件事的代码。它是一个 boxed kernel,以 m.fallback 形式注册到 Python key 上(还有几个相关的 key)。文件:aten/src/ATen/core/PythonFallbackKernel.cpp。

注册段(第 151 行附近):

// 把 pythonFallback 这个函数注册成 Python key 的 boxed 兜底 kernel
// 注册到下划线 namespace = "所有 namespace 都生效"
TORCH_LIBRARY_IMPL(_, Python, m) {
  m.fallback(torch::CppFunction::makeFromBoxedFunction<&pythonFallback>());
}

// 还有三个相关的 key 也注册了类似的 fallback(细节见 §4)
TORCH_LIBRARY_IMPL(_, PythonDispatcher, m) {
  m.fallback(torch::CppFunction::makeFromBoxedFunction<&pythonDispatcherFallback>());
}
TORCH_LIBRARY_IMPL(_, PythonTLSSnapshot, m) {
  m.fallback(torch::CppFunction::makeFromBoxedFunction<&pythonTLSSnapshotFallback>());
}
TORCH_LIBRARY_IMPL(_, PreDispatch, m) {
  m.fallback(torch::CppFunction::makeFromBoxedFunction<&preDispatchFallback>());
}

所以 dispatcher 看到位图最高位是 Python 时,就会调用 pythonFallback。函数体是这一切的"中央枢纽"(第 52 行附近,我把关键部分注释加密):

// aten/src/ATen/core/PythonFallbackKernel.cpp
void pythonFallback(const c10::OperatorHandle& op,
                    c10::DispatchKeySet dispatch_keys,
                    torch::jit::Stack* stack) {

  // === ① 进 guard: 标记"我们已经在 Python fallback 里了"
  //       + 把 Python 及以下的所有 key 加入 TLS exclude
  //       —— 这样 Python 内部调 func(*args) 时不会再触发本 fallback,
  //         实现 2.5 节要讲的"不递归"语义。
  at::impl::MaybeSetTLSOnEntryGuard tls_guard;
  c10::impl::ExcludeDispatchKeyGuard exclude_guard(after_Python_keyset);

  const auto& schema = op.schema();
  const auto num_arguments = schema.arguments().size();

  // === ② 路径 A: 模式栈(mode stack)非空 → 用栈顶的 mode 处理
  //
  // TorchDispatchModeTLS 是 C++ 端的一个线程局部模式栈;
  // 每次 Python 端进 `with SomeMode():` 就 push 一个 mode 对象到这。
  // pythonFallback 拿栈顶的 mode,调它的 pyinterpreter()->dispatch(...)。
  const auto mode_stack_len = c10::impl::TorchDispatchModeTLS::stack_len();
  if (mode_stack_len > 0) {
    const auto& cur_mode = c10::impl::TorchDispatchModeTLS::get_stack_at(mode_stack_len - 1);
    cur_mode->pyinterpreter()->dispatch(op, stack);
    // ↑ 通过 vtable 跳到 Python,真正调 mode.__torch_dispatch__(...)
    return;
  }

  // === ③ 路径 B: 没有模式,但有 tensor subclass 在输入里
  //
  // 走完输入参数,找:① 是否真有 tensor 带 Python key;② 任意一个有 pyobj_slot 的 tensor
  // 用作 PyInterpreter 入口。
  bool tensors_with_python_key_present = false;
  c10::impl::PyInterpreter* interpreter = nullptr;
  for (const auto& ivalue : torch::jit::last(*stack, num_arguments)) {
    if (ivalue.isTensor()) {
      auto* t = ivalue.unsafeToTensorImpl();
      if (t->key_set().has(c10::DispatchKey::Python)) {
        tensors_with_python_key_present = true;
      }
      if (!interpreter && t->pyobj_slot()->load_pyobj()) {
        interpreter = c10::impl::getGlobalPyInterpreter();
      }
    }
    // ... List / Optional 形式参数也走相同逻辑(略)
  }

  if (interpreter) {
    if (tensors_with_python_key_present) {
      // 真有 subclass 想拦截这次调用 → 跳 Python
      (*interpreter)->dispatch(op, stack);
    } else {
      // 输入里有 PyInterpreter 但都不带 Python key:把 Python 位 mask 掉,
      // 让 dispatcher 用 redispatch 继续走下一层 —— 没人想拦,正常算就行
      c10::DispatchKeySet keyset = dispatch_keys.remove(c10::DispatchKey::Python);
      c10::impl::LocalDispatchKeySet local_keyset = c10::impl::tls_local_dispatch_key_set();
      c10::impl::ForceDispatchKeyGuard no_python_guard(
        local_keyset.included_.remove(c10::DispatchKey::Python),
        local_keyset.excluded_);
      op.redispatchBoxed(keyset, stack);
    }
    return;
  }

  TORCH_INTERNAL_ASSERT(0, "Hit Python dispatch key but no arguments had PyInterpreter");
}

这个函数体浓缩了 "C++→Python 跳转" 的全部决策。逐段读一遍:

  1. ① guard 段:进 fallback 立刻把"Python 及更低优先级"全部 mask 掉(after_Python_keyset 是预算好的常量,2.5 节展开)。这保证后续 Python 代码体内的 op 调用不会再触发自己 —— 否则就是无限递归。
  2. ② 路径 A:mode stack 非空。这是 FakeTensorMode、ProxyTorchDispatchMode 这些 mode 的入口。它们在 __enter__ 时把自己 push 进 TorchDispatchModeTLS,__exit__ 时 pop。pythonFallback 只取栈顶 —— LIFO 优先级。
  3. ③ 路径 B:输入张量带 Python key。这是 tensor subclass 路径(如 FakeTensor 作为参数传给某 op)。tensors_with_python_key_present 检查 key,interpreter 找一个有 pyobj_slot 的张量作为入口。两个条件都满足才真跳。
  4. 路径 B 的"假阳性"分支:如果 input 有 PyInterpreter 但都不带 Python key(意味着这些 tensor 的 Python 包装存在但没人指定要拦截),就把 Python 位 mask 掉,redispatch —— 这处理 "有 Python 包装但不需要拦截"的常见情况(普通 torch.Tensor 都在这个分支)。
边栏 2.4 — 模式栈优先级 vs 子类优先级 观察上面代码:mode stack 路径写在前面,subclass 路径写在后面。这是关键的优先级决定 —— 如果同时存在 mode 和带 Python key 的 tensor 参数,mode 赢。

这是合理的设计选择。Mode 是"用户主动声明的全局变换"(with FakeTensorMode():),subclass 是"输入数据本身的属性"。当两者冲突时,主动声明的语义更强、更显式 —— 用户写 with FakeTensorMode(): 期望所有 op 都被假化,即使输入里有别的 subclass。

2.5 ExcludeDispatchKeyGuard 与 after_Python_keyset

2.4 节的 guard:

c10::impl::ExcludeDispatchKeyGuard exclude_guard(after_Python_keyset);

这是整个机制最巧妙的设计。它防止"Python 函数体内调 op → 又回到 PythonFallback → 死循环"。

after_Python_keyset 在 PythonFallbackKernel.cpp 第 27 行附近定义:

constexpr c10::DispatchKeySet after_Python_keyset =
    c10::DispatchKeySet(c10::DispatchKeySet::FULL)
    ^ (c10::DispatchKeySet(c10::DispatchKeySet::FULL_AFTER, c10::DispatchKey::Python)
       | c10::DispatchKeySet(c10::DispatchKey::Python));

这个表达式比较绕,但意思是:"Python 这一位 + 比它优先级低的所有位"。XOR 一下,得到的就是"Python 这一位 + 它下面所有位"组成的 mask。

ExcludeDispatchKeyGuard 是个 RAII 对象(类似第一篇里讲过的 AutoDispatchBelowAutograd),构造时把这些位加进 TLS exclude,析构时还原。所以:

当 Python __torch_dispatch__ 体内调 func(*args)(意味着"我处理完了,让 dispatcher 继续走"),这个调用过 at::* API 进 Dispatcher::call,算位图时会自动减掉 exclude 集合,Python 位被清零 —— 最高位变成 Autograd* 或更下面的 backend key,直接落到真实实现,不会再触发 pythonFallback。

小例 2.5 — 一次完整的 mode 拦截 + 透传 追踪 with MyMode(): z = x + y 的位图:

1. 顶层 dispatcher::call:位图 = {Python, AutogradCUDA, CUDA}(因为进了 mode → TLS include 里有 Python)。最高位 = Python,跳 pythonFallback。
2. pythonFallback 内:ExcludeDispatchKeyGuard 把 {Python, Autograd*, CUDA, ...} 全加进 TLS exclude。然后调 mode 的 __torch_dispatch__(func=aten.add.Tensor, args=(x, y))。
3. Python __torch_dispatch__ 体内:用户写 return func(*args, **kwargs)(意思:我啥都不做,让你继续走)。这调到 at::add 进顶层 Dispatcher::call。
4. 嵌套 dispatcher::call:位图初始 = {Python, AutogradCUDA, CUDA}(还在 mode 里),但 TLS exclude 减掉了 {Python, Autograd*, CUDA, ...}。运算后:最高位 = ???
5. ★ 关键:exclude 集合也包含 Autograd 和 CUDA 等位!所以剩下的位图是空的?

其实不是 —— 仔细看 after_Python_keyset 的定义:FULL_AFTER, Python 是 "Python 之后" 的位(更低优先级)。XOR 出来的是 "Python 本身 + Python 之后" 的位,也就是 Python 及以下。所以 exclude 包含 {Python, ADInplaceOrView, Autograd*, ..., CUDA, ...}。
6. 减掉后,从 {Python, AutogradCUDA, CUDA} 剩下空集 —— 是不是没法 dispatch?

答案藏在 path B:实际场景里 x 和 y 不带 Python key(它们是普通 CUDA tensor),进 pythonFallback 的第二次调用会走到 "no tensor with Python key" 分支,把 Python 位 mask 掉、强制 redispatch,这次最高位变成 AutogradCUDA —— 链条正确继续。

对于 tensor subclass 路径:输入张量带 Python key,但 __torch_dispatch__ 内通常先调 unwrap(x) 把张量"剥成"普通 tensor,然后再 func(unwrapped_args)。普通 tensor 不带 Python key,后续 redispatch 同样落到 backend。

2.6 ConcretePyInterpreterVTable::dispatch:跨越边界

pythonFallback 在路径 A 调 cur_mode->pyinterpreter()->dispatch(op, stack)。这个 dispatch 就是 2.2 节那个虚函数 —— 真正实现在 torch/csrc/PyInterpreter.cpp 第 177-218 行:

// torch/csrc/PyInterpreter.cpp
void ConcretePyInterpreterVTable::dispatch(
    const c10::OperatorHandle& op, torch::jit::Stack* stack) const {

  const auto& schema = op.schema();
  const auto num_arguments = schema.arguments().size();
  auto arguments = torch::jit::pop(*stack, num_arguments);   // 从 stack 弹出参数

  // === ① 获取 GIL —— 跨界第一步
  py::gil_scoped_acquire g;

  // === ② 找 op 对应的 Python API 函数(例如 torch.ops.aten.add.Tensor)
  std::vector<PyObject*> overloaded_args;
  py::handle torch_api_function_overload = getTorchApiFunction(op);

  // === ③ 扫描 IValue 参数,收集"带 Python key 的 tensor"(subclass)
  for (const auto idx : c10::irange(arguments.size())) {
    const auto& ivalue = arguments[idx];
    if (ivalue.isTensor()) {
      const auto& tensor = ivalue.toTensor();
      if (isPythonTensor(tensor)) {
        append_overloaded_tensor(&overloaded_args, py::cast(tensor).ptr());
      }
    }
    // 还处理 list / optional-list 内的 tensor(略)
  }

  // === ④ 把所有 IValue 转成 PyObject 形式的 (args, kwargs)
  auto args_kwargs = parseIValuesToPyArgsKwargs(op, arguments);
  auto args = std::move(args_kwargs.first);
  auto kwargs = std::move(args_kwargs.second);

  // === ⑤ 调用 Python: 找 __torch_dispatch__,带 mode stack 优先级 + types MRO
  PyObject* obj = handle_torch_function_no_python_arg_parser(
      overloaded_args, args.ptr(), kwargs.ptr(), nullptr,
      torch_api_function_overload.ptr(), nullptr,
      &op, &arguments, TorchFunctionName::TorchDispatch);

  // === ⑥ 把 Python 返回值转回 IValue 并压回 stack
  pushPyOutToStack(op, stack,
      py::reinterpret_steal<py::object>(obj),
      "__torch_dispatch__");
}

这是真正跨界的 6 步。逐段解释:

  1. ① py::gil_scoped_acquire g;:获取 GIL。这是把 C++ 线程拉进 Python 解释器的"门票" —— 没有 GIL,你不能安全调任何 CPython API。RAII 对象,出函数自动释放。
  2. ② getTorchApiFunction(op):从 op 名("aten::add.Tensor")找对应的 Python 函数对象(torch.ops.aten.add.Tensor)。这个函数对象后续被传给 Python 端,Python 端 __torch_dispatch__ 的 func 参数就是它。
  3. ③ 扫描"覆盖"列表:走一遍 IValue 输入,找带 Python key 的张量,收集到 overloaded_args。Python 端需要这个列表才能按 MRO 顺序找哪个 subclass 的 __torch_dispatch__ 应该被调。
  4. ④ parseIValuesToPyArgsKwargs:把所有 IValue 转成 Python 对象(args 是 tuple、kwargs 是 dict)。对 tensor 参数,py::cast(tensor) 会优先从 pyobj_slot_ 取已有的 Python 包装,没有才新建 —— 这是 2.3 节那个反向指针的用武之地。
  5. ⑤ handle_torch_function_no_python_arg_parser:实际"调 Python" 的步骤。它走 TorchDispatchModeTLS 栈 + overloaded_args 的 MRO,找第一个有 __torch_dispatch__ 的对象,把 (func, types, args, kwargs) 传进去。具体的调度优先级:
       (a) Mode stack 从顶到底依次尝试,直到一个返回非 NotImplemented;
       (b) Subclass 按 MRO 顺序尝试 overloaded_args 里的对象。
    具体 mode vs subclass 的优先级,见 §3.3。
  6. ⑥ pushPyOutToStack:Python 返回了一个 PyObject(可能是张量、tuple、scalar 等)。把它转回 IValue,push 到 stack 上。dispatcher 顶层从 stack 取就是最终结果。

这 6 步执行一次的开销:GIL 获取 + 几十次 PyObject 创建 + 一次 Python 函数调用 + 反向转换。经验值 5-15µs(对比纯 C++ dispatch 的 50ns,慢了 100-300 倍)。这就是 §7 要讨论的"什么时候不该用 __torch_dispatch__"的代价。

本章带走的三个心智模型:
① vtable 是 c10/Python 解耦的关键。libc10 不依赖 CPython,通过 vtable 把"调 Python"这件事抽象成虚函数,运行时由 libtorch_python 填充。
② pyobj_slot 是反向指针。每个 TensorImpl 8 字节存对应 Python 包装的指针。Tensor subclass 路径靠它找到 __torch_dispatch__。
③ ExcludeDispatchKeyGuard 防递归。Python 函数体内调的 op 都自动跳过 Python kernel,直接落到 backend —— 实现"我处理完了,你继续"的递归语义。
