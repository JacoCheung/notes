PyTorch Linear 到 cuBLAS: shape、layout 与 reduction

torch.nn.Linear 的官方定义是对输入做 \(y=xA^T+b\)。在代码里通常把权重记成 \(W\),所以本文写成 \(Y=XW^T+b\)。PyTorch 存的 weight shape 是 (out_features, in_features),也就是下文的 \(W\in\mathbb{R}^{N\times K}\)。

1. PyTorch 里的 shape

把 batch、sequence 等前导维度都展平成 \(M\)。Linear 的三个主要张量是:

\[ X\in\mathbb{R}^{M\times K},\quad W\in\mathbb{R}^{N\times K},\quad Y\in\mathbb{R}^{M\times N} \]
  • \(M\): batch/token 维度,来自输入最后一维之前的所有维度乘积。
  • \(K\): in_features,输入特征维度。
  • \(N\): out_features,输出特征维度。

忽略 bias 后,forward 和 backward 的矩阵形式是:

\[ Y=XW^T,\qquad dX=dY\,W,\qquad dW=dY^T X \]

也可以把 weight gradient 写成 \(dW^T=X^T dY\)。这个写法很适合对照 classic cuBLAS,因为 row-major 的 \(dW\) buffer 在 cuBLAS 眼里就是 \(dW^T\)。

2. cuBLAS 的视角为什么会转置

classic cuBLAS 按 column-major 矩阵理解传入的指针。一个连续 row-major buffer \(R_{r\times c}\),如果不搬数据,直接交给 column-major API,数学上会被看成:

\[ \operatorname{cm}(R_{r\times c}) = R^T_{c\times r} \]

cuBLAS GEMM 的接口模型是:

\[ C=\alpha\,\operatorname{op}(A)\operatorname{op}(B)+\beta C,\qquad \operatorname{op}(A)\in\mathbb{R}^{m\times k},\quad \operatorname{op}(B)\in\mathbb{R}^{k\times n},\quad C\in\mathbb{R}^{m\times n} \]

所以同一块 row-major 输出内存,cuBLAS 写的不是 PyTorch 视角的 \(Y\),而是它自己的 column-major view。对 forward 来说,这块内存在 PyTorch 里是 \(Y_{M\times N}\),在 classic cuBLAS 里就是 \(Y^T_{N\times M}\)。这就是表格里公式看起来"反着写"的原因。

表里的 layout 指 (transA, transB),不是 PyTorch Tensor 的 memory format。TN 表示 transA=CUBLAS_OP_T,transB=CUBLAS_OP_N。

3. forward / dW / dX 对照表

4. reduction 维度怎么读

对 GEMM 本身,求和规约的维度永远是 cuBLAS 的 \(k\):

\[ C_{i,j}=\sum_{\ell=1}^{k}\operatorname{op}(A)_{i,\ell}\operatorname{op}(B)_{\ell,j} \]

容易混淆的是:三次 Linear 相关 GEMM 的 cuBLAS \(k\) 对应的逻辑维度不一样。

  • forward: \(k=K\),沿 in_features 规约。
  • dW: \(k=M\),沿 batch/token 维度规约。参数梯度把所有样本对同一个权重元素的贡献加起来。
  • dX: \(k=N\),沿 out_features 规约。每个输入特征的梯度来自所有输出通道的反向贡献。

这也是做 FP8 / MXFP8 scaling 或者看 kernel descriptor 时必须盯住的地方。不要只记 \(XW^T\) 这条公式,要问这次 GEMM 的 reduction axis 在当前 layout 里落到哪里。

5. cuBLASLt 的另一种写法

cuBLASLt 可以用 matrix layout descriptor 描述矩阵的 rows、cols、leading dimension,也可以通过 layout order 表达 row-major 数据。用这种接口时,你可能看到更接近 PyTorch 公式的 transA/transB:

目标 A / B / C row-major descriptor 下常见 op reduction
\(Y=XW^T\) \(A=X,\ B=W,\ C=Y\) NT \(K\)
\(dW=dY^T X\) \(A=dY,\ B=X,\ C=dW\) TN \(M\)
\(dX=dY W\) \(A=dY,\ B=W,\ C=dX\) NN \(N\)

所以读代码时先确认是哪一层抽象:classic cuBLAS column-major wrapper、cuBLASLt row-major descriptor,还是框架内部又包了一层自己的命名。否则 TN 和 NT 很容易被看反。

PyTorch Linear to cuBLAS: shapes, layout, and reduction

torch.nn.Linear applies \(y=xA^T+b\). This note uses \(W\) for the stored weight, so the forward formula is \(Y=XW^T+b\). PyTorch stores weight as (out_features, in_features), written here as \(W\in\mathbb{R}^{N\times K}\).

1. PyTorch shapes

Flatten all leading batch or sequence dimensions into \(M\):

\[ X\in\mathbb{R}^{M\times K},\quad W\in\mathbb{R}^{N\times K},\quad Y\in\mathbb{R}^{M\times N} \]
  • \(M\): batch/token dimension.
  • \(K\): in_features.
  • \(N\): out_features.

Ignoring bias, the forward and backward GEMMs are:

\[ Y=XW^T,\qquad dX=dY\,W,\qquad dW=dY^T X \]

The weight gradient can also be written as \(dW^T=X^T dY\), which lines up with the classic cuBLAS view of a row-major dW buffer.

2. Why the cuBLAS view is transposed

The classic cuBLAS API interprets matrix pointers as column-major. A contiguous row-major buffer \(R_{r\times c}\), passed to cuBLAS without moving data, is seen mathematically as:

\[ \operatorname{cm}(R_{r\times c}) = R^T_{c\times r} \]

The GEMM contract is:

\[ C=\alpha\,\operatorname{op}(A)\operatorname{op}(B)+\beta C,\qquad \operatorname{op}(A)\in\mathbb{R}^{m\times k},\quad \operatorname{op}(B)\in\mathbb{R}^{k\times n},\quad C\in\mathbb{R}^{m\times n} \]

That is why the same output memory is named differently. The PyTorch forward output is \(Y[M,N]\), but classic cuBLAS sees that buffer as \(Y^T[N,M]\).

In the tables below, layout means (transA, transB). It is not the PyTorch tensor memory format.

3. Forward / dW / dX table

Pass PyTorch view cuBLAS C view A / B / C buffer layout cuBLAS \(m,n,k\) reduction Note
forward \(Y=XW^T\) \(Y^T=W X^T\) \(A=W,\ B=X,\ C=Y\) TN \(m=N,\ n=M,\ k=K\) \(k\), logically \(K\) cuBLAS sees row-major \(Y[M,N]\) as column-major \(Y^T[N,M]\).
dW \(dW=dY^T X\), or \(dW^T=X^T dY\) \(dW^T=X^T dY\) \(A=X,\ B=dY,\ C=dW\) NT \(m=K,\ n=N,\ k=M\) \(k\), logically \(M\) The desired gradient is \(dW[N,K]\), while the cuBLAS C view is \(dW^T[K,N]\).
dX \(dX=dY\,W^{TT}=dY\,W\) \(dX^T=W^T dY^T\) \(A=W,\ B=dY,\ C=dX\) NN \(m=K,\ n=M,\ k=N\) \(k\), logically \(N\) \(W^{TT}=W\). This keeps the link to the forward \(W^T\) explicit.

4. Reading the reduction dimension

For GEMM, the summation dimension is always cuBLAS \(k\):

\[ C_{i,j}=\sum_{\ell=1}^{k}\operatorname{op}(A)_{i,\ell}\operatorname{op}(B)_{\ell,j} \]

The logical dimension behind that \(k\) changes by pass: forward reduces over \(K\), weight gradient over \(M\), and input gradient over \(N\). This distinction matters when reading quantization scales, block layouts, and kernel descriptors.

5. The cuBLASLt variant

cuBLASLt matrix layout descriptors can describe rows, columns, leading dimension, and layout order more directly. With a row-major descriptor, the operation flags often look closer to the PyTorch formula:

Target A / B / C Common row-major op reduction
\(Y=XW^T\) \(A=X,\ B=W,\ C=Y\) NT \(K\)
\(dW=dY^T X\) \(A=dY,\ B=X,\ C=dW\) TN \(M\)
\(dX=dY W\) \(A=dY,\ B=W,\ C=dX\) NN \(N\)

When reading framework code, first identify the abstraction level: classic cuBLAS column-major wrapper, cuBLASLt row-major descriptor, or a framework-specific wrapper around either one.
